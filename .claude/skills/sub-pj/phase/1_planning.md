# Phase 1: Planning

plan.md を探索ループで詳細化し、2 つのゲートを通過して実行可能な計画に仕上げる。
全体像は `references/overview.md` を参照。

---

## 最初にやること

plan.md の現在の status を確認しろ:

- `exploring`（または骨格のみ）→ 探索ループから開始
- 戦略が確定済み → Gate 1 チェックへ
- task 順序も確定済み → Gate 2 チェックへ

前セッションで情報が揃っている場合、ユーザーに聞け:

```
前セッションの情報を確認します。以下はどこまで進んでいますか？

- [ ] 対象の調査（情報収集）
- [ ] Vision 確定（README に反映済み）
- [ ] 方針の候補出し（戦略立案）
- [ ] 技術選定
- [ ] task の洗い出し

揃っている部分は飛ばして、足りない部分から始めます。
```

---

## 探索ループ

以下の活動を対話でぐるぐる回せ。順序固定ではない。ユーザーの発言に応じて適切な活動に入れ。

### 情報収集

対象を調査しろ。文献サーチ、既存コード読み、外部ツール調査を行え。
結果は plan.md の「探索ログ」に記録しろ:

```markdown
### {日付}: {トピック}
- **調査結果**: {分かったこと}
- **判断**: {決めたこと}
- **未決**: {まだ決まらないこと}
```

### 対象理解（作業者向け必須情報）

plan.md の「作業者向け必須情報」セクションを埋めろ。以下を必ず埋めろ:

- **対象**: 何を扱うか。対象の構成（ディレクトリツリー等）、触るファイル/要素一覧（優先度 P0/P1/P2 付き）
- **ドメイン固有の注意点**: 見落としがちな前提や制約
- **前提知識**: PJ 参加に必要な最低限の知識（箇条書き）
- **関連スキル**: 使う M3E スキル一覧
- **作業環境注意事項**: 環境の制約・未整備事項

ブレスト素材（idea/ 配下）があれば読み、要約を反映しろ。

### 戦略立案

方針の選択肢を出せ。複数案がある場合は pros/cons を並べろ。

選択肢を出したら、自己検討しろ。各案のリスク・実現可能性・工数を評価し、
明確に差があるなら **メインプラン**（推奨）と **サブプラン**（代替）の形で提示しろ。
ユーザーが「これでいい」と即決できる形にすることが目的。

```
**メインプラン**: {案A} — {推奨理由}
**サブプラン**: {案B} — {メインが不適切な場合の代替}
```

差がはっきりしない場合や、観点がバラバラで一概に比較できない場合は、
無理にランク付けせず並列で提示してよい。ユーザーが判断できる材料を提示しろ。

### 技術選定サーチ

ツール・ライブラリ・手法の候補を調査しろ。
各候補の特徴・制約・採用理由を探索ログに記録しろ。

### task ブレスト

やるべきことを洗い出せ。この段階では順序は問わない。
漏れがないか、ユーザーに「他にやるべきことはありますか？」と確認しろ。

---

## ★ Gate 1: 戦略決定

以下がすべて揃っているか確認しろ:

- [ ] Vision が明確（README の問題・完了像・スコープが具体的）
- [ ] 方針が 1 つに絞られている
- [ ] 技術選定が完了
- [ ] task が洗い出されている（順序は未定でよい）
- [ ] 作業者向け必須情報が埋まっている

すべて揃っていたら、ユーザーに確認しろ:

```
Gate 1（戦略決定）の確認です:

- Vision: {1行}
- 方針: {1行}
- 技術: {列挙}
- task数: {N}個

これで戦略を確定し、実行計画の策定に進みますか？
```

ユーザーが承認したら、plan.md の「確定事項」セクションに記録しろ:

```markdown
## 確定事項（{日付} akaghef 確認済み）

- {方針}: {理由}
- {技術選定}: {理由}
- {スコープ絞り込み}: {理由}
```

承認されるまで探索ループに戻れ。

## 実行計画策定（Gate 1 通過後）

以下を順序通りに進めろ。

### 1. task 依存関係と実行順序を決めろ

洗い出された task の依存関係を整理し、実行順序を決めろ。

- task 間の依存関係を明示しろ（「B は A の結果が必要」等）
- 並列可能な task を特定しろ
- 実行順序を決定しろ

`sort-task` スキルが使える場合はそれを活用しろ。

### 2. Phase 設計をしろ

実行順序が決まった task 群を Phase に分割しろ。原則:

- Phase 0 は「設計と最小検証」。実装量を最小にしろ
- Phase 粒度は 1〜2 週間で完了できるサイズにしろ
- 後半 Phase は粗い粒度でよい

各 Phase に以下を決めろ:
- **名前と目的** — 1 行
- **完了条件** — 遷移判定に使う具体基準
- **タスク** — Phase 0 のみ具体化。Phase 1+ は粗くてよい

### 3. Phase 0 の着手手順を具体化しろ

最初の Phase だけ時間見積もり付きの着手手順を作れ:

```markdown
### Phase 0: {名前}（{推定期間}）

**目的**: {1行}

**着手手順**:
- [ ] Step 0: {環境確認・準備}（{時間}）
- [ ] Step 1: {最初のタスク}（{時間}）
- [ ] Step 2: {次のタスク}（{時間}）
- [ ] Step N: 振り返りと次セッション準備（15 分）

**完了条件**: {遷移判定の基準}
```

### 4. facet 設計をしろ（マップ使用 PJ のみ）

M3E マップを使う PJ の場合のみ実施。対象をどう分解するか設計しろ。

ユーザーに「この対象をどんな切り口で見たいか？」と聞け。
最初は 3〜5 facet に絞れ。全部定義するな。

各 facet に以下を定義しろ:

- **役割**: この facet で何が分かるか
- **構造**: ノードの親子関係・粒度
- **ノード属性**: attribute 名と値の例
- **具体ノード例**: 最低 1 個、attribute 値込み
- **使う瞬間**: どの Phase のどのタスクで参照するか

facet 設計と合わせてリンク種別と最小ノードフォーマットも定義しろ。

**scope をまたぐ link は張るな。** 異なる scope 間の関係は alias で示せ。
link は同一 scope 内でのみ使え。これは全 facet 共通のルール（protocol.md §4）。

PJ01 の 12 facets を参考にしたい場合は `references/examples.md` を読め。
ただし PJ ごとに facet は全く異なる。PJ01 のコピーを作るな。

facet タイプ別のスコープ粒度・レイアウトルールは `references/facet/` にある。
該当する facet タイプがあれば読み、plan.md に粒度ルールを転記しろ。
なければ PJ 固有のルールをユーザーと対話で決めろ。

### 5. 運用ルールを策定しろ

PJ 固有の作業ルールを plan.md に追加しろ:

- ノード作成・リンクの規約（マップを使う場合）
- Phase 遷移の判定基準（具体的に）
- 中断時の保全ルール

### 6. runtime readiness を決めろ（既定必須）

M3E を実行画面に使う PJ は、Gate 2 の前に runtime の正本を決めろ。
既定では **3-view runtime を持つ前提** とし、以下を plan.md または README に明記しろ:

- Progress Board の map / scope
- Review の map / scope
- Active Workspace の map / scope
- traceability の張り方（task → review / workspace）

runtime を使わない場合のみ、README と plan.md の両方に
`runtime_opt_out: {理由}` を書いて opt-out しろ。
理由がない限り、「あとで考える」は認めるな。

---

## ★ Gate 2: 実行計画確定

以下がすべて揃っているか確認しろ:

- [ ] task の依存関係と実行順序が決まっている
- [ ] Phase 設計が完了（各 Phase に名前・目的・完了条件）
- [ ] Phase 0 の着手手順が具体的（時間見積もり付き）
- [ ] facet 設計が完了（マップ使用 PJ のみ）
- [ ] 運用ルールが策定されている
- [ ] runtime readiness が決まっている
  - 3-view runtime を使う
  - もしくは `runtime_opt_out` が README / plan.md の両方にある

すべて揃っていたら、ユーザーに確認しろ:

```
Gate 2（実行計画確定）の確認です:

- Phase 設計: {N} 段階
- Phase 0 の着手手順: {M} ステップ
- facet: {K} 個（マップ使用時）
- 運用ルール: 策定済み
- runtime: {3-view runtime / opt-out}

これで実行計画を確定し、作業に入りますか？
OKなら `/sub-pj start` でセッション開始です。
```

ユーザーが承認したら、plan.md の status を `exploring` → `ready` に変更しろ。

## PJ01 からの教訓（詳細は references/lessons.md）

- マップ構造は粗く始めろ。scope/facet は最初 3〜5 個
- ノード例を具体的に書け。サブエージェント指示には attribute 値込みの例を載せろ
- フロー構造の軸を明示しろ。depth=時間, breadth=並列
- 設計レビューは ★1-5 形式。pros/cons + 【推奨】タグ
