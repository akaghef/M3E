# sub-PJ ライフサイクル概観

M3E 内のサブプロジェクト（sub-PJ）がどのような段階を経て進むかの全体像。
人間（akaghef）が PJ の現在地を把握し、次に何をすべきか判断するための文書。

---

## 全体フロー

```
【探索フェーズ】反復的・対話的 — 何周もする

  アイデア → 情報収集 → Vision決定 → 戦略立案 / 技術選定 / taskブレスト
     ↑________________________________↓  ぐるぐる回る
                                        │
                                  ★ Gate 1: 戦略決定
                                  「何をやるか」が固まった
                                        │
  task依存関係整理 → 実行順序決定 → 実行計画確定
                                        │
                                  ★ Gate 2: 実行計画確定
                                  「どの順でやるか」まで決まった
                                        ↓
【実行フェーズ】確定的 — 決めた通りに進む

  map構築 → 作業サイクル → Phase遷移 → … → PJ完了・振り返り
```

---

## 各段階の意味

### 探索フェーズ

PJ の方向性を固める段階。答えがない状態から対話で収束させる。

| 活動 | 何をしているか | 成果物 |
|------|---------------|--------|
| **アイデア** | 「これやりたい」の種。idea/ のメモや会話の断片 | idea/ 配下のメモ |
| **情報収集** | 対象の調査、文献サーチ、既存コード読み | 探索ログ（plan.md 内） |
| **Vision決定** | 問題・完了像・スコープを言語化 | README.md |
| **戦略立案** | 方針の選択肢を出す。複数案を比較 | 探索ログ |
| **技術選定サーチ** | ツール・ライブラリ・手法の候補調査 | 探索ログ |
| **taskブレスト** | やるべきことの洗い出し（順序は問わない） | task リスト |

これらは順序固定ではない。情報を集めたら Vision が変わり、Vision が変われば戦略が変わる。
「もう一周回ったほうがいい」と思ったら回る。

**前セッションで情報が揃っている場合は途中参入する。** 全部やり直す必要はない。

### Gate 1: 戦略決定

探索ループの出口。以下が揃って通過:

- Vision が明確（README の問題・完了像・スコープが具体的）
- 方針が 1 つに絞られている
- 技術選定が完了
- task が洗い出されている（順序は未定でよい）
- 作業者向け必須情報が plan.md に書かれている

ここで plan.md に「確定事項」を記録する。

### 実行計画策定

Gate 1 を越えたら、順序通りに進める段階に入る:

1. **task 依存関係整理** — task 間の前後関係を明示
2. **実行順序決定** — どの順で着手するか決定
3. **Phase 設計** — task 群を Phase（1〜2 週間単位）に分割
4. **Phase 0 着手手順** — 最初の Phase だけ時間見積もり付きで具体化
5. **facet 設計** — M3E マップを使う PJ のみ。対象をどう分解するか
6. **運用ルール** — PJ 固有の作業規約

### Gate 2: 実行計画確定

実行計画の出口。以下が揃って通過:

- task の依存関係と実行順序が決まっている
- Phase 設計が完了（各 Phase に名前・目的・完了条件）
- Phase 0 の着手手順が具体的
- 運用ルールが策定されている

ここで plan.md の status が `exploring` → `ready` に変わる。

### 実行フェーズ

計画に従って作業を進める段階。作業中は **資源管理**（protocol §6）が常に稼働する。

| 活動 | 何をしているか |
|------|---------------|
| **セッション開始** | 現在地確認、今日のタスク特定 |
| **作業サイクル** | タスク実行、コミット、資源（進捗・負債・決定）を正本に記録 |
| **セッション終了** | ハンドオフ報告（変更・検証・次タスク）+ 会話ログ生成 |
| **Phase 遷移** | 完了条件チェック → 人間が判断 → 次 Phase 具体化 |
| **PJ 完了** | 振り返り → 教訓抽出 → クローズ |

### 資源管理

PJ の遂行には成果物だけでなく、判断・知見・状態といった**無形の資源**が不可欠。
放置すると散逸し、再構築コストが膨大になる。
M3E の Resource 概念（SSOT・型別振る舞い・ロールアップ）と同じ思想で管理する。

- 各資源には**正本**が 1 箇所ある（SSOT）
- 資源の種類ごとに記録タイミング・場所が決まっている
- PJ の現在状態は plan.md を読めば導出できる（ロールアップ）
- 不確実なものは pool して進める。止まるな（容量超過 = 警告、停止ではない）

資源レジストリの全量は protocol §6 を参照。

## PJ の状態遷移

plan.md の `status` フィールドで追跡:

```
exploring  ──Gate 1──→  exploring  ──Gate 2──→  ready  ──start──→  active  ──complete──→  done
                         (戦略確定)              (計画確定)         (実行中)              (完了)
```

README.md の `status` は `active` / `paused` / `done` の 3 値。

## なぜこの設計か

### 探索と実行の分離

PJ01 の教訓: 探索（何をやるか決める）と実行（決めたことをやる）は本質的に異なる。
探索は反復的で、戻りがある。実行は直線的で、計画通りに進む。
これを同じ phase 構造で扱うと、「今どこにいるか」が曖昧になる。

### 2 つのゲート

Gate が 1 つだと「方針は決まったが順序が決まっていない」状態で実行に入ってしまう。
Gate 1（何を）と Gate 2（どの順で）を分けることで、
計画が不完全なまま作業を始める手戻りを防ぐ。

### 途中参入

PJ のアイデアは会話の中で自然に生まれる。
前セッションで情報収集や Vision 決定が済んでいることも多い。
「最初から全部やれ」ではなく「今どこまで進んでいるか確認して、足りない部分だけやる」設計。

### facet は PJ 固有

PJ01 の 12 facets は MATLAB→Julia 移植に特化したもの。
文献整理 PJ なら「著者」「時系列」「主張」「反論関係」等、全く異なる facet になる。
テンプレートで埋めようとせず、対話で PJ ごとに設計する。

## facet タイプ一覧

PJ01 の 12 facets を汎用化したパターン集。PJ ごとに必要なものだけ選ぶ。

| タイプ | 何を分解するか | 使う場面 |
|--------|---------------|---------|
| **実装構造** | クラス、モジュール、ファイルの内部構造 | コード解析、移植 |
| **処理フロー** | 時系列の処理手順、呼び出しチェーン | アルゴリズム理解、デバッグ |
| **依存グラフ** | 静的な依存関係（import、参照） | 影響範囲分析、移植順序 |
| **概念モデル** | ドメインの概念と関係 | 世界モデル構築、オントロジー |
| **型・階層** | 継承、分類体系、is-a 関係 | 型設計、分類整理 |
| **マッピング** | 2 つの世界の対応関係 | 概念↔実装、理論↔実験 |
| **データフロー** | データの変換・流れ | ETL、計算グラフ |
| **検証** | テスト、ground truth、ベンチマーク | 品質保証 |
| **設計** | ターゲットのアーキテクチャ・API | 新規設計 |
| **タスク管理** | 作業項目、進捗、マイルストーン | プロジェクト管理 |

### PJ タイプ別の推奨組み合わせ

| PJ タイプ | 推奨 facet |
|----------|-----------|
| コード移植 | 実装構造 + 依存グラフ + 概念モデル + 処理フロー |
| 文献整理 | 概念モデル + 型・階層（分類）+ マッピング（主張↔根拠） |
| 新規開発 | 設計 + タスク管理 + 検証 |
| 調査・分析 | 概念モデル + データフロー |

**共通ルール**: 最初は 3〜5 facet に絞る。scope をまたぐ link は張らない（alias で示す）。ノードには必ず summary を付ける。

各タイプの詳細なスコープ粒度・レイアウトルールは `facet/` ディレクトリにある（実装構造・処理フロー・依存グラフ・概念モデルの 4 タイプが整備済み）。

## 実行フェーズの標準 runtime

M3E を実行画面に使う PJ は、探索が終わったら **3-view runtime** に落とすのが標準。
これは **既定で必須** とし、使わない場合のみ `runtime_opt_out` を README / plan.md に明記して外す。

| view | 主 facet | 役割 |
|------|----------|------|
| **Progress Board** | タスク管理 + 依存グラフ | 今どの phase か、何が blocked / review / ready か |
| **Review** | reviews | 判断負債の batch review |
| **Active Workspace** | PJ に応じた実 facet | 実ファイル・実マップ・詳細根拠に降りる |

### 重要原則

- 3 view は **同一 master map の 3 scope** として作るのを第一選択とする
- Progress Board は summary でも、**必ず現物へ辿れる**こと
  - task → review Q
  - task → workspace
  - gate / blocker → 根拠
- scope をまたぐ関係は link ではなく alias で示す
- map から chat に送れない間は、人間が checkpoint を手動通知する
- `次フェーズへ` は人間の Go サイン
