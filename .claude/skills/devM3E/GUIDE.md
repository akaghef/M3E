# devM3E ユーザーガイド

エージェントとの協働セッションで、あなた（人間開発者）が何をすればいいかのガイド。

---

## セッションの始め方

Coworkで「開発」「次のタスク」「今日の作業」等と入力すると devM3E がトリガーされる。
エージェントが Assess phase を自動実行して現在地サマリーを提示するので、そこから始める。

**やること**: サマリーを確認し、次のいずれかで応答する。

| あなたの発言 | エージェントの動き |
|-------------|-----------------|
| 「〇〇を実装して」 | 指定タスクの Plan → Execute 開始 |
| 「推奨でいい」 | エージェント提案のタスクを実行 |
| 「まず△△を議論したい」 | 設計議論モードに入る |
| 「状況だけ見せて」 | Assess 結果のみ表示、作業開始しない |

---

## タスク完了のマーク方法

devM3E では以下の3条件を満たしたときに「完了」となる。

### 1. コミット
エージェントがコードをコミットする。あなたが確認後に：
- **LGTM** → エージェントが次に進む
- **修正して** → Execute に戻る

### 2. Daily Note
エージェント（doc-updater）が `dev-docs/daily/YYMMDD.md` を自動更新する。
内容に不足があれば「〇〇も追記して」と指示する。

### 3. Current Status
統合ロールのセッションではエージェントが `Current_Status.md` を更新する。
あなたが最終確認して問題なければその時点で完了。

**ショートカット**: エージェントがレビュー結果を提示した際に **「完了」** と一言返せば、上記3条件の残りを自動で処理する。

---

## レビューフェーズでの応答パターン

エージェントが変更の差分サマリー・テスト結果・判断ポイントを提示したら、以下のどれかで返す。

| 応答 | 意味 | エージェントの動き |
|------|------|-----------------|
| `LGTM` / `ok` / 無言で次の指示 | 承認 | Integrate → 次タスクへ |
| 「ここ直して：〇〇」 | 修正指示 | Execute に戻って再実装 |
| 「方針変えたい：△△」 | 方針転換 | Decision Pool に記録 → Plan に戻る |
| 「今日はここまで」 | セッション終了 | 終了手順を実行 |

---

## 設計判断が出たとき

会話中に設計判断が生まれた場合、エージェントが Decision Pool への記録を提案する。

**あなたがやること**:
- 判断に同意 → 「それで」（エージェントが Decision Pool に記録）
- 保留したい → 「保留で」（status=`blocked` で記録）
- 却下 → 「却下。理由は〇〇」（記録しない or superseded で記録）

正式な仕様（Spec / Architecture / ADR）への昇格は、あなたの明示的な承認が必要。エージェントが勝手に昇格することはない。

---

## 新しいタスクの追加

セッション中に気づいたタスクは、口頭で伝えるだけでいい。

- 「〇〇もやらないとだめだ」 → エージェントが Todo Pool に `pooled` で追加
- 「〇〇を次やって」 → Todo Pool に `ready` で追加し、次サイクルで着手
- 「〇〇は急ぎ」 → 現タスク完了後に即着手

Todo Pool のstate遷移:
```
pooled → ready → doing → done
                       → blocked
```

あなたが手動で Todo Pool を編集する必要はない。エージェントに伝えれば反映される。

## ブランチとPR

### Coworkセッション（統合ロール）
エージェントが `dev-beta` で直接作業する。PR は不要。

### Claude Code / Copilot セッション（visual / data ロール）
1. エージェントが担当ブランチ（`dev-beta-visual` / `dev-beta-data`）で作業
2. 完了時にPRを自動作成（base: `dev-beta`）
3. **あなたがPRをレビューしてマージ**（または統合ロールのエージェントに依頼）

### あなたが直接コードを書く場合
普通にブランチを切ってコミットする。エージェントにレビューを頼むなら「これレビューして」でreviewer エージェントが起動する。

## よく使うフレーズ集

| フレーズ | 効果 |
|---------|------|
| 「開発」「今日の作業」 | devM3E 起動 + Assess |
| 「次のタスク」 | Todo Pool から推奨タスクを提示 |
| 「〇〇を実装して」 | 指定タスクを Plan → Execute |
| 「テストして」 | verifier エージェント起動 |
| 「レビューして」 | reviewer エージェント起動 |
| 「daily書いて」 | doc-updater で daily note 更新 |
| 「status更新して」 | Current_Status.md 更新 |
| 「LGTM」「ok」 | レビュー承認 → 次へ |
| 「完了」 | 3条件を自動処理してタスク完了 |
| 「今日はここまで」 | セッション終了手順 |
| 「〇〇を覚えて」 | メモリに保存 |
| 「Todo Poolを見せて」 | 現在のプール一覧を表示 |

---

## セッション終了

「今日はここまで」と伝えれば、エージェントが以下を自動実行する。

1. 未コミット変更の確認（あれば警告）
2. daily note の最終更新
3. Todo Pool に残タスクを追記
4. Current_Status の更新

終了後、次のセッションでは前回の続きから Assess が始まる。

## ホワイトボード（マップ共有）

M3Eマップの `_wb` フォルダがエージェントとの共有ホワイトボードになる。
M3Eを起動していれば、エージェントがAPIで直接読み書きする。あなたはマップのUIで同じツリーをリアルタイムに見られる。

### 何が載るか

| フォルダ | 内容 |
|---------|------|
| `_wb/tasks` | タスクボード（doing/ready/done）。Todo Pool と自動同期 |
| `_wb/design` | 設計議論のツリー。選択肢・根拠・結論を構造化 |
| `_wb/brainstorm` | アイデア出し。収束後にdesignやtasksに移動 |
| `_wb/scratch` | 一時メモ。セッション間で残るが整理対象 |

### よく使うフレーズ

| フレーズ | 効果 |
|---------|------|
| 「マップ見せて」 | `_wb` のツリーをテキストで表示 |
| 「マップに書いて」 | 会話の内容をマップに構造化して追加 |
| 「タスクボード見せて」 | `_wb/tasks` の状態をサマリー表示 |
| 「〇〇を完了にして」 | `tasks/doing → tasks/done` に移動 + attributes更新 |
| 「ブレストしよう」 | `_wb/brainstorm` にアイデアノードを連続追加 |
| 「まとめて」 | brainstormのノードをグルーピング→designに移動 |
| 「Todo Poolと同期して」 | マップとTodo Pool.mdの双方向同期 |

### ホワイトボードとマップUIの関係

エージェントが `_wb` 配下にノードを追加すると、M3EのUI上にもリアルタイムで反映される（次回のマップ読み込み時）。あなたがUI上で `_wb` 配下のノードを編集しても、エージェントは次回読み込み時にその変更を認識する。

ただし `_wb` の外のツリーをエージェントが変更することはない（読み取り専用）。

## トラブル時

| 状況 | 対処 |
|------|------|
| エージェントが間違った方向に進んでいる | 「止めて」で停止。方針を説明し直す |
| テストが壊れた | 「テスト直して」で verifier → implementer が修正 |
| ブランチがconflictした | 「conflict解消して」でエージェントが対処。自動解消できなければエスカレーション |
| 仕様と実装が食い違っている | 「spec確認して」で reviewer が整合チェック |
| エージェントの判断が信用できない | 「根拠を見せて」でspec/コード根拠を提示させる |
