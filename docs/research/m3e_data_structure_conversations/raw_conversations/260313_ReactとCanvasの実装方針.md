**ChatGPT**: <sub>2026-03-13 20:35</sub>

その感覚は妥当です。

Canvasを捨てる、というより、**UIの主戦場をReact componentに置く**方向がよい、という整理です。M3Eの現在の前提でも、責務分離はすでに Model / View / Controller / ViewState に分かれており、描画とデータ管理を分離する方針なので、React中心に寄せやすいです。fileciteturn0file3

整理すると、次です。

まず、**CanvasはUI構造の代替ではない**です。Canvasは描画面であって、状態管理、入力制御、パネル構成、モーダル、ツールバー、検索、プロパティ編集、スコープ切替のようなアプリ全体のUIを綺麗に扱うのには向きません。そこはReact componentのほうが明確に強いです。

次に、あなたのアプリは単なる無限平面ではなく、**スコープ、帯域、選択状態、ドラッグ中プレビュー、Undo/Redo、AI提案承認**のような状態が多いです。これはReactで component 単位に分けたほうが破綻しにくいです。M3E仕様でも View状態と構造状態を分ける前提になっているので、その相性が良いです。fileciteturn0file2

したがって、実装方針としては、

**ReactをUIの骨格にする**
- AppShell
- ScopePane
- CanvasViewport
- NodeLayer
- Inspector
- Breadcrumb
- CommandPalette
- AIProposalPanel

**描画だけを専用層に閉じ込める**
- 少数ノードなら HTML/SVG
- 多数ノードや高速パンなら Canvas / Konva / Pixi

この二層構成が一番自然です。

特に重要なのは、**ドラッグ中の見た目を ViewState で持ち、drop確定時だけ Model を更新する**という方針です。これはすでに整理されていて、Reactとの相性が良いです。React側は drag preview と hover target を持ち、確定時に Command を発行するだけで済みます。fileciteturn0file0

実務上の結論だけ言うと、今はこう置くのがよいです。

- UI全体は React
- 状態管理は Zustand か Redux Toolkit
- Model は React外の純粋TSモジュール
- 操作は Command パターン
- 描画は最初 SVG か絶対配置 HTML
- 重くなった部分だけ Canvas 化

つまり、**React first、Canvas partial**です。

これなら、
- 仕様変更に強い
- AI提案UIを後付けしやすい
- スコープやプロパティパネルを自然に増やせる
- 独自エディタを進める場合も、Freeplaneラップに寄せる場合も資産を流用しやすい

という利点があります。方針転換の文脈とも整合します。fileciteturn0file4

逆に、最初から全面Canvas中心にすると、
- UI部品の分離が弱い
- 状態遷移の可視性が落ちる
- 入力系と描画系が密結合しやすい
- AI提案や補助パネルを載せたときに苦しくなる

ので、今の段階では得策ではないです。

結論は明確で、**M3Eのようなアプリは Canvas app ではなく React app として作り、その中の一部に canvas-like viewport を持つ**のがよいです。

はい。一般にそのほうがよいです。

正確には、**債務が局所化されている**ほうがよい、であり、その代表形が「コンポーネントにまとまっている」です。

理由は単純で、債務が全体ににじむと、
変更影響範囲が読めない、
壊れる場所が増える、
置き換え単位がなくなる、
という問題が出ます。

React component にまとまっていると、
その部品を境界として
- 後で作り直せる
- 代替実装に差し替えやすい
- 不具合の出所を追いやすい
- テスト範囲を切りやすい

ので、債務としてはかなり健全です。

ただし、少し補正が必要です。
**見た目のコンポーネントにまとまっているだけでは不十分**です。
本当に良いのは、次の単位で債務が閉じていることです。

- UI債務は UI component に閉じる
- 描画債務は renderer に閉じる
- 状態管理の債務は store / command 層に閉じる
- 外部アプリ依存の債務は adapter に閉じる

逆に悪いのは、
一つの component の中に
UI、
データ変換、
永続化、
外部連携、
操作ルール
が全部入っている状態です。
これは見た目は component にまとまっていても、実際は債務が圧縮されて埋まっているだけです。

なので結論は、

**債務はコンポーネント境界やモジュール境界に閉じ込めるべきであり、アプリ全体に拡散するより圧倒的によい。**
ただし、React component だけに押し込まず、責務ごとの境界に分けるのが本筋です。

M3E文脈なら、例えばこうです。

- ScopePane の都合は ScopePane に閉じる
- ノード描画の妥協は NodeRenderer に閉じる
- Freeplane連携の雑さは FreeplaneAdapter に閉じる
- 一時的なAI提案UIの雑さは ProposalPanel に閉じる

これなら後で一枚ずつ張り替えられます。

一番避けるべきなのは、**今だけ動く分岐が全体に散ること**です。これは最悪の債務です。

理由は3つです。

1つ目は、今の主要リスクが UI フレームワーク選定ではなく、**操作整合性**だからです。
M3Eで難しいのは、ドラッグ中の一時状態、確定時の反映、Undo/Redo、選択状態、スコープ切替です。ここは React を入れても自動では解決しません。

3つ目は、**負債の閉じ込め先が明確になる**からです。
Core、Renderer、UI Adapter を分ければ、たとえば UI だけ React にして、描画は Canvas のまま、という構成が自然にできます。

- 最優先: Core の境界設計
- 次: Renderer の責務分離
- 最後: UI を React にするかどうか

**Lit / Web Components > React > 素の DOM 直書き**
