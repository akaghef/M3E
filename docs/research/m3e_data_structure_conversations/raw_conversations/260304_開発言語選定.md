第二に、「消失バグなし最優先」「コマンドとしてログ化」は、状態遷移を純粋な操作列として再生できることが肝です。描画ロジックが実体状態を持つ設計だと、再生（リプレイ）で同じ結果を保証しにくくなります。逆に、描画はモデルのスナップショットから決定的に生成するだけにすると、復元検証が容易になります。

1) Core：Node/Folder/Alias制約、操作コマンド、Undo/Redo、スナップショット、保存形式

3) Renderer：ViewModelを入力にCanvasへ描画（副作用は描画のみに限定）

**グラフの接続構造（親子関係など）が実際に変わる瞬間だけ Model を更新する**設計は、MVC＋Command の標準的な運用と一致します。

操作を段階で整理すると次のようになります。

ドラッグ中（構造はまだ変わらない）

pointermove  
→ Controller  
→ ViewState 更新（ドラッグ位置、候補親など）  
→ View 再描画  

この段階では **Modelは不変**です。  
接続構造も変わっていません。

ドロップ確定（構造が変わる）

pointerup  
→ Controller  
→ ReparentNodeCommand 作成  
→ Model.apply(command)  
→ 接続構造更新  
→ history更新  
→ View再描画

ここで初めて

```
parent(A) = P → P'
```

という **グラフ構造の変更**が起きます。

このタイミングだけ Model を書き換えると、次の利点があります。

・Undo/Redo が単純になる  
・履歴が「意味のある操作単位」になる  
・パフォーマンスが安定する  
・ドラッグキャンセルが容易

エディタ系ソフトでは一般に

**preview phase**  
（ドラッグ中、Model不変）

**commit phase**  
（ドロップ時、Model更新）

という二段階に分けます。

KGFの場合も同じ構造になります。

preview  
・ノードの仮位置  
・候補親のハイライト  

commit  
・親変更  
・alias生成  
・昇格など

したがって  
「グラフの接続構造が変わるタイミングでだけ Model を更新する」という方針は、設計として妥当です。
