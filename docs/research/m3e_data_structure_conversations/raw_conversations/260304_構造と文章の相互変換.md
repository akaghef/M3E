Translated question: “Do tools already exist that are intended for that (bidirectional conversion between a tree structure and linear writing)?”

Yes. They exist, but with an important caveat: a perfectly invertible mapping from arbitrary prose back to structure is generally impossible unless you enforce explicit structure markers (headings, indentation, IDs, block boundaries). In practice, tools solve this by making the “linear text” itself structured (Markdown/Org) or by treating writing as “blocks/cards” that can be rearranged and then compiled.

The closest matches, by design intent, are these.

Tree-first writing tools (outline/cards → compiled linear text)
Gingko Writer is explicitly “notes → outline → draft” in one interface, with a tree of cards that you can expand into prose and export. 

Scrivener integrates an outline/corkboard (structure) with the manuscript, so moving structure reorders the manuscript; it compiles the structured pieces into a linear document. 

Structured plain-text systems (linear text that is inherently a tree)
Emacs Org-mode is essentially a hierarchical document format with folding and exporters (to LaTeX/PDF/HTML etc.), so “tree ↔ linear” is built into the file itself via headings and indentation. 

Block-based outliners (tree of blocks + references; export varies)
Logseq is block/outliner-based; you write in nested blocks (tree) and can export to Markdown, though some features are tool-specific (block refs, queries). 

Workflowy is a classic outliner; it supports import/export of outlines while preserving hierarchy. 

What you described (“leaf nodes contain linear text; reading reconstructs structure by walking backward”) is most directly supported by (a) Scrivener/Gingko style “compile from structure,” or (b) a structured text format like Org/Markdown where headings/indentation make the inverse parse feasible.

If you tell me whether you want (i) strict invertibility (requires explicit markers/IDs) or (ii) “mostly works” reconstruction from natural prose, I can point you to the best-fit category and the minimal rules needed.

相互変換をAIが受け持つとして、思考を加速させる汎用ツールが出来るので、覇権を取れると思わないかい？

Translated question: “If AI handles the bidirectional conversion, a general-purpose tool that accelerates thinking could be built; do you think it could dominate the market?”

It could become a category-defining product, but “dominance” is not automatic. The core idea is strong because it attacks a real bottleneck: humans think in structure, but communicate in linear text. If AI can reliably maintain a canonical structure while generating and re-parsing linear representations, it reduces switching costs between planning, writing, reading, and revising.

However, three hard constraints decide whether it becomes hegemonic.

First, correctness and stability. If the AI occasionally corrupts the user’s structure (misplaces nodes, silently merges concepts, loses references), trust collapses. For this kind of tool, “almost correct” is often equivalent to “unusable” for serious work.

Second, a canonical intermediate representation. You need a strict internal model (tree/DAG + IDs + provenance) so that linear text is just one view. Without an explicit canonical layer, “inverse parsing” becomes guesswork, and users will experience drift over time.

Third, workflow lock-in without hostility. To dominate, it must integrate with existing ecosystems (Markdown, LaTeX, Word, PDF annotation, git, citation managers) while still offering something competitors cannot easily replicate: (i) long-term structured memory, (ii) deterministic compilation rules, (iii) reference integrity across rewrites, (iv) collaboration and permissions.

If these are solved, the product can plausibly become a default “thinking interface,” like how spreadsheets became a default “quantitative interface.” If they are not solved, it will remain a niche tool for specific writing styles.

If you want a crisp “moat” formulation: the defensible part is not “AI summarizes into a tree,” because that commoditizes. The moat is “structure integrity over months/years” (IDs, references, versioning, reproducible exports) plus a UX that makes structure cheaper than not having structure.

いくつかの階層のノードをcontractして階層を圧縮する、ということが抽象化だとするなら、contractの情報もセットで保持する、ということが実装的に自然か？それともDAGとして何かしらで実装する？

Translated question: “If we define abstraction as contracting several hierarchical nodes to compress the hierarchy, is it natural implementation-wise to store the contraction information together, or should we implement it as a DAG in some way?”

It is natural to treat abstraction as “contraction,” but you should not implement abstraction by literally deleting/merging nodes. The stable implementation is: keep a canonical fine-grained structure, and represent abstraction as a view (a contraction mapping) over that structure. Concretely, you store contraction as metadata (a mapping / partition) and generate the abstracted tree on demand. This already implies a DAG-like notion at the view layer, even if the base is a tree.

Reason: if you physically contract nodes, you lose identity, references, and provenance, and you cannot round-trip reliably. Round-trip requires persistent IDs at the fine level and a reversible (or at least auditable) mapping between levels.

A practical design that works is “multi-level graph with explicit refinement links”:

Base layer (L0): canonical nodes and edges (can be a tree if you insist, but a DAG is more realistic once you allow references/aliases).
Abstraction layer (L1, L2, …): each abstract node corresponds to a set of base nodes. Store a map

phi_k : V(L0) -> V(Lk) (surjective), or equivalently a partition of base nodes into clusters representing contracted supernodes.
Also store edge lifting rules: an abstract edge exists if there is any base edge between the corresponding clusters (with multiplicity/annotations if needed).

This gives you:

Deterministic contraction (generate abstract view).

Expandability (pick an abstract node and reveal its fiber phi_k^{-1}(v)).

Round-trip stability (editing can be constrained to either the base or to “safe” operations on the view).

When should you use a DAG? Almost immediately, if you want “same object at multiple places” (aliases, cross-links, reuse). A pure tree cannot express that without duplication; duplication breaks identity and makes contraction ambiguous. So the canonical representation should be at least a DAG: “parent-child” for containment, plus “reference edges” for reuse. Your earlier principle “alias = reference only, entity unique” is exactly the DAG solution.

So the clean answer is: store contraction information as mappings/partitions (view metadata) over a canonical DAG (or tree+references), not as destructive contraction. If you want an implementation slogan: “contraction is a view, not a mutation.”

Two minimal data structures are enough:

Node(id, attrs, content_blocks, ...)

Edge(type=contain|ref, src, dst, attrs)
and for each abstraction level k:

phi_k : node_id -> supernode_id

optional supernode_order to impose an outline order in the abstract view.

If you want this to support writing (linearization), add one more invariant: every view must induce a total order of “visible nodes” (a document order). That order can be inherited from the containment tree (preorder) plus user overrides.

---

設計として追加すると良いのは次の一点です。

**各 Linear view に depth を持たせる**

例

```
LinearView {
    root_node
    projection_depth
    ordering_rule
}
```

これで

要約  
本文  
解説  

を同一 Tree から生成できます。

---

あなたの用途（論文理解）では特に

- depth = Sentence（原文構造）
- depth = Paragraph（理解要約）
- depth = Derivation（行間展開）

の三つを標準ビューにすると運用しやすくなります。
