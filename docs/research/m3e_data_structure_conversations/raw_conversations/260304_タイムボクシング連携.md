連携の実装方針は、まず「KGF側が時間箱を持つ」か「外部ツールの時間箱にKGFを寄せる」かで分けるのが簡潔です。どちらでも、主構造（Node/親子）は増やさず、タスクは text node と属性（期限・見積・実績）で表現し、ビュー（Rapid）で「今日の箱」を切るのが安全です。MVC/Command/Undoの前提とも整合します（時間箱の開始・終了・移動をコマンド化）。fileciteturn0file1

機能としては最小でも次が効きます。
1) 「タイムボックス開始」：選択ノード（またはInboxの種）を実行中にする。開始時刻を記録。
2) 「タイムボックス終了」：終了時刻と所要時間を記録し、必要ならFlash→Rapid昇格を同時に促す。
3) 「当日プラン生成」：Rapidスコープ内から、見積時間と優先度で今日の箱（例：25分×N）を自動提案。ただし確定は承認制。
4) 「ログの再利用」：Deep側で、実績が溜まった作業の標準所要時間（自分専用）を推定し、見積の初期値に反映。

設計上の注意点は、時間計測は「構造」ではなく「ログ」として扱うことです。つまり、ノードの親子関係は不変のまま、別テーブル（WorkLog）に `nodeId, start, end, boxId, memo` を積む。これなら見えすぎ回避（スコープ外のログを既定で見せない）にも乗ります。fileciteturn0file0

ワークフロー定義（「収集→選別→箱詰め→実行→反映」）
1. 収集（Inbox化）
- 入力源：未処理メモ、思いつき、メール、会議メモ、締切イベント、昨日の未完了。
- 出力：Flash(Inbox)に CandidateTask の種を追加（DeepのTaskに直結させない）。

2. 正規化（候補化）
- 各種を「次アクション（動詞＋対象）」に整形し、参照先（Project/Goal/Role）を1つ以上付与。
- 見積（例：5/15/30/60分）とエネルギー要求を付与。
- 依存関係があるものは “blockedBy” を付与し、ブロック元も候補として生成。

3. 今日の可処分枠の抽出（予定グラフから）
- 予定グラフから当日の Availability を抽出。
- 可処分枠を「箱（例：25分、50分）」に離散化し、箱の連結（連続枠）も保持。

4. 選別（候補集合の確定）
- ルール例（固定の優先順位で十分です）
  - 締切が近い Project の候補を優先
  - blocked のものはブロック解消候補を先に
  - 高エネルギー枠（午前など）には高エネルギー要求の候補を割当
  - 小タスクは隙間箱に詰める
- 出力：今日の CandidateTask 集合（Rapidの一時ビュー）

5. 箱詰め（タイムボクシング計画の生成）
- Availability の箱列に CandidateTask を割当し、各箱に「1タスクのみ」を原則とする。
- 例外：5分タスクは「雑務箱」としてまとめるが、箱自体は1つのタスク扱いにする。
- 出力：TimeBoxPlan（箱の列）。これは「ビュー状態」であり、深い構造ではない。

6. 実行（計測・中断管理）
- 箱開始：activeBoxId と startTime を記録（WorkLog）。
- 箱終了：endTime、成果メモ、次の一手（継続/分割/保留）を記録。
- 中断：割り込みはFlashに投げ、現在箱は「中断理由」とともにログ化して再配置候補に戻す。

7. 反映（Deepへのフィードバック）
- 当日終了時に、WorkLogを集計してDeepへ反映（ただし自動更新は最小限）
  - Taskの標準見積の更新（移動平均など）
  - Project進捗の更新（完了カウント、残工数推定）
  - Constraintの更新（集中可能時間帯の推定など）
- 重要：反映は「提案」として出し、確定は手動にする（自己像の破壊を防ぐ）。

実装上の境界（重要）
- Deep：人物・目標・プロジェクト・制約・予定（事実／準事実の層）
- Rapid：今日の候補と箱計画（運用の層、可逆）
- Flash：割り込みと未整理（入力の層）

この定義で、あなたをグラフで表現しつつ、予定もグラフで管理し、そこから今日の候補を収集して箱詰めして実行し、ログをDeepに還流できます。

- TaskはDeepに「標準分解」まで持つか、DeepはProjectまででTaskはRapidに寄せるか。

   - Block解除Task

- date  
- availableBlocks  
- candidateSet  
- proposedBoxes

Planningは「構造を変更しない」。  
DeepのProjectやTaskを書き換えず、Rapidに対して実行案を生成するだけです。

1日の情報流  
Deep / Schedule  
↓  
Flash.Planning（15分）  
↓  
Rapid（実行）  
↓  
Flash.Review  
↓

**ChatGPT**: <sub>2026-03-04 17:51</sub>

理論的には可能ですが、「すべて」は現実的には困難です。理由は三つあります。

第一に、スケジュールは分散して存在します。  
カレンダー（Google、Outlook）、メール、チャット、タスク管理、研究ノート、個人的メモなど、情報源が多数あります。APIで取得できるものは統合可能ですが、非構造情報（メール本文、会話、頭の中の予定）は完全自動収集できません。

第二に、予定には「確定度の差」があります。  
実際の予定には次の層があります。  
- 固定予定（会議、授業）  
- 準固定予定（締切、提出）  
- 希望予定（やりたい作業）  
- 潜在予定（思いつき、未整理タスク）

多くのツールは固定予定しか扱いませんが、あなたが構想しているグラフはこれらすべてを扱う必要があります。これは可能ですが、入力の負担をどう減らすかが問題になります。

第三に、時間は唯一の資源ではありません。  
実行可能性は時間だけで決まりません。  
- エネルギー状態  
- 集中可能な環境  
- 依存関係（誰かの返信待ちなど）  
- 必要資料の有無  

そのため「スケジュール統合」は実際には **状態管理グラフ** に近くなります。

ただし、あなたの設計（Deep / Rapid / Flash）では、かなり現実的な形にできます。重要なのは「完全収集」を目指さないことです。実装としては次の三層が実用的です。

1. 固定予定の自動収集  
カレンダーAPIから取得してScheduleグラフに反映。

2. 準固定予定の半自動抽出  
メールやメモから締切やイベントをPlanning時に取り込む。

3. 潜在予定のFlash入力  
思いつきや割り込みはFlashに捕捉し、Planningで候補化。

この構造なら、実質的には「生活の大部分の予定」はグラフで管理できます。完全ではありませんが、実用上はほぼ十分です。

むしろ本質的な問題は別にあります。  
それは **「いつ何をやるか」を決めるアルゴリズム** です。  

スケジュールを集めること自体は技術的に難しくありませんが、  
Deepの目標・締切・エネルギー・依存関係から「今日の最適行動」を出す部分が最も難しい設計になります。
