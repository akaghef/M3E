**ChatGPT**: <sub>2026-03-13 18:13</sub>

結論だけ述べる。  
用途によって最適な方式が異なる。大きく4系統ある。

**1. ツリー型（アウトライナー）**  
例：Freeplane、Logseq、Workflowy。  
構造は親子関係のみ。編集速度が速い。  
思考整理や議論メモには適している。  
弱点は、同一ノードの再利用や多重参照が弱いこと。

**2. グラフ型（リンクベース）**  
例：Obsidian、Roam Research。  
ノートを相互リンクする。  
文書間の関係は表現しやすい。  
ただし、構造が曖昧になりやすく、長期的に整理が崩れやすい。

**3. データベース型**  
例：Notion、Airtable。  
各ドキュメントをレコードとして扱う。  
属性・フィルタ・クエリが可能。  
構造は明確だが、文章編集はやや遅い。

**4. AST／構文木型（ソースコード型）**  
例：Markdown＋Git、LaTeX、Org-mode。  
ドキュメントを構文木として管理する。  
差分管理、生成、再構成が容易。  
ただし視覚操作は弱い。

実務では次の組み合わせが多い。

- **執筆**：Markdown / LaTeX  
- **構造整理**：アウトライナー  
- **横断検索**：リンク型  
- **メタ情報管理**：DB

あなたの発言を見る限り、  
- scope  
- 単一ID  
- ノード再利用  
- command更新  

を重視している。  
この場合、通常のツリー型では足りない。  
**DAG（有向非巡回グラフ）＋階層ビュー**が適切である。

実装モデルは次の形になる。

```
Node
 ├ id
 ├ content
 ├ attributes
 └ children (ordered)

Edge
 ├ parent
 └ child

View
 ├ root
 └ layout
```

重要なのは次の原則。

- 実体は **Node一つだけ**
- 表示構造は **Viewで決まる**
- 編集は **Commandログ**

この方式は

- IDE
- ASTエディタ
- 知識グラフ

で広く使われている。

要するに、  
**ドキュメント＝ファイルではなく、ノード集合として扱う設計**が最も安定する。
