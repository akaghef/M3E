主な制限は次の5つ。

1つ目は、**技術基盤が Java Swing + OSGi のデスクトップアプリ**であること。

**ChatGPT**: <sub>2026-03-13 23:48</sub>

可能。ただし条件付き。

まず前提として、**Freeplane は GPL ライセンス（GPLv2 以降）**で公開されている。したがって、ソースコード自体は取得・改変・再配布が可能である。

ただし GPL には次の制約がある。

1. **派生物は GPL で公開する必要がある**  
   Freeplane のコードをコピー・改変して自分のアプリに組み込むと、そのアプリ全体も GPL で公開しなければならない。

2. **バイナリ配布時はソース公開義務がある**  
   修正版を配布する場合、対応するソースコードも公開する必要がある。

3. **リンクも派生物と見なされる可能性がある**  
   Freeplane のクラスを直接リンクして自分のプログラムに組み込む場合、通常は GPL の対象になる。

したがって、実装方式ごとに扱いは次のようになる。

**問題が小さい方法**
- Freeplane をそのまま別プロセスとして起動する  
- `.mm` ファイルを読み書きするだけ  
- UI 操作を外部から自動化する  
- IPC やファイル連携で通信する  

この場合、通常は派生物と見なされない。

**GPLを受ける可能性が高い方法**
- Freeplane の Java コードをコピーして使用  
- Freeplane をフォークして自分のアプリとして配布  
- Freeplane の内部 API に直接リンクするプラグインを同梱配布  

この場合、あなたのコードも GPL になる可能性が高い。

整理すると次の三段階になる。

1. **外部ツールとして使う**  
   → ライセンス問題はほぼ発生しない。

2. **プラグインとして拡張する**  
   → GPL に従う必要が出る可能性がある。

3. **コードを流用して新しいアプリを作る**  
   → アプリ全体を GPL で公開する必要がある。

したがって、M3E の現在の方針（既存アプリを使いながら外側に思考支援層を作る）なら、**Freeplane のソースを流用する必要はなく、むしろ流用しない方が安全**である。

主に5通りです。

1. 外部アプリとしてそのまま起動する  
自作側は別プロセスで動き、ファイル入出力や起動引数だけ使う。最も安全です。

2. ファイル形式だけ使う  
既存アプリの保存形式を読み書きし、自作側で生成・変換・差分管理を行う。UIは既存アプリ、知能は自作側に置けます。

3. プラグインで拡張する  
既存アプリが拡張機構を持つなら、その上に機能を足す。内部状態に触れやすいが、保守とライセンスの制約が重くなります。

4. スクリプトやAPIで操作する  
公開API、マクロ、スクリプト機構があれば、ノード作成や編集を外部から呼ぶ。改造より軽いです。

5. UI自動操作で使う  
クリック、キー入力、画面認識で外側から操作する。最後の手段です。壊れやすいです。

実務上の切り分けは次です。

- 最も軽い: ファイル連携
- 使いやすい: API / スクリプト
- 深く入る: プラグイン
- 危ない: UI自動操作
- 最も重い: ソース改変

Freeplane なら、現実的なのはこの順です。

- `.mm` を読む・書く
- 必要ならスクリプトや拡張点を使う
- それで足りなければプラグイン
- ソース流用や本体改造は最後

つまり、既存アプリは普通は「中身を借りる」のではなく、  
**UI、保存形式、編集機能のどれかを借りる**  
という使い方になります。
