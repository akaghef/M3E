# 01. メタ機能とは何か — コンセプトとユースケース

M3E の **メタ機能** = 「M3E が M3E 自身を扱うための機能」。
具体的には、M3E の開発・改善・拡張・観察 を M3E のマップ上で行う仕組み群。

## なぜメタ機能を考えるのか

### 1. 既に部分的に実装されている
- DEV map で M3E 開発自体を管理している（`feedback_agent_status_routine`）
- agent と人間が同じマップを canvas として共有（`canvas-protocol` skill）
- triage / scratch / Todo_Pool が思考プールとして稼働
- → **完全な無からの設計ではなく、既存パターンの拡張**

### 2. 自己改善の複利効果
- M3E が自分の不便を観察 → 改善
- 改善された M3E が次の不便をより速く観察 → さらに改善
- project_projection_vision（半年で実用化）に直結する **加速装置**

### 3. メタ機能特有の制約
- 「akaghef 個人の生産性ツール」という出自（OSS化は副次目的）
- policy_privacy（機微情報を LLM に送らない）と矛盾しない設計が必要
- 「メタばかり考えて本業が止まる罠」（feedback_no_interpretation 系）

### 4. 既存メモリとの直結
- `project_alglibmove_dogfood`: M3E lacks cognitive layer; D-first valid; 21+ feature gaps
  → 自己ドッグフードで発見された 21+ ギャップが既に存在
- `feedback_agent_status_routine`: マップ更新の儀式が既存
- `feedback_todo_single_file`: TODO 集約パターン
  → これらをメタ機能の基礎として活かせる

## 前提となる二面性

メタ機能は常にこの二軸の上に置かれる:

| 軸 | 個人ツール側 | 公開プロダクト側 |
|---|---|---|
| **目的** | akaghef の生産性最大化 | 他者が触れる土台 |
| **設計の重力** | 「自分が便利」 | 「他人が壊さない」 |
| **典型機能** | DEV map ドッグフード | フィードバック UI / マーケット |
| **時間軸** | 即時（来週使う） | 長期（半年〜数年） |
| **失敗時のコスト** | 自分が困る | 信頼を失う |

## ユースケース 5+

### UC-1. akaghef が研究中に「M3E のここが不便」と気づく（個人ツール側）
- 朝 8:00、論文を書きながら別ノードを参照したい
- 現状: 別ウィンドウで開かないと並列参照できない
- 望み: その場で「不便メモ」を投げて、後で自分で対処
- 関連論点: Mt-C1（改善提案の入口）, Mt-D2（収集トリガ）

### UC-2. AI agent が DEV map のタスクを完了する（既存パターン）
- 23:00 に夜間 agent タスクが走る
- agent が「テスト追加」「バグ修正」を実装、Agent Status を更新
- 翌朝 akaghef が確認、レビュー、マージ
- 関連論点: Mt-B3（agent 自律タスクとマップ表現）, Mt-C3（人間ゲート位置）

### UC-3. M3E 自身が「最近遅い」と気づく（自己観察）
- マップが 1000 ノード超で操作レイテンシが体感悪化
- M3E が自分のレンダ時間を計測、akaghef にアラート
- 関連論点: Mt-E1（何を測るか）, Mt-E4（自己観察ダッシュボード）, M1（巨大マップ対応）

### UC-4. 共同研究者が試用、フィードバックを返したい（公開側）
- akaghef が共同研究者に M3E を渡す
- 「ここでつまづいた」「この機能が欲しい」を共同研究者が送る
- akaghef は自分の DEV map にそれが届く
- 関連論点: Mt-D1（チャネル）, Mt-D3（粒度: snapshot 添付？）

### UC-5. 別研究室が M3E を fork して教育用にカスタマイズ（公開側 / 遠未来）
- ラベル変更・暗号化機能 OFF・ガイド付き起動
- core を触らずに上書きできる構造があるか
- 関連論点: Mt-G1（config と core 分離）, Mt-F1（拡張ポイント形式）

### UC-6. akaghef が「自分が M3E をどう使ってきたか」を振り返る
- 半年使った結果、最も触ったノード/最も改善したカテゴリ/最も停滞した領域
- 自己定量化レポート
- 関連論点: Mt-E4, Mt-B4（ドッグフード度合い指標）, I3（差分アニメ）

### UC-7. M3E プラグイン作者が「自分のプラグイン」をマーケットに登録
- 公式 store なのか、git URL のコピペなのか、npm install なのか
- 関連論点: Mt-F2, Mt-F3

## 既存ツールとの差分

| 既存ツール | M3E メタ機能との差 |
|---|---|
| **Obsidian + コミュニティプラグイン** | プラグイン文化は強いが「自己改善ループ」は人間頼り |
| **Roam / Logseq の Datalog** | 自己クエリは強いが UI 改変まではいかない |
| **Notion の Database + Webhook** | 自己観察できるが、AI agent との二人三脚はない |
| **VSCode + GitHub Copilot** | コードへの自己改変は AI 化しているが、マップ UI/UX 自体は固定 |
| **Linear / Jira (自社利用)** | 開発タスク管理は強いが「自分自身を改善する道具」ではない |

M3E のメタ機能の **特異点**:
- マップ = canvas という既存パターン上に **agent と人間が同居**
- 改善対象（M3E自身）と改善の道具（M3E）が同一物
- akaghef 個人の研究目標（科研費等）と直結している

## 「メタ過剰」の警戒

メタ機能はそれ自体が魅力的すぎて、本業（研究）を侵食する危険がある。
具体的な失敗シナリオ:
- 自己観察ダッシュボードを作るのに 2 週間 → その間研究は止まる
- 改善提案ノードが 100 件溜まる → どれも実装されないまま心理的負担
- プラグイン仕様策定に没頭 → 実利用者は自分1人だけ

論点 **Mt-A3** （メタ過剰の回避線）として、メタ機能には予算（時間・複雑度）の上限を最初から組み込むべき、という視点を 05_meta_traps_and_strategy.md で深掘る。
