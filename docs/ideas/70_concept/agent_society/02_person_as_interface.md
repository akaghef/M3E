# As2: 人 = クラスターのインターフェース、agent 組織 = カプセル化された private 実装

## 命題

> 「人間が上流化する」とは、人間が偉くなることではなく、
> **人間がその人の agent クラスターへの唯一の公開インターフェースになる**ということ。
> クラスター内部の agent 組織構成は、他者から見れば実装詳細であって良い。

## ソフトウェア設計との対応

これはオブジェクト指向のカプセル化そのものである:

| OOP 概念 | 組織への写像 |
|---|---|
| public interface | 人（対外的にやり取りする唯一の窓口） |
| private field / internal state | その人の agent クラスター構成・使っているモデル・内部の会話 |
| method call | inko 経由の依頼・引き継ぎ |
| encapsulation | 「他人の agent に直接指図しない」という不文律の構造化 |
| information hiding | RQ4 の帰属境界（backend は A-sys、公開面は projection のみ） |

**今日確定した要求定義(RQ4)が、まさにこのカプセル化の最小実証だった。**
「監視 backend は個人システムと密結合であり A-sys に帰属。M3E には projection
契約のみを出す」という決定は、1 人分のクラスターに対して「内部実装は隠し、
契約だけを外部に見せる」という OOP の原則を適用した結果に他ならない。

## なぜカプセル化が必要か（対立命題への応答）

対立命題: 「全部見える方が透明性が高く、組織として健全ではないか」。

応答:

1. **全裸の透明性は監視コストを可視化する側に押し付ける。** 誰かの agent 組織の
   全生ログを他人が読める状態は、読む側に「読み解いて判断する」という新しい
   負荷を生む。今日の要求定義で「概要欄は手書きメモでいい」と決めたのは、
   まさにこの負荷を「本人が要約して渡す」側に置く選択だった
2. **agent 組織の構成は頻繁に変わる。** モデル切替、prompt 調整、内部の試行錯誤。
   これを公開契約にすると、内部の実験のたびに外部との契約が壊れる。
   カプセル化は「内部の自由な変更」と「外部との安定した契約」を両立させる、
   ソフトウェア工学で最も基本的な理由がそのまま当てはまる
3. **失敗の可視化は本人の選択であるべき。** agent が迷走した過程を全部晒すか、
   結論だけ晒すかは、信頼関係の段階に応じて本人が調整できるべき
   ([50_collab/privacy_security](../../50_collab/privacy_security/README.md) の脅威モデルと直結)

## クラスター内部の構造（参考: agent 側で既に確定している semantic model）

codex-agent-mapping スレの結論をそのまま人単位に外挿できる:

- **Agent = Session** の 1:1 対応（機械採番）が、クラスター内部では「その時点で
  誰が働いているか」の実体になる
- **(team, agent_name) は個体識別ではなくラベル** ── これは会社スケールでも同じ形で
  現れるはず。「今日の akaghef 番」担当が実は別の agent instance に交代していても、
  外部の interface（人）が変わらなければクラスター外は気づく必要がない
- **edge は runtime 層(mailbox接続=現在状態)と semantic 層(会話=不変履歴)の二種**。
  クラスター外に見せるべきは semantic 層の要約だけで、runtime 層(誰が今どの
  session を張っているか)は内部実装

## 代替モデル（対立案）

| 案 | 構造 | 弱点 |
|---|---|---|
| **フラット直結** | 全 agent が全 agent と直接会話可能 | As4 で扱う致命的問題。責任の所在が消える |
| **完全階層（従来の組織図）** | 人の下に agent がぶら下がる木構造、報告は上にのみ | 群個体の水平協業(agent同士の効率的な連携)を殺す。今のsubagent階層モデルの限界そのもの |
| **人=界面モデル（採用候補）** | 人がクラスターの唯一の公開 API、内部は自由 | 界面のスループットが人間の処理速度に制約される（後述のボトルネック） |

## 残る設計課題: 界面のスループット

人が唯一のインターフェースだと、クラスター間のやり取り速度は結局
人間の読み書き速度に制約される。これは意図的なボトルネックか、解消すべき制約か?

- 意図的だとする立場: 人間の判断が介在する速度が、そのまま組織の意思決定速度の
  上限であるべき（暴走の歯止め）
- 解消すべきとする立場: 定型的なやり取り（進捗確認・日程調整）は人を介さず
  agent 間で完結してよく、**非定型（コミットメントの変更・優先度判断）だけが
  人を通る**べき

後者を取る場合、「どこまでが定型か」の線引きが As4 の scoped channel 問題と同じ形になる。
この 2 ファイルは実質同じ問題を異なる角度（個体の視点 vs 会社構造の視点）から見ている。
