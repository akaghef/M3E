# As1: 3 層プロトコルスタックの一般化（inko / agmsg / sync）

## 命題

> 通信・状態複製・意味伝達を 1 枚のチャットに潰すのが今の組織の標準形であり、
> それこそが「チャットが沼になる」構造的原因である。層を分離すると、
> 各層が独立に壊れ・独立に直り・独立にスケールする。

## 現状の 3 層（akaghef のスタックでの実例）

| 層 | 役割 | 実例 | 性質 |
|---|---|---|---|
| **sync（状態層）** | ファイル・DB の複製、正本の配布 | Syncthing、git、Hermes state.db | 冪等・差分・衝突解決が主戦場。**メッセージという概念を持たない** |
| **agmsg（搬送層）** | 宛先へメッセージを届ける。mailbox は個体識別ではなくラベル | agmsg の (team, agent_name) 設計 | **宛先が不在でも送れる**。配送保証が主戦場。中身の意味は関知しない |
| **inko（意味層）** | 指示・引き継ぎ・コミットメントという「意味を持った行為」 | inko 通信スキル | 「何を頼んだか」「何を約束したか」が主戦場。配送方法は agmsg に委譲 |

## なぜこの分離が一般解になり得るか

**1. Slack/Teams 型チャットは 3 層を 1 層に潰している。** 1 メッセージが同時に
「状態更新の通知」「配送」「約束の表明」を兼ねる。結果、検索しても「何が今の正本か」
「何を誰が約束したか」が本文の自然言語からしか復元できない。これがチャットが
ナレッジベースとして機能しない根本理由。

**2. 層を分けると「不在」への耐性が層ごとに最適化できる。** sync は不在の間も
複製が進む(non-interactive)。agmsg は不在でも mailbox に届く(store-and-forward)。
inko だけが「相手の応答」を要求する同期的な意味を持つ ── **人間は agent よりずっと
不在時間が長い**ので、この3層構造は agent より人間でこそ効く（akaghef の直観の核心）。

**3. 障害が層内に閉じる。** sync が止まっても(オフライン)、後で追いつける。
agmsg の mailbox が積み上がっても、意味(inko)の緊急性とは独立に処理できる。
1 層に潰れていると、1 つの障害が「今の状態が分からない」「頼まれごとが分からない」
の両方を同時に引き起こす。

## 代替トポロジー（対立案）

| 案 | 構造 | 弱点 |
|---|---|---|
| **T-flat（現状の大多数の組織）** | チャット 1 層に全部乗せる | 検索不能、正本不在、約束が流れて消える |
| **T-2layer** | sync + inko のみ(agmsg省略、直接 push) | 宛先不在に弱い。相手が offline だと詰む |
| **T-3layer（採用候補）** | sync / agmsg / inko の完全分離 | 層間の翻訳コストがある。3つのツールを跨ぐ学習コスト |
| **T-mesh** | 層を作らず、全ノードが3役を持つ P2P | 統制不能。会社スケールでは非現実的（研究ネットワーク向き） |

T-3layer が現状の最有力候補。理由は「今日の M3E で実際に動いている」こと ──
仮説ではなく akaghef の日常運用の記述である点が他の思想ブレストと異なる。

## Track U / ADR_009 への接続

Track U の合流契約（[handoff_track_u_orchestration_seam_260719.md](../../../tasks/handoff_track_u_orchestration_seam_260719.md) §2）
は、実は同じ 3 層分離を計画平面に適用したものだった:

- **projection JSON = sync 層**（状態の複製・配布。意味を持たない静的スナップショット）
- **graph operation 形式 = agmsg 層**（宛先=substrate に対する配送。中身の意味は問わない）
- **Goal/Task/Gate の意味論(DECOMPOSES/ASSIGNED_TO/BLOCKED_BY) = inko 層**（何を約束し、
  何が誰にブロックされているか）

つまり Track U を設計した時点で、無自覚にこの 3 層モデルを踏襲していた。
As1 はこれを**意図的な設計原則として明文化**する提案でもある。

## 実装可能性（参考、最終決定ではない）

- 短期: 何もしない。既存 3 ツールがそのまま実証環境
- 中期: agmsg の message 実体に「これは inko 由来か」のタグを持たせ、
  意味層の trace を搬送層のログから復元可能にする
- 長期: sync/agmsg/inko の 3 層を明示的なプロトコルスタックとして文書化し、
  第三者(チーム・別プロダクト)が同じ層構造を再実装できる仕様に格上げする
