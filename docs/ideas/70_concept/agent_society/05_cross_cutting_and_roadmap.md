# 横断観察・接続表・ロードマップ・未決質問

## 横断観察

**1. この思想は「予言」ではなく「今日すでに部分実装されている」。** 他の
70_concept/philosophical の各命題（未完成原則・意図的忘却など）は理念が先行し
実装が追う構造だが、As0〜As4 は逆で、**RQ4(backend=A-sys/表示=M3E)・Track U の
gate 意味論・codex-agent-mapping の semantic model が先に存在し、この文書が
それを事後的に一般化した**。これは強い実証性を持つ一方、「たまたま今週やった
作業を大げさに理論化しただけ」というリスクも併せ持つ（05_open_questionsで扱う）。

**2. As1〜As4 は 1 本の依存鎖になっている。** 層分離(As1)が無ければ人=界面の
カプセル化(As2)は技術的に支えられず、As2 が成立しないと会社マップの edge が
「約束」として意味を持たない(As3 は組織図に退化する)。As4 の scoped channel問題は
As2 のカプセル化を守ったまま効率化するための、唯一残った調整弁。**どれか1つを
外すと残り全部が別の理論になる。**

**3. M3E 自身のスコープ判断と整合する。** [50_collab/collaboration/README.md]
(../../50_collab/collaboration/README.md) は既に「Google Docs 級の完全協調エディタは
範囲外」「M3E のスコープはハブ役・非同期レビュー基盤・引き継ぎ場に絞る」と
方針化している。As0〜As4 はこの方針の**理論的裏付け**になっている:
M3E が目指すべきは「全員が同時編集する場」ではなく「約束が記録され、
境界壁越しに読める場」であり、これはまさに非同期ハブの定義そのもの。

## 未決質問（次回このテーマに戻る時のために）

1. **Q-As-1**: この理論は「今週やった作業の後付け一般化」ではなく、実際に
   6ヶ月後・1年後の製品判断を左右する強度を持つか。判断材料が増えるまで
   保留でよいが、次に Track U/D の大きな意思決定をする際に **明示的にこの
   文書を参照して検証**すること
2. **Q-As-2**: As2 の「界面のスループット」問題（人間の処理速度が組織速度の
   上限になる）は、意図的な歯止めとして残すべきか、scoped channel で
   積極的に緩和すべきか — akaghef 自身の運用実感が溜まるまでは判断材料不足
3. **Q-As-3**: 会社マップ(As3)の scope 設計 — 単一 flat graph + scope 折りたたみで
   本当にスケールするか。M3E 本体の viewer が現状どこまでの node 数で
   実用的かの実測(708 node の U1 board 実データが既に手元にある)を、
   会社規模の想定 node 数と照合する検証がまだ行われていない
