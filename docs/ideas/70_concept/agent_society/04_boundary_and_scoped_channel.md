# As4: projection を「境界壁」として一般化、クラスター横断直結という未決問題

## 命題

> projection 契約は単なる技術的なデータ形式ではなく、
> **個人の主権と組織の可視性を両立させる境界壁の実装形**である。

## 境界壁としての projection

今日の要求定義(RQ4)で確定した「backend は A-sys 帰属、M3E には projection のみ」は、
一見するとホスティング先の都合(密結合・プライバシー)の話に見えるが、
As2 のカプセル化を技術的に実現する **唯一の実装ポイント**でもある。

境界壁が機能するための 3 条件（今回の決定が満たしている条件を明示）:

1. **境界の内側は境界の外側の形式に縛られない**（A-sys 内部の backend 実装は自由）
2. **境界を通過するのは契約化されたデータだけ**（projection JSON、schema 凍結）
3. **境界の維持者(本人)が何を通すかを決められる**（概要欄=本人の手書きメモ、
   RQ2 の介入ラダー L0 が「読む」から始まり本人の許可なく上位に進まない）

これが成立する限り、**クラスターが何個に増えても境界壁の形は変わらない**。
1 人分で実証した契約が N 人に複製可能というのが、As0 の中核命題の技術的根拠になる。

## 未決の設計問題: クラスター横断の agent 直結

効率化の圧力は必ず「なぜ人を介して伝言ゲームするのか、agent 同士で直接
話せばいいのに」という要求を生む。これを無条件に許すと:

- As2 のカプセル化が崩れる（他人の agent 組織の内部に直接アクセスすることになる）
- As3 の「約束グラフが正本」という前提が崩れる（人を介さない合意は、誰の
  コミットメントとして graph に記録されるべきか不明になる）
- 責任の所在が消える（agent が agent に約束した、は法的にも運用的にも誰の約束か
  曖昧）

対立命題への配慮: 完全に禁止すると、As2 で触れた「界面のスループット」問題
（人間の処理速度が組織の意思決定速度の上限になる）が悪化する。定型的な
やり取りまで人を介すのは非効率。

## 選択肢（対立案、決定はしない）

| 案 | 仕組み | 評価 |
|---|---|---|
| **完全禁止** | agent は自クラスター内でのみ通信可能 | 安全だが As2 の「界面スループット」問題を悪化させる |
| **無制限直結** | agent 同士が自由に mailbox を持つ | As2/As3 が崩壊。今日の議論全体の前提と矛盾するので不採用が濃厚 |
| **scoped channel（有力候補）** | 人の承認で一時的に開通する channel。開通の事実・範囲・期限が
  inko 層のコミットメントとして graph に記録される | 「開通ログ」自体が As3 の約束グラフの
  一部になるため一貫性が保てる。承認 UI・失効設計が新規に要る |
| **proxy agent** | 各クラスターに「対外交渉担当 agent」を 1 体置き、それ同士だけが
  直結できる | 人間界面の負荷は下がるが、「対外交渉 agent」自体の信頼境界をどう
  設計するかという同型の問題が 1 段階下にずれるだけ |

**scoped channel が現時点の最有力**。理由: 「開通の事実自体を約束として記録する」
という発想が、As1〜As3 で確立した「graph が正本」という原則と矛盾しない唯一の案。
proxy agent 案は問題を先送りするだけで解決にならない。

## Track U board との接続（実装可能性の芽）

Track U の凍結語彙に将来追加され得る node/edge の候補（**現時点では追加しない**。
schema v0 は Track D との凍結契約であり、変更には両者合意が要る — ADR_009 §5）:

- edge 候補: `SCOPED_CHANNEL_OPENED` / `SCOPED_CHANNEL_EXPIRED`
  （クラスター間の一時直結を約束グラフに記録する形）
- gate 候補: `channel approval gate`（Track U の GATE node と同型。
  承認前は BLOCKED、承認後 ASSIGNED_TO 相当で開通）

この対応関係が成立するのは偶然ではない。Track U の gate 意味論を設計した時点で
既に「承認によって何かが解放される」という一般構造を作っており、
scoped channel はその構造の再利用先の一つに過ぎない。

## 実装可能性ロードマップ（参考、決定ではない）

- 短期: 何もしない。今は RQ4 の 1 クラスター分の境界壁を運用で確かめる段階
- 中期: agmsg に「一時的な相互解放」を宣言する API があれば、
  scoped channel の最小実装になり得る（agmsg 自体の拡張は別途評価要）
- 長期: Track U の projection schema に channel 系語彙を正式追加する提案を
  Track D と協議（schema v1 の議題候補として記録するに留める）
