## M2. 既存 attribute との衝突回避

### M2.1 既存属性の確認が先
- M3E 現状: status / priority / freshness / last_touched 等が既にある（要確認）
- 物理隠喩の物性は **既存属性の派生** として実装するのが安全
- 新規属性追加は最小限に

### M2.2 統合パターン
| パターン | 説明 |
|----------|------|
| M2.2.a | **派生のみ** 既存属性から計算、永続化しない |
| M2.2.b | **キャッシュ** 派生だが計算結果を保存 |
| M2.2.c | **新規属性追加** physics.gravity 等を新設 |
| M2.2.d | **名前空間分離** physics 系を別 namespace |

→ 推し: M2.2.a → M2.2.b （新規属性は最後の手段）

### M2.3 名前付けの注意
- 既存 freshness と新規 temperature が被るなら、片方廃止 or 統合
- "温度" "鮮度" "熱度" 等の用語選定はユーザー側の好みで決める

## M3. 段階導入ロードマップ

### Phase 0: 観察・基盤
- 既存属性の棚卸し
- ノードイベントログ（編集・閲覧）の収集基盤

### Phase 1: 静的物性のみ（描画系）
- ノードに色・サイズで物性を表現

### Phase 3: 創発現象の1つを実装
- 朝の天気予報（M1.c）が最有力

### Phase 4: 地震・季節・生態系を順次追加
- 通知設計を慎重に
- 各機能はオプトインで

### Phase 5: 化石・発掘・time travel 統合
- I1（time travel）とセットで

→ Phase 1〜2 までで止めても価値は出る。Phase 3 以降は遊び寄り。

## M5. 横断観察（ファイルを跨いだ気づき）

### M5.1 メタファ系は「観察ツール」が本命
- 物理隠喩でレイアウトを最適化することは目的ではない
- **マップ全体の状態を読み取る** のが本命価値
- → 創発現象（04）が中核、物性（02）と挙動（03）はそのための手段

### M5.2 物性 = 永続、現象 = 派生
- 物性（温度・重力等）は属性として保存する余地あり
- 創発現象（気象予報・地震）は永続化不要、計算結果だけ通知に使う
- アーキテクチャとしては「物性ストア + 現象計算エンジン」

### M5.3 二層構造で生き残る
- 表層 = 遊び（季節・噴火・化石）
- 深層 = マップ衛生（沈降・冷却・stale 検出）
- 表層が無くても深層だけで価値あり、表層は楽しさの追加
- → 段階導入の Phase 1〜2 だけでも実用、Phase 3+ は遊び

### M5.4 既存 stale / freshness 機能の "上位互換" を狙う
- 物理隠喩は完全新規ではなく、既存マップ衛生機能の **言語と挙動の刷新**
- "stale" と言うより "化石化" と呼ぶことで体験が変わる
- 機能本体ではなく **言葉づかい** が新規性

### M5.5 メタファの一貫性が崩れると体験崩壊
- 「重力 = 重要度」と決めたら他の意味で使わない
- 用語を統一する責任が大きい
- "用語辞書" を作る価値あり

### M5.6 オフ機能の "対称性"
- 物理 ON / OFF を簡単に切り替えられる
- OFF 時も創発現象（気象予報）は別パネルで見られる
- 視覚効果と分析機能を分離可能に

### M5.7 他ブレストとの接続
- D1（重複検出） + 物性 = 重複ノードは "磁力で引き寄せ" → merge 提案 UI
- C5（insight 抽出） + 創発現象 = AI が "今週の地震は重要決定でした" と物語化
- I1（time travel） + 化石化 = 過去の地層を掘り返す体験
- J5（メタ認知ログ） + 温度 = 「沸騰している = 確信が揺らいでいる」可視化
- C10（自走 AI） + 物性 = 夜間整理のターゲット選定に物性を使う

### M5.8 メタファ選定の自由度
- "思考の物理" は1解釈にすぎない
- 「庭園」「都市」「水族館」「天体」など他メタファも並列に検討余地あり
- ユーザー設定で選べると理想（ただし実装コスト高）
