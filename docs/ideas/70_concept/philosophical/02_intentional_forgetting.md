# Q2. 「忘れる」ことの設計（古い情報の意図的減衰）

「アーカイブ＝完全保存」「削除＝完全消去」の二択に縛られず、
**「ぼやける」「思い出せる」「だが見えない」** という第三の状態を設計する。

time_history I4「忘却」/ cognitive_science J2「忘却曲線」/ thought_physics P2「鮮度」
と直接接続する。

## なぜ忘却を設計するか

- 人間の記憶は **能動的な忘却** で機能している（重要度フィルタ）
- マップが大きくなると **「全部見える」が逆にノイズ** になる
- 削除には心理的抵抗がある → 結果としてマップが汚れ続ける
- 思い出せる「忘却」があれば **安心して見えなくできる**
- 忘却された後に **再発見した時の価値** が高い（セレンディピティ）

## ユースケース

- **U1.** 1 年前の捨てきれないアイデアを「見ないが残す」状態に置きたい
- **U2.** scratch 階層の古いメモを集中時に視界から外したい
- **U3.** 共著者にマップを見せる時、未消化の私的メモを一時的に薄くしたい
- **U4.** 季節タスク（毎年 4 月の予算申請）を 11 ヶ月間は忘れていたい
- **U5.** 失敗した実験ログを「いつか参照可能だが今は忘却」にしたい
- **U6.** 過去の自分の感情ノートを「見ないが残す」状態に
- **U7.** 学生卒論プロジェクト終了後、構造を残して詳細をぼかしたい

## サブ論点

### Fg1. 忘却の表現メタファ

| 案 | メタファ | 視覚化 |
|---|---|---|
| Fg1.a | 透明度 | opacity が時間と共に低下（最終 5%） |
| Fg1.b | 距離 | "意識の中心" から遠ざかる（縮小 + 周辺移動） |
| Fg1.c | 霧 / 霞 | ノード周囲にぼかしフィルタ |
| Fg1.d | モノクロ化 | 色が抜け、グレースケールに |
| Fg1.e | 文字小型化 | text サイズが縮む |
| Fg1.f | レイヤー分離 | "前景" / "背景" レイヤーに移動 |
| Fg1.g | 折り畳み | subtree が折りたたまれた状態に |
| Fg1.h | 沈み込み | z 軸方向に「沈む」（3D） |

組み合わせ可。推し: **Fg1.a + Fg1.b + Fg1.d** の 3 段階（薄→遠→灰）。

### Fg2. 忘却のトリガー

| 案 | トリガー |
|---|---|
| Fg2.a | 時間ベース（最終編集から N 日） |
| Fg2.b | アクセスベース（最終閲覧から N 日） |
| Fg2.c | コンテキストベース（特定タグの時のみ表示） |
| Fg2.d | 手動（"忘れる" ボタン） |
| Fg2.e | 関連度ベース（現在編集中ノードと類似度が低い） |
| Fg2.f | エネルギーベース（気持ち低い時は楽しいノードのみ表示） |
| Fg2.g | 周期的（毎月 1 日にリフレッシュ） |
| Fg2.h | AI 提案（「これ忘れていいんじゃない？」） |

組み合わせ. 推し: **Fg2.b（アクセスベース）+ Fg2.d（手動）** が最も「自然」。

### Fg3. 忘却の段階

| 段階 | 名称案 | 状態 | 復帰方法 |
|---|---|---|---|
| 0 | active | 通常表示 | - |
| 1 | dormant | 薄く表示 | クリックで覚醒 |
| 2 | sleeping | 検索でのみヒット | 検索で復活 |
| 3 | hibernating | 検索もヒットしない、リンクからのみ | リンク辿りで復活 |
| 4 | forgotten | 完全に隠蔽、「忘却モード ON」でのみ | 明示的なモード切替 |
| 5 | gone | 削除（Q5 死へ移行） | 復元不能 or backup 経由 |

推し: **3 段階構成（active / dormant / forgotten）** から始め、必要なら細分化。

### Fg4. 「思い出す」体験

忘却機能の本質は「思い出せる」こと。

| 案 | 仕組み |
|---|---|
| Fg4.a | 検索で完全一致したら復活（Fg3 段階 2 まで） |
| Fg4.b | リンクされた他ノードを開いた時、関連忘却ノードが薄く現れる |
| Fg4.c | "今日この日に書いたメモ" 的なリマインド（年単位） |
| Fg4.d | 偶然の re-discovery（ランダム表示） |
| Fg4.e | AI が「忘れている重要ノード」を週次で 1 個提示 |
| Fg4.f | "Throwback" モード（毎月 1 日に 1 年前のマップを薄く重ね表示） |

→ Fg4.c + Fg4.e は automation_obstacles N14「待つ時間」と接続。
"思い出した時" の価値を演出する仕掛け。

### Fg5. 忘却 vs アーカイブ vs 削除

| 状態 | 表示 | 検索 | リンク辿り | 復元 | データ保持 |
|---|---|---|---|---|---|
| アーカイブ | 別 view のみ | ヒット | 可 | 即時 | 完全 |
| 忘却（dormant） | 薄く表示 | ヒット | 可 | 即時 | 完全 |
| 忘却（hibernating） | 不可視 | 不可 | 可 | リンク経由 | 完全 |
| 削除（soft） | 不可視 | 不可 | 不可 | backup 経由 | バックアップのみ |
| 削除（hard） | 不可視 | 不可 | 不可 | 不能 | なし |

→ 忘却は **「データは残るが認知資源を奪わない」** 範囲を定義する。

### Fg6. ユーザーの意図表現

「自動忘却」は怖い。「自分で意図して忘れる」体験を設計する。

- Fg6.a. ノードに `forget_after: 30d` を設定可能に
- Fg6.b. 「重要」フラグで永続フォーカス（忘却対象外）
- Fg6.c. 「いつか役立つかも」フラグで dormant 移行を加速
- Fg6.d. 範囲選択 → 「これらをまとめて忘れる」操作
- Fg6.e. ノード作成時に "賞味期限" デフォルト設定（カテゴリ別）

### Fg7. プライバシーとの接続

policy_privacy 文脈で重要。

- Fg7.a. 機微情報ノードは時間で自動 forgotten（共有時の事故防止）
- Fg7.b. 共有マップでは自分の dormant ノードを表示しない
- Fg7.c. 「忘却」=「他人には最初から無かったように見せる」も含むか

### Fg8. 忘却が実装に効く副次効果

- ノード列挙 API が「active のみ」をデフォルトにできる → クエリ高速化
- AI に渡す context を絞れる → token 節約
- マップレンダリングが軽くなる（dormant は LOD で簡素描画）
- M1（巨大マップ対応）の根本解になる

## 反対意見

- **R1.** 「全部見える」が原則だから検索が成立する。隠すと混乱
- **R2.** 過去の自分への敬意を欠く（書いた瞬間は重要だった）
- **R3.** 復元手順が分かりにくいと「消えた」と誤解する
- **R4.** 自動減衰は予測不可能で恐怖
- **R5.** 「忘れていい」と判断する基準が曖昧

→ 答え:
- R1 → 検索は段階別（dormant までヒット、hibernating は明示モードで）
- R2 → 削除ではなく "ぼかし" であることをアニメで明示
- R3 → 「忘却の鍵」を毎月メールするなど復元導線
- R4 → 全自動禁止、提案のみ（Fg2.h）
- R5 → ユーザー手動を基本（Fg2.d）

## 既存ツール差分

| ツール | 忘却の扱い |
|---|---|
| Roam | 全保存・検索一覧、忘却概念なし |
| Obsidian | アーカイブフォルダで隔離 |
| Notion | データベースフィルタで「非表示」 |
| Anki | 復習周期で「忘却曲線」を扱う（記憶側） |
| M3E | 表示 × 検索 × リンクの 3 軸で段階化（差別化） |

## キーメッセージ

- **「忘れる」≠「消す」。** UI 表現の違いだけで、データは保持される
- **段階的忘却（active → dormant → forgotten）** が破壊感を吸収する
- **思い出した時の価値** こそが忘却機能の本体
- **手動忘却を基本に** すべき（自動化は信頼を蓄積してから）
- **巨大マップ対応の副次効果** が大きく、実用面の価値も高い
