# 04. 検索・取り込み系 — C6 / C8

「研究秘書」として AI が文献を取り込み、過去ノードを横断検索する候補。
project_projection_vision（世界モデル→射影→科研費出力）の **入力側** を支える。

## C6. research assistant フロー

「このテーマで論文探して」→ 文献ノード自動投入。

### R1. 情報源

| 案 | 概要 | 良い点 | 悪い点 |
|---|---|---|---|
| R1-a 一般 Web 検索 | LLM の Web 検索機能 | 汎用 | 学術品質ばらつき |
| R1-b arXiv API | 直接 arXiv 検索 | 学術特化、無料 | プレプリント中心 |
| R1-c Semantic Scholar / OpenAlex | API で構造化文献検索 | 引用ネットワーク取得可 | 分野偏り |
| R1-d Google Scholar スクレイプ | 網羅性 | 規約違反リスク |
| R1-e Zotero ライブラリ参照 | 既に持っている文献の中から | プライベート、整合性高 | カバレッジ限定 |
| R1-f 内部マップ検索 | 「マップにある関連ノード」を先に出す | 知識資産の活用 | 外部新発見ゼロ |
| R1-g マルチソース合成 | 上記を複数組み合わせ | 強い | 統合の品質保証が難 |

**観察**: **R1-f → R1-b/c → R1-a** の段階検索（内→外）が研究者文化に合う。
「マップに既に答えがある」のに外部を叩くのは無駄＋雑音。

### R2. 文献ノードのスキーマ

文献ノードに必須・推奨フィールド:

| フィールド | 必須 | 内容 |
|---|---|---|
| title | ◎ | 論文タイトル |
| authors | ◎ | 著者リスト |
| year | ◎ | 出版年 |
| doi / url | ◎ | 識別子 |
| venue | ○ | ジャーナル / 会議 |
| abstract | ○ | 抄録（原文） |
| ai_summary | ○ | AI 要約（原文と区別） |
| ai_relevance | ○ | なぜこのテーマに関連か（AI 説明） |
| ai_confidence | ○ | 関連度スコア |
| ai_caveat | △ | AI による注意喚起（「未読・幻覚可能性」） |
| my_note | △ | 人間メモ |
| read_status | △ | 未読 / 流し読み / 精読 |
| critique | △ | 批判 / 反例（C1 と接続） |

**痕跡保存（P9 / G10）**: AI 由来フィールドは必ず `ai_*` プレフィックスで原文と分離。

### R3. 自動投入 vs 提案 inbox

| 案 | 自動レベル | 良い点 | 悪い点 |
|---|---|---|---|
| R3-a 提案 inbox | L1 | 安全、選別可 | 手間 |
| R3-b 仮ノード（pending） | L2 | 視覚的、ワンクリック確定 | inbox 膨張 |
| R3-c 自動投入＋確認 | L3 | スピード | 誤投入の山 |
| R3-d 完全自動 | L4 | 最速 | 幻覚論文のリスク（B1） |

**推し**: R3-a / R3-b。L4 は **B1 幻覚** リスクが致命的（架空論文の引用は研究者として死活問題）。
**P5 検証スキャフォールド**: DOI 解決可能性チェックを必須にする。

### R6 と接続: 「このテーマで」の解釈

「このテーマで論文探して」の **「このテーマ」** をどう特定するか:
- 選択中ノードの text + 親 path
- 開いている subtree 全体
- ユーザが追加で 1 行説明
- マップの最近編集ノード群から推定

→ ユーザに **どう解釈したかを必ず確認** する step が必要（誤解検索を防ぐ）。

## C8. 会話型クエリ

「先週決めたこと教えて」→ AI が該当ノードを提示する。

### R4. 入力 UI

| 案 | 概要 | 良い点 | 悪い点 |
|---|---|---|---|
| R4-a チャット欄常設 | 画面下に常時 | 慣れたメタファ | 場所占有 |
| R4-b コマンドパレット | `Ctrl+K` で開く | 既存 UI 拡張 | 「会話」感が薄い |
| R4-c フローティング | フキダシ風 | 邪魔にならない | 発見性低 |
| R4-d 音声 | マイクで | ハンズフリー | 環境依存 |
| R4-e サイドペイン | 右側にパネル | 履歴も見える | 場所占有 |

### R5. 結果の提示

| 案 | 概要 |
|---|---|
| R5-a ノード強調＋自動フォーカス | 該当ノードを画面に集めて点滅 |
| R5-b 別ペインに list | サイドパネルに「ヒット 5 件」 |
| R5-c inline カード | チャット欄内にカードで返す |
| R5-d 一時 overlay | 該当だけ濃く、他をフェード |
| R5-e マップ＋ペインの併用 | カード＋マップ強調を同時 |

### R6. 「先週」「○○について」の意味解釈

| 解釈軸 | 例 | 必要データ |
|---|---|---|
| 時間 | 「先週」「3 日前」「昨日の朝」 | 編集タイムスタンプ |
| 主題 | 「重力波のこと」 | semantic 検索 |
| 動作 | 「決めたこと」「迷ったこと」 | ノード type / status |
| 関係 | 「Aさんと話した内容」 | author 属性 / コメント履歴 |
| 場所 | 「○○の subtree」 | 構造 |
| 状態遷移 | 「却下されたやつ」 | 履歴／status 履歴（I4） |

→ 1 クエリに複数解釈が成立しうる。**AI は複数解釈を提示し、人間が選ぶ** べき。

### R7. クエリ履歴の資産化

よく聞くクエリは **保存** しておくと便利:
- 「今週やったこと」「未決事項」「○○の進捗」
- マップ内 `query:` ノードに保存（G7 プロンプト資産化と同型）
- ホーム画面に「保存クエリ 5 件」を貼れる

### R8. 検索 vs 生成の境界

重要原則: **マップ内に答えがある時は、生成しない**。
- 該当ノードが見つかれば、それを **そのまま提示**
- 見つからない時のみ「該当なし。関連するのは...」と推論
- 「決まっていない」ことを **「決まっています」と捏造しない**（幻覚 B1）

→ **検索 first, 生成 fallback** の二段構成が安全。

## C6 / C8 の依存関係

```
C8 会話型クエリ ──┐
                  ├→ どちらも「マップ全体を AI に読ませる」基盤が必要
C5 insight ──────┘
                                   ↓
                          → C7 要約があるとコスト激減
```

C6 は外向き（外部文献）、C8 は内向き（内部マップ）と方向が逆。
両方とも **検閲（G3）を必ず通す**：
- C6: クエリ自体に機微情報が含まれていないか
- C8: 検索結果として返したノードが機微指定されていないか

## ベンダ切替の論点（G4 適用）

| 機能 | 候補 LLM | 理由 |
|---|---|---|
| C6 文献分類・要約 | Claude long-context / GPT | 長文処理 |
| C6 引用整形 | local LLM | 形式変換、軽量 |
| C8 クエリ意図解釈 | Haiku / 軽量モデル | 短文、高頻度 |
| C8 結果ランク付け | embedding model | semantic 類似度 |

→ **同じ機能内でも複数 LLM をパイプライン** に並べる設計が必要。

## 既存実装との接続

- 既存 `topic-suggest` サブエージェントの拡張で C6 のミニ版が作れる
- `flash_ingest.ts` / `md_reader.ts` は C6 の文献ノード投入の素地
- `vault_importer.ts` / `vault_exporter.ts` の延長で文献データベース連携

## 比較: C6 と C8 の使い分け

| | C6 research assistant | C8 会話型クエリ |
|---|---|---|
| 方向 | 外向き（外部 → マップ） | 内向き（マップ内検索） |
| リスク | 幻覚論文 | プライバシー漏えい |
| 必須機能 | DOI 検証、出典 | 時間範囲解釈、複数解釈提示 |
| 期待頻度 | 週次 | 日次 |
| 主用途 | 文献調査、引用補強 | 「あれどこだっけ」想起補助 |
