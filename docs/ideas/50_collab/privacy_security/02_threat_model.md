# 02. 脅威モデル — 何から守るか、どこまで諦めるか

防御は脅威モデルなしには設計できない。
M3E が **想定する** 脅威と **諦める** 脅威を明示し、機微度の段階分けを定義する。

## T1-T12. 脅威の網羅リスト

### 物理 / 端末層

| ID | 脅威 | 想定攻撃者 | 既存対策 | 追加検討 |
|---|---|---|---|---|
| **T1** | 端末紛失・盗難 | 拾った第三者 | OS ログイン | M3E 起動時パスフレーズ |
| **T2** | スクショ・画面撮影 | 同僚・SNS 流出 | OS 機能 | 機密ノード blur / 撮影禁止フラグ |
| **T3** | ショルダーハッキング | 隣の人 | なし | panic ロック / blur on focus loss |
| **T4** | OS マルウェア | RAT / ステーラー | OS 防御 | 平文インデックス縮小、メモリ即解放 |
| **T5** | ブラウザ拡張漏洩 | 悪意ある拡張 | CSP | iframe 隔離、機密ノードは別 origin |
| **T6** | クリップボード読取 | 別アプリ | OS 機能 | コピー時タイマー消去 |

### 通信 / 同期層

| ID | 脅威 | 想定攻撃者 | 既存対策 | 追加検討 |
|---|---|---|---|---|
| **T7** | sync サーバー漏洩 | サーバー管理者・侵入者 | **AES-256-GCM E2E（policy_privacy）** | 既決、変更しない |
| **T8** | 通信傍受 | MITM | TLS + E2E | 既決 |
| **T9** | sync key 流出 | キーファイル盗難 | ローカルファイル権限 | 起動時パスフレーズ追加？ |

### AI / クラウド層

| ID | 脅威 | 想定攻撃者 | 既存対策 | 追加検討 |
|---|---|---|---|---|
| **T10** | LLM プロバイダーの学習利用 | 善意のプロバイダー | API の opt-out 設定 | デフォルト送信禁止モード |
| **T11** | LLM プロバイダー漏洩・召喚状 | プロバイダー / 第三者 | なし | ローカル LLM、または送信禁止 |
| **T12** | プロンプトインジェクション逆流 | 悪意ある外部テキスト | なし | sanitization / context 隔離 |

### 内部 / 誤操作層（しばしば最大リスク）

| ID | 脅威 | 想定攻撃者 | 既存対策 | 追加検討 |
|---|---|---|---|---|
| **T13** | 誤レーン投入 | 自分 | なし | 投入時警告 / レーン推定 |
| **T14** | export 時の機密混入 | 自分 | なし | export ガード / preview |
| **T15** | search index に機密が混入 | 自分（検索時） | なし | index 暗号化 / 機密 opt-out |
| **T16** | undo / backup に過去機密が残る | 自分 | backup あり | 機密ノードの履歴追放 |

T13-T16 が **意外と最頻発・最大被害** のパターンになることが多い。
高度な暗号より、誤操作防止 UX の方が ROI 高いことを意識。

## 諦める脅威（Out of Scope 候補）

| ID | 脅威 | 諦める理由 |
|---|---|---|
| **OS1** | 国家レベル攻撃（NSA / 物理ハック） | 個人ツールの責任範囲外 |
| **OS2** | 端末そのものへのキーロガー | OS / 物理層の問題 |
| **OS3** | 自分の脳の漏洩（口頭で喋る） | ツール責任範囲外 |
| **OS4** | 同僚に直接見せる | 自己責任 |
| **OS5** | プリンタ経由漏洩 | 印刷機能制限で対応程度 |

これらを諦めることで設計が単純化する。明示しないとスコープ膨張する。

## S1-S5. 機微度の5段階分類

ノード / vault / workspace に **機微度ラベル** を付ける案。
全ノードに付けるのは負荷が大きいので **デフォルト + override** モデル。

| ID | ラベル | 例 | At-Rest 暗号化 | AI 送信 | 同期 | 表示 |
|---|---|---|---|---|---|---|
| **S1 公開可** | open | 文献まとめ、公知 | 不要 | 自由 | 通常 | 通常 |
| **S2 準公開** | semi | 自分の意見、未公開アイデア | 任意 | 警告付き許可 | 通常 | 薄い色帯 |
| **S3 内部** | internal | 共同研究者氏名、所属、雑感 | 推奨 | マスク後送信 | 暗号化 | 黄色帯 |
| **S4 機密** | confidential | 査読中、被験者、予算 | 必須 | ローカル LLM のみ | 暗号化 | オレンジ帯 + 鍵 |
| **S5 極秘** | secret | API キー、パスワード、人間関係 | 必須 | 送信禁止 | 暗号化 | 赤帯 + 鍵 + blur |

### S1-S5 設計上の論点

- **段階数の妥当性**: 3段階（公開/中間/機密）で十分か、5段階か、7段階か
- **デフォルト**: 新規ノードのデフォルトは S1 か S2 か（安全側 vs 使いやすさ）
- **継承**: 子ノードは親の機微度を継承するか
- **昇格・降格**: 機微度変更時の挙動（既存暗号化はどう扱う）
- **複合度**: 1ノード内に複数機微度のテキストが混在する場合（field 単位 vs ノード単位）

| 案 | 段階数 | 良い点 | 悪い点 |
|---|---|---|---|
| **S-a 3段階** | open / internal / secret | シンプル、選びやすい | 査読/予算/被験者が同レベルで雑 |
| **S-b 5段階** | 上記 | 表現力高い | 選択コスト |
| **S-c 7段階** | 細かく | 完全 | 誰も使えない |
| **S-d 2軸** | 機微度 × カテゴリ | 表現力高い | 複雑 |
| **S-e ラベル自由** | タグ運用 | 柔軟 | 統制不能 |

推し: **S-b 5段階 + デフォルト S2**（安全側、慣れたら S1 に降格）

## レーン分離原則

機微度を分けたとして、**実装上どこで分離するか** の選択肢:

| 粒度 | 内容 | 防御強度 | 実装コスト |
|---|---|---|---|
| **L-a ノード単位** | attribute で `sensitivity: S4` | 低（同じファイルに混在） | 低 |
| **L-b subtree 単位** | 特定 subtree 配下を全部暗号化 | 中 | 中 |
| **L-c vault 単位** | workspace 内の vault を分離（タグ namespace） | 中 | 中 |
| **L-d workspace 単位** | 別 workspace ファイル / 別キー | 高 | 中 |
| **L-e device 単位** | 機密データは特定端末のみ | 最高 | 高 |
| **L-f session 単位** | 起動時にどの vault を開くか選ぶ | 高 | 中 |

組合せで使う:
- S1-S2: L-a ノード単位（メイン workspace）
- S3: L-b subtree（メイン workspace 内の隔離 subtree）
- S4-S5: L-d workspace（別ファイル + 別キー）
- 認証情報: L-e device（同期しない、特定端末のみ）

詳細は `05_vault_separation.md` で展開。

## 脅威 × 機微度 マトリクス（簡易）

| / | T1 紛失 | T2 スクショ | T7 sync 漏洩 | T10 LLM 学習 | T11 LLM 漏洩 | T13 誤投入 |
|---|---|---|---|---|---|---|
| **S1 公開** | 影響なし | 影響なし | 影響なし | 影響なし | 影響なし | 影響なし |
| **S2 準公開** | 軽 | 軽 | 軽 | 中 | 軽 | 中 |
| **S3 内部** | 中 | 中 | 中（暗号で防御） | 中 | 中 | 大 |
| **S4 機密** | 大（暗号で防御） | 大 | 中（暗号で防御） | 大 | 大 | 大 |
| **S5 極秘** | 大（暗号で防御） | 大 | 中（暗号で防御） | 致命 | 致命 | 致命 |

観察:
- **S5 × T10/T11** が致命 → AI 経路完全遮断（送信禁止フラグ）必須
- **T13 誤投入** が S3 以上で大被害 → UX 防御の重要性
- **T2 スクショ** は暗号で防げない → 表示時 blur / panic / 撮影禁止が必要

## 脅威モデル選択肢のまとめ

ブレストとして「どこまで守るか」の上位方針候補:

| 案 | 内容 | 良い点 | 悪い点 |
|---|---|---|---|
| **TM-a 最小限** | T7-T9（sync）のみ守る、policy_privacy 既存だけ | 実装最小 | 内部脅威に無防備 |
| **TM-b 中庸** | T1-T16 を扱うが、防御は緩く UX 主体 | バランス | 一部脅威に脆弱 |
| **TM-c 重装** | 全脅威に技術防御、二要素・パスフレーズ・hardware key | 強固 | 使いにくい |
| **TM-d 段階適用** | S1-S5 で防御強度を変える、S5 のみ重装 | 合理的 | 設計複雑 |

推し: **TM-d 段階適用**（5段階機微度と整合）

## キーポイント

- T13-T16 内部 / 誤操作層が **最頻発・最大被害**、UX 防御の優先度高い
- S5 × T10/T11 が致命、AI 経路完全遮断フラグ必須
- スクショ T2 は暗号で防げない、表示時防御が別途必要
- 諦める脅威を明示しないとスコープが膨張する
- 段階適用（TM-d × S-b）が筋の良い基本方針候補
