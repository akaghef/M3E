# 01. Concept — なぜ協調機能が必要か、何を作るか

M3E に「協調・コラボレーション」を入れる意義と、入れる範囲・入れない範囲を
**複数案** で整理する。採否は決めない。

## Cn1. M3E にとっての「協調」の射程

M3E は単独の研究思考ツールとして始まったが、現実には:

- 共著者・指導教員・査読者とマップを共有したい場面がある
- AI subagent（visual/data/team）が並列稼働しており、それ自体が「複数人協働」
- 自分自身が複数 PC・複数アカウントで触る「自己協働」

これらをどこまで一つの機構でカバーするかが論点。

| 案 | 射程 | 良い点 | 悪い点 | 推し度 |
|----|----|------|------|------|
| Cn1-A | 自己協働のみ（マルチデバイス同期） | 安全、scope 小 | 共同研究に届かない | △ |
| Cn1-B | 自己 + AI agent（人間共著はやらない） | dogfood 価値高 | 研究室での共同利用に弱い | ○ |
| Cn1-C | 自己 + AI + 少人数共著（〜5人） | 研究現場に届く | 同期/権限が複雑化 | ◎ |
| Cn1-D | 一般 SaaS 並（Google Docs 級） | 市場大 | NM4 と矛盾、戦線拡大 | × |

推し: Cn1-B または Cn1-C を上限に。

## Cn2. 「やらない」境界線（automation_obstacles/05 整合）

automation_obstacles/05_m3e_strategy.md に明示された **NM4: リアルタイム協調エディタ
（Google Docs 級）はやらない**。本ブレストはこの境界を尊重する。

| 領域 | やる | やらない |
|----|----|----|
| ノードの同時編集 | ScopeLock で「順番に」編集 | 文字単位の OT/CRDT 同編集 |
| プレゼンス | 「誰がどのノードにいる」 | 文字単位カーソル位置トラッキング |
| コメント | ノード添付の議論 | リッチテキスト編集の本格 inline コメント |
| レビュー | 非同期依頼→集約 | 同期レビューワークフロー |
| 引き継ぎ | agent ↔ human の trail | リアルタイム共同タスク管理 SaaS |

論点: 「ノード単位ロック」が UX 上どこまで許容されるか。
文字単位の同時編集を諦めるかわりに、ノードを開いている間は他者が触れない設計。

## Cn3. 想定ユーザー組と協調シーン

具体ユースケース（最低3-5を意識）:

1. **共著論文の執筆準備** — 共著者がアブスト案ノードにコメント、akaghef が解決
2. **査読依頼** — 指導教員にマップ全体の read-only リンクを共有、コメントだけ受け付け
3. **研究班ミーティング** — 1人がプレゼンタとして発言権を持ち、他は閲覧追従
4. **dev-beta 開発** — visual/data/team の3 subagent がマップ上で進捗を共有
5. **agent dispatch → akaghef レビュー** — エージェントが終わったら DEV map に成果を吊し、
   akaghef が朝に受信トレイで承認/差し戻し
6. **複数 PC 跨ぎの自己協働** — 同じ akaghef が研究室と自宅で別々に触り、競合解決
7. **学生との合同マップ** — 指導学生に subtree を渡してそこだけ編集してもらう

## Cn4. 3軸の独立性

協調を **一つの機能として束ねるか、軸ごとに分けるか**。

| 案 | 統合度 | 良い点 | 悪い点 |
|----|----|------|------|
| Cn4-A | 3軸を1つの「協調パネル」に統合 | 学習1回で済む | 過密、要件の違いが埋もれる |
| Cn4-B | 軸ごとに別 UI（人↔人 / 人↔AI / AI↔AI） | 各軸を最適化 | UI 数が増える |
| Cn4-C | データモデルは共通、UI だけ分離 | 整合・最適化両立 | 設計コスト中 |

推し: Cn4-C。collab.ts の entity/role モデルは既に統合的、UX 層だけ分けるのが現実的。

## Cn5. 既存ツールとの差別化

| ツール | 強み | M3E が真似ない点 | M3E が勝てる点 |
|----|----|----|----|
| Google Docs | 文字単位同編集 | リッチ文書編集 | ノード構造ベースの議論 |
| Notion | DB + ページ + 共有 | 一般用途 | 思考プロセス特化 |
| Miro / Mural | フリーキャンバス共同編集 | 自由配置 | 構造化された議論 |
| Figma | デザイン共同編集 | 視覚特化 | 思考の出典・履歴 |
| Slack スレッド | 議論 | チャンネル管理 | ノード文脈と直結 |
| GitHub PR レビュー | 差分レビュー | コード前提 | 思考の差分レビュー |

M3E の差別化軸候補:
- **ノード構造に紐付いた議論** — 議論が必ず思考の文脈に乗る
- **AI と人間が同じ場で動く** — agent が議論参加者になれる
- **ローカルファースト** — vault 内で完結、選択的に共有

## Cn6. 成功指標の候補

| 案 | 指標 | 測りやすさ |
|----|----|----|
| Cn6-A | 引き継ぎ漏れ件数 / 月 | 中 |
| Cn6-B | レビュー往復の中央値（依頼〜解決まで） | 高 |
| Cn6-C | プレゼンス可視化満足度（自己申告） | 低 |
| Cn6-D | コメント解決率 | 高 |
| Cn6-E | 共著者の月次アクティブ数 | 高 |
| Cn6-F | agent ↔ human ハンドオフ成功率 | 中 |

複数指標の組み合わせが筋。Cn6-B + Cn6-D + Cn6-F の3点セットを推し候補に。

## Cn7. 「協調なし」のベースライン

協調機能を **入れない** とどうなるかも検討する（フェアな比較のため）:

- 共著者は zip / git で渡し、Slack でコメント → M3E は自己専用ツール
- AI agent との連携は docs/backlog の md 経由
- 引き継ぎは Agent Status ノードへの手動更新

利点: シンプル、戦線が広がらない、外部ツールの強みを活かせる
欠点: 「思考の場」が分断される、議論の出典が散る、agent 完了の見落とし

つまり「やらない選択肢」も論理的には妥当。
**やる場合は「ハブ役」「非同期レビュー」「ハンドオフ trail」の3点に集中** が筋がよい。

## Cn8. 「人間 ↔ 人間」「人間 ↔ AI」「AI ↔ AI」要件の違い

| 軸 | 主な要件 | レイテンシ要求 | 主な UI 要素 |
|----|----|----|----|
| 人 ↔ 人 | 議論・レビュー・プレゼン | 中（数秒〜分） | コメント、フォロー、発言権 |
| 人 ↔ AI | dispatch / 受け取り / 承認 | 中（タスク完了通知） | 受信トレイ、承認ボタン |
| AI ↔ AI | 並列タスク調整、相互参照 | 低（リアルタイム不要） | Status ボード、ロック |

3軸で UI 要素が違う。同じデータモデルに乗せても UI は分けるのが筋。
