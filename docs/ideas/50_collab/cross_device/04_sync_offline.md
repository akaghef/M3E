# 04 Sync / Offline — オフライン同期と競合解決

クロスデバイスの本丸。
PC とモバイルで同じマップを触る前提のデータ整合性、conflict_backup 連動、オフライン UX。

既存資産: `vault_watch.ts`, `vault_path.ts`, `conflict_backup.ts`。
これらを「**マルチデバイス**」に拡張する場合の選択肢を並べる。

---

## Sy1. オフライン編集モデル

ネット接続なしでもモバイル側で編集できるか。

| 案 | 良い点 | 悪い点 | 推し度 |
|---|---|---|---|
| **完全オフライン**（ローカル DB に保持） | 飛行機・地下鉄 OK | 競合複雑化 | ◎ |
| **読み専用オフライン**（編集不可） | 競合ゼロ | 移動中編集不可 | ◎ |
| **キュー方式**（編集はキュー、再接続時 push） | 編集可能で安全 | キュー管理 UX 必要 | ○ |
| **オフライン不可**（ネット必須） | 実装簡単 | モバイル価値半減 | △ |

**推し**: **読み専用オフライン**から始め、**キュー方式**に拡張。

---

## Sy2. クラウドリレー方式

マップデータをどこに置くか。

| 案 | 良い点 | 悪い点 | privacy リスク |
|---|---|---|---|
| **自前サーバー** | 完全制御 | 運用コスト | 制御可 |
| **Dropbox/iCloud Drive** | 既存利用、設定簡単 | 暗号化必須、API 制約 | 中（クライアント暗号化で OK） |
| **Google Drive** | API 充実 | privacy 懸念 | 中 |
| **GitHub（git）** | 履歴自然、無料 | バイナリ弱い、競合 git 解決 | 低（プライベート repo） |
| **WebDAV / Nextcloud** | 自前ホスト可 | 設定難 | 制御可 |
| **Firebase / Supabase** | リアルタイム同期 | ベンダロック | 中 |
| **CRDT 専用（Yjs/Automerge）** | 競合自動解決 | 既存 vault 設計と齟齬 | 中 |

**推し**: **iCloud Drive**（akaghef 単独運用想定）+ **クライアント暗号化**。
将来複数人なら **Yjs** + **自前 relay**。

---

## Sy3. P2P 同期（同 Wi-Fi）

PC とスマホが同 Wi-Fi にいる時のみ同期。

| 案 | 良い点 | 悪い点 |
|---|---|---|
| **WebRTC P2P** | クラウド不要 | NAT 越え難 |
| **mDNS + HTTP** | LAN 内シンプル | Wi-Fi 同一前提 |
| **AirDrop / Nearby Share** | OS 標準 | 1 回限り、自動化弱 |
| **同期しない（クラウド一択）** | 単純 | LAN の高速性活かせない |

**推し**: **mDNS + HTTP**（既存 dev サーバーの延長）。
LAN 内同期は研究室・自宅で強い。

---

## Sy4. 競合解決戦略

PC とモバイルで同じノードを編集してしまった時。

| 戦略 | 概要 | 良い点 | 悪い点 |
|---|---|---|---|
| **last-write-wins** | 後勝ち、簡単 | 実装最小 | データ消失 |
| **conflict_backup 流用** | 競合時に両方を別ファイルに保存 | 既存資産活用、消失なし | UX が技術的 |
| **手動 merge** | ユーザーに 3-way diff を見せる | 安全 | UX 重い |
| **CRDT** | 自動マージ（テキスト merge） | 自動 | 実装複雑、構造変更弱 |
| **ノード ロック** | 編集中ロック | 競合発生せず | オフライン時困る |
| **field 単位 LWW** | text と attribute を別管理 | 細粒度 | 設計複雑 |

**推し**: **conflict_backup 流用 + field 単位 LWW** のハイブリッド。
既存資産を活かしつつ、テキストと attribute は別扱い。

---

## Sy5. 同期粒度

何を単位に同期するか。

| 粒度 | 例 | 良い点 | 悪い点 |
|---|---|---|---|
| **マップ全体** | m3e-map.json 全体 | シンプル | 大きいと遅い |
| **scope 単位** | subtree ごと | 部分同期可 | 境界管理 |
| **ノード単位** | 個別 PUT/PATCH | 細粒度 | API 多発 |
| **field 単位** | text のみ等 | 最小転送 | 整合性管理 |
| **diff 単位** | 差分のみ | 効率的 | diff 計算コスト |

**推し**: **マップ全体（MVP）→ ノード単位（拡張）**。
最初は m3e-map.json 全体送受で十分。

---

## Sy6. 暗号化

policy_privacy: 機微情報は平文不可。モバイルストレージは特に注意。

| 案 | 暗号化範囲 | 鍵管理 | 良い点 | 悪い点 |
|---|---|---|---|---|
| **未暗号化** | なし | - | 簡単 | privacy 違反 |
| **クラウド側のみ E2E** | 通信 + クラウド保存 | パスフレーズ | クラウド見えない | 端末ローカルは平文 |
| **端末ローカルも暗号化** | 全端末 + クラウド | パスフレーズ | 完全防御 | 性能影響、UX 重い |
| **選択的暗号化** | 機微マークノードのみ | per-node 鍵 | バランス | UX 複雑 |
| **ハードウェア鍵** | TPM / Secure Enclave | OS 管理 | 強力 | プラットフォーム依存 |

**推し**: **クラウド側 E2E + 端末ローカル選択的暗号化**。
研究データの一部のみ機微、それを per-node マーク。

## Sy7. 同期方向

| 方向 | 用途 | 良い点 | 悪い点 |
|---|---|---|---|
| **PC → mobile のみ** | 読み専用モバイル | 競合なし | モバイル編集不可 |
| **mobile → PC のみ** | capture 専用 | 競合最小 | 既存ノード変更不可 |
| **双方向** | 全機能 | 完全 | 競合解決必要 |
| **モード別**（capture は片方向、編集は双方向） | 用途に応じ | 安全と機能の両立 | 実装複雑 |

**推し**: **モード別**。capture は片方向、閲覧 / 編集は双方向。

---

## Sy8. 帯域・電池・ストレージ

モバイル特有の制約。

| 観点 | 配慮 |
|---|---|
| **モバイル回線** | 大マップ全体送信は痛い → diff 同期 |
| **電池** | バックグラウンド同期 vs 手動 → 設定可 |
| **ストレージ** | 全マップ vs 部分のみ → scope 同期 |
| **画像** | 添付画像は別 fetch on-demand |
| **同期頻度** | リアルタイム / 30 秒 / 手動 → 選択 |

**推し**: 手動同期 + scope 部分のみ（MVP）→ バックグラウンド同期（拡張）。

---

## Sy9. 「読むだけスナップショット」モード

編集を捨て、PC マップの読み専用コピーをモバイルに置く。

| 案 | 配信形式 | 良い点 | 悪い点 |
|---|---|---|---|
| **静的 HTML スナップショット** | HTML/CSS/JS | 軽量、CDN 可 | リアルタイム性なし |
| **JSON スナップショット** | m3e-map.json コピー | 既存 viewer で動く | 容量 |
| **Markdown スナップショット** | md ファイル群 | 他ツールでも読める | 構造失う |
| **PDF/epub スナップショット** | 電子書籍化 | E-Ink 端末対応 | 編集不可、構造失う |

**推し**: **JSON スナップショット**（MVP）+ 将来 epub。
既存 viewer がそのまま使える。

## Sy10. workspace（vault）切替 UX のモバイル版

PC では workspace（vault）を切り替えられる。モバイルでも必要か。

| 案 | UX |
|---|---|
| **切替なし**（1 vault 固定） | 設定不要、シンプル |
| **設定で切替** | 起動時選択 |
| **複数 vault 同時** | タブで並列、混乱 |
| **vault ごとに別アプリインストール** | 完全分離、管理面倒 |

**推し**: **設定で切替**（MVP）。複数 vault は研究/プライベート分離に有用。

## conflict_backup.ts 連動の具体案

既存 `conflict_backup.ts` の振る舞い:
- ファイル競合検知時、片方を `*.conflict-YYYYMMDD-HHMMSS.json` として退避
- 直近の正規ファイルは残す

モバイル sync への拡張案:

| 拡張 | 内容 |
|---|---|
| **Cb1. デバイス ID 付与** | 競合バックアップに `device_id` 付与（PC vs スマホ vs タブレット） |
| **Cb2. UI 表示** | 競合発生時にモバイル UI で通知 + 解決画面 |
| **Cb3. 自動 merge トライ** | テキストは git-style 3-way merge を試行、不可なら退避 |
| **Cb4. 競合ノード一覧ビュー** | マップ上で「競合あり」マーカー、解決待ちキュー |
| **Cb5. ロールバック** | バックアップから復元する UI |

## 横断的観察

- **オフライン同期は「やらない」も選択肢** — モバイルを読み専用に割り切れば 90% の問題消える。
- **conflict_backup.ts は思った以上に効く** — モバイル側でも同じパターンが使える。
- **CRDT は魅力的だが過剰投資** — 単独研究者運用なら LWW + backup で十分。
- **暗号化は MVP 後でよいが、設計は MVP 段階で考慮** — 後から差し込むと痛い。
- **クラウドリレーの選択は宗教戦争** — iCloud / Dropbox / 自前、akaghef の好みで決める。
- **同期粒度は M3E のマップサイズ次第** — feedback_dev_map_size に従えば全体送信で十分。
