# 05. K6 メタ視点 と K7 アンチパターン

K1〜K5 の個別案を超えて、ゲーム化の **方針論** と **やってはいけない設計** を整理する。

## K6: メタ視点（なぜゲーム化するのか／しないのか）

### K6.1 ゲーム化を入れる根拠（pro 側）

| ID | 根拠 | 強さ |
|----|------|----|
| K6.1a | 倦怠期に小さな手応えが復帰を助ける | 中 |
| K6.1b | M3E ユーザー層が広がった時、研究者以外も使う | 弱（現状想定外） |
| K6.1c | 「育てる楽しさ」は研究行為と矛盾しない | 強（K4 で示した） |
| K6.1d | 認知科学（J 系）と統合すると学習価値が出る | 強（K5）|
| K6.1e | M3E 自身が「面白い道具」になる差別化 | 中 |
| K6.1f | akaghef 個人の好み | 不明、要確認 |

### K6.2 ゲーム化を入れない根拠（con 側）

| ID | 根拠 | 強さ |
|----|------|----|
| K6.2a | 研究者の知的真面目さを毀損する | 強 |
| K6.2b | 外発的動機が内発的動機を破壊する（既存研究多数）| 強 |
| K6.2c | UI 複雑度が増し、本筋（思考支援）から外れる | 中 |
| K6.2d | 開発リソースを project_projection_vision から奪う | 強 |
| K6.2e | OFF にされる機能を作る無駄 | 中 |
| K6.2f | M3E のブランドイメージに合わない | 中 |

### K6.3 「生き物感」という中間アプローチ

「ゲーム化」という言葉を使わず、「マップが生きている感」だけを設計する案。
00_topic_pool.md K6 そのもの。

| 観点 | 通常のゲーム化 | 生き物感アプローチ |
|------|----------|-------------|
| 報酬 | バッジ・XP | 触ると元気・触らないと萎れる |
| 競争 | ランキング | なし |
| 物語 | クエスト | マップ自体が応答する |
| 真面目度 | 損なう傾向 | むしろ親密度が上がる |
| 既存例 | Habitica | たまごっち、植物育成アプリ |

→ 「生き物感」は K4 の延長として最も研究者向きの落とし所。

### K6.4 段階導入の哲学

| 案 | 哲学 |
|----|------|
| K6.4a | 全部入れる、ON/OFF で個別選択 | 自由度最大、UI 複雑 |
| K6.4b | 副作用最小のものから入れる（K4 → K5 → K2 → K1 → K3）| 安全 |
| K6.4c | 「ゲーム化しない」を最初から方針として宣言 | 純度最大 |
| K6.4d | テーマパック（「研究者向けミニマル」「ライト育成」「フル」）| プリセット化 |
| K6.4e | 入れずに、後から「実装する人がいたら任せる」| 拡張プラグイン化 |

→ akaghef の判断待ちの典型論点。

## K7: アンチパターン（やってはいけない設計）

### K7.1 「触れば触るほど報酬」の罠

| アンチ | 何が起きるか |
|-------|---------|
| K7.1a | ノード数で評価 → 薄いノードを乱造 |
| K7.1b | 編集回数で評価 → 意味なく編集ボタンを押す |
| K7.1c | リンク数で評価 → 関係ないノード同士を繋ぐ |
| K7.1d | 滞在時間で評価 → 開いたまま放置 |

→ 対策: K2.4c（**整理活動を評価**）、K2.3b（行動の多様性で評価）、
   AI による「内容の質」の参照（K4.1h）

### K7.2 「切れる恐怖」の罠

| アンチ | 何が起きるか |
|-------|---------|
| K7.2a | ストリーク切れ警告 → 旅行先でも触らないと罪悪感 |
| K7.2b | 「あと1日触らないとリセット」通知 → 強迫的 |
| K7.2c | バッジ達成期限 → 焦って雑に達成 |

→ 対策: K1.3f（休息日）、K1.6a（休暇宣言）、K1.4a（通知 OFF）

### K7.3 「子供っぽさ」の罠

| アンチ | 何が起きるか |
|-------|---------|
| K7.3a | RPG 物語演出（K3.6c）→ 査読者にデモ見せられない |
| K7.3b | 派手な達成アニメ + 効果音 → 真面目作業中に萎える |
| K7.3c | キャラクター登場 → ブランド毀損 |
| K7.3d | キラキラエフェクト多用 → 視覚疲労 |

→ 対策: 演出強度スライダー（K4.7d）、トーン選択（K3.6d）

### K7.4 「データ的搾取」の罠

| アンチ | 何が起きるか |
|-------|---------|
| K7.4a | 操作履歴を全保存 → privacy 制約と衝突 |
| K7.4b | 詳細な行動分析 → policy_privacy 違反リスク |
| K7.4c | 達成データを外部送信 → 研究内容流出懸念 |

→ 対策: 全てローカル保存、外部送信ゼロ前提（既存 M3E 方針通り）

### K7.5 「機能膨張」の罠

| アンチ | 何が起きるか |
|-------|---------|
| K7.5a | K1〜K5 全部実装 → コードベース複雑化、保守不能 |
| K7.5b | 設定項目が増えすぎ → 新規ユーザーが迷う |
| K7.5c | ゲーム要素のバグ修正に時間取られる → 本筋停滞 |

→ 対策: 段階導入（K6.4b）、または不採用（K6.4c）

### K7.6 「研究プロセスの歪み」の罠（最大の懸念）

| アンチ | 何が起きるか |
|-------|---------|
| K7.6a | クエストに合わせて研究方向を曲げる | 研究的誠実さ毀損 |
| K7.6b | バッジ目当てで「終わってない」を done 化 | 記録の正確性毀損 |
| K7.6c | クイズで答えを覚えてる風に振る舞う | 自己欺瞞 |

→ 対策: 評価軸を「やった量」でなく「振り返りの質」に置く（K2.3a, K3.4e）

| 候補 | 副作用リスク | 真面目度両立 | 推奨度 |

| K4 ノード進化（K4.2a + K4.3b） | 低 | ◎ | A |
| K5 デイリークイズ（K5.2a + J系統合）| 低 | ◎ | A |
| K2 バッジ（K2.4c 整理活動評価 + K2.2c 静か）| 中 | ○ | B |

| K3 クエスト（K3.2c AI 提案 + K3.6a 演出なし）| 中 | ○ | B |
| K1 ストリーク（K1.3f 休息日 + K1.4a 通知OFF）| 中〜高 | △ | C |
| K2 量バッジ（K2.4a ノード数等）| 高 | × | D 不採用候補 |

→ A 推奨 = MVP 候補
→ D 不採用候補 = ブレストとしては記録するが、実装しないことを明示推奨

## 横断的気づき（K6/K7）

- **「OFF にできれば良い」は逃げ口上にならない**。
  デフォルト ON のものは多くのユーザーが触らずに済ませる
  → デフォルト OFF + 明示的 ON が原則
- **アンチパターンを README にも置く** 価値がある。
  実装側エージェントが知らずに踏むのを防ぐ
- **「整理活動を評価する」軸が M3E 独自性の核**。
  K2.4c はゲーム化と D 系（hygiene）の橋渡しで、
  単なる「集める系ゲーム」を超える設計余地
- **「生き物感」（K6.3）は『ゲーム化』というラベルを外して提示** することで
  研究者の心理的抵抗を最小化できる。命名戦略の問題
- **K1 と K3 の物語演出は最も滑りやすい**。
  M3E では「真面目モード」が標準であり、ノリの強い演出は浮く
