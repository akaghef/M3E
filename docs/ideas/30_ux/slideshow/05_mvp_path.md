# 05. MVP 設計案と未決の質問

レビューが入る前の **デフォルト案**（暫定）。
全部「決め打たず、論点ID で後から差し替え可能」を基本に置いた。

## 暫定 MVP（最小で動く形）

### スコープ

- 1マップに1ツアー（論点D=D1）
- ステップ = nodeId の配列（論点A=A1, B=B3, C=C1）
- カメラはパン演出（論点E=E3）+ 周辺フェード（論点F=F2）
- UI は下部オーバーレイバー（論点G=G1）
- 操作は矢印キーと Prev/Next ボタン（論点I=I1+I4）
- 注釈は ノードの details 流用（論点H=H1）
- 永続化はマップ JSON 内 `state.tour`（論点J=J1, 単一なので配列1本）
- エクスポートなし、URL params のみ（論点K=K6）

### データ追加（マップ JSON）

```json
"state": {
  "rootId": "...",
  "nodes": { ... },
  "links": { ... },
  "tour": {              // ← 追加（単一ツアー）
    "steps": ["n_xxx", "n_yyy", "n_zzz"],
    "updatedAt": "2026-04-16T..."
  }
}
```

### UI 要素

- ヘッダーに「▶ Play tour」ボタン
- 再生中は下部に固定バー
  - `‹ Prev` `Next ›` `× Exit`
  - `3 / 12` 進捗
  - 現在ノードの details テキスト（折り返し2行まで）
- 矢印キー: ← → でステップ移動、Esc で終了

### 編集 UX（最小）

- ノード右クリック → 「ツアーに追加（末尾）」
- ノード右クリック → 「ツアー番号を表示」（ステップ何番か）
- ツアー一覧は当初は無し（編集はマップ JSON 直書き or REST 経由）
- 並び替え UI は次フェーズ

### 推定実装ボリューム

- 純フロント変更: viewer.ts に ~150 行、CSS に ~50 行、HTML に控えめなバー要素
- データ層: rapid_mvp.ts のスキーマに `tour` フィールド追加、保存・読込パス1個
- 既存の `centerOnNode` をそのまま使えば、目玉のカメラ移動はほぼ書かなくて良い
- 新規ファイル: たぶん不要（viewer.ts に閉じ込められる）

これで「動く」最小形になる。

## 未決の質問（akaghef レビュー待ち）

優先度順:

1. **ツアーは1マップ1本で良いか、複数欲しいか**（論点D）
   - 戦略レビュー用 / 教員用 / 自分振り返り用 を別個に持ちたいなら D2 を最初から入れる
2. **同一マップで複数ツアーを使うとして、どこから切替えたいか**
   - サイドメニュー / URL params / 専用パレット
3. **発表用フルスクリーンは初期から欲しいか**（論点G2）
   - 欲しいなら CSS 追加、不要なら G1 だけで進める
4. **注釈テキストは details 流用で十分か、ツアー固有の note が要るか**（論点H）
   - details が技術メモ寄りだとプレゼン文として読みにくい可能性
5. **ステップ間のパン演出は最初から欲しいか、瞬時で十分か**（論点E）
   - パン入れるとサブツリーの大きさで距離計算が要る、瞬時なら今日にも書ける
6. **URL 共有リンクで再生開始できる必要はあるか**（論点K=K6）
   - 必要なら最初から URL params パーサに `tour` `step` を追加
7. **ノード削除時の挙動**（論点K2/K3）
   - 黙って外す / 警告する
8. **ツアー編集はマップ直書きで耐えられるか、最初から GUI が要るか**
   - 自分用なら直書きで耐えられる。共有用なら GUI 必須

## 段階導入ロードマップ案

```
Phase 1 (MVP)         : 上記の暫定スコープ全部
Phase 2 (Polish)      : 並び替え UI / ツアー編集モード / 注釈フィールド分離
Phase 3 (Multi)       : 名前付き複数ツアー / 切替 UI
Phase 4 (Present)     : フルスクリーン / カメラ演出 / リモコン
Phase 5 (Export)      : PDF / 共有リンク / 静的 HTML
Phase 6 (Smart)       : エージェント自動生成ツアー / クエリ駆動ツアー
```

各 Phase は前 Phase の上に積めるよう設計する（後方互換）。
最低でも Phase 1 が独立で価値を持つように切ってある。

## 開発時の留意点（実装に入るとき思い出す用）

- 既存ホットキーと衝突しない（slideshow モードでキー横取り）
- collapsed なノードがステップに来たら自動展開（戻り時は元に戻すかは別決定）
- viewState への state 追加は最小限（再生中フラグ + 現在ステップ index）
- ツアー1件のスキーマ拡張は前方互換に。`tour` 未定義のマップで壊れない
- agent (LLM) が「ツアー読んで作業順序にする」用途を最初から想定 → 構造はシンプルに
