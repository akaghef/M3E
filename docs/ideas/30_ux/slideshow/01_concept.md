# 01. コンセプト

## 何を作るか

M3E マップ上のノードを **作者が指定した順序で1つずつフォーカス** し、
視聴者は「次／前」で進めるだけでマップの読み筋を辿れる機能。

PowerPoint 的なスライド単位ではなく、**マップ上のノードに「視点」を当てていく**形を基本とする。
マップ自体が舞台で、ツアーはその上のカメラワーク。

## 想定ユースケース

| # | シーン | 誰が見る | 期待される効果 |
|---|--------|---------|---------------|
| 1 | 研究の意思決定ログを指導教員に説明 | 教員1名 | 何を・なぜ決めたかが時系列で伝わる |
| 2 | 新メンバーへのプロジェクトオンボーディング | チーム新規参加者 | マップ全体を「迷子にならず」歩ける |
| 3 | 自分用の振り返り（過去の判断の再訪） | 本人 | 当時の判断列を再生して再評価 |
| 4 | 学会・セミナー発表 | 聴衆多数 | スライドを別途作らず M3E から直接発表 |
| 5 | LLM/エージェントへの説明 | エージェント | ツアー順序＝重要度として参照 |
| 6 | 進捗レビュー（strategy ノードを順番に） | 自分 / akaghef | 戦略の現状を順番に確認 |

ユースケース毎に必要な機能の重みが違う。例えば
- (1)(4) → 演出・カメラ移動・全画面が重要
- (2)(3) → 編集中も再生でき、自由に分岐できる方がよい
- (5) → JSON で機械可読な順序情報が必要
- (6) → 既存ノードに後付けで「順序」を振れる軽さが重要

## 既存ツールとの差分（あえて M3E でやる理由）

- **Reveal.js / PowerPoint**: スライド本体を別途作る必要があり、マップとの二重管理。
- **Freeplane の presentation 機能**: ノード単位フォーカスはあるが、編集体験が分離。
- **Obsidian のスライド系プラグイン**: テキスト前提でグラフ構造を活かせない。

M3E でやる価値は **「マップが資料そのものであること」** に尽きる。
資料を別に作らない＝メンテコストゼロ、というのが訴求点。

## やらないこと（初期スコープ外候補）

- アニメーション付きトランジション（パン以上の演出）
- 視聴者側の編集ロック・権限制御
- リアルタイム協調プレゼン（presenter→viewer 同期）
- 音声ナレーション
- スライド単位の独立レイアウト（マップから切り離すと差分が破綻する）

これらは「あったら嬉しいが、本質ではない」ので別フェーズ。

## 論点

- **論点A. ステップの粒度**
  - A1: 1ステップ = 1ノード（最小・直感的）
  - A2: 1ステップ = (ノード, カメラ設定) のペア（同じノードで違う見せ方が可能）
  - A3: 1ステップ = 完全独立スライド（マップから切り離せるが二重管理）
  - 推し: A1 ベース、必要に応じて A2 を後付け
- **論点D. 複数ツアーの管理**
  - D1: 1マップに1ツアーのみ
  - D2: 名前付き複数ツアー（onboarding / weekly-review / for-advisor 等）
  - 推し: 最初は D1、設計上は D2 に拡張可能な構造で
