# J3. 複利的知識構築（前提→応用の自動推薦）

ノードを **「前提（prerequisite）→ 応用（application）」** の DAG として扱い、
新ノードに対して「これを理解するために必要な過去ノード」「このノードから派生できる応用」を自動推薦する。

研究マップが **複利的に育つ**（Andy Matuschak 流 Compounding Notes）ようにする土台。

## なぜ要るか

研究は積み上げ。だが M3E は今のところ:
- 親子（hierarchy）と任意 link しか持たない
- 「Aを知らないとBを理解できない」依存関係を扱う仕組みがない
- 結果、**既に書いた前提の再発見** が手動

→ 過去資産を **次の思考に再投入** する仕組みが必要。これが「半年で世界モデル → 射影」を支える基盤。

## ユースケース具体化

1. **新仮説を書いた瞬間** — 「この仮説の前提となる過去ノード3個」が即推薦される
2. **論文の章を書く時** — 「この章の主張を支える既存ノード群」を一覧表示
3. **ブレスト中** — 新ノードを置いた瞬間「関連する過去ブレスト3件」が表示
4. **指導学生に教える時** — 「この概念を教える前に、学生が理解すべき前提群」を整理
5. **過去資産の活性化** — 「3ヶ月前のあのアイデアが今のテーマに使えそう」を AI が指摘
6. **科研費申請時** — 「申請書の各主張に対応する既存ノード」自動マッピング（projection に直結）

## J3.A. 前提関係の取り方

| 案 | 仕組み | 精度 | 手間 | 推し度 |
|---|---|---|---|---|
| J3.A1 明示リンク | `prereq` 種別の link を手動付与 | 高 | 高 | ★★ |
| J3.A2 親子流用 | hierarchy をそのまま前提と見做す | 中 | 低 | ★★ |
| J3.A3 タグ依存 | タグ間の依存定義 | 中 | 中 | ★ |
| J3.A4 AI 推論 | LLM が text/details から判定 | 中〜高 | 低（自動） | ★★★ |
| J3.A5 編集履歴 | 「Aを書いた直後にBを書いた」を依存と推定 | 低 | 低 | ★ |
| J3.A6 ハイブリッド | A1 + A4（明示が無ければ AI 推論） | 高 | 低 | ★★★ |

**観察**: J3.A6 が現実解。明示と AI を両立。

## J3.B. 推薦タイミング

| 案 | When | 邪魔さ | 推し度 |
|---|---|---|---|
| J3.B1 ノード作成直後 | 新ノード保存時に「関連3件」表示 | 中、流れを断つ | ★★ |
| J3.B2 編集中ライブ | 入力中に随時推薦パネル更新 | 高、注意散漫 | ★ |
| J3.B3 明示要求 | 「関連を探す」ボタン | 低、能動的 | ★★★ |
| J3.B4 バックグラウンド | 別 view で「今日の関連発見」サマリ | 弱、見落とし | ★★ |
| J3.B5 AI agent 経由 | サブエージェントが朝に整理して提示 | 低、非同期 | ★★ |
| J3.B6 hover 時 | ノードホバーで関連が浮かぶ | 中 | ★ |

**観察**: J3.B3 + J3.B5 の組み合わせ（明示要求と非同期サマリ）が研究フローに馴染む。

## J3.C. 推薦の表示形式

| 案 | 形式 | 推し度 |
|---|---|---|
| J3.C1 サイドリスト | 右ペインに関連ノード一覧 | ★★ |
| J3.C2 マップ上強調 | 既存マップ上で関連ノードがハイライト | ★★★ |
| J3.C3 仮想リンク表示 | 推薦された関連を点線リンクで表示 | ★★ |
| J3.C4 カード | 「あなたの 2025-09 のノード覚えてる？」カード | ★★ |
| J3.C5 サブツリー展開 | 推薦ノードを subtree として一時表示 | ★ |
| J3.C6 自動 group化 | 関連ノードを一時的にグループ化 | ★ |

**観察**: J3.C2 + J3.C3 が「マップ上で関係が見える」点で M3E らしい。

## J3.D. 「複利」の指標

複利的に育っている / 育っていないをどう測るか:

| 案 | 指標 | 解釈 |
|---|---|---|
| J3.D1 平均リンク数 | ノードあたり被リンク数 | 高い=再利用されている |
| J3.D2 古ノード再活性度 | N日前以前のノードへの新リンク率 | 高い=過去資産活用 |
| J3.D3 DAG 深さ | 前提チェーンの最大長 | 深い=積み上がっている |
| J3.D4 AI 評価 | 「マップから新発見できる量」を LLM 評価 | 主観的 |
| J3.D5 「呼び戻し率」 | SRS（J1）で再露出されたノードの活用率 | J1 連動 |

**観察**: J3.D2 が「複利が回っているか」の本質指標。ダッシュボード化候補。

## J3.E. 推薦の質をどう保つか

ノイズ推薦は無視されてやがて機能停止する。
| 案 | 質保証 | 推し度 |
|---|---|---|
| J3.E1 ユーザフィードバック | 推薦に「採用 / 不採用」マーク、学習 | ★★★ |
| J3.E2 thresholding | 信頼度 ≥ N のみ表示 | ★★ |
| J3.E3 個数制限 | 上位 3〜5 件のみ | ★★★ |
| J3.E4 種別表示 | 「前提として / 応用として / 類似」を明示 | ★★ |
| J3.E5 「なぜ推薦か」説明 | LLM の根拠を1文表示 | ★★ |

**観察**: J3.E3 + J3.E5 はセットでないとノイズ化する。

## J3.F. 「Compounding Notes」原則

Andy Matuschak の永続ノート原則を M3E に翻訳:

| 原則 | M3E への翻訳 | 該当論点 |
|---|---|---|
| Atomic | 1ノード1概念 | D7 巨大ノード警告 |
| Concept-oriented | テーマ単位、トピック単位ではない | J3.E4 種別 |
| Densely linked | 多重リンク推奨 | J3.D1 |
| Personal | 自分の言葉で再表現 | J1.G1 SRS再露出 |
| Evergreen | 何度も書き直される | J2 鮮度回復 |

**観察**: J3 は単独機能ではなく、J1/J2 と相互依存する原則の **実装層** に位置する。

## J3.G. 「応用」推薦（projection vision との接続）

「過去ノード → 申請書/論文の主張」のマッピングを自動化:

| 案 | 仕組み | 推し度 |
|---|---|---|
| J3.G1 主張テンプレ | 「この主張に使える過去ノード」を AI が抽出 | ★★★ |
| J3.G2 章ごとマッピング | 論文章 → 関連ノード群の対応表 | ★★ |
| J3.G3 申請書テンプレ駆動 | 科研費の各セクションに紐付ける専用属性 | ★★★ |
| J3.G4 「このノード使える論点リスト」 | 既存ノードを開くと「使える書類セクション」表示 | ★★ |

**観察**: J3.G3 は project_projection_vision の本丸。B2「科研費テンプレ生成」と直結。
最優先候補。
