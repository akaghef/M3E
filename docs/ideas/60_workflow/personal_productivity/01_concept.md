# 01. コンセプト: なぜ M3E を生活の中心にするのか

「研究思考支援ツール」を週1回の道具で終わらせるか、毎日の足場にするか。
本トピックは後者を狙う体験設計の素材を集める。

## 1. 解こうとしている問題

### Pr1. 研究は「気が向いた時にやる」と進まない
論文を書く、データを見る、仮説を練る。
これらは **特定の認知状態** が要る作業で、忙しい日常でその窓を逃す。
- 関連: automation_obstacles A5 エネルギー管理

### Pr2. 「やる気」だけに頼るとボラティリティが大きい
モチベーションは天候のようなもの。仕組みで補わないと月単位で停滞する。
- 関連: J1 燃え尽き

### Pr3. M3E に書いたことを「思い出さない」と腐る
1ヶ月後に見ると「これ何だっけ」になる。
触り続ける動線がないと、過去の自分の知が他人化する。
- 関連: D3 stale 検出 / J2 忘却曲線

### Pr4. 自己欺瞞を放置すると無意味な作業が増える
「論文 50 本読んでる ≠ 論文書いてる」を可視化できないと、
忙しさに溺れて成果ゼロの月が生まれる。
- 関連: automation_obstacles A4 自己欺瞞

### Pr5. 自動化しすぎると思考の場が消える
カレンダーや TODO アプリで全部管理すると、思考の余白が削られる。
M3E は「未構造のまま置ける場所」であり続けるべき。
- 関連: automation_obstacles N7 創発の場 / N9 自己観察

## 2. 研究者 akaghef の典型的な1日 (仮)

ブレスト用ペルソナ。実際のリズムに後で合わせる前提。

```
 6:30  起床 / コーヒー
 7:00  M3E を開く ← ここに何が入るか?
 9:00  講義 / 学生対応
12:00  昼食 / メール処理
13:30  研究時間 (深い思考が要る時間帯)
16:00  会議 / 雑務
18:30  帰宅 / 家庭時間
21:00  軽い読書 / 振り返り ← ここにも M3E?
23:00  就寝
```

**観察**:
- 7:00 と 21:00 が M3E にとっての「ゴールデンタイム」
- 13:30 の研究時間こそ集中モードを要する
- 12:00 / 16:00 は「散らかったまま放置される時間」

## 3. ユースケース (具体5シーン)

### UC1. 朝のキックオフ
コーヒー片手に M3E を開く。
昨日 done になったノードと今日触るべきノードが3〜5個提示される。
何を「今日やる」と宣言するかを1分で決める。
→ 関連: A1 朝チェックイン / M23 Sprint Plan

### UC2. 研究時間の Pomodoro
1ノード = 1ポモドーロを宣言。
画面が他ノードをフェードアウトし、集中を物理的に保護。
25分後に「進んだ」「進まなかった」を1キーで残す。
→ 関連: F1 / G1 集中フェード / G4 集中破壊ログ

### UC3. 会議直後の捕捉
会議の決まり事を Triage mode で scratch に投げ込む。
**儀式化されていないが、儀式の隙間** に成立する捕捉。
→ 関連: keyboard_modes M02 Triage

### UC4. 夜の振り返り
3問だけ答える: 「今日進んだこと」「詰まったこと」「明日の最初の一手」。
それぞれ1〜2行で十分。書き続けると Journal mode (M26) に堆積。
→ 関連: A3 夜チェックイン / B3 Retro 統合

### UC5. 週末の Retro
1週間を振り返って Keep/Problem/Try。
日次の Journal が自動でクラスタリング候補に。
→ 関連: M25 Retro mode / B2 KPT

## 4. 既存ツールとの差分

| ツール | 強み | M3E との違い |
|-------|------|------------|
| Notion / Obsidian | ノートと DB の柔軟性 | M3E はグラフ構造が一級市民 |
| TickTick / Todoist | タスクのスケジューリング | M3E は **未構造のまま置ける** |
| Toggl / Clockify | 時間記録に特化 | M3E は思考の文脈とリンクできる |
| Habitica / Streaks | 習慣のゲーム化 | M3E はゲーム化を **強制しない** |
| Calendar | 時刻と予定 | M3E は時刻を主軸にしない (J3) |
| RescueTime | 自動計測 | M3E は能動記録を価値とする |

**仮説**: 本トピックの肝は「上記ツールを M3E が置き換える」ではなく、
**思考と時間/エネルギー/習慣が同じ画面で見える** ことの新規性。

## 5. 設計哲学 (本トピック共通の鉄則)

### Ph1. 強制しない、サポートする
- 未記入を責めない
- スキップしてもバッジが消えない
- Streak が切れても続きが書ける
- 関連: K6 マップの「生き物感」(育てるが枯らす自由もある)

### Ph2. 自動化 < 効率化
- ワンキーで書ける
- だが書く判断は人間
- AI は「今書くべきか」を判定しない
- 関連: automation_obstacles N9 自己観察

### Ph3. 既存生活を侵食しない
- 散歩 / 雑談 / 読書 を奪わない
- 通知でユーザーを呼ばない (G7 集中モードと裏返し)
- 関連: N7 創発の場

### Ph4. データを見たくない時の自由
- ログを取ったが「見せない選択」を持つ
- ヒートマップを「閉じておく」設定
- 関連: I4 / Q2 無視・忘却の意図的設計

### Ph5. 研究者という非定型労働者向け
- 9-17時の固定リズムを前提にしない
- 締切ドリブンの波を許容
- 燃え尽き予測より、燃え尽き後の復帰支援
- 関連: J1 燃え尽き

## 6. 採否しない選択肢の俯瞰 (パターン軸)

本トピックを掘る上での選択軸:

| 軸 | 一方の極 | 他方の極 |
|---|---------|---------|
| トリガー | 時刻通知 | 手動起動 |
| 粒度 | 1ノード単位 | プロジェクト単位 |
| 計測 | 自動 | 手動申告 |
| 可視化 | 常時表示 | オンデマンド |
| ゲーム化 | あり (Streak/バッジ) | なし (静かな記録) |
| AI 介在 | 強 (提案・要約) | 弱 (記録のみ) |
| 共有 | 公開可 | 完全プライベート |
| 強制度 | 未記入アラート | 完全任意 |

各論点を後続ファイルで複数案として展開する。
