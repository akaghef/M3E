# 01. コンセプト — 構造スニペットとは何か

## C1. 定義

**構造スニペット** = M3E のノードに対して、**子ノード構造を一発展開する小さなテンプレート**。

例:
- 親ノード「新規事業 X」にカーソル → `SWOT` スニペット呼び出し → 子に `Strengths / Weaknesses / Opportunities / Threats` の4ノードが即生成
- 親「研究テーマ Y」→ `5W1H` → 子に `Who / What / When / Where / Why / How` の6ノード

**内容は空**で構造だけ展開される。中身はユーザーが後から手で（または AI で）埋める。

## C2. なぜ必要か

### 手で毎回書く苦痛

研究・思考・分析で繰り返し使うフレームワークが多数ある（5W1H、SWOT、MECE、ロジックツリー...）。
これらを毎回手で書くのは摩擦が大きい:
- スペルミス・項目漏れ・順序のブレ
- 「あれ、SWOT って SW と OT どっちが内部要因だっけ」
- 集中が切れる

### フレームワーク想起コスト

「こういう時にどのフレームワーク使えばいいか」が思い出せない問題もある。
カタログがあれば逆引きできる:
- **「分析が必要 → PEST か SWOT か 5Forces」**
- **「アイデアを広げたい → SCAMPER か 6色ハット」**
- **「意思決定したい → RICE か MoSCoW か決定マトリクス」**

### 構造化の強制力

スニペットで「先に構造」を置くと、**空欄を埋める**作業に変わる。
これは認知負荷を下げる（生成的思考 → 反応的思考）。
特に熟練していない領域で効果大。

## C3. ユースケース（誰が、いつ、どんな時）

具体シーンを最低 5 個:

### U1. 研究者の論文構想フェーズ
- 新しい研究テーマを思いついた時、親ノードに `PICO` スニペットを当てて対象・介入・比較・アウトカムを強制的に埋める
- リサーチクエスチョンを `5W1H` で分解

### U2. 事業企画の初期検討
- 新規事業アイデアのノードに `SWOT` と `5Forces` を両方当てて多角的に検討
- `リーンキャンバス` スニペットで 9 項目を一気に展開

### U3. 会議の議事骨格
- 会議開始時に親ノード「定例 2026-04-17」に `議題/決定/TODO/ブロッカー` の4項子スニペット
- 議論中に `なぜなぜ` スニペットで深堀りモード起動

### U4. 意思決定の場面
- 「A と B どっち？」という親ノードに `決定マトリクス` スニペット → 評価軸の行と選択肢の列を自動生成
- `プレモーテム`（失敗前提の予想）スニペットで反対視点を強制

### U5. 学習ノート整理
- 教科書の章を読み終えたら `要点/例/疑問/応用` スニペットで振り返り
- `ファインマンテクニック` スニペット（説明/混乱/再学習/類推）

### U6. バグ調査・トラブルシュート
- 障害ノードに `なぜなぜ分析` スニペットで 5 階層の why を事前展開
- `Kepner-Tregoe`（問題分析の定型）スニペット

### U7. クリエイティブ発想
- アイデア詰まりに `SCAMPER`（置換/結合/適応/修正/他用途/除去/逆転）スニペットで強制分岐
- `6色ハット` スニペットで多視点レビュー

### U8. 面接・評価
- 面接記録ノードに `STAR`（状況/タスク/行動/結果）スニペットで整理
- 人事評価に `360度`（自己/上司/同僚/部下）スニペット

## C4. 既存ツールとの差分

| ツール | 何ができる | M3E との差分 |
|---|---|---|
| Notion テンプレ | ページ単位でテンプレ展開 | M3E は**ノード（木の一部）単位**で展開 |
| Obsidian Templater | ファイル単位、変数展開が強力 | ファイル単位ではない、**構造展開**が本質 |
| Roam Research | スラッシュコマンドでブロック展開 | 近いが、Roam は箇条書きベース、M3E はツリー |
| Miro/FigJam | 付箋グループテンプレ（SWOT 等） | 自由配置 vs **構造化ツリー**、M3E の方が検索・再利用に強い |
| Airtable テンプレ | データベース構造のテンプレ | レコード指向、ツリー展開にはならない |
| Workflowy/Dynalist | アウトライン、テンプレはプラグイン頼み | ネイティブ機能として位置づけ |
| Notion Database Template | 特定データベースの行テンプレ | M3E は「どのノードにも当てられる汎用スニペット」 |
| VSCode snippets | コード断片を挿入 | **思考構造**を挿入する、という転用発想 |

**M3E の強み**: ツリー構造の操作（親→子一括生成、再配置、折りたたみ）がネイティブなので、
構造スニペットと親和性が極めて高い。

## C5. 「構造」と「内容」の分離

### 構造 = スニペットが提供
- 子ノードのラベル（例: `Strengths`, `Weaknesses`...）
- 階層（1階層 / 2階層 / 再帰的）
- ノード種別（質問/タスク/データ）やタグ
- 場合によってはリンク関係（X は Y の原因、等）

### 内容 = ユーザー or AI が埋める
- ラベル配下の本文
- ノードの詳細（text/details）
- 判断・評価・具体例

**この分離により**:
- スニペット側を交換可能にできる（SWOT → SWOTN、TOWS に差し替え等）
- 同じ内容を別の構造で捉え直すのが楽（視点の切替）
- AI に「この構造の空欄を埋めて」と頼める

## 補足: スニペットの粒度

スニペットは以下の粒度を混在させうる:

| 粒度 | 例 | 特徴 |
|---|---|---|
| **原子** | 5W1H の 6 項目だけ | 組み替え自由、再利用性高 |
| **複合** | リーンキャンバス 9 項目 | 1ドメインを網羅、独立性高 |
| **パラメタ化** | なぜなぜ（深さN） | 呼び出し時に変数指定 |
| **連鎖** | PDCA（P→D→C→A）+ 各段階の詳細テンプレ | 多段階、時間軸を含む |
| **組合せ** | SWOT + TOWS の連続展開 | ワークフロー的 |

→ データモデル（07 参照）で粒度を許容する設計が必要。
