# 01. 評価軸

数学 PDF OCR ツールを比較するための **軸** を並べる。個別ツールはこの軸で評価する（02〜04 で参照）。

## Ax1. 数式保持形式

| 形式 | 典型出力 | 編集容易性 | 意味的アクセス | M3E 取り込み |
|---|---|---|---|---|
| **LaTeX** | `\int_0^1 f(x)\,dx` | 高 | △（パース要） | KaTeX で表示しやすい |
| **MathML** | `<mrow><mi>f</mi>...</mrow>` | 中 | ◎（構造化済み） | 表示は MathJax |
| **Unicode plain** | `∫₀¹ f(x) dx` | 高 | × | シンプルだが限界 |
| **画像 (PNG/SVG)** | `<img>` | 低 | × | 表示のみ可能 |
| **AsciiMath** | `int_0^1 f(x) dx` | 高 | △ | マイナー |

→ M3E 基本方針: **LaTeX を正として保存、表示は KaTeX、将来的に MathML 併記可**

## Ax2. 出力の構造粒度

- **A2-1. フラット Markdown**: ページ単位の一本文字列。Marker / Nougat の基本モード
- **A2-2. ブロック配列 (JSON)**: 見出し・段落・数式・図をブロックに分割。MinerU / Docling 系
- **A2-3. 文書全体ツリー**: 章→節→段落の階層 JSON。Unstructured / LlamaParse 系
- **A2-4. レイアウト保持**: 座標付き要素配列（画像・数式・テキストを位置関係ごと）

→ オントロジー化には **A2-2 or A2-3 が望ましい**（M3E の木構造に直接マップ可能）

## Ax3. 図版・表の扱い

- **A3-1. 無視**（テキストのみ抽出）
- **A3-2. bbox 出力**（座標を返すだけ、画像抽出は別ツール）
- **A3-3. ラスタ抽出**（PNG 切り出し）
- **A3-4. ベクタ抽出**（SVG 切り出し — pdf2svg / mutool 併用）
- **A3-5. 表を表 (markdown table/HTML table) 化**

→ M3E ノードに図を貼る前提なら **A3-3 or A3-4 必須**。前の会話で SVG ノード採用方針なので A3-4 優先

## Ax4. OCR 必要性 (テキスト層の有無)

- **A4-1. 電子組版 PDF**: テキスト層あり、数式も LaTeX から生成された PDF。一番楽
- **A4-2. スキャン PDF**: 画像のみ、完全 OCR が必要
- **A4-3. ハイブリッド**: テキスト層ありだが数式は画像化されている（多くの教科書がこれ）
- **A4-4. 手書き**: 黒板写真等、超高難度

→ 対象が A4-2 / A4-3 中心なら **OCR+数式認識両方必要**。A4-1 中心なら PyMuPDF + pix2tex 程度で足りる可能性

## Ax5. 実行環境・ホスティング

| 環境 | メリット | デメリット |
|---|---|---|
| ローカル CPU | 無料・プライバシー◎ | 遅い（1 冊で時間単位） |
| ローカル GPU | 高速・プライバシー◎ | VRAM 要求（7GB〜） |
| クラウド API | 運用楽・最新モデル | 従量課金・プライバシー懸念 |
| セルフホスト API | バランス | インフラ維持コスト |

→ M3E は個人研究用途なので **ローカル GPU 優先、商用 API はスポット利用** が基本線（project_cloud_sync_solo と整合）

## Ax6. ライセンス

- **A6-1. MIT / Apache**: 自由に使える
- **A6-2. BSD**: 同上
- **A6-3. GPL / AGPL**: 組み込みに制約。M3E は個人利用主体なので致命ではない
- **A6-4. 非商用限定 (CC BY-NC 等)**: 趣味利用なら OK
- **A6-5. 商用ライセンス必須**: 個人購入可否と金額次第

→ 基本 A6-1/A6-2 優先、A6-3 は個人使用なら許容、A6-4/A6-5 は **1 冊単位で使うかどうか** で判断

## Ax7. 言語対応

- **A7-1. 英語のみ**: 洋書メイン
- **A7-2. 多言語 (日中仏独等)**: 広範な対応
- **A7-3. 日本語特化**: 和文数学書に強い
- **A7-4. CJK 対応**: 中国・韓国発ツールは CJK が強い傾向

→ 岩波・共立・裳華房等の和書を取り込むなら **A7-2 or A7-3 必須**

## Ax8. コスト (商用 API のみ)

1 冊 500 ページ想定で比較:

| 料金単位 | 500 ページ換算 | 年 20 冊なら |
|---|---|---|
| $0.002/page | $1 | $20 |
| $0.01/page | $5 | $100 |
| $0.05/page | $25 | $500 |
| $0.10/page | $50 | $1000 |

→ **$0.01/page 前後** が「趣味でも許せる」ライン。Mathpix は歴史的にやや高い (ページ/行/式で課金差)

## Ax9. 構造タグの自動検出

- **A9-1. なし**（テキストだけ）
- **A9-2. 見出しレベルのみ**（h1/h2/h3）
- **A9-3. 数式ブロック判定**（inline/display 区別）
- **A9-4. 定理環境 (thm/lem/prop/cor) 検出**
- **A9-5. 参照 (式番号・定理番号) 抽出**

→ A9-4/A9-5 は 2026 年時点どのツールも怪しい。**LLM 後処理に回すのが現実的**

## Ax10. M3E ノード化の相性

- **A10-1. 1 ページ = 1 ノード**: 粗すぎる。使わない
- **A10-2. 1 節 = 1 ノード、子に段落・数式**: 一般的
- **A10-3. 1 定理/定義 = 1 ノード**: オントロジー指向の正解
- **A10-4. 1 概念 (名詞) = 1 ノード**: 最も抽象、LLM 後処理必須

→ 段階的に **A10-2 → A10-3 → A10-4** と精緻化するロードマップが自然

## 論点 (このファイル発生分)

- **論点 X1**: 数式の正本を LaTeX と MathML のどちらにするか（両方持つか）
- **論点 X2**: 図版を SVG にするか PNG にするか（前の会話で SVG 方針だが大きい図版は PNG の方が軽い）
- **論点 X3**: ローカル GPU をどこまで要求するか（7GB / 16GB / 24GB の閾値でツール選択が変わる）
- **論点 X4**: 和書と洋書で別パイプラインを用意するか、単一で済ませるか
- **論点 X5**: 構造タグ (Definition/Theorem) は OCR 側で取るか LLM 側で取るか
