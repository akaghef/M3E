# 03. Processing Pipeline — 取り込み後の処理

チャネルを通って入ってきた raw データを、どう scratch ノードまで加工するか。
**「入った後」の体験設計**。AI を挟むか挟まないか、どこまで自動分類するか。

---

## P0. パイプライン全体像（候補）

```
[チャネル] → [前処理] → [構造化] → [メタ付与] → [scratch 投入] → [triage キュー]
              ↓             ↓           ↓             ↓              ↓
            P1 分割      P2 分類      P5 リンク     P10 投入規則    P11 失敗時
            P7 翻訳      P3 要約      P6 引用       P12 レート      
            P8 OCR       P9 AI パス
```

各段階に複数の選択肢がある。**全段階を AI 任せにする案** と **人間が触る前は加工しない案** の
両極端の間に多数のグラデーションがある。

---

## P1. 句点・段落での自動分割

長文 capture を複数ノードに分けるか、1ノードに入れるか。

### 案 P1-a: 1 capture = 1 ノード（分割しない）
- 利点: シンプル、原文性保持
- 欠点: 後で分割する手間

### 案 P1-b: 句点（。.）で自動分割
- 利点: 思考の単位に近い
- 欠点: 文脈が切れる

### 案 P1-c: 段落（空行）で分割
- 利点: 著者の意図を尊重
- 欠点: 段落が無いとスッキリ分けられない

### 案 P1-d: AI に「意味単位」で分割させる
- 利点: 質が高い
- 欠点: 速度・コスト・privacy

### 案 P1-e: チャネル別に方針を変える
- 音声: 句点分割（A1-c）
- PDF: 段落分割（A3-c）
- AI 履歴: turn 分割（A8）
- メール: 1メール=1ノード（A5）

**推し度**: P1-e（チャネル別最適化）。一律ルールは多分破綻する。

---

## P2. 自動タグ付け / カテゴリ分類

### 案 P2-a: 一切タグ付けしない（生のまま）
- 利点: 速い、間違わない
- 欠点: 後で探しにくい

### 案 P2-b: チャネル名をタグに
- `tag: voice` `tag: pdf` 等
- 利点: 出自が分かる
- 欠点: 内容ベース検索には弱い

### 案 P2-c: 既存タグから類推
- 既存マップのタグ語彙を学習し近いものを suggest
- 利点: 一貫性
- 欠点: 学習データ不足だと精度低い

### 案 P2-d: 固定カテゴリ（idea/task/reference/observation）
- 利点: triage が楽
- 欠点: 当てはまらないものが出る

### 案 P2-e: AI フリータグ
- LLM に「このノードに合うタグ3つ」を生成させる
- 利点: 柔軟
- 欠点: タグ語彙発散

### 案 P2-f: ハイブリッド: 必ずチャネルタグ + opt-in で AI タグ
- **推し**

---

## P3. 要約・タイトル自動生成

scratch ノードのタイトル（text フィールド）をどう決めるか。

| 案 | タイトル生成 | 速度 | コスト |
|---|---|---|---|
| P3-a 先頭N文字 | 先頭40字 | 即時 | 0 |
| P3-b 先頭1文 | 句点まで | 即時 | 0 |
| P3-c AI 1行要約 | LLM | 数秒 | API |
| P3-d ユーザー入力 | capture 時に手入力 | 摩擦増 | 0 |
| P3-e タイトル無し（空） | 後で付ける | 即時 | 0 |
| P3-f source ベース | URL/ファイル名 | 即時 | 0 |

**推し度**: P3-a を default、P3-c を opt-in。Drafts と同じ思想。

---

## P4. 重複検出

新規 capture が既存ノードと類似していたら？

### 案 P4-a: 重複検出しない
- 全部新規ノード作成

### 案 P4-b: 完全一致のみ検出
- 同じ text の既存ノードがあれば「重複の可能性」フラグ

### 案 P4-c: fuzzy 一致
- Levenshtein 距離 / cosine 類似で suggest

### 案 P4-d: AI 意味類似
- embedding で「意味的に近い既存ノード」を提示

### 案 P4-e: 即マージ提案
- capture 直後に「既存 X と統合しますか」UI

### 案 P4-f: triage 段階で検出
- capture 時は何もせず、triage で「これ似てない？」

**推し度**: P4-f が低摩擦。即時検出は焦りを生む。  
00_topic_pool の D1（重複検出）と直接接続。

---

## P5. 関連ノード自動 link 提案

### 案 P5-a: link 提案しない
### 案 P5-b: タグ一致で suggest
### 案 P5-c: AI semantic で suggest
### 案 P5-d: 親ノード自動推定（capture 時）
### 案 P5-e: triage 時に「この scratch、どのノード配下？」を AI が提案

**推し度**: P5-a を default、P5-e を opt-in。  
00_topic_pool C4（自動 link 提案）と接続。

---

## P6. 引用・出典のメタデータ抽出

PDF や Web ページ取り込み時。

### 抽出対象
- title
- author(s)
- date / year
- DOI / arXiv ID
- URL
- bibtex
- citation key

### 案 P6-a: 抽出しない（本文のみ）
### 案 P6-b: 自動抽出（GROBID / arXiv API / DOI lookup）
### 案 P6-c: 別ノード（引用情報専用 child node）
### 案 P6-d: メタフィールド（attribute として埋め込み）
### 案 P6-e: 別 store（Zotero ライブラリと同期、H4）

**推し度**: P6-d を基本、Zotero ユーザーは P6-e。

---

## P7. 言語検出 / 翻訳

### 案 P7-a: 何もしない
### 案 P7-b: 言語検出のみ（lang 属性）
### 案 P7-c: 翻訳ペア保存（原文 + 訳）
### 案 P7-d: マップの言語に合わせて翻訳統一
### 案 P7-e: opt-in 翻訳

**推し度**: P7-b + P7-e。原文性を尊重しつつ後で訳せる。

---

## P8. OCR / Whisper 等の前処理

A1（音声）/ A4（画像）/ A3（PDF）のための前処理レイヤー。

### 配置案
| 案 | 処理場所 | 利点 | 欠点 |
|---|---|---|---|
| P8-a クライアント完結 | ブラウザ WASM Whisper | privacy ◎ | 重い |
| P8-b ローカルサーバ | Node 上で実行 | バランス | セットアップ |
| P8-c クラウド API | OpenAI/Google | 速い・楽 | privacy ✕ |
| P8-d ハイブリッド | 機微: ローカル / 公開: クラウド | 柔軟 | 複雑 |
| P8-e オフライン蓄積 | capture 時は raw、後で batch 処理 | 即時応答 | 待ち時間 |

**推し度**: P8-d（ハイブリッド）。L1/L2 と整合。

---

## P9. AI 要約パスを通すか通さないか

全 capture を AI に通すと贅沢だが privacy/コスト/速度が痛む。

### 案 P9-a: 通さない（純粋 capture）
### 案 P9-b: 全部通す
### 案 P9-c: チャネル別（PDF だけ通す等）
### 案 P9-d: ユーザー指定 tag があれば通す（`#ai`）
### 案 P9-e: triage 時に「これ AI に投げる？」ボタン
### 案 P9-f: 夜間 batch（C10 自走 AI）

**推し度**: P9-c + P9-e。即時 AI は要らない。

---

## P10. triage キューへの投入規則

scratch に入れた後、triage 待ち行列にどう並ぶか。

### 案 P10-a: 単一キュー（FIFO）
- 全 capture が時系列で1列

### 案 P10-b: チャネル別キュー
- voice キュー、pdf キュー、mail キュー...

### 案 P10-c: 優先度キュー
- AI が「これ重要そう」を上に

### 案 P10-d: タグ別キュー
- `#urgent` は別キュー

### 案 P10-e: キューを作らず scratch に入れっぱなし
- 既存 m3e-scratch の挙動延長

### 案 P10-f: 「未 triage」フラグだけ
- 表示時にフィルタ

**推し度**: P10-f が既存 scratch と整合。

---

## P11. 失敗時の挙動

### 案 P11-a: 沈黙失敗 + ログ
### 案 P11-b: 即通知 + 再試行ボタン
### 案 P11-c: ローカル退避 + バックグラウンド再送
### 案 P11-d: dead letter ノード（失敗データ専用 scratch）
### 案 P11-e: ユーザーに「保存できなかった、コピーしたから貼り直して」

**推し度**: P11-c + P11-d の組み合わせ。データロス絶対回避（C7 と整合）。

---

## P12. 取り込み量レート制限

爆撃的取り込みからマップを守る。

### 案 P12-a: 制限なし
### 案 P12-b: 1分N件まで
### 案 P12-c: 1日N件まで（爆発検出）
### 案 P12-d: 自動グルーピング（短時間連続を1ノードに）
### 案 P12-e: バーストは別 scratch に「群」として保存

**推し度**: P12-d（自動グルーピング）。音声分割と相性良い。

## パイプライン構成パターン

複数の P を組み合わせた典型構成。

### Pattern PP1: ミニマル（即 scratch、加工ゼロ）
- P1-a + P2-b + P3-a + P4-f + P5-a + P9-a + P11-c
- 速い、privacy ◎、後でユーザーが全部やる
- → m3e-scratch skill の延長

### Pattern PP2: AI 全乗せ
- P1-d + P2-e + P3-c + P4-d + P5-c + P9-b
- 楽、ただし AI 任せ、コスト高、privacy リスク

### Pattern PP3: 研究者向けバランス
- P1-e + P2-f + P3-a + P4-f + P5-e + P6-d + P9-c
- チャネル別最適、AI は opt-in

### Pattern PP4: 夜間自走
- 即時は P1-a + P3-a + 投入のみ
- 夜間 batch で P2-e + P3-c + P4-d + P5-c
- C10 自走 AI と連動

### Pattern PP5: privacy 最優先
- P8-a + P9-a + P11-c
- 全部ローカル、AI 一切無し

---

## パイプライン横断比較

| 構成 | 速度 | コスト | 品質 | privacy | 推し対象 |
|---|---|---|---|---|---|
| PP1 ミニマル | ◎ | 0 | 低 | ◎ | MVP |
| PP2 AI 全部 | ✕ | 高 | 高 | ✕ | 一部研究 |
| PP3 バランス | ○ | 中 | 中 | ○ | 推奨 |
| PP4 夜間 | ◎ | 中 | 高（後で） | ○ | 自走派 |
| PP5 privacy | ○ | 低 | 中 | ◎ | 機微情報 |

## P13. 既存 ai_subagent.ts との関係

M3E には既に AI subagent 基盤がある。capture pipeline はそれを再利用するか別系統か。

### 案 P13-a: 完全別経路
- capture 専用パイプライン
- 既存 subagent と分離

### 案 P13-b: subagent を呼ぶ
- 既存基盤再利用
- subagent の queue に capture task を入れる

### 案 P13-c: capture 専用 subagent を新設
- 既存基盤の上に capture-agent

**推し度**: P13-c（capture-agent 新設）。役割明確化。

---

## P14. 取り込み履歴の保持

「何月何日に何が capture されたか」を遡れるか。

### 案 P14-a: 履歴保持しない
### 案 P14-b: 全 capture log を別ファイル
### 案 P14-c: 各 scratch ノードに timestamp + source
### 案 P14-d: capture history ビュー（時系列）
### 案 P14-e: 月次バックアップに含める

**推し度**: P14-c + P14-d。後で「あれいつ拾ったんだっけ」問題を解く。
