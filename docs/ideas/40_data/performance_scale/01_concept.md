# 01. Concept — なぜ「性能/スケール」を今ブレストするか

## 1. 背景: 規模の壁はいつ来るか

M3E は「個人の研究思考世界モデル」を目指す。
すなわちユーザー akaghef の **生涯マップ** が想定範囲。

10 年使い続ければ:
- 数万ノード（1 日 10 ノード × 365 × 10 = 36,500）
- 数百〜数千の workspace / vault
- 数千スナップショット（backup.ts）

現状: 1 マップ数百ノード規模、scope param で部分表示で凌いでいる
（feedback_dev_map_size）。
半年で「世界モデル→射影」実用化を狙う以上
（project_projection_vision）、年内のどこかで巨大マップは現実になる。

「いずれ」ではなく **時期が見えている問題**。

## 2. 痛みのシーン（具体ユースケース）

抽象論を避けて、研究者 akaghef の実シーンで考える。

### Cn1.1 朝、PC 起動 → マップを開く

- 開く時間が 1 秒 → 5 秒 → 30 秒に劣化していく
- 「重いから別のマップを使う」逃避が始まると終わり

### Cn1.2 検索

- 「あのアイデア、どこに書いたっけ」
- 現状マップ内 Ctrl+F に相当するもの（ない）
- workspace を跨いだら絶望
- backup スナップショット内も検索したい

### Cn1.3 ノード追加時の反応性

- m3e-scratch スキルで scratch に追加 → マップに反映の遅延
- 入力中の体感が落ちると capture rate が落ちる
- 入力チャネルの劣化は致命的

### Cn1.4 ズームアウトで全体像

- 数千ノードを「俯瞰」したい時、全部描こうとして固まる
- 縮小サムネが出ない
- 「群」が見えるべきなのに「1個1個の点」しか見えない

### Cn1.5 workspace 切替

- 別の研究テーマ workspace に飛びたい
- 名前を全部覚えていない（10 個超えると無理）
- ブラウザのタブと同じ問題

### Cn1.6 オフライン執筆

- 移動中・出張中に編集 → 帰宅で同期で衝突
- conflict_backup ファイルが溜まる
- ローカルに full copy が無いと出先で何もできない

### Cn1.7 履歴の発掘

- 「3 ヶ月前にどう考えていたか知りたい」
- backup.ts のスナップショットを開いて、また検索...
- 横断インデックスがあれば 1 クエリで終わる

### Cn1.8 公開前のチェック

- マップ全体を「機微情報あるノード」で grep したい
- L5 検閲と同じ問題
- インデックスが無いと毎回全 walk

## 3. 現状の延命策と限界

### 既存の凌ぎ方

- **scope param** — 部分表示で UI 負荷を回避（feedback_dev_map_size）
- **DEV map ~50-300 ノード** — マップ自体を分割
- **vault watch** — ファイル変更検知で差分更新

これらは今のところ機能している。
ただし以下に弱い:

| 凌ぎ方 | 効く場面 | 効かない場面 |
|---|---|---|
| scope 切替 | 1 マップ内の局所表示 | マップが本質的に大きい場面（俯瞰） |
| マップ分割 | 別テーマで完全分離 | 横断検索・横断ナビ |
| vault watch | ファイル変化反映 | 初回 full load・検索インデックス |

つまり scope は **巨大マップを「見ない」ことで延命** している。
本質的に巨大マップを **「見る・触る」** ことを諦めている。

それを「見れる・触れる・検索できる」にするのが本ブレストの動機。

## 4. 「巨大マップは健全か」の哲学的論点

### Cn4.1 そもそも分割すべき派

- 1 マップ = 1 主題、巨大化したら subgraph として split
- Obsidian 的な思想: ノートは小さく、リンクで繋ぐ
- M3E でも「ファイル＝1マップ」の拡張で似た構造を作れる

### Cn4.2 1 マップに統合すべき派

- 横断検索・俯瞰・関連発見は 1 マップ内が圧倒的に楽
- 分割は人間の整理コストを増やす
- 「全部入れる」が capture rate を最大化する

### Cn4.3 ハイブリッド派

- workspace = 主題、その中に 1 巨大マップ
- workspace 横断は弱リンク + 検索で繋ぐ
- 現状の M3E の方向性に近い

→ 本ブレストは **Cn4.3 を仮置き** で進めるが、
   M2/M3/M4 が結局 Cn4.1 寄りに引き戻す可能性も残す。

## 5. 性能投資の機会費用

半年の射程で 「性能」 vs 「機能」 を比較。

| 投資先 | 半年での効果 | リスク |
|---|---|---|
| 性能（本ブレスト） | 巨大マップに耐える、長期運用前提に | 今の数百ノードでは効果見えにくい |
| B1 論文ドラフト生成 | project_projection_vision の本丸 | LLM 接続・出力品質に依存 |
| C1 Devil's Advocate | 研究の質直接向上 | UX 設計コスト |
| D1 重複検出 | マップ衛生 | インデックス基盤を本ブレストと共有 |
| F2 スマホ capture | 入力チャネル拡大 | モバイル UI 設計コスト |

**観察**: D1 / F2 は性能基盤と接続している。
本ブレストの基盤投資は他機能の前提でもある。
特に **検索インデックス（Sr1-Sr14）** は B1 / C1 / D1 の全てが間接利用する。

## 6. ベンチマーク基準（Cn6）

何を「性能達成」とみなすか。

### Cn6.1 60fps 信奉

- スクロール・ズームが滑らか
- 入力遅延 < 100ms
- アニメーションが自然

### Cn6.2 30fps 妥協

- 「触れる」レベル、但しスクロールはカクつく
- 入力遅延 100-300ms 許容
- 多くの開発ツールはこのライン

### Cn6.3 「触れる」だけ

- 操作が即座に反応すればよい、見た目の滑らかさは捨てる
- 入力遅延 < 500ms
- VS Code の巨大ファイルもこのライン

### Cn6.4 「死なない」だけ

- 開けば描画はされる、操作は重くてもよい
- フリーズしないことだけを保証

→ 推し（軽い見立て）: **Cn6.2 ~ Cn6.3 のハイブリッド**。
   描画は 30fps、操作（クリック・キー）は < 200ms。
   60fps は必要な場面（ドラッグ中など）だけ目指す。

## 7. M3E に固有の制約・前提

- **1 ユーザー運用** — 多人数同時編集の最適化は不要
- **vault = ローカルファイル** — クラウド前提のサービス設計とは違う
- **policy_privacy** — クラウドに平文置けない、暗号化前提
- **React+TS+SVG** — フレームワーク変更は半年で無理
- **AlgLibMove dogfood** — 認知層が無い、AI による「賢い間引き」は未着手
- **feedback_no_interpretation** — 勝手に「分割すべき」と決めず、両派並列

## 8. このブレストが扱わないこと

- 具体的なライブラリ選定（02 で複数並べるが採否決めない）
- 実装スケジュール
- 性能テストフレームワーク選定
- DB 選定（Sr3-Sr5 で並列に並べるが決めない）
- マルチテナント・SaaS 化（1 ユーザー運用前提）
- GPU 計算（Rd6 WebGL の延長で触れるのみ）
