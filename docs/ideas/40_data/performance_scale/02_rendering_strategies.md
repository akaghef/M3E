# 02. Rendering Strategies — 10k+ ノード描画の選択肢

M1「巨大マップ対応」の **描画方式** を網羅的に並べる。
viewer.ts は約 7800 行で SVG 全描画ベース。
ここを **どの方向に進化させるか** の選択肢を採否なしで列挙。

## 0. 議論の前提

- M3E は **テキスト主体ノード + ノード間リンク** のグラフ
- ノードはクリック・ドラッグ・編集対象
- 既存はすべて DOM/SVG エレメントとして存在
- Markdown レンダリング（markdown_renderer.ts）が各ノード内で走る
- ズーム/パン/フォーカスが頻繁

## 1. Rd1: SVG 全描画（現状）

| 項目 | 内容 |
|---|---|
| 概要 | 全ノードを `<g>` `<rect>` `<text>` として常時 DOM に配置 |
| 良い点 | 実装シンプル、CSS で見た目調整、イベント標準、a11y |
| 悪い点 | 数千ノードで paint/composite が破綻、Markdown 高コスト |
| 限界目安 | 〜800-1500 ノードで体感劣化、〜3000 でフリーズ |
| 推し度 | 維持 + 部分置換が現実解 |

## 2. Rd2: SVG + viewport カリング

| 項目 | 内容 |
|---|---|
| 概要 | 画面に映らないノードは DOM から外す（or `display:none`） |
| 良い点 | 実装影響小、現状コードと共存可能、ROI 最大 |
| 悪い点 | スクロール/ズーム時の出入りで paint コスト |
| 限界目安 | 1 画面の表示数 = 数百なら 10k 全体でも耐える |
| 推し度 | **MVP 第一候補** |
| 接続 | `feedback_dev_map_size` の延長線、scope と直交 |

派生:
- Rd2.1 完全 unmount（DOM から消す）
- Rd2.2 `display:none` 残し（位置情報は保持）
- Rd2.3 buffer 余裕（画面外 +N px まで描く）
- Rd2.4 ズームレベル別カリング閾値

## 3. Rd3: SVG + LOD（Level of Detail）

| 項目 | 内容 |
|---|---|
| 概要 | ズームアウト時にテキスト省略・詳細削減・複数ノードを 1 矩形化 |
| 良い点 | 「俯瞰時に死なない」体験、認知負荷も下がる |
| 悪い点 | 何を省略するか設計が要る、状態遷移バグ温床 |
| 限界目安 | LOD 段階数 × 戦略の質に依存 |
| 推し度 | Rd2 の次に効く |

派生:
- Rd3.1 ズーム < 0.5 で text を消す
- Rd3.2 ズーム < 0.3 で詳細パネルを矩形のみに
- Rd3.3 ズーム < 0.1 で複数ノードを 1 dot にクラスタ化（→ Rd8）
- Rd3.4 アクティブノードのみ常に詳細表示
- Rd3.5 「重要度」属性ベースの LOD（重要ノードは消えない）

## 4. Rd4: Canvas 2D 全描画

| 項目 | 内容 |
|---|---|
| 概要 | `<canvas>` に全ノードを 2D 描画、DOM は持たない |
| 良い点 | 数千〜数万ノード OK、描画速度安定 |
| 悪い点 | クリック判定・ドラッグ・テキスト編集を全部自作 |
| 限界目安 | 〜数万ノードまで実用 |
| 推し度 | テキスト編集との相性悪く、フル移行は大手術 |

派生:
- Rd4.1 全 Canvas（編集も Canvas 上で）
- Rd4.2 Canvas + 編集中ノードのみ DOM オーバーレイ
- Rd4.3 react-konva 等のライブラリ採用

## 5. Rd5: Canvas + SVG ハイブリッド

| 項目 | 内容 |
|---|---|
| 概要 | 大量ノードは Canvas、操作対象（選択中・編集中）は SVG |
| 良い点 | Rd4 の操作問題を回避しつつ大量描画可 |
| 悪い点 | 二重管理、座標系の同期コスト |
| 限界目安 | 〜10k ノード、選択 1-数個ならスムーズ |
| 推し度 | **大規模化の本命候補** |

派生:
- Rd5.1 Canvas は背景描画専用
- Rd5.2 hover でだけ SVG プロモート
- Rd5.3 選択時 SVG プロモート + 編集も SVG

## 6. Rd6: WebGL（PixiJS / regl / sigma.js）

| 項目 | 内容 |
|---|---|
| 概要 | GPU で描画、数万-数十万ノードも可 |
| 良い点 | 圧倒的性能、Force-directed レイアウトとも相性良 |
| 悪い点 | 学習コスト、テキスト描画が苦手、デバッグ困難 |
| 限界目安 | 数十万ノード（理論上） |
| 推し度 | M3E の用途（テキスト中心）には過剰、長期オプション |

派生:
- Rd6.1 sigma.js（グラフ専用）採用
- Rd6.2 PixiJS（汎用 2D）採用
- Rd6.3 自作 WebGL（最小コア）

## 7. Rd7: DOM 仮想化（react-window 風）

| 項目 | 内容 |
|---|---|
| 概要 | リスト的に並ぶ場合のみ有効な仮想化技法 |
| 良い点 | リスト UI（outline / list view）で完璧 |
| 悪い点 | グラフ・自由配置には適用不可 |
| 限界目安 | リスト系ビューには無限 |
| 推し度 | map_views の outline モード等で部分採用可 |

→ Rd2 viewport カリングのリスト版とみなせる。

## 8. Rd8: クラスタリング

| 項目 | 内容 |
|---|---|
| 概要 | ズームアウト時、近接ノード群を 1 cluster ノードに集約 |
| 良い点 | 俯瞰の認知負荷が劇的に下がる、性能と UX 両得 |
| 悪い点 | クラスタ境界の意味付けが難しい、点滅問題 |
| 限界目安 | 数万ノード規模で有効 |
| 推し度 | Rd3 LOD の最深段階として組み合わせ |

派生:
- Rd8.1 空間クラスタ（座標近傍）
- Rd8.2 構造クラスタ（同じ親・同じタグ）
- Rd8.3 意味クラスタ（embedding 近傍、Sr6 接続）
- Rd8.4 ユーザー定義クラスタ（手動グループ化）

## 9. Rd9: 段階ロード（progressive loading）

| 項目 | 内容 |
|---|---|
| 概要 | 起点ノードから N hop だけ表示、奥は折りたたみ |
| 良い点 | 初期表示が常に軽い |
| 悪い点 | 「全体像」が見えない、認知ストレス |
| 限界目安 | UX の許容次第 |
| 推し度 | scope の自動運用版とみなせる |

派生:
- Rd9.1 ルートから 3 hop デフォルト
- Rd9.2 「展開ボタン」で奥を読む
- Rd9.3 検索 hit 時は周辺を自動展開
- Rd9.4 「最近触ったノードの周辺」を優先ロード

## 10. Rd10: WebWorker でレイアウト計算

| 項目 | 内容 |
|---|---|
| 概要 | force-directed や clustering を Worker で実行 |
| 良い点 | UI スレッドが固まらない |
| 悪い点 | Worker ↔ Main の data transfer コスト |
| 限界目安 | 計算量が大きいレイアウトで効く |
| 推し度 | Rd5/Rd6 採用時に必須化 |

## 11. Rd11: OffscreenCanvas

| 項目 | 内容 |
|---|---|
| 概要 | Worker 内で Canvas 描画完結、Main はビットマップ受領のみ |
| 良い点 | 描画も含めて UI スレッドから外せる |
| 悪い点 | ブラウザサポート（Safari 弱い）、デバッグ |
| 限界目安 | Rd4/Rd5 系の延長で大型化に効く |
| 推し度 | Rd6 と組み合わせる長期案 |

## 12. Rd12: 物理エンジン分離（WASM 化）

| 項目 | 内容 |
|---|---|
| 概要 | d3-force / Verlet を WASM 化、または物理エンジン採用 |
| 良い点 | レイアウト時間が桁で短縮 |
| 悪い点 | 必要な人にしか効かない、自動レイアウトを使わない場合は無意味 |
| 限界目安 | 自動配置使用時のみ |
| 推し度 | P1 思考の物理エンジンと連動 |

## 14. 組み合わせの相性

- **Rd2 + Rd3** : viewport カリング + LOD は相補的、最初に並走させる価値高
- **Rd5 + Rd8** : Canvas+SVG ハイブリッドにクラスタを乗せると 10k+ 楽勝
- **Rd5 + Rd10 + Rd11** : Worker でレイアウト計算 + OffscreenCanvas で描画
- **Rd9 + Sr1**（03 で詳述）: 段階ロード前提で初期は検索からエントリー
- **Rd2 + Rd9** : viewport カリングと段階ロードの境界が曖昧、設計時に統合検討

## 15. 採用判断は決めない

このブレストは **「採否を決めない」** が原則。
ただし軽い見立てとして:

- **短期**: Rd2 + Rd3 が ROI 最高、現状コードを壊さず 5-10x の改善
- **中期**: Rd9 段階ロード、Rd8 クラスタリングで 10k 級に踏み込む
- **長期**: Rd5 ハイブリッドが本命、Rd6 WebGL は不採用でも構わない
