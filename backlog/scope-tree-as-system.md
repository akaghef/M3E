# Scope Tree = System Architecture

## Core Insight

M3Eの木は文書整理ツールではなく、**システムそのもの**を木として構築する。
ファイルシステムのように、ルートからユーザー・プロジェクト・共有領域まで
すべてがスコープ木の中に存在する。

## スコープ木の構造イメージ

```
/                              <- ルートスコープ
├── system/
│   ├── config/
│   └── plugins/
├── users/
│   ├── akaghef/
│   │   ├── sandbox/           <- ドキュメント境界
│   │   ├── projects/
│   │   │   ├── M3E-dev/       <- ドキュメント境界
│   │   │   └── research/      <- ドキュメント境界
│   │   └── scratch/
│   └── collaborator-x/
└── shared/
    └── team-alpha/            <- ドキュメント境界
```

## 3層の整理

| 概念 | 役割 | 例え |
|------|------|------|
| **スコープ** | 木全体のアドレス体系（パス） | ファイルシステムのパス |
| **ドキュメント** | スコープ木上の**境界ノード**。データ・権限・同期の単位 | マウントポイント / パーティション |
| **docId** | 境界ノードのID（内部用） | inode |

## ドキュメント境界の責務

ドキュメント = スコープ木の中で明確な境界を持つノード。以下の単位になる:

- **永続化** — SQLiteの1レコード、cloud syncの1単位
- **権限** — 誰が読み書きできるか
- **バージョン管理** — docVersionはこの境界内
- **collabロック** — スコープロックはドキュメント境界内で動く

## URI設計

- URIは**スコープパスのみ**で表現する
- docIdはURIに出さない（内部でスコープパスから解決）
- 「どのドキュメントにいるか」はUIが境界ノードを表示して示す

### URI形式の候補

```
m3e:///users/akaghef/projects/M3E-dev/phase2
```

### 内部解決

- 人間向け: スコープパス（リネームで変わりうる）
- 永続リンク: UUID（内部解決用、URIには出さない）
- どちらを正とするかは未決

## 現状の問題

- リンク（URL）にドキュメントIDが明示されていない
- どのドキュメントを表示中かUIで常時見えない（メタから復元が必要）
- scopeとdocIdが並列パラメータとしてAPIに混在している

## 決定事項

- [ ] スコープ木がシステム全体を表現するアーキテクチャを採用
- [ ] ドキュメントはスコープ木上の境界ノードとして定義
- [ ] URIはスコープベース、docIdは内部実装に降格
- [ ] ドキュメント名はUIで常時表示（パンくず等）

## 未決事項

- パスURI vs UUID URI の正本問題（リネーム耐性 vs 可読性）
- 境界ノードのマーキング方式（ノードプロパティ？ type フィールド？）
- マルチドキュメント間のリンク表現
