# 02. K1 ストリーク と K2 バッジの選択肢

「日課カウンタ」と「達成バッジ」を、研究者用途で副作用を抑える方向に複数案で並べる。

## K1: ストリーク／日課カウンタ

### K1.1 何を「触った」とカウントするか

| 案 | 定義 | 良い点 | 悪い点 | 研究者向け推し度 |
|----|------|------|------|------------|
| K1.1a | アプリを起動した | 簡単 | 開いただけで稼げる、形骸化 | 低 |
| K1.1b | 1ノード以上 edit した | 妥当 | 一文字直しで稼げる | 中 |
| K1.1c | N文字以上 input した | 質を担保 | 過剰書き込み誘発 | 中 |
| K1.1d | リンク or 構造変更を1回 | 思考的行為のみカウント | UI 操作とテキスト編集の区別必要 | 高 |
| K1.1e | done に1ノード送った | 完了駆動 | 何でも done 化が起きる | 中 |
| K1.1f | scratch から1個でも整理した | triage 推奨と整合 | 実装複雑 | 高 |

→ 推し: K1.1d または K1.1f（質的行為に紐づく）。
→ 副作用最大: K1.1a（形骸化）。

### K1.2 表示形式

| 案 | 表示 | 圧 | 既存ツール例 |
|----|------|---|-----------|
| K1.2a | 「7日連続」数字のみ | 弱 | GitHub |
| K1.2b | 炎マーク + 日数 | 中 | Duolingo |
| K1.2c | ヒートマップ（カレンダー） | 弱 | GitHub, Anki |
| K1.2d | 進捗バー + Lv | 強 | RPG系 |
| K1.2e | 表示なし、内部記録のみで後で振り返り | 最小 | （独自） |

→ 研究者向け推し: K1.2c（ヒートマップ）または K1.2e（不可視）。
→ K1.2b/d は萎える可能性が高い。

### K1.3 ストリーク切れ時の挙動（最重要副作用ポイント）

| 案 | 切れた時 | 心理的副作用 |
|----|---------|----------|
| K1.3a | 0 にリセット、警告 | 強い喪失感、強迫を生む |
| K1.3b | 0 にリセット、無言 | やや強い喪失感 |
| K1.3c | -1 されるだけ、最大値は記録 | 緩衝材になる |
| K1.3d | 「凍結チケット」を週1付与（Duolingo 風） | 安全弁、ただし複雑 |
| K1.3e | そもそも切れない（累計のみ） | 喪失感ゼロ、緊張感もゼロ |
| K1.3f | 自動休息日（土日 or ユーザー指定曜日はカウント外） | 健全、推し |

→ 推し: K1.3f（休息日設定）。研究者は週末を休む権利があるべき。
→ 警告枠: K1.3a を入れたら絶対に研究者は萎える。

### K1.4 通知／プッシュの強度

| 案 | 強度 | 研究時の害 |
|----|------|---------|
| K1.4a | 全部 OFF（自分で見に行く） | ゼロ |
| K1.4b | アプリ内バナーのみ | 小 |
| K1.4c | OS 通知（夜21時等） | 中 |
| K1.4d | メール / Slack | 中 |
| K1.4e | しつこい再通知 | 大、Duolingo 化 |

→ 推し: K1.4a または K1.4b。M3E はローカル研究ツール、プッシュ最小化が筋。

### K1.5 ストリークの範囲（個人 vs マップ）

| 案 | 範囲 | 想定状況 |
|----|------|--------|
| K1.5a | 全 workspace 横断、ユーザー単位 | 個人ダッシュボード向き |
| K1.5b | workspace（マップ）単位 | 複数研究テーマでバランスを見たい時 |
| K1.5c | 特定タグ／subtree 単位 | 「論文Aだけのストリーク」 |
| K1.5d | カテゴリ別（書く／整理する／レビューする等） | 行動分類で見たい時 |

→ 全部実装するのは過剰、K1.5a を基本、K1.5b/c はオプションが落とし所か。

### K1.6 「健全な逸脱」を許す設計

研究者の生活には学会出張、原稿締切徹夜、家族の事情、病気がある。
**ストリーク機能の本質的問い**: ライフイベントで切れる罪悪感をどう消すか。

| 案 | 機構 | 解説 |
|----|------|------|
| K1.6a | 「休暇宣言」ボタン（期間指定） | 該当期間はストリークに含めない |
| K1.6b | 自動検出（連続不在 N 日 → 休暇扱い） | UI なし、心理負荷ゼロ |
| K1.6c | 学会／予定連動（Calendar 統合 H2） | 自動で「会議中」を尊重 |
| K1.6d | そもそもストリーク機能を入れない | K0.1 不要派の解 |

## K2: 達成バッジ

### K2.1 バッジの粒度

| 案 | バッジ単位 | 数 | 研究者向け |
|----|---------|----|-----------|
| K2.1a | ノード数（10/100/1000）| 数個 | 形だけ、薄い |
| K2.1b | リンク数（10/100/1000）| 数個 | 構造化を促す、◎ |
| K2.1c | カテゴリ別到達（Insight/Question/Done 各 N 個）| 中 | 行動分類が育つ、◎ |
| K2.1d | 期間達成（30日連続、半年継続）| 少 | K1 と統合 |
| K2.1e | 質的バッジ（subtree が深さ7に到達等）| 中 | 構造観察で面白い |
| K2.1f | コラボ系（誰かにマップ共有した等）| 少 | E 系（協調）と接続 |

### K2.2 バッジの「見せ方」

| 案 | 見せ方 | 真面目さとの両立 |
|----|------|--------------|
| K2.2a | 達成時に派手なポップアップ + 効果音 | × 萎える |
| K2.2b | 達成時に小さなトースト（数秒で消える） | ○ 邪魔少 |
| K2.2c | 達成時はサイレント、後で「アチーブメントビュー」に並ぶ | ◎ 研究者向け |
| K2.2d | 達成時に該当ノードに小さなアイコンが付く | ○ 文脈に紐付く |
| K2.2e | バッジを集める専用ページ／プロフィール | コレクター心くすぐる、ただし本業逸脱リスク |

→ 推し: K2.2c + K2.2d 併用。

### K2.3 「外発的動機の毒」をどう抑えるか

研究の質はバッジ目当てで上げると **必ず** 下がる（既存研究: Deci & Ryan 自己決定理論）。
副作用を抑える設計案：

| 案 | 機構 |
|----|------|
| K2.3a | バッジを「行動の結果」ではなく「振り返り時の発見」にする | 達成済みでも本人未通知、レビュー時に「実はこれ達成してました」と知る |
| K2.3b | バッジを「数」ではなく「行動の多様性」にひもづける | 「3種類の異なる subtree を更新した」等 |
| K2.3c | 「失敗バッジ」も並べる | 「このノードに3回戻ってきた」等、価値中立にする |
| K2.3d | バッジに数値スコアを付けない | XP、レベル化を排除 |

### K2.4 達成イベントの種類（何を評価するか）

| ID | 評価対象 | M3E 既存との接続 |
|----|--------|----------------|
| K2.4a | 量的成長（ノード数・リンク数） | maintenance_hygiene/D5 |
| K2.4b | 構造的進化（深いツリー、密なクラスタ） | map_views, P 系 |
| K2.4c | 整理活動（archive、merge、cleanup mode） | D 系全般 |
| K2.4d | 思考活動（Insight タグ、Question 解決） | personal_productivity/G 系 |
| K2.4e | 継続（K1 と統合） | K1 |
| K2.4f | 自己振り返り（週次レビュー実施回数） | personal_productivity/G2 |

→ K2.4c（整理活動）を評価する設計が **M3E 独自の差別化** になりうる。
   既存ゲーム化ツールは「集める」しか評価しない。**捨てる／整理する** を称賛するのは新規。

### K2.5 バッジ専用ビュー

| 案 | 形式 |
|----|------|
| K2.5a | グリッド表示（一般ゲーム風） |
| K2.5b | タイムライン表示（達成順、振り返り向き） |
| K2.5c | マップに重畳（達成バッジが該当ノードに小さく表示） |
| K2.5d | 隠し（明示的に開かないと見えない） |

→ 推し: K2.5b または K2.5d。

## K1 × K2 比較表

| 観点 | K1 ストリーク | K2 バッジ |
|------|------------|---------|
| 強迫性 | 高い（毎日切れる） | 低い（達成は一度きり） |
| 質への影響 | 「触っただけ」になりがち | 量だけ追うと薄いノードを増やす |
| 研究者好感度 | 低〜中 | 中 |
| OFF にしやすさ | OFF 必須 | OFF しなくても許容範囲 |
| 既存ツール例 | Duolingo, Anki | GitHub, Steam, Habitica |
| MVP 難易度 | 中（休息日／凍結が必要） | 低（達成判定 + 一覧 UI） |
| M3E への適合度 | △（要慎重設計） | ○（K2.4c 整理活動評価で独自性） |

## 横断的気づき（K1/K2）

- **K1 と K2 は補完関係**: K1 は「今日触った？」、K2 は「累計で何を達成した？」
- **両方入れるなら、K2 を主・K1 を従** にすべき。
  K1 主だと Duolingo 化して研究者から逃げられる
- **「整理を評価する」軸（K2.4c）は M3E 独自の意義**。
  集める方向だけでなく、捨てる方向も称賛する設計
- **休息日設計（K1.3f, K1.6a-c）は最初から組み込む** こと。
  後付けは不可能（罪悪感は最初の数週間で植え付けられてしまう）
