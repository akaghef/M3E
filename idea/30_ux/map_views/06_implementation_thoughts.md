# 06. ビュー実装・横断設計の論点

100以上のビュー候補を「実際にどう M3E に組み込むか」の横断的考察。

## ビューの共通アーキテクチャ案

ビューを **「マップ → 配置への射影」** と捉えると、共通骨格が見える:

```
View = {
  name: string
  description: string
  required_attributes: string[]   // 必要な属性（無いノードは「未配置」へ）
  layout: (nodes: Node[]) => { id: string, x: number, y: number, ... }[]
  decoration: {                    // 軸線、象限名、ラベル
    axes?: AxisSpec[]
    regions?: RegionSpec[]
    background?: "grid" | "heatmap" | "none"
  }
  interactions: {                  // ノードドラッグ時の挙動
    onMoveNode?: (id, newPos) => AttributeUpdate[]
  }
}
```

すべてのビューがこの形に収まれば、新ビュー追加 = `View` 1個書くだけ。
（slideshow / keyboard_modes と同様、抽象は実装してから抽出 — Rule of Three）

## ビュー切替の論点

### Sw1. 切替方法

- メニュー選択
- ホットキー（`v` プレフィックス + ビュー番号）
- URL params (`?view=eisenhower`)
- 右上のドロップダウン
- スワイプ／ピンチ的ジェスチャ
- 「ビューパレット」（Cmd+P 風）で fuzzy 検索

### Sw2. 切替時の演出

- 瞬時切替（速い、混乱の可能性）
- フェード（自然）
- ノードがアニメーションで移動（強烈、学習効果あり、計算重い）
- ビュー間で位置補間（同ノードが新位置にスーッと飛ぶ）

### Sw3. 元のビューへの戻り

- 「前のビュー」ボタン
- ビュー履歴スタック
- どこから来たかを覚えておく
- F5 で初期ビュー

### Sw4. 同時表示

- 単一ビューのみ
- 2分割（左右）
- 複数タブ
- ピクチャインピクチャ（小窓）

## データソース論点

### Ds1. ノード属性の正規化

- 各ノードに priority/urgency/effort/impact 等を持たせる必要
- 無い場合のデフォルト: undefined / 0 / 中央
- ビュー間で属性名が衝突しないよう命名規約必要

### Ds2. 計算属性 vs 直接属性

- 直接: ユーザが明示的に値を入れた
- 計算: created_at から age を算出、子ノード数から weight 算出
- ビュー側は両方を区別なく使えるべき

### Ds3. 属性の型

- 数値（整数/小数/範囲）
- カテゴリ（enum）
- ブール
- 日付
- テキスト（NLP 派生）
- ベクトル（埋め込み）

### Ds4. AI 派生属性

- LLM がノード内容から属性を推定（urgency=8、cynefin=Complex 等）
- 自動 vs 提案表示 vs オフ
- コストと精度のトレードオフ

## 編集インタラクション

### Ed1. ビュー上での属性更新

- ドラッグで軸位置 = 属性値が変化（直感的）
- ドラッグで象限移動 = 該当属性のカテゴリ変化
- ドラッグ無効、ビューは閲覧専用

### Ed2. ビュー上でのノード作成

- 空白クリックで新ノード（その位置に対応する属性値で）
- ビューでは作成不可、元ビューで作って戻る

### Ed3. ビュー上でのノード削除・編集

- 通常マップと同じキーで触れる
- ビュー固有の操作セット（M01 Review 等のキューモードに引き継ぐ）

## パフォーマンス論点

### Pf1. ノード数の上限

- 100 ノード未満: 全て描画OK
- 〜1000: 仮想スクロール / クラスタリング
- 1000+: サンプリング表示 / フィルタ強制

### Pf2. レイアウト計算のタイミング

- ビュー切替時に1度
- ノード変更ごとに再計算
- ユーザ要求時のみ
- バックグラウンドでプリ計算

### Pf3. キャッシュ

- ビュー毎の座標をキャッシュ
- 属性変更時に invalidate
- 頻繁切替に強い

## ビュー組み合わせ・ハイブリッド

### Hy1. 階層的ビュー

- 外側ビュー = Eisenhower マトリクス
- 各象限内 = 通常 mind map
- 用途: 全体俯瞰しつつ詳細閲覧

### Hy2. レンズ機構

- ベースは tree ビュー
- 「重要度レンズ」を被せると重要度の高いノードがピカる
- 「期限レンズ」を被せると期限近いノードがピカる
- 複数レンズ重ね掛け可能

### Hy3. オーバーレイ装飾

- ビューはマップそのまま、軸線を **オーバーレイ** として被せる
- 「ノードを動かさず、判定基準だけ可視化」
- 用途: 既存マップのノード位置を尊重しつつ Eisenhower 風判定

### Hy4. フィルタ + ビュー

- ビュー前にフィルタ（done を除く / 自分担当のみ）を適用
- ビュー後の結果が見やすくなる
- 用途: 大マップで意味あるビューを得る

### Hy5. 比較ビュー

- 2つのフィルタ結果を同じビューで色分け表示
- 「先月 vs 今月」「自分 vs 他者」
- 用途: 変化や差異の発見

## ビュー定義の保存

### Sv1. ハードコードのみ

- 既定ビュー10個程度をコードで提供、それ以外は無し
- 軽い、すぐ作れる

### Sv2. ユーザ定義ビュー

- マップ毎に「このビューを記憶」
- 軸選択・象限ラベルをカスタマイズ
- 用途: 個別研究プロジェクトに最適化

### Sv3. ビューライブラリ

- コミュニティでビュー定義を共有
- json / プラグイン形式
- 重い設計、後段

### Sv4. AI 生成ビュー

- 「今このマップに最適なビュー」を AI が提案
- 例: 「未着手タスクが多いので Now/Next/Later にしては？」

## 採用判断の優先付け（暫定）

100ビュー全部は無理なので、最初に作るべきは何か。
**判断基準**: M3E の主用途（研究思考支援）× 実装コスト × 他ビューの基盤になりやすさ

| 順位 | ビュー | 理由 |
|-----|-------|------|
| 1 | V20 ユーザ定義の自由2軸 | 1個で多くを吸収、Eisenhower 等の上位互換 |
| 2 | V21 Timeline | createdAt が既にある、最少コスト |
| 3 | V27 Kanban | status 属性が既にある、即実用 |
| 4 | V01 Eisenhower（V20 のプリセット） | 認知度高い、研究タスクにも適用可 |
| 5 | V41 Sunburst（既存ツリーの別表現） | 大マップ俯瞰、計算簡単 |
| 6 | V67 Hypothesis Cycle | 研究固有、project_projection_vision に直結 |

7位以降は需要が出てから。

## 共通の論点

- **Im1. ビューは「マップを変える」か「マップの見せ方だけ変える」か** — 大原則
- **Im2. ビュー数の制御** — 増やしすぎると選べない（Hick's law）、3〜10個を厳選
- **Im3. ビューの命名・カテゴリ** — ユーザが「いま欲しいビュー」を見つけられるか
- **Im4. デフォルトビュー** — 起動時のビューは何か、マップ毎に記憶するか
- **Im5. ビューと slideshow / keyboard_modes との交差** — Read mode + Eisenhower view、等
- **Im6. ビュー切替のロギング** — どのビューが使われているか測って取捨選択

## 次のステップ案

4. Timeline / Kanban を別ビューとして追加
5. ビュー切替の UX を磨く（ホットキー / アニメーション）
6. 必要に応じて Hy3 オーバーレイ機構を追加
