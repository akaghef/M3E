# 03. Cross-Map Search & History Index — 横断検索と履歴インデックス

M2「複数マップ間リンク・統合検索」と M4「検索インデックス」を統合してブレスト。
理由: バックエンド・データモデル・UX が同根であるため。

## 0. 議論の前提

- M3E の vault は **複数 workspace × 複数 map × 複数 backup snapshot** の階層
- 1 workspace 内マップは数十、生涯で数千マップになる想定
- backup.ts により過去バージョンも全て手元にある
- LLM 接続は ai_subagent で部分実装、embedding は未着手
- policy_privacy: クラウド embedding は機微情報リスク

## 1. インデックスの粒度（Sr1 / Sr2）

### Sr1. 全マップを 1 インデックスに

| 項目 | 内容 |
|---|---|
| 概要 | 全 workspace 全 map を単一インデックスに登録 |
| 良い点 | グローバル検索が 1 クエリ、最強の発見性 |
| 悪い点 | プライバシー境界が曖昧、メモリ消費大 |
| 推し度 | ★★★ |

### Sr2. マップごとインデックス + 串刺しクエリ

| 項目 | 内容 |
|---|---|
| 概要 | マップごとにインデックスを持ち、クエリ時に並列 fan-out |
| 良い点 | 分離保てる、マップ単位 evict 可能 |
| 悪い点 | クエリレイテンシ増、ranking が難しい |
| 推し度 | ★★ |

→ 実は **「インデックスファイルは 1 個 / 内部 namespace 分離」** がハイブリッド解。
  これも選択肢として残す（Sr1.1）。

## 2. インデックスバックエンド（Sr3-Sr5）

### Sr3. client-side fuzzy（Fuse.js / MiniSearch）

| 項目 | 内容 |
|---|---|
| 概要 | ブラウザに JS で全インデックスをロード、in-memory 検索 |
| 良い点 | 実装最短、サーバー不要、即動く |
| 悪い点 | 数千行で遅延、起動時メモリ消費、起動時間長 |
| 限界目安 | 〜数千ノード、〜数十マップ |
| 推し度 | ★★★（MVP 第一候補） |

### Sr4. client-side embedded DB（SQLite WASM / DuckDB WASM）

| 項目 | 内容 |
|---|---|
| 概要 | ブラウザ内 DB エンジンに FTS インデックスを置く |
| 良い点 | 数万行 OK、SQL でフィルタ自由、ファイル永続化可 |
| 悪い点 | バンドルサイズ +数 MB、初期化コスト |
| 限界目安 | 〜数十万ノード |
| 推し度 | ★★★（中期本命） |

### Sr5. server-side インデックス（local node プロセス）

| 項目 | 内容 |
|---|---|
| 概要 | rapid_mvp.ts に検索 API を生やす、ファイルシステム索引化 |
| 良い点 | サイズ無制限、grep 系ツール（ripgrep）流用可、起動軽い |
| 悪い点 | サーバー前提、PWA/オフラインと矛盾 |
| 限界目安 | ディスクサイズ依存（事実上無限） |
| 推し度 | ★★（既存基盤と相性は良い） |

派生:
- Sr5.1 ripgrep 直叩き（インデックス無し）
- Sr5.2 SQLite + FTS5 をサーバー側で
- Sr5.3 自前 inverted index（Tantivy 風）
- Sr5.4 既存 vault_watch でインクリメンタル更新

## 3. semantic search（Sr6）

### Sr6. embedding ベース意味検索

| 項目 | 内容 |
|---|---|
| 概要 | 各ノードを vector 化、cosine 類似で検索 |
| 良い点 | 「似た意味」が拾える、自然文クエリ可 |
| 悪い点 | LLM API コスト、policy_privacy（クラウド送信）、index 鮮度管理 |
| 推し度 | ★★（中長期、L5 検閲とセット） |

派生:
- Sr6.1 OpenAI embedding API（クラウド）
- Sr6.2 ローカル embedding（sentence-transformers / Ollama）
- Sr6.3 ハイブリッド（fuzzy + semantic の rank fusion）
- Sr6.4 「機微フラグ無し」ノードのみ embedding

→ 「fuzzy が main, semantic が optional 補助」が現実解。

## 4. 増分インデックス（Sr7）

### Sr7. インクリメンタル更新

| 項目 | 内容 |
|---|---|
| 概要 | vault_watch でファイル変更検知 → 該当ノードのみ再 index |
| 良い点 | 起動時 full scan 回避、リアルタイム性 |
| 悪い点 | 状態管理バグ、index と vault の乖離リスク |

## 11. データモデル統合の論点

### Sr.D1 ノード ID の global uniqueness

- 現状: マップ内 unique、workspace 横断は未保証
- 横断検索には **global ID 生成** が前提条件
- → vault path + nodeId のハッシュで仮想 global ID を作る案も

### Sr.D2 リンクの workspace 横断表現

- 「マップ A のノード X が、別マップ B のノード Y を参照」
- 既存リンクは同マップ内前提
- スキーマ拡張要

### Sr.D3 「リンク」と「検索 hit」の境界

- 明示リンク（手動）vs 暗黙関連（検索/類似度）
- UI で区別すべきか

## 12. 横断的気づき

- **「検索」と「ナビ」は本来分離可能だが、UX 上は連続体** — Sr8.4 専用画面派は
  04 の workspace nav と統合して「検索＝ナビ」にする発想もアリ。
- **Sr10 履歴インデックスは M3E 固有の宝** — 他ツールに無い差別化要因。
  「研究の自分史検索」は科研費申請書の「これまでの研究」セクションでも活きる。
- **fuzzy + semantic は対立しない** — fuzzy で recall、semantic で precision を補完。
  Sr6.3 rank fusion が最終形。
- **policy_privacy が semantic の足を引っ張る** — ローカル embedding 環境
  （Ollama 等）整備が前提条件になる。
- **検索バックエンドは D1 重複検出と共有可能** — D1 は「類似ノード列挙」
  なので semantic search のサブセット。基盤統合の経済性高い。
