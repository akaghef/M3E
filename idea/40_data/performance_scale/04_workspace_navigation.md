# 04. Workspace Navigation — 横断ナビゲーション UX

M3「workspace 切替 UX」の選択肢を網羅。
複数マップ・複数 workspace を **どう人間が往き来するか** に集中。
検索バックエンド（03）とは独立に、UX 層だけ並べる。

## 0. 議論の前提

- 既存 URL params: `ws / map / scope` の 3 軸統合済（コミット d0b5173）
- Standalone Home 画面あり（home.ts）
- workspace = vault dir、map = .json/.md ファイル
- 1 ユーザーの workspace は数十、生涯で数百規模を想定
- ブラウザの「タブ」「履歴」「ブックマーク」が比較対象

## 1. Nv1: 最近の workspace リスト（MRU）

| 項目 | 内容 |
|---|---|
| 概要 | 最近使った N 個を Home or 切替 UI に表示 |
| 良い点 | 90% のシーンで最有効、実装も軽い |
| 悪い点 | 「忘れていた古い workspace」を見つけるには弱い |
| 推し度 | ★★★（MVP 必須） |

派生:
- Nv1.1 N=5 / N=10 / N=20 の表示数
- Nv1.2 「今日触った」「今週触った」のグループ分け
- Nv1.3 各エントリにサムネ（Nv4 と統合）
- Nv1.4 切替時刻 / 滞在時間表示

## 2. Nv2: ピン留め workspace

## 11. 「ナビの哲学」並行論点

### Nv.P1 「常駐 UI」 vs 「召喚 UI」

- 常駐: タブバー / サイドペイン / Home へ常時戻る
- 召喚: Cmd-K palette、キーで一瞬出現
- akaghef のキーボード重視運用なら **召喚優位**

### Nv.P2 「明示」 vs 「暗黙」

- 明示: ピン・ブックマーク（ユーザーが選ぶ）
- 暗黙: MRU・hot map・推薦（システムが選ぶ）
- ハイブリッドが現実解

### Nv.P3 「workspace」 vs 「context」

- workspace = ファイル境界（vault ディレクトリ）
- context = 作業文脈（執筆 / レビュー / 振り返り）
- これらは直交概念、UI で同等に扱うか分離するか

## 13. 横断的気づき

- **MVP 三本柱は Nv1 + Nv6 + Nv7** — MRU リスト + Cmd-K palette + URL 統合の
  3 点が揃えば数十 workspace まで快適。
- **Nv4 サムネは「巨大マップ」と相性悪い** — 描画コストが本ブレストの主題と
  矛盾する。サムネ生成側でも Rd2-Rd3 の最適化が必要。
- **Nv5 タブバーは PC 限定** — モバイル（cross_device）では別 UI 必須、
  ここで設計を分岐する必要あり。
- **Nv6 Cmd-K palette は keyboard_modes と接続** — 既存ブレストの
  キーバインド統一指針に従って設計すべき。
- **「ナビ＝検索」統合（Nv6 + Sr8.2）は強力** — 1 つの palette で
  workspace/map/node/action 全部に到達できる体験は他ツールでも称賛される
  デザインパターン（VS Code, Linear, Raycast）。
