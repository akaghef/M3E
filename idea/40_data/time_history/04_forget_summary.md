# 04 — I4 忘却機能 と I5 履歴 AI 要約

時間軸を **「操作する側」** の機能。
I4 は "見せない" を機能化し、I5 は "言葉で振り返る" を機能化する。
両者ともユーザーの **認知負荷** を下げる方向に効く。

---

## I4. 忘却機能（古いノードを意図的にぼかす／隠す）

### Fg1. "忘却" の意味の選択肢

| ID | 解釈 | 動作 | 不可逆性 |
|---|---|---|---|
| Fg1.A | 視覚的フェード（推し） | 透明度を下げる | 0（表示変更のみ） |
| Fg1.B | 距離化 | 物理的に外周に追いやる | 0（位置変更） |
| Fg1.C | 完全非表示 | display: none | 0（フィルタ） |
| Fg1.D | グレースケール化 | 色を抜く | 0 |
| Fg1.E | 縮小 | サイズを小さくする | 0 |
| Fg1.F | "霧" エフェクト | ガウスぼかし | 0 |
| Fg1.G | "深層に沈める" | 別レイヤー（z-index 下）| 0 |
| Fg1.H | アーカイブ移動 | archive 親に移動 | 中（位置が変わる） |

→ 全部 **可逆性ゼロまたは低い** = 安全に試せる。
   推しは Fg1.A + Fg1.E（透明＋縮小の組み合わせ）。

### Fg2. 忘却の発火条件

| ID | 条件 | 例 |
|---|---|---|
| Fg2.a | 経過時間（最終編集から N 日）| 30 日 / 90 日 / 180 日 |
| Fg2.b | アクセス頻度（直近 N 回開かれていない） | 10 回 |
| Fg2.c | ステータス（done / archived）| done から N 日 |
| Fg2.d | リンク密度（孤立しているほど早く忘れる）| 0 リンクで即 |
| Fg2.e | ユーザー手動指定（"忘れる" ボタン）| 任意 |
| Fg2.f | テーマ別（"このプロジェクトは一旦忘れる"）| サブツリー単位 |
| Fg2.g | 集中モード（このセッションだけ忘れる）| 一時的 |

### Fg3. 忘却の強度（段階）

単純 ON/OFF ではなく **グラデーション** が研究者向き:

| 段階 | 透明度 | 説明 |
|---|---|---|
| F0 普通 | 100% | 通常 |
| F1 少し古い | 80% | 30 日触っていない |
| F2 結構古い | 60% | 90 日 |
| F3 ほぼ忘却 | 30% | 半年 |
| F4 霞 | 10% | 1 年（まだ存在感あり） |
| F5 完全に霞 | 5% | hover で復活 |

→ "忘れた感" を **連続的な視覚** で表現することが鍵。
   バイナリな hide だと「消えた」になり、復活発見が困難。

### Fg4. 思い出しトリガ（復活）

「忘れていたが、必要になった」場面の救済:

| ID | トリガ | 動作 |
|---|---|---|
| Fg4.A | hover で一時復活 | マウスを近づけると 100% に |
| Fg4.B | 検索ヒットで復活 | 検索結果に出ると一時的に 100% |
| Fg4.C | リンク先からの参照で復活 | 関連ノードから飛んだら復活 |
| Fg4.D | "思い出す" ボタン | 明示的に 100% に戻す |
| Fg4.E | AI 提案による復活 | "このノード、関連しそうです" |
| Fg4.F | 完全復活（タイマーリセット） | "また使う" 宣言 |

### Fg5. ユーザー設定の粒度

- **Fg5.a** グローバル ON/OFF
- **Fg5.b** タグ別設定（"reference は忘却しない" など）
- **Fg5.c** プロジェクト別設定
- **Fg5.d** ノード単位の "忘却対象外" マーク（pin）

### Fg6. メタ視覚化："忘却地図"

忘却の状況自体を可視化する案:

- マップ全景の "鮮度ヒートマップ"（緑＝活きてる、青＝忘れかけ）
- 忘却率の時系列グラフ
- "今週復活したノード TOP 5"

### Fg7. 哲学的論点

- **Fg7.a** 忘却は **整理** か **演出** か
  → 演出寄り（実体は消さない）が研究文脈では安全
- **Fg7.b** 忘却を **AI に任せる** か **ユーザー裁量** か
  → 自動演出 + ピン留めでカスタマイズが王道
- **Fg7.c** **「忘れる」と「捨てる」の違い** を UI で明示する必要
- **Fg7.d** 共同研究者が見たマップでも忘却は反映する？（共有時の挙動）

### Fg × D3 stale 検出 との関係

D3 stale 検出（maintenance_hygiene/02_detectors.md）は **削除候補を出す**。
I4 忘却は **削除しないで霞ます**。**両者は相補的**:

```
[古いノード]
   ├→ D3 stale 検出 → archive 候補（実体移動）
   └→ I4 忘却     → 視覚的に霞む（実体維持）
```

→ **同じ年齢ノードに対し、D3 はバッジ、I4 は透明度** で別役割を担うとクリーン。

## I5. 履歴 AI 要約（"先週何があった" 自動生成）

### Su1. 要約の対象範囲

| ID | 範囲 | 例 |
|---|---|---|
| Su1.A | 時間範囲 | 直近 1 週間 / 1 ヶ月 |
| Su1.B | サブツリー範囲 | 特定プロジェクト配下 |
| Su1.C | タグ範囲 | "research/X" タグ付き |
| Su1.D | 操作種別 | 追加だけ / 編集だけ |
| Su1.E | エージェント別 | "Claude が触った範囲だけ" |
| Su1.F | ノード単位（1 ノードの履歴要約）| 1 ノードの変遷 |

### Su2. 要約の粒度

| ID | 粒度 | 例 |
|---|---|---|
| Su2.A | 1 行（"今週は X を進めた"）| ダッシュボード用 |
| Su2.B | 3 行 | スライド 1 枚 |
| Su2.C | 1 段落 | 週次レポート |
| Su2.D | 詳細レポート（1 ページ）| 月次総括 |
| Su2.E | 構造化（JSON: 進捗／課題／次手）| 自動連携用 |

### Su3. 要約のフォーマット

| ID | 形式 | 用途 |
|---|---|---|
| Su3.A | 平文要約 | 読む用 |
| Su3.B | 箇条書き | スライド・メール |
| Su3.C | "進捗・課題・次手" 3 区分 | 週次レビューテンプレ |
| Su3.D | "5W1H" | 報告書 |
| Su3.E | "今週のハイライト + 注意点" | カジュアル |
| Su3.F | "感情つき" 振り返り（"今週は試行錯誤多め"）| 個人ジャーナル |
| Su3.G | 査読対応形式（"査読 1 への対応として X を変更"）| 査読返答 |

### Su4. 起動タイミング

- **Su4.a** 手動（"今週要約して" コマンド）
- **Su4.b** 定期（毎週月曜 9:00 自動生成）
- **Su4.c** イベント駆動（査読受領 → 査読対応要約）
- **Su4.d** 退出時（M3E を閉じる前に "今日の要約" を表示）
- **Su4.e** ノード作成時（"このノードまでの経緯を要約" を AI に依頼）

### Su5. 要約結果の保管先

| ID | 保管先 | 良い点 | 悪い点 |
|---|---|---|---|
| Su5.A | マップ内ノード（"week_2026-W14" など）| 検索可・履歴になる | マップ肥大化 |
| Su5.B | 別ストア（要約専用 DB）| マップを汚さない | アクセス導線必要 |
| Su5.C | Markdown ファイル（dev-docs/weekly/）| 既存資産活用 | マップとの連携弱い |
| Su5.D | 揮発（毎回再生成）| 鮮度ある | 過去要約見れない |

→ **Su5.A + Su5.D ハイブリッド**（重要なものだけノード化）が筋が良さそう。

### Su6. 入力データソース

要約 AI が読むもの:

- audit_log.ts のイベントログ
- 各ノードの text 変更差分（diff）
- backup snapshot 間の構造変化
- ユーザーのコメントノード（思考メモ）
- エージェント発言ログ

### Su7. プライバシー（policy_privacy 接続）

要約は AI に送信される → **L5 エージェント送信検閲** と直交:

- **Su7.a** 機微タグ付きノードは要約対象から除外
- **Su7.b** 要約前に自動マスキング
- **Su7.c** ローカル LLM 利用オプション
- **Su7.d** 要約結果も privacy フラグを継承

### Su8. AI への問いかけパターン

研究者が "週報 AI" に投げる典型プロンプト:

- "先週何があった？"
- "今月の主要な進捗 3 つ教えて"
- "この 2 週間で行き詰まりがあるとしたらどこ？"
- "査読対応で何を変えた？"
- "今週は思考のテーマがどう動いた？"
- "このノードの誕生から現在までを教えて"

→ **問いかけ自体をテンプレ化**（コマンドパレット）すると入口が広がる。

### Su9. 要約の検証

AI 要約の **誤り検知**:

- **Su9.a** 元データへの引用リンク必須（クリックで該当ノードへ）
- **Su9.b** "信頼度" 表示（曖昧な部分を AI が自己申告）
- **Su9.c** ユーザーが手動で訂正可能、訂正は次回要約に反映
- **Su9.d** 二重要約（同じ範囲を 2 回要約して食い違いをハイライト）

### Su10. 出力先連携

- **Su10.a** Slack / Discord に自動投稿
- **Su10.b** メール送信（自分宛 / 共同研究者宛）
- **Su10.c** Notion / Obsidian 連携
- **Su10.d** B3 Slideshow の素材として
- **Su10.e** B1 論文ドラフト生成 への入力

## I4 × I5 の組み合わせ

| 組み合わせ | 効果 |
|---|---|
| **Fg4.E + Su 提案** | "忘れていたが今のテーマと関連" を AI が指摘して復活 |
| **Su1.B + Fg2.f** | "プロジェクト要約を生成 → そのプロジェクトを忘却" の引退セット |
| **Su2.A + Fg6** | 1 行要約をヒートマップに重ねる |
| **Fg5.d pin + Su1.E** | "AI 要約からは外す（pin） + ユーザー専用領域" の保護 |

## 横断観察

- **I4 は "視覚の時間操作"、I5 は "テキストの時間操作"。** 両者は別系統だが、
  「過剰な情報量を整理する」目的では同じ動機
- **どちらも "可逆性が高い" = 安全機能**。だから自動 ON で問題が少ない
- **I5 は大規模 LLM 必須**、I4 は計算コスト極小（CSS のみで実現可能）→
  実装難度は I4 << I5
- **I5 を毎週月曜に動かすだけで研究者の Pomodoro 比約 5% 改善** 仮説
  （振り返り時間の自動化）
