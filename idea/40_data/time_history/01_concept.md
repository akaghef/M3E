# 01 — コンセプト：時間軸／履歴とは何を指すか

5 つのサブトピック（I1〜I5）の **共通の問い** と **それぞれの違い** を整理。
ユースケース・既存ツール差分・5 機能の関係図を含む。

## C1. 共通の問い：研究者にとって "時間" は何か

研究の現場で「時間」が問題になる典型場面:

- **C1a.** 「2 週間前、自分はどう考えていたか」を確認したい（査読返却・進捗報告で頻出）
- **C1b.** 「あの時 A 案を選んだが、B 案だったら今どうなっていたか」を試したい
- **C1c.** 「先月から今月で、自分のテーマはどう動いたか」を 1 枚で見せたい（ラボミーティング）
- **C1d.** 「3 ヶ月前のメモを思い出したい／今は不要なので隠したい」
- **C1e.** 「先週何があったか／今月の主要トピック」を AI に纏めて欲しい

これらに対し、I1〜I5 がそれぞれ違う角度で応える。

## C2. 5 サブトピックの差異マップ

| ID | 名前 | 動詞 | 操作対象 | 不可逆性 |
|---|---|---|---|---|
| I1 | タイムトラベル | "戻る" | 全マップ状態 | 0（読み取り専用復元） |
| I2 | What-if 分岐 | "試す" | 仮想ブランチ | 0（サンドボックス） |
| I3 | Diff アニメ | "再生する" | 時間範囲の変化 | 0（演出のみ） |
| I4 | 忘却 | "ぼかす" | ノード可視性 | 低（復活可能） |
| I5 | AI 要約 | "語らせる" | 履歴テキスト | 0（読み取り） |

→ **どれも "破壊的でない" 機能。** これは大事な共通項で、
   後の 04_safety_irreversibility 的な議論はほぼ不要にできる。

## C3. ユースケース具体化（最低 3、できれば 5）

### C3a. I1 タイムトラベル

- **科研費申請書の前回バージョンを参照**: 提出版マップを 2025-10 時点に巻き戻す
- **査読コメント返答時の文脈再現**: 投稿時点のマップ状態で「当時こう考えていた」を確認
- **共同研究者の混乱解消**: 「あの会議の翌日のマップを見せて」
- **ロールバック**: 整理しすぎて壊した構造を 30 分前に戻す
- **思考の自分史**: 1 年前の自分がどんな関心を持っていたかを再現

### C3b. I2 What-if 分岐

- **採択しなかった研究テーマを仮想展開**: 「もし B テーマを進めていたら今どこまで」
- **構造の試作**: 「scratch 配下を全部別マップにしたら、どう見える？」のドライラン
- **論文構成案の比較**: A 構成と B 構成を別マップで作って横並べ
- **査読対応の事前検討**: 「査読者の指摘どおり書き換えたら全体はどう変わるか」を試す
- **削除前の試行**: 「このサブツリー無くして大丈夫？」を仮想的に消して見渡す

### C3c. I3 Diff アニメ

- **進捗報告の説得力アップ**: 「この 1 ヶ月でこれだけ動きました」を 10 秒動画
- **共同研究者へのキャッチアップ**: 「先週から今日までの差分を再生」
- **自己振り返り**: 1 週間の動きを時短で見て「何に時間を使ったか」を体感
- **指導教員への可視化**: 「学生が今週どこを触ったか」を一目で
- **ラボミーティング**: 短い差分動画 → 議論時間を本論に集中

### C3d. I4 忘却機能

- **古い未着手案件のフェード**: 半年触っていない backlog をぼかして視界クリア
- **完了済みの archive 演出**: done 後 30 日でフェード、存在は残る
- **集中モード補助**: 関係ないノードをわざと薄くして当該テーマに集中
- **新規ユーザーへの段階開示**: 古い議論を最初は隠し、必要時に呼び出し
- **ストレス低減**: 「今は見たくない」テーマを一時的にミュート

### C3e. I5 履歴 AI 要約

- **月次／週次レビュー自動化**: 「2026-04 第 2 週の活動」を毎週月曜に生成
- **査読回答の下書き**: 「査読 1 への対応で何を変えたか」を AI が要約
- **共同研究者への週報**: マップ差分から週報メールの草稿を生成
- **ふりかえりノート**: 月末に「今月の発見・行き詰まり・次月の課題」を AI に書かせる
- **進捗報告スライドの素材**: I3 と組み合わせて「動画＋テキスト要約」のセット出力

## C4. 既存ツール／概念との差分

| 既存 | できること | M3E I1〜I5 が足すもの |
|---|---|---|
| Git | 差分・ブランチ | グラフ構造の可視化、研究文脈での演出 |
| Obsidian "File Recovery" | 単一ファイル復元 | マップ全体の時点復元・what-if |
| Notion "Page History" | ページ単位履歴 | 構造（リンク／親子）も含めた履歴 |
| Roam "Daily Notes" | 日次のテキスト履歴 | 構造変化のアニメ表示 |
| Google Docs Version History | 線形履歴 | 分岐 (what-if) サポート |
| Granola/Otter (議事録 AI) | 会話の AI 要約 | マップ差分の要約（I5 がここを狙う） |
| Time Machine (macOS) | スナップショット復元 | グラフ視点での "触れる" 体験 |

→ **M3E ならではの差分**:
  - グラフ構造／リンク／親子関係まで含めて時間遷移する
  - 研究文脈（科研費・査読・進捗報告）に直結
  - 既存 backup.ts/audit_log.ts でデータは半分揃っている

依存関係:

- I5 は I3 の **テキスト版** とも見れる（同じ「時間範囲の変化」を扱う）
- I4 は I3／I5 と独立（"忘却" は時間軸 UI ではなく可視性 UI）

## C6. 何を "履歴" として保持するか

履歴の対象には複数のレイヤがある:

| レイヤ | 例 | 既存資産 |
|---|---|---|
| **L1 マップ全体スナップショット** | DB 全体 | backup.ts |
| **L2 操作ログ** | add/edit/delete/move | audit_log.ts |
| **L3 ノード単位履歴** | text の変更差分 | （未整備） |
| **L4 構造履歴** | 親子・リンク変更 | audit に含まれる |
| **L5 メタ履歴** | エージェント発言・思考メモ | （未整備） |

→ I1 は L1 で最低限実現可能、I3 は L1〜L4、I5 は L1〜L5 が理想。
   **既存資産で 70% カバー** されているのが時間軸テーマの強み。

## C7. このカテゴリ全体のスタンス

- **「過去を触れる」を当たり前にする** → 思考の継続性が増す
- **「未来を試す」もカジュアルに** → 意思決定の質が上がる
- **「見せ方」は研究者の認知に最適化** → タイムスタンプではなく "今週／今月" 感覚
- **AI は語り部として** → 履歴を勝手に編集しない、要約と提示だけ
