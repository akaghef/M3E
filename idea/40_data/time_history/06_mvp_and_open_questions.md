# 06 — MVP / 横断観察 / 組み合わせ / 未決質問

5 機能（I1〜I5）を **どこから掘ると効くか**、**何が組み合わせで強いか**、
**まだ決められない論点は何か** を最終整理。

## Mv1. MVP の最小構成（採否は決めない、複数案）

「半年で実用化」（project_projection_vision）に対し、
**先週の活動を週次レビューする** ループだけ回せれば最大価値、という前提で 3 案:

### Mv1.A 「読み取り MVP」案（最小・1〜2 週）

- I1 タイムトラベル: backup snapshot 一覧から選んで TT3.E 別タブ表示
- I3 Diff アニメ: 直近 24h、フェードのみ、固定速度 10 秒
- I5 AI 要約: 手動コマンド `/summary 1week`、Su2.B 3 行版

→ 既存 backup.ts / audit_log.ts のみで動く。
   仮想ブランチも忘却もなし。最小実装。

### Mv1.B 「振り返り MVP」案（中・3〜5 週）

- Mv1.A に加え、I3 を audit ベースに格上げ（細かいアニメ）
- I5 を Su4.b 定期実行（毎週月曜 9:00 自動）
- I5 結果を Su5.A マップ内ノード化（"week_2026-W14"）
- I4 忘却: Fg1.A 透明度のみ、Fg2.a 経過時間ベース

→ 週次レビューが「儀式化」される。研究者向けに強い。

### Mv1.C 「実験 MVP」案（大・6〜10 週）

- Mv1.B に加え、I2 What-if: WI1.B サンドボックス、Mv1.D 仮想差分メモリ版
- I1 を TT5.c イベントベース（"査読受領後" など）まで拡張
- I5 を WI6.a と統合（仮想空間の AI 要約）

→ 思考実験が日常化。研究の質に効く。

### Mv1 案の比較

| 案 | 工数 | 価値 | リスク |
|---|---|---|---|
| Mv1.A 読み取り | 1〜2 週 | 中 | 低 |
| Mv1.B 振り返り | 3〜5 週 | 高 | 低 |
| Mv1.C 実験 | 6〜10 週 | 極高 | 中（仮想差分の整合性） |

## Mv3. 横断観察（このブレストの主要気づき）

### Mv3.1 既存資産で 70% カバー

backup.ts と audit_log.ts は **時間軸機能の土台として既に十分**。
追加で必要なのは:
- ノード単位 text_diff（Dm7）
- 仮想差分ストア（Dm5）
- 忘却メタ（Dm9.a/b/f）
- AI 要約のキャッシュとプロンプトテンプレ

これだけで I1〜I5 が一通り動かせる。

### Mv3.2 "破壊的" 機能はほぼ無い

5 機能のほぼ全てが **読み取り or 演出**。
唯一 destructive なのは I2 What-if のマージ（WI3.E / WI7）。
→ **段階導入の心理的ハードルが極端に低い** カテゴリ。

### Mv3.3 "週次レビュー" を中心に組み立てると筋がいい

研究者の自然な周期は **1 週間 / 1 ヶ月 / 査読サイクル**。
- I3 アニメ: "先週どう動いた"
- I5 要約: "先週何があった"
- I4 忘却: "古いものは霞ます"
- I1: "先週月曜に戻る"
- I2: "今週の判断を試す"

→ **週次レビュー儀式（G2）と統合する** と全機能が同時に意味を持つ。

### Mv3.4 UI メタファは "別物" として設計するほうがいい

5 機能で共通基盤（履歴ストア）は同じだが、UI は:
- I1: タイムスライダ／カレンダー
- I2: オーバーレイ／別タブ
- I3: 再生コントロール（タイムバー）
- I4: 透明度（演出のみ、UI 露出最小）
- I5: テキスト出力／チャット

→ **共通の "時間 UI" を作ろうとすると失敗** する。
   各機能ごとに最適メタファを選び、データ層だけ共通化する。

### Mv3.5 AI コストの偏り

| 機能 | AI 使用度 |
|---|---|
| I1 | 0（TT5.c イベント抽出時のみ） |
| I2 | 中（WI6 で活躍） |
| I3 | 0（演出のみ） |
| I4 | 低（自動忘却なら ML 不要） |
| I5 | **極大**（要約タスク） |

→ I5 の **LLM コスト試算** が必要。
   ローカル LLM（Ollama 等）でも要約品質が出るかが論点。

### Mv3.6 "保持" "再現" "見せ方" を分離

| 層 | 責務 | 既存／新規 |
|---|---|---|
| 保持層 | 履歴データの永続化 | backup + audit + node_history（新） |
| 再現層 | 任意時点の状態を構築 | history_store（新） |
| 見せ方層 | UI / アニメ / 要約テキスト | 各機能ごと |

→ この 3 層分離が **拡張性の鍵**。

## Mv6. このブレストのキーメッセージ（最終）

1. **既存 backup.ts + audit_log.ts は時間軸機能の土台として 70% 完成。**
   足すべきは ノード単位 text_diff・仮想差分・忘却メタ・AI 要約だけ。

2. **5 機能はほぼ全部 "破壊的でない"。** 段階導入の心理ハードルが極端に低い。
   What-if のマージだけ慎重設計が必要。

3. **週次レビュー（G2）に全機能を集約すると最大効果。**
   「先週何があった（I5）→ アニメで再生（I3）→ 古い backlog 忘却（I4）→
    あの時に戻る（I1）→ 今週の判断を試す（I2）」が一連のフロー。

4. **共通基盤は履歴ストアのみ、UI は各機能で別物として設計せよ。**
   "統一時間 UI" を作ろうとすると失敗する。

5. **I5 + B1 論文ドラフト生成 が projection_vision の本丸組み合わせ。**
   履歴要約から論文・科研費草稿へ繋がる軸が、半年実用化の最短経路。

6. **I4 忘却は "整理" ではなく "演出"。** 実体は消さず、透明度のグラデーションで
   「忘れている感」を出す。可逆性が必須。

7. **I2 What-if は "サンドボックス" メタファ一択。** Git の branch を持ち込むと
   研究者の認知が壊れる。"ドラフト世界" のような可逆メタファが筋。
