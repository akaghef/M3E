# 03. アクション系（Actions）

検出器（02）が見つけた「汚れ候補」に対して **何をするか** の選択肢列挙。
M11 Cleanup mode（既存）で 1 キー操作可能にする想定だが、
ここでは **アクションそのものの設計** を扱う。

---

## act_archive（アーカイブ）

### Ac1. archive の物理形態

| 案 | 形態 | 良い点 | 悪い点 |
|---|---|---|---|
| Ac1-A | **フラグ立て**（`archived: true` 属性追加） | データは残る、可逆 | マップ上では見える/隠す UI が要 |
| Ac1-B | **専用 archive 親ノードへ移動**（`/archive/2026-04/...`） | ツリー上で隠れる | 元の親情報を失う |
| Ac1-C | **別 vault / 別 workspace へ移動** | 完全に視界から消える | 取り戻し操作が重い |
| Ac1-D | **フラグ立て + 折りたたみ自動化**（Ac1-A の UX 拡張） | 可逆性◎、視覚的にも消える | UI 実装必要 |
| Ac1-E | **「凍結」概念の導入**（編集不可になる） | 完全保存 | 概念が増える |

**観察**: Ac1-A or Ac1-D が筋が良さそう。Ac1-B は元の親を `archived_from` に保存すれば救える。

### Ac5. D4 自動アーカイブの発火条件

| 案 | 条件 | リスク |
|---|---|---|
| Ac5-A | **status = done から N 日経過**（topic pool の D4 原型） | done を取り消したくなった時に困る |
| Ac5-B | **status = done かつ N 日触られていない** | より安全 |
| Ac5-C | **status = done かつ親も done** | "完全に終わった" subtree のみ |
| Ac5-D | **手動承認のみ**（自動化しない） | 一番安全だが面倒 |
| Ac5-E | **AI 判断**（"これはもう触らないだろう"） | コスト |

### N の設定
- 30 日 / 60 日 / 90 日 / 6 ヶ月
- ノード種別ごとに変えるか（todo は短く、idea は長く）

## act_merge（マージ）

### Ac2. merge の方式

| 案 | 方式 | 良い点 | 悪い点 |
|---|---|---|---|
| Ac2-A | **片寄せ**（A に B を統合、B 削除） | シンプル | どちらを残すか判断要 |
| Ac2-B | **新ノード生成**（A+B の合成 C を作り、A,B は archive） | 履歴保持 | ノード数増える |
| Ac2-C | **リンク化に逃がす**（merge せず "see also" リンクで繋ぐ） | 不可逆操作を回避 | 重複は残る |
| Ac2-D | **AI 合成**（LLM が両方読んで最適なマージ案を提示） | 賢い | 操作前にレビュー必須 |

### Merge 時の保持データ
- `merged_from: [node_id_b]` を残し、誰と誰を merge したか追跡可能に
- B の attribute / link / 子ノードはどうする？
  - 子: A の子に移す
  - link: A に移す
  - attribute: 衝突したら A 優先 / B 優先 / 確認

### 誤操作対策
- merge は **常に preview**（合成後のノードを表示してから confirm）
- 直近 N 件の merge を **undo 可能**

## act_split（分割）

### Ac3. split の単位

| 案 | 単位 | 良い点 | 悪い点 |
|---|---|---|---|
| Ac3-A | **段落ごと**（空行で区切り） | 機械的、シンプル | 過剰分割 |
| Ac3-B | **見出しごと**（H2/H3 検出） | 構造的 | 見出し無いと使えない |
| Ac3-C | **トピックごと**（LLM 推定） | 意味的 | コスト |
| Ac3-D | **手動**（ユーザーが切り目を指定） | 完全制御 | 手間 |
| Ac3-E | **AI 提案 + 人間承認**（Ac3-C + 確認） | バランス | 操作 2 段階 |

### Split 後の構造
- 元ノード A を残し、子として A1/A2/A3 を生成
- 元ノード A は削除し、A の親に A1/A2/A3 を並列で生成
- どちらも選べるオプション

### Split で気をつけること
- リンクは新ノードに自動振り分けできるか？（テキストに含まれる ID を解析）
- 元の text を **完全に保持**（split 後に "元のテキスト" を `original_text` に保存）

## act_quarantine（隔離）

「削除するか残すかすぐ決められない」ノードを一時的に **隔離** する状態。

### Ac4. quarantine の存在意義

| 案 | 設計 | 良い点 | 悪い点 |
|---|---|---|---|
| Ac4-A | **専用 quarantine 親ノード**（`/quarantine/`） | 視覚的に分離 | 移動が物理的 |
| Ac4-B | **quarantine フラグ**（`quarantined: true`） | 元の場所に残る | 検出器の対象から外す処理要 |
| Ac4-C | **「N 日後に再判定」**（`quarantined_until: 2026-05-01`） | 自動再評価 | 期限管理 |
| Ac4-D | **quarantine 不要**（trash + keep フラグで十分） | コンセプト最小化 | 中間状態を表現できない |

**観察**: Ac4-D が最小主義。Ac4-B + Ac4-C のセットが「保留中」を綺麗に表現。

## act_keep（除外マーク）

検出器が誤検出した、または意図的に残したいノードを **検出対象から外す**。

### バリエーション
- 全検出器から keep（`hygiene_keep: true`）
- 特定検出器のみ（`hygiene_keep: ["det_duplicate"]`）
- 期限付き keep（`keep_until: ...`） — Gl5 と同じ
- keep の理由メモ（`keep_reason: "..."`）

### keep の挑戦
- keep されたノードはダッシュボードでどう見えるか？
  - 完全に非表示 / 別タブで一覧 / 普通に表示するが薄く
- keep の取り消し UX

---

## act_inspect（精査表示）

破壊操作の前に **詳細を見る** だけのアクション。

### 見るべきこと
- ノードの全 text / details / attribute
- リンク先・被リンク
- 履歴（過去のステータス変化、編集タイムスタンプ）
- 検出器がなぜこれを候補にしたか（理由表示）
- merge 候補なら相手側ノードと並列表示

### Inspect から飛べる先
- archive / merge / split / keep / 編集 / 戻る

---

## act_skip / act_defer（後回し）

判断を保留する。

### 違い
- **skip**: 今のセッションでだけスキップ（次回再表示）
- **defer**: N 日後まで非表示（`deferred_until: ...`）

---

## act_restore（復元）

trash や archive からノードを取り戻す。

### Restore の射程
- trash → 元の場所
- archive → 元の場所 / 任意の場所
- merge 取り消し（`merged_from` を逆算して B を復元）— **これは難しい**

### Time-travel との接続
- I1 time travel で「2 日前のスナップショット」を呼び出して該当ノードだけ復元
- これがあれば merge undo は不要かもしれない

## アクション × 安全装置 マトリクス

| アクション | 不可逆性 | 推奨安全装置 |
|---|---|---|
| act_archive | 低（フラグなら可逆） | 軽（即実行 OK） |
| act_merge | **高** | preview + undo 必須 |
| act_split | 中（合成すれば戻せる） | preview 必須 |
| act_quarantine | 低 | 軽 |
| act_keep | ゼロ | 不要 |
| act_inspect | ゼロ | 不要 |
| act_delete | **最高** | 葉ノード絶対禁止 + 確認ダイアログ + trash 経由 |
| act_restore | ゼロ | 不要 |

詳細は [04_safety_irreversibility.md](04_safety_irreversibility.md)。

## Ac6. アクション結果の通知

| 案 | 通知方法 | 良い点 | 悪い点 |
|---|---|---|---|
| Ac6-A | **サイレント** | 邪魔にならない | 何が起きたか不明 |
| Ac6-B | **トースト**（"3 件 archive しました [Undo]"） | 軽量、undo 可能 | 流れて消える |
| Ac6-C | **履歴ノードを生成**（`/system/hygiene_log/2026-04-16` に追記） | 履歴永続 | ノード数増える |
| Ac6-D | **マップ上でアニメ**（archive されたノードがフェードアウト） | 視覚的フィードバック | マップ更新負荷 |

**観察**: Ac6-B + Ac6-C の併用が良さそう。Ac6-D は楽しい（K 系ゲーム化と接続）。
