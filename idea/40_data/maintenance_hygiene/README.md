# メンテナンス・衛生（Hygiene）

マップが大きくなった時の **腐敗防止** を扱うブレスト。
50〜300 ノード規模を維持し、孤立ノード・重複・stale・巨大化などの
「マップが汚れる」現象に対し、**検出系**（気づかせる）と **アクション系**（直す）を
両輪で並べて整理する。Pattern B（候補列挙＋共通基盤）を採用。

`idea/keyboard_modes/` の **M11 Cleanup mode** が「掃除を実行する1キー操作」を扱うのに対し、
こちらは **「何を掃除すべきと判定するか」「どこまで自動でやるか」「不可逆性をどう抑えるか」**
の上位設計を扱う。

## 方針

- 採否判断はしない、複数案を並べる
- **検出系（detector）と アクション系（action）を分離** して書く
  - 同じ「検出器」でも、それに応じたアクションは複数あり得る
  - 同じ「アクション」でも、複数の検出器がトリガになり得る
- **「自動 vs 手動」「警告 vs 強制」「即時 vs バッチ」** の対比軸を毎回明示
- **削除の不可逆性に対する安全装置** を必ず触れる（葉ノード保護・trash 経由・undo 可能性）
- 既存 M11 Cleanup mode との接続点を各章で明示
- M3E 規約遵守: **葉ノードは絶対削除しない**（m3e-scratch スキル規約）

## 全体俯瞰

検出器（**Detector**）→ 候補プール → アクション（**Action**） の流れで考える。

```
[ノード集合]
   ↓ Detector（D1〜D7+α）
[候補プール（"汚れ"フラグ）]
   ↓ ユーザー or AI レビュー
[Action（archive / merge / split / quarantine / keep）]
   ↓ 安全装置（trash / undo / 葉保護 / 確認ダイアログ）
[実行 or 取り消し]
```

## 論点 ID 一覧

### Global（共通設計） — Gl
- **Gl1.** 検出器とアクションを分離するか、結合するか
- **Gl2.** 検出のタイミング（保存時 / 起動時 / バッチ / 手動）
- **Gl3.** 検出結果の表示場所（バッジ / Cleanup queue / ダッシュボード / 通知）
- **Gl4.** ユーザーへの介入度（警告のみ / 提案 / 半自動 / 全自動）
- **Gl5.** 「掃除対象から外す」マーク（keep flag）の扱い
- **Gl6.** 検出ロジックの実装場所（クライアント JS / Node 側 / AI 委託）
- **Gl7.** 命名規約（detector_id / action_id の名前付け）

### Detectors — Dt
- **Dt1.** D1 類似度判定方式（テキスト一致 / embedding / LLM 判定）
- **Dt2.** D1 マージ候補の表示 UX（並列表示 / diff / 投票）
- **Dt3.** D2 孤立の定義（リンク無し / 親無し / scratch 配下）
- **Dt4.** D3 stale の閾値設定（固定 N ヶ月 / ノード種別ごと / ユーザー設定）
- **Dt5.** D5 健全性スコアの内訳設計（重み付け）
- **Dt6.** D6 過深ツリー閾値（7 固定 / 可変 / カテゴリ別）
- **Dt7.** D7 巨大ノードの基準（文字数 / 行数 / トピック数）
- **Dt8.** 検出結果のキャッシュ vs 都度計算

### Actions — Ac
- **Ac1.** archive の物理形態（フラグ / 別ノード親へ移動 / 別 vault）
- **Ac2.** merge の方式（片寄せ / 新ノード生成 / リンク化に逃がす）
- **Ac3.** split の単位（節ごと / 段落ごと / トピックごと）
- **Ac4.** quarantine（隔離）状態の存在意義
- **Ac5.** D4 自動アーカイブの発火条件
- **Ac6.** アクション結果の通知（サイレント / トースト / 履歴ノード生成）

### Safety / Irreversibility — Sf
- **Sf1.** trash 経由の保持期間（30 日 / 90 日 / 無期限）
- **Sf2.** 葉ノード絶対保護の実装（型レベル / 実行時チェック / UI 上のみ）
- **Sf3.** undo の射程（直近 1 操作 / セッション内 / 全履歴 time-travel）
- **Sf4.** 確認ダイアログの段階（無し / 1 段 / 重要度別 2〜3 段）
- **Sf5.** バックアップとの連動（操作前スナップショット）
- **Sf6.** AI による自動実行の禁止領域

### M11 接続 — M11
- **M11a.** Cleanup mode のキューに何を流すか（検出器のフィルタ）
- **M11b.** keep フラグの永続化と Cleanup mode 側の除外
- **M11c.** Cleanup mode の上位ダッシュボード（D5 スコア表示）

## キーメッセージ

- **検出とアクションは分離設計が筋がいい。** 同じ検出器でも複数アクションが選べ、
  同じアクションでも複数検出器がトリガになる。1:1 で結ぶと拡張性が死ぬ
- **「警告」と「強制」の中間に "提案+keep フラグ" のレイヤーがある。**
  これを置かないと、ユーザーは警告を無視するか機能を切るかの二択に追い込まれる
- **不可逆性は「ノード種別 × 操作」のマトリクスで考える。**
  葉ノードの delete だけは型レベルで禁止、それ以外は trash + undo + バックアップで救う
- **D5 健全性スコアは "数字" であってはならない。** 単一スコアは「最適化対象」
  になってしまい、無理に分割・archive する歪みを生む。**色 / 顔文字 / 天気** のような
  曖昧表示が向いている可能性
- **AI に任せる範囲は "候補出し" まで。** 実行は人間が押す（半自動）。
  C10 自走 AI（夜間整理）の文脈でも、destructive 操作だけは人間承認を残す案が安全
