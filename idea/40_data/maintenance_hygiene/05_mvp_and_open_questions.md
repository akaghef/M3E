## MVP 案（3 段階）

「採否は決めない」原則を保ちつつ、もし作るとしたらどの順序か、を **複数案** で並べる。

### MVP 案 1: 「健康診断」型（観測重視）

最初は「**何が汚れているか見せるだけ**」の Lv0-1 検出のみ実装。

#### 含むもの
- det_duplicate（Dt1-B 簡易: 正規化文字列一致のみ）
- det_orphan（Dt3-C: scratch 配下の N 日滞留のみ）
- det_stale（Dt4-A: 固定 3 ヶ月）
- det_health_score（Dt5-C: 顔文字表示）
- ノードバッジ（Gl3-A）+ 簡易ダッシュボード（Gl3-C）
- act_keep（除外マーク）と act_inspect のみ
- **破壊的アクションは実装しない**

#### 良い点
- 安全（破壊操作なし）
- 検出器の精度を測れる（誤検出をユーザーが手で見つけられる）
- M11 Cleanup mode は既存のまま、入力ソースが増えるだけ

#### 悪い点
- 「で、どうしろと？」になる
- ユーザーが手で archive/merge する手間は変わらない

#### 推定実装規模
- viewer.ts に検出ロジック追加: ~150 行
- バッジ表示 UI: ~80 行
- ダッシュボード（顔文字 1 つ）: ~50 行
- 計 **~280 行**

---

### MVP 案 2: 「半自動清掃」型（M11 直結）

検出 → M11 Cleanup queue → 1 キー操作の流れを完成させる。

#### 含むもの
- MVP 案 1 全部
- det_duplicate を embedding 版に強化（Dt1-E）
- act_archive（フラグ立て + 折りたたみ Ac1-D）
- act_merge（preview + confirm、Ac2-A 片寄せ）
- M11 Cleanup mode との接続（M11a, M11b）
- 安全装置: 葉ノード保護（Sf2-B）+ trash 30 日（Sf1-A）+ Undo 直近 1 操作（Sf3-A）

#### 良い点
- 実用性◎、汚れを実際に減らせる
- M11 既存設計の再利用
- 安全装置最低限を備える

#### 悪い点
- merge は不可逆性高、preview UI を作る必要
- embedding 計算のコスト（local model or 既存埋め込みサーバ要）

#### 推定実装規模
- MVP 案 1 を含めて **~600 行**

---

### MVP 案 3: 「自走 AI」型（夜間整理）

C10「自走 AI による夜間整理」と同時実装する。

#### 含むもの
- MVP 案 2 全部
- AI 委託（Gl6-C）の検出 + 候補ランキング
- D4 自動アーカイブ（Ac5-B）
- AI 操作の人間レビューキュー（Sf6）
- アクション前スナップショット（Sf5-A）
- time-travel との接続（Sf3-D）

#### 良い点
- 「マップを触っていなくても綺麗になる」体験
- C10 と歩調を合わせて統合

#### 悪い点
- AI コスト
- 信頼性問題（AI が暴走したら）
- 大型実装

#### 推定実装規模
- **~1500 行 + バックエンド整備**

---

### MVP 推奨: **案 1 → 案 2 → 案 3 の段階導入**

理由:
- 案 1 で検出精度を測れる（誤検出率を可視化してから案 2 へ）
- 案 2 で「触れる衛生」になる（dogfood で実用性確認）
- 案 3 は C10 ブレストと連動するので時期を合わせる

## 横断的気づき（Pattern B 共通基盤の観察）

### 1. 「検出器プラグイン化」が筋がいい

検出器を **増やしやすい設計** にしておくと、後から派生検出器（unfinished / broken_link / status_inconsistent / 命名違反 ...）を追加しやすい。

```
interface Detector {
  id: string;            // "det_duplicate"
  scope: "node" | "tree" | "map";
  priority: "low" | "med" | "high";
  detect(map): Candidate[];
  defaultActions: ActionId[];
}
```

これが Gl1-A の前提条件。Gl1-B（結合型）を採るとプラグイン性が死ぬ。

### 2. アクションも同様にプラグイン化できる

```
interface Action {
  id: string;            // "act_archive"
  irreversibility: 0 | 1 | 2 | 3;
  needsPreview: boolean;
  needsConfirm: boolean;
  apply(node, opts): Result;
  undo?(result): void;
}
```

不可逆性レベルが型の中に入ると、04 の安全装置が**自動で発動**できる。

### 3. M11 Cleanup mode は「キュー消化器」に純化

M11 は **既に "1 キーで操作する" UI** なので、こちらに「キューに何を流すか」だけ
プラグインで差し込めば、M11 自体を変える必要は少ない。

```
M11 queue source = list of detectors enabled for cleanup
```

この設計だと、ユーザーが「今日は重複だけ片付けたい」のような **ターゲット設定** も可能。

### 4. D5 健全性スコアは「対立軸」を内蔵すべき

単一スコアは最適化対象になり歪む（Goodhart's law）。
**複数の対立指標**（多すぎ警告 ↔ 少なすぎ警告 / 深い ↔ 浅い）を並べると、
ユーザーが「どっちにも振れ過ぎない」ことを学習できる。

例:
- ノード数: 50 未満（少なすぎ） / 300 超（多すぎ）の両方を警告
- 平均深さ: 2 未満（浅すぎ＝整理されてない） / 7 超（深すぎ＝細分化しすぎ）

### 5. 「触らないことが正しい」ノードの扱い

stale 検出の最大の課題は **「完成して触る必要が無いノード」を誤検出** すること。
これは全検出器に共通のテーマ:
- 用語集 / 参照表 / 引用全文 / 季節性ノード
- これらを keep フラグで除外する運用が現実的

→ **プリセット keep** の概念があると便利
- "/glossary/" 配下は全部 keep
- attribute `kind: reference` は全部 keep

### 6. 「掃除する楽しさ」の設計余地

Pattern B 単独ではなく、K（ゲーム化）と組み合わせると **「掃除が楽しい」** 体験になる:
- 掃除した量の visualization（"今週 12 ノード整理しました"）
- マップが綺麗になったアニメ（K6 マップの生き物感）
- 連続記録（K1 streak）

→ 衛生は「義務」ではなく「ご褒美」にできる可能性。

### 7. プライバシー × AI × 衛生の三角形

- LLM で検出するなら、機微情報ノードを送れない
- 機微情報を除外すると、機微情報の重複が見つけられない
- 機微情報こそ重複・stale が放置されがち（触りたくないから）

→ **暗号化ノードは「ローカル軽量検出のみ」** の二層化が筋。
   - 平文ノード: フル検出（Dt1-G 段階適用 + LLM）
   - 暗号化ノード: 文字列一致 + 統計的シグナルのみ（Dt1-B）

### 8. 「衛生を促進する UX」と「衛生を強制する UX」は別物

- 促進: 提案・バッジ・スコア（Lv1-2）
- 強制: 自動 archive・自動 merge（Lv4-5）

両者を **同じノブで** 制御すると、ユーザーが急に "強制" 側に振った時に事故る。
→ **検出器ごとに介入度を独立設定** できることが望ましい（Gl4 + Gl5-D）。

## 未決質問（Open Questions）

### Q1. ノード種別をマップに導入するか？
- 葉ノード保護を型レベルで実装するなら、ノード種別の導入が必要
- ただし既存マップとの互換性、マイグレーション負荷
- 「種別導入する / しない」は M3E 全体の設計判断

### Q2. archive と trash と quarantine、本当に 3 つ必要か？
- 概念の冗長性
- 1 つに統合できる（フラグで状態区別）か、3 つの物理的分離が UX に効くか
- 認知負荷とのトレードオフ

### Q3. D5 健全性スコアは「あるべき」か？
- 数値化＝最適化対象化のリスク
- 「健康診断」UI 自体を作らない選択肢もある
- ユーザーが見たがる vs 見ない方が良い、の哲学的判断

### Q4. AI 自走（C10）と手動 cleanup（M11）の住み分け
- 全部 AI に任せる時代なら M11 不要？
- 逆に、AI を信用できないなら M11 を強化？
- 中庸案: AI が候補出し、人間が M11 で確認

### Q5. embedding 計算の実行場所
- ローカルで sentence-transformers を動かす（重いが privacy 守れる）
- API（OpenAI 等）で計算（速いが privacy 漏れる）
- M3E のスタンス（policy_privacy）的にはローカル推奨だが、計算機要件が増える

### Q6. 自動アーカイブ（D4）のデフォルト ON / OFF
- ON: 自然に衛生される、ただし「勝手に消えた」恐怖
- OFF: ユーザーが意図的に有効化、習慣化障壁
- 初期は OFF + チュートリアルで紹介が無難？

### Q7. 「マップ規模 50-300 ノード」目標との整合
- そもそも 300 ノード以下なら衛生機能の必要性は薄い？
- 衛生機能が要るのは 1000+ になってから？
- → MVP は 300 ノード規模を維持するための「予防」として設計、
   1000+ になった時の「治療」は別フェーズ？

### Q8. AlgLibMove dogfood との接続
- M3E 自身の開発マップで衛生機能を最初に dogfood する？
- 「自分のマップが汚いから作る」モチベーション駆動
- project_alglibmove_dogfood に「D 系」を追加する案

### Q9. 既存 backup.ts との API 整合
- アクション前スナップショット（Sf5-A）は既存バックアップ機能で足りるか
- 「特定アクション直前」のフックを backup.ts に追加する必要があるか

### Q10. 「衛生」という言葉自体の是非
- "Hygiene" は技術用語として正しいが、UX 的に冷たい
- 「整える」「整理する」「片付ける」「お掃除」など、
  ユーザー向け表記は別途検討余地
