# 07. MVP 案・横断観察・戦略・未決質問

ここまで N1〜N5 を個別に並べた。最後に **横断的気づき** と **MVP 案**、
M3E 教育 vertical の戦略選択肢、未決質問を整理する。

## 1. 横断的気づき

### X.1. N1 と N4 は同じ問題の別フォーマット
- 入力: 教科書 PDF（N1） vs 講義 PPTX（N4）
- 出口: M3E マップ
- → 共通基盤「ドキュメント→マップ変換」を抽象化できる
- → 実装順は **N4 → N1** がよい（PPTX のほうが構造保持率高い）

### X.2. N2（クイズ）と N5（進度）は閉ループ
- クイズ正答率 → 進度に反映
- 進度の弱点 → 次のクイズが選ばれる
- → 単独実装すると価値半減、セット設計推奨

### X.3. N3（教師ビュー）はプライバシーが本質
- N3 を素朴に作ると policy_privacy 違反、L1〜L5 連携必須
- MVP では N3 を **やらない** または「教材だけ管理」（N3.A4）に絞るのが安全
- → 教師ビューは Phase 2 以降

### X.4. M3E 既存スキルの流用余地が大きい
| 既存 | 教育用途への流用 |
|---|---|
| slideshow ブレスト | N1/N4 閲覧 UI、ツアー定義 |
| keyboard_modes | 学習者高速入力（クイズ回答） |
| cognitive_science | N2/N5 SRS、忘却曲線 |
| collaboration | N3 教師ビューの共有基盤 |
| export_publish | N1 教材エクスポート |
| automation_obstacles | やらない領域の指針 |
| capture_ingest | A3 PDF 取り込み再利用 |

→ **教育機能の 60〜70% は既存ブレストの組み合わせで構築可能**

### X.5. 「自習者個人」から始めるのが MVP として正解
- 教員→学生群は運用が重い（人数管理・成績連携・プライバシー）
- akaghef 自身が **自分の研究学習** に使う段階を最初に通すと安全
- dogfood しやすい

### X.6. 「教育用途」と言いつつ akaghef の研究学習にも有用
- 論文を読む = N1 教科書マップ化
- 論文の主張を覚える = N2 クイズ・N5 SRS
- 学生指導記録 = N3 の縮小版
- → 教育機能 = akaghef の研究学習機能の流用と捉えると一石二鳥

### X.7. 教育市場参入の本気度判断
- 真に教育 vertical を狙うなら独立サブブランド（M3E Edu）
- 副作用的なら本体に薄く統合
- akaghef の優先（科研費）を踏まえると **後者** が現実的

## 2. MVP 案（複数）

### MVP-α: 自習者シングル（最小・最安全）
- N4.B1（PPTX → 1スライド1ノード）
- N1.A1（PDF → 章節）
- N2.A1（穴埋め）+ N2.B1（AI 生成）
- N5.A2 + N5.B2（ステータス + ヒートマップ）
- N3 含めない
- 実装規模: 〜2週間
- 推し: ⭐⭐⭐

### MVP-β: 教員主導（教材作成支援）
- N4 全機能
- N1 章節までは深掘り
- N2 は教員レビュー必須（N2.B4）
- N3.A4（教材だけ管理、学生マップ非関与）
- N5 含めない
- 実装規模: 〜1ヶ月
- 推し: ⭐⭐

### MVP-γ: ゼミ運用（小集団・opt-in）
- N1〜N5 全部薄く
- N3.A1（5人以内、opt-in）
- N3.E2（学期末提出）
- 暗号化・ACL 含む
- 実装規模: 〜2ヶ月
- 推し: ⭐

### MVP-δ: 既存研究フローの教育流用（akaghef 用）
- akaghef 自身の論文学習を「教科書マップ化」と見立てる
- N1 + N2 + J1 SRS だけ
- 教育用途の検証は二の次、研究学習の効率化が一次
- 推し: ⭐⭐⭐（dogfood 観点）

## 3. 段階導入ロードマップ案

```
Phase 0 (現在): 教育機能なし、研究用途のみ
Phase 1 (1ヶ月): MVP-α または MVP-δ。自習者個人。
Phase 2 (3ヶ月): MVP-β。教材作成支援。教員ベータ。
Phase 3 (6ヶ月): N3 を限定的に開放（小規模ゼミ）。
Phase 4 (1年〜): 大規模クラス対応（プライバシー基盤完成後）。
```

ただし **Phase 2 以降は科研費目標と競合** する可能性。
project_projection_vision の進捗次第で判断。

## 4. 戦略選択肢

### S1. 本体統合派
- M3E に教育機能を組み込む
- 単一プロダクト
- 利点: 開発・維持コスト低
- 欠点: 研究者向け純粋性が薄まる

### S2. フォーク派（M3E Edu）
- 教育専用ブランチ
- 利点: 教育向け UI 大胆に変えられる
- 欠点: 二重メンテ

### S3. プラグイン派
- M3E は薄く、教育機能は plugin として
- 利点: モジュラー
- 欠点: plugin システム自体の設計コスト

### S4. テンプレート派
- M3E 本体に手を入れず、教育用「マップテンプレート」と AI prompt 集として配布
- 利点: 開発コスト最小
- 欠点: 体験が断片的

推し: **S1 + S4 ハイブリッド**（小機能は本体、大物はテンプレ・指南書）

| リスク | 影響 | 緩和 |
|---|---|---|
| プライバシー違反 | 信頼喪失 | N3 を限定、暗号化必須 |
| 著作権侵害（教科書取り込み） | 法的 | ローカル限定、共有禁止 |
| 評価ツール化 | 学生監視批判 | N3.D6 評価切り離し |
| AI 採点誤り | 学習効果損 | 教員レビュー必須、自己採点優先 |
| akaghef のリソース分散 | 研究遅延 | Phase 1 を MVP-δ で兼用 |
| 教育機関のシステム制約 | 普及阻害 | LMS 連携 / SSO 後回し |

## 6. 未決質問（最重要）

レビュー時に akaghef が判断すべき問い:

### Q1. **教育 vertical を本気で取るか？**
- Yes → S1〜S3、Phase 2 以降に投資
- No → MVP-δ だけで dogfood、本体には薄く

### Q2. **akaghef 自身の教員業務を M3E で回すか？**
- Yes → MVP-β または MVP-δ から
- No → MVP-α、純粋に「自習者向けプロダクト」として

### Q3. プライバシー設計を初期から入れるか、後付けか
- 初期: N3 が遅れる
- 後付け: 設計やり直し負債

### Q4. クイズ生成 AI のコスト負担
- ローカル LLM / クラウド API / 教員手動

### Q5. 著作権問題への対処方針
- 完全ローカル / 公開禁止 / fair use 主張

### Q6. 既存 LMS（Moodle 等）との連携
- 連携する: 普及加速、開発コスト
- しない: 独立性、孤立

### Q7. 多言語対応の優先度
- 日本語のみ / 日英バイリンガル / 多言語
