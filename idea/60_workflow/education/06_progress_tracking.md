# 06. N5: 学習進度トラッキング

学習者の進度を可視化し、適切な介入タイミングを見つける。
N2（クイズ）と閉ループを形成し、cognitive_science / J1 SRS と連携。

`automation_obstacles/04_what_not_to_automate.md` の N5「学習」と区別:
あちらは「学習そのものは自動化しない」、こちらは「学習の **進度管理は** 自動化してよい」。

## 目的

- 学習者が自分の進度を客観視
- 教員が学生の遅れを早期発見
- 復習タイミングを最適化（SRS）
- モチベーション維持

## 1. 進度の定義（N5.A）

「進度」とは何を測るか:

### N5.A1. ノードの読了数
- 最も簡単
- 「読んだ」≠「理解した」問題

### N5.A2. ステータス遷移（未着手→読了→演習→説明可能）
- 段階的、M3E ネイティブ
- 学習者の自己申告に依存

### N5.A3. クイズ正答率（N2 連携）
- 客観的、N2 質に依存

### N5.A4. 滞在時間
- 受動的計測可能
- プライバシー強い、品質低い

### N5.A5. 編集量（追加ノード・リンク）
- アウトプット型の進度
- 推し: ⭐⭐

### N5.A6. SRS 復習成績（J1 連携）
- 長期記憶への定着度
- 推し: ⭐⭐⭐

### N5.A7. ハイブリッド（複数指標を加重）
- 推し: ⭐⭐⭐
- 重み付けが難問

## 2. 可視化（N5.B）

### N5.B1. 進度バー（章ごと）
- 「第3章: 60% 完了」
- シンプル、教科書ライクな安心感

### N5.B2. マップ全体ヒートマップ
- 未完了ノード = 赤、完了 = 緑
- 推し: ⭐⭐⭐

### N5.B3. カレンダー（GitHub 風 contribution graph）
- 日々の学習ログ
- ゲーミフィケーション

### N5.B4. 折れ線グラフ
- 進度の時系列推移

### N5.B5. レーダーチャート
- 章別 / 領域別の理解バランス

### N5.B6. 樹形図（教材マップそのまま）
- 既存マップビューの再利用

### N5.B7. 「次の一手」レコメンド
- 「次はこのノードを読むと進度＋10%」
- 推し: ⭐⭐

## 3. 介入トリガー（N5.C）

進度データから何を引き起こすか:

### N5.C1. リマインド通知
- 「3日触っていません」

### N5.C2. SRS 復習通知
- 忘却曲線に沿って自動

### N5.C3. 教員への警告
- 「この学生が2週間進度ゼロ」

### N5.C4. 学習計画の自動再調整
- 試験までの残り日数 × 残り範囲 → 1日あたりノルマ提示

### N5.C5. 励ましメッセージ
- ゲーミフィケーション K1 連携

### N5.C6. ペースダウン警告
- 「進度が早すぎ → 理解度確認推奨」

### N5.C7. 介入なし（観察のみ）
- 学習者の自由を尊重

## 4. 学習計画（N5.D）

### N5.D1. 教員が事前作成
- 「第1週は1章、第2週は2章」を教員がマップ上に配置

### N5.D2. 学習者が自分で作成
- 自由ペース

### N5.D3. AI 自動生成
- 試験日 + マップ → 学習計画

### N5.D4. ハイブリッド（教員雛形＋学習者調整）
- 推し: ⭐⭐⭐

## 5. 失念・忘却の扱い（N5.E）

- 一度「読了」ステータスでも、N日経つと「鮮度低」表示
- J2 忘却曲線と連携
- N5.E1. ステータスを自動戻す（強制）
- N5.E2. 警告だけ（学習者が判断）
- N5.E3. SRS 復習対象に追加（推し: ⭐⭐⭐）

## 6. 学習履歴の永続化（N5.F）

### N5.F1. ノード属性として埋め込み
- ステータス・完了日・正答率
- マップと一体

### N5.F2. 別 DB / イベントログ
- 詳細イベント（クリック・滞在）まで記録
- 重い、プライバシー強

### N5.F3. 集計値だけ保存
- 「先週の学習時間 5h」
- 軽い

### N5.F4. 永続化しない（セッション中のみ）

## 7. 学習者へのフィードバック粒度（N5.G）

### N5.G1. 即時（クイズ直後）
- ドーパミン的、ゲーミフィケーション

### N5.G2. デイリー（毎晩のサマリー）
- 内省的

### N5.G3. ウィークリー（週次レビュー）
- G2 personal_productivity と連携

### N5.G4. 学期末（成績風）
- 重い、評価ストレス

## 8. ユースケース

### UC1. 学部生の試験準備
- 試験2週間前にマップ全体を「対象範囲」として進度トラッキング開始
- N5.B2 ヒートマップで弱点可視化
- N5.D3 AI 計画で1日あたり負荷を逆算

### UC2. 大学院生の論文準備
- 「投稿締切 = 試験」、関連論文ノード = 試験範囲
- 既読/未読/精読を進度管理

### UC3. オンライン講座受講者
- 動画視聴 → ノード読了 → クイズ → 進度バー
- N5.C5 励ましで継続率向上

### UC4. 教員の学期運営
- N3 連携で全学生の進度俯瞰
- 遅れ学生に個別連絡

## 9. アンチパターン

- 進度を「数値ノルマ」化しすぎる → モチベ低下
- 教員監視ツール化 → 学生の自主性喪失（automation_obstacles N5 警告）
- ゲーミフィケーション過剰 → 本来の学習目的逸脱
- プライバシー過剰収集

## 論点 ID

- N5.A. 進度の定義（読了 / ステータス / 正答率 / 滞在 / 編集量 / SRS / 加重）
- N5.B. 可視化（バー / ヒートマップ / カレンダー / 折線 / レーダー / 樹形 / 推奨）
- N5.C. 介入トリガー（リマインド / SRS / 教員警告 / 計画再調整 / 励まし / 観察のみ）
- N5.D. 学習計画（教員作成 / 学生作成 / AI 生成 / ハイブリッド）
- N5.E. 忘却扱い（自動戻し / 警告 / SRS）
- N5.F. 履歴永続化（属性 / DB / 集計 / なし）
- N5.G. フィードバック粒度（即時 / 日次 / 週次 / 学期末）
- N5.H. ゲーミフィケーション統合度
- N5.I. 学習者の opt-out 権（トラッキング拒否）
