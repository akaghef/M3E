# 07. データモデル・MVP・戦略

スニペットをデータとして表現し、最初に何を作るかを整理。
他章と異なり、**最小実装案**を含む（ただし「確定」ではない）。

## M1. スニペット定義スキーマ

提案 A. JSON / YAML ベース（ファイル）

```jsonc
{
  "id": "swot",
  "label": { "ja": "SWOT 分析", "en": "SWOT Analysis" },
  "category": "analysis",  // analysis / thinking / research / creative / decision
  "tags": ["business", "strategic"],
  "description": "内外環境を強弱 × 好機脅威で分類",
  "aliases": ["swot", "スウォット", "強み弱み"],
  "children": [
    { "label": { "ja": "強み", "en": "Strengths" }, "nodeType": "default" },
    { "label": { "ja": "弱み", "en": "Weaknesses" } },
    { "label": { "ja": "機会", "en": "Opportunities" } },
    { "label": { "ja": "脅威", "en": "Threats" } }
  ],
  "params": [],
  "source": "built-in",
  "version": "1.0"
}
```

提案 B. 定義は M3E のノード自身（メタマップに置く）

- `system/snippets/` スコープ下にスニペット定義ノードを置く
- スニペット自体も M3E で編集可能（ドッグフード）
- アクセス: `m3e:///system/snippets/swot`

**推し**: B が M3E らしい。「M3E を M3E で拡張する」`scope-tree-as-system` の思想と一致。
ただし MVP は A（ファイル）で始めて後で移行が現実的。

## M2. 組み込み vs ユーザー定義

3 層構造:

| 層 | 例 | 更新 | 共有 |
|---|---|---|---|
| **Built-in** | 本ブレストの 60-70 個 | M3E 本体のバージョンアップ | 全ユーザー共通 |
| **Workspace** | 自分のワークスペース専用 | ユーザーが編集 | ワークスペース内共有 |
| **Personal** | 個人の好み | ユーザーが編集 | 本人のみ |

**論点 M2a**: オーバーライド
- Personal が同名の Built-in を上書きできるか
- Workspace が Personal に優先するか

**論点 M2b**: 組み込みの編集
- Built-in を「派生してカスタマイズ」できる（編集→自動で Personal 化）

## M3. パラメタ化（バリエーション）

スニペット定義にパラメタを持たせる例:

```jsonc
{
  "id": "why-tree",
  "label": { "ja": "なぜなぜ分析" },
  "params": [
    { "name": "depth", "type": "number", "default": 5, "min": 1, "max": 10 }
  ],
  "generate": "depth 階層の Why を連鎖生成"
}
```

**呼び出し時**:
- パレット: `why-tree 3` で深さ 3
- キーバインド: 数字キーで深さ指定

**対象スニペット**: なぜなぜ（T3）、ロジックツリー（T2）、KPI ツリー（A20）、ブレインライティング（T16）の人数指定、など。

## M4. メタ情報

スニペットに付けたいメタ:
- `category` — 4〜5 大分類（analysis/thinking/research/creative/decision）
- `tags` — 複数のフリータグ（business/academic/personal/medical ...）
- `language` — 原語（ja/en/...）
- `source` — 出典（教科書・提唱者・リンク）
- `when_to_use` — 使いどころの 1 行
- `compatibility` — 「この前後で使うと良い」他スニペットへの参照（例: SWOT → TOWS）
- `popularity` — 全ユーザー集計の使用頻度（オプトイン）

## M5. MVP の最小セット

「まず何を動かすか」の提案 3 案:

### MVP-A: 超ミニマル（1 週間）
- 組み込み **10 個**だけ（5W1H, SWOT, MECE, Why5, PDCA, SCAMPER, 6色ハット, RICE, MoSCoW, IMRaD）
- 呼び出し: コマンドパレット（`/` 起動）のみ
- 定義: ハードコード（`src/shared/snippets.ts`）
- ユーザー定義なし
- 空ノード展開のみ、内容埋めは手動

### MVP-B: 中規模（2〜3 週間）
- 組み込み **30 個**
- 呼び出し: パレット + 右クリックメニュー
- 定義: JSON ファイル（ビルド時バンドル）
- ユーザー定義: JSON 手動編集（まだ UI なし）
- 日英ラベル、recently used 機能あり

### MVP-C: フル機能（1〜2 ヶ月）
- 組み込み **60+ 個**（本ブレスト網羅）
- 呼び出し: パレット / キー / 右クリック / AI
- 定義: メタマップ上のノード（M1 提案 B）
- ユーザー定義 UI, 共有機能、パラメタ化
- AI 提案、コンテキスト依存ランキング

**推し**: MVP-A から始めて段階的に拡張。最初の 10 個で 80% のユースケースをカバー。

## M6. 共有・エクスポート

**論点 M6a**: ユーザー定義スニペットの共有経路
- Git リポジトリ（パブリック）
- M3E Hub のようなサーバー（将来）
- Gist / Pastebin 的ペースト
- ワークスペースごと丸ごと

**論点 M6b**: プライバシー
- 自作スニペットに機微情報（「弊社専用」「研究室内のみ」）が混じる可能性
- 共有前の自動マスキング（topic_pool L2）

## M7. バージョン管理

- Built-in は M3E バージョンに紐づく
- ユーザー定義は個別 version フィールド
- **論点 M7a**: 破壊的変更時の既展開ノードへの影響
  - 既に展開済みのノードは変更されない（安全）
  - 新規展開時のみ新バージョンが適用される

## M8. 段階導入ロードマップ

```
Phase 0 （調査・ブレスト）
  ✓ 本ブレスト（idea/60_workflow/structure_snippets/）
  ✓ 類似既存メモ探索（結果: なし）

Phase 1 （MVP-A, 1 週間）
  - 10 個の組み込みスニペット
  - コマンドパレット実装
  - 展開ロジック（親ノード選択 → 子を一括追加）

Phase 2 （MVP-B, 2-3 週間）
  - 30 個まで拡張
  - 右クリックメニュー追加

  - 日英ラベル

Phase 3 （MVP-C 前半, 1 ヶ月）
  - 60+ 個まで拡張
  - ユーザー定義（JSON ファイル）
  - キーバインド（keyboard_modes 統合）

Phase 4 （MVP-C 後半, 1 ヶ月）
  - メタマップベース定義に移行
  - AI 呼び出し（自然言語）
  - パラメタ化（なぜなぜ深さ等）
  - コンテキスト依存ランキング

Phase 5 （将来）
  - 共有機能
  - マーケットプレイス的なもの
  - プラグイン型拡張
```

## 未決質問

1. **Q1. 既展開ノードの扱い** — スニペット展開後に親ノードにすでに子がいたらどうする？
   - 追加（既存子の後ろに）/ 警告 / 新しい中間ノードを作ってその下に置く

2. **Q2. 同じスニペットを 2 回当てる** — SWOT を 2 回当てたら重複する？
   - 冪等（再実行は無視）/ 毎回追加 / 差分適用

3. **Q3. スニペットのネスト** — SWOT の Strengths の下に 5W1H を当てる、の連鎖
   - 制限なし（ユーザー判断）が自然

4. **Q4. undo 動作** — スニペット展開を 1 ステップで undo できるか
   - できるべき（複数ノード生成を 1 トランザクション）

5. **Q5. モバイル UI** — スマホで 60 種類をどう見せる？
   - カテゴリ → リスト の 2 段階、またはお気に入りのみ

6. **Q6. 多人数同時展開** — collab モードで A がスニペット展開中に B も別スニペットを展開した場合の競合
   - CRDT 的には順番が決まれば OK。scope-lock で同一親ロック

7. **Q7. AI 提案の信頼性** — 「このノードには SWOT が合う」の AI 判定の根拠をどう見せるか
   - 「理由」テキストを生成して提案に付ける

8. **Q8. カテゴリ分類の粒度** — 5 大カテゴリ（本ブレスト案）で足りるか
   - サブカテゴリ必要（例: analysis/marketing, analysis/strategic）

## 全体の横断気づき（このブレスト全体のまとめ）

1. **M3E のツリー構造と極めて相性が良い** — 子ノード一括生成はネイティブ機能。他ツール（Notion/Roam）より自然
2. **60-70 種類のロングテール戦略が現実的** — 定番 10 個で 80%、残りはニッチだが専門分野には必須
3. **「構造だけ展開・内容は別」の分離が本質** — AI と手動の両方で内容を埋められる
4. **呼び出し UI の多重化が鍵** — 初心者（メニュー）/熟練（キー）/AI依存（自然言語）を並列サポート
5. **分類軸 5 つ（analysis/thinking/research/creative/decision）で大半カバー** — 将来的にサブカテゴリ追加
6. **連鎖スニペットが強力** — SWOT → TOWS、ブレスト → KJ法、Why 5 → 対策立案 等の定型フローをパッケージ化
7. **M3E 自身のメタ改善にも使える** — 本ブレストを書く過程で 60 個カタログ化したが、これ自体が「スニペット化前の思考プロセス」の記録。将来 M3E にスニペット機能が入ったら、このブレスト自体をスニペットで再構築可能（dogfood）
