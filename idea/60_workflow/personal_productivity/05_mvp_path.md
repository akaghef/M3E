# 05. 横断観察 / MVP / 未決質問

01〜04 を横断する観察と、最小実装の素描、未決の質問を集める。
**MVP は1案ではなく、いくつかの「最初の一歩」を併記** する。

## PP. 横断観察 (フォルダ全体)

### PP-1. 4層モデルが共通フレーム
README の「リズム / 方向 / 計測 / 状態」4層が、本トピックの暗黙の構造。
G1〜G8 はこの層に分散していて、層を意識すると優先順位が見える。

### PP-2. 「強制しない、サポートする」 (Ph1) が判定基準
全選択肢を Ph1 で再評価できる:
- α 案 (強制系) は短期効果高いが、長期離脱率も高い (推測)
- γ 案 (任意系) は離脱率低いが、効果が見えにくい
- ハイブリッドが落としどころ

### PP-3. 自動化と能動記録の対立
時間記録 (V) や状態変化 (DD) は **自動化したくなる** が、
「自分で意識して記録する」こと自体が思考のトリガーになる場面が多い。
automation_obstacles A8 学習機会喪失と直結。

### PP-4. データを「取る」と「見せる」は別問題
エネルギーログ (Z-5) / 中断ログ (II-5) / 時間記録 (W-4) すべてに
**「見ない権利」** を残す設計案がある。Ph4 の一貫した適用。

### PP-5. 既存 keyboard_modes との結合点
- M23 Sprint Plan ↔ G3 OKR (週次の方向決め)
- M24 Standup ↔ G1 朝チェックイン
- M25 Retro ↔ G2 週次レビュー
- M26 Journal ↔ G1 夜チェックイン (D-γ)
- M02 Triage ↔ UC3 会議直後の捕捉
- M30 Focus ↔ G7 集中モード

本トピックは UX/概念層、keyboard_modes は実装層。
**両者を切り離して進化** させられる構造。

### PP-6. automation_obstacles との関係
- A4 自己欺瞞 → 時間記録 (T〜W) で「読書時間 vs 執筆時間」が見える
- A5 エネルギー管理 → G8 エネルギーログで時間帯最適化
- J1 燃え尽き → R-3 / F-4 / NN-4 の「保護モード」
- N7 創発の場 → G7 集中モードと両立する設計 (NN-3 散らかしモード)
- N9 自己観察 → G8 / D-α 夜チェックインで支援

### PP-7. ガミフィケーション (K 系) との緊張
- Streak (P-α) は K1 と直結だが Ph1 違反リスク
- 達成バー (Q) は K2 achievement と接続するが研究者向きでない可能性
- 推し: K 系を **オプトイン** で完全分離、デフォルトは静かな記録

### PP-8. プライバシー (L 系) との交差
- 時間記録 (W) / エネルギー (Z) / Journal (D-γ) すべて **機微情報**
- 暗号化 + マルチ Vault 切替 (L3) の設計が前提
- 共有時のマスキング (L2) も必要

## QQ. MVP 案 (4つの「最初の一歩」)

採否決めず、**どの順で実装してもよい4案** を並べる。

### MVP-1. デイリー儀式 MVP
**最小実装**: 朝/夜の3問テンプレを `daily/2026-MM-DD` 日付ノードに自動生成。

```
- A1-β (日付ノード生成型)
- B-1 (3問テンプレ)
- D-α (3問: 進んだ/詰まった/明日)
- E (手動起動 + 起動時に「今日の朝チェックイン?」軽い提案)
- F-1 (Streak 概念なし、完全許容)
```

**実装ボリューム感**: 50〜150行程度 (ノード生成スクリプト + テンプレ展開)。
keyboard_modes M26 Journal mode と統合可能なら相互流用。

**価値**: PP-5 の M24 Standup / M26 Journal の前段、
「毎日 M3E を開く理由」の最低限。

### MVP-2. エネルギーログ MVP
**最小実装**: ノード入力時に「今のエネルギー」を絵文字5択で1キー記録。

```
- X-γ (絵文字 😴😐🙂😄🚀)
- Y (Pomodoro 終了時 OR マップ起動時)
- Z-2 (曜日 × 時刻ヒートマップ)
- AA-3 (オンデマンド AI 提案)
```

**実装ボリューム感**: 100〜200行程度 (記録 attribute + 簡易可視化)。
ヒートマップは外部ライブラリで省力化。

**価値**: PP-6 A5 エネルギー管理 / N9 自己観察に直接効く。
記録摩擦が小さい (絵文字1キー) のが鍵。

### MVP-3. Pomodoro + 集中モード MVP
**最小実装**: 1ノード選択 → Pomodoro 開始 → 他ノード 80% フェード → 25分後解除。

```
- BB-β (ノード選択推奨)
- CC-1 (古典 25/5)
- DD-γ (ログのみ、status 自動変更なし)
- EE-2 (中断もログ、責めない)
- FF-β (80% フェード)
- GG-1 (M3E 内通知のみ抑制、OS は触らない)
- HH-δ (時間 OR 手動 OR 完了で解除)
- OO-β (集中中も scratch 1キー投入可)
```

**実装ボリューム感**: 200〜400行程度 (タイマー + フェード CSS + キーバインド)。
keyboard_modes M30 Focus mode の前身として位置付け。

**価値**: PP-2 Ph1 を保ちつつ、M3E の「ノード = 単位作業」構造を最大活用。

### MVP-4. 週次レビュー MVP
**最小実装**: 週末に「今週触ったノード」一覧を表示し、KPT を簡易記録。

```
- G-β (今週触ったノードのみ対象)
- H-1 (KPT)
- I (緩い接続: M25 Retro mode へのリンク表示)
- J-β (手動コピー、来週の sprint 候補に貼る)
```

**実装ボリューム感**: 100〜250行 (一覧抽出 + KPT 記録ノード生成)。
M25 Retro mode が実装されていれば、それを呼び出すだけで成立。

**価値**: PP-5 M25 と直結、長期的なマップ衛生 (D 系) にも貢献。

## RR. MVP 間の依存関係

```
MVP-1 (儀式) ──┐
              ├─→ 日付ノード基盤を共有
MVP-4 (週次) ──┘

MVP-2 (エネルギー) ──→ MVP-3 (集中) と組合せで Pomodoro 完了時に記録 (MM-2)

MVP-3 (集中) ──→ keyboard_modes M30 へ進化
```

**推し順**: MVP-1 → MVP-2 → MVP-3 → MVP-4
理由: 摩擦小・効果見える順、MVP-2 が最も投資対効果高い (推測)。

ただし「Pomodoro が一番欲しい」というユーザー判断なら MVP-3 から開始可。

## TT. 未決の質問 (要レビュー)

### TT-Q1. Streak の扱い (推し度わかれる)
**論点**: P-α (Streak あり) vs P-γ (緩い表示) vs P-β (なし) のどれを採用するか。
**懸念**: ユーザー akaghef の性格依存。Streak が動機になるか、プレッシャーになるか。
**選択肢**:
- (a) P-γ デフォルト、α/β はユーザー切替 (推し)
- (b) P-β デフォルト、α は完全オプトイン
- (c) ユーザーに最初に聞いて決める (オンボーディング質問)

### TT-Q2. 日付ノードの場所と量 (爆発リスク)
**論点**: A1-β で日付ノード生成すると **365ノード/年** 増える。マップ汚染。
**選択肢**:
- (a) `daily/2026/04/16` のように年月で階層化、月次 archive
- (b) 別 workspace `journal-2026` に分離 (K-3)
- (c) 日付ノードは仮想化 (ファイル化せずビューだけ生成)

### TT-Q3. Toggl 連携の優先度
**論点**: T-α (外部連携) を Phase 5 に置いたが、akaghef が既に Toggl 使っているなら Phase 2 に上げるべき。
**選択肢**:
- (a) Phase 5 のまま、まず内蔵で試す
- (b) Phase 2 に前倒し
- (c) 連携自体を諦め、手動入力 (T-γ) のみ

### TT-Q4. AI 提案の範囲 (能動 vs 受動)
**論点**: AA / MM-3 / NN-4 で AI が能動的に提案するか、聞かれた時だけ答えるか。
**選択肢**:
- (a) 全部能動 (UX 強引、Ph1 違反気味)
- (b) 全部受動 (オンデマンドのみ、Q4 整合)
- (c) ユーザー設定 (デフォルトは受動)

### TT-Q5. プライバシー Vault の分離粒度
**論点**: Journal / エネルギー / 時間記録 を別 Vault に分けるか、同 Vault で暗号化レベルだけ変えるか。
**選択肢**:
- (a) 全部同 Vault、ノード単位暗号化 (L1)
- (b) Journal だけ別 Vault (L3)
- (c) 機能毎に Vault (3〜4 Vault に肥大)

### TT-Q6. 4層モデル (PP-1) を実装上どう表現するか
**論点**: README の L1〜L4 は概念整理だが、UI で層を見せるか隠すか。
**選択肢**:
- (a) 隠す (内部設計のみ)
- (b) ダッシュボードで層別ビュー
- (c) ノードに `layer: rhythm/direction/measure/state` タグ付け

## UU. レビュー時の判断手順 (案)

ユーザーが採否を決める時の推奨フロー:

1. **MVP 4案** (QQ-1〜4) のうち最初に着手するものを1つ選ぶ
2. その MVP に含まれる **論点 ID** (例: A1-β / B-1 / E など) を確認
3. 採用 ID を README の「論点一覧」にチェック
4. **TT 未決質問** のうち選んだ MVP に関係するものに先に答える
5. 段階導入 (SS) の Phase をどこから始めるか決定

## WW. キーメッセージ (本フォルダ最終結論)

1. **「毎日開く理由」は状態 (エネルギー / 集中) から作る** ほうがタスクから作るより継続率高い (推測)
2. **Ph1 強制しない設計** が長期維持の鍵、Streak 系は両刃
3. **MVP-2 エネルギーログが最小投資・最大効果** の有力候補
4. keyboard_modes M23-M26 と本トピックは **UX 層 ↔ 実装層** の関係、両輪で進化
5. **automation_obstacles の N7 創発の場 / N9 自己観察** を侵さない設計が、研究者向き M3E のアイデンティティ
