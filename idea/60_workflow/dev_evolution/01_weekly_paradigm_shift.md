# 開発スタイルの週次脱皮と、残る希少スキル

2026-04-17 の対話ダンプ。
「M3E は 1 週間ごとに開発スタイル自体が劇的に変わっている」という観察を起点に、
外挿と、人間に残る希少スキルの正体までを一気に通した思考の記録。

## 起点の観察

`docs/daily/` を 3/30 から 4/15 まで眺めた結果、各週で**作ったもの**ではなく
**作り方そのもの**が別物になっている、というパターンが見えた。

## 実績: 3世代のスタイル

### W1 (3/30–4/05) 単独爆走スタイル
- akaghef が全部書く。1 日 20 件の「追加実施」
- MVP として何が成立するかを自分の指で確かめる週
- 意思決定は 1 人で完結

### W2 (4/06–4/12) worktree 分業 + PR ゲートスタイル
- `dev-beta-visual` / `dev-beta-data` の worktree 分離
- codex1 / codex2 の役割固定
- 部下 push → 上司 merge → 部下 rebase の強制フロー
- AI を「別人格の同僚」として扱う運用が定着

### W3 (4/13–4/17 進行中) TDD + Canvas 駆動スタイル
- vitest / playwright が議論より先に答えを出す
- M3E マップ自身が strategy board / task pool / agent status になる
  (開発対象 == 開発管理面、dogfood の完成形)
- devM3E / pr-beta / map-update 等の skill が workflow を構造化

## 再帰性の発見

各週のスタイルが**次週のスタイルを可能にする道具**になっている。

- W2 の worktree 分業がないと W3 の Canvas 駆動は成立しない
  (複数主体が同じ面を見る必要がある)
- W3 の TDD + Canvas がないと W4 のジャーニー駆動は回らない
  (ジャーニーの詰まりを自動検出する基盤が要る)

**開発スタイルが自己進化するループ** に入っている。

## 外挿: 次の 3 週間（当初 24 週と予想 → 実際は 3 週間）

### W4 (4/18–4/24) ジャーニー駆動開発
- 評価軸が「機能が動く」から「akaghef のタスクが詰まらず走る」に変わる
- 痛点ログが PR を駆動する
- コードを書くのはジャーニーが詰まった時だけ
- 開発の起点が「計画」から「実走の破綻」に移る

### W5 (4/25–5/01) cloud 製品化 + オントロジー研究
- cloud: Supabase 連携が「同期」から「マルチテナント可能な提供形」へ
- ontology: node / relation / scope に意味論スキーマが入る
- 射影の前提となる「世界モデルの語彙」が形式化される
- 実装と理論が並走する学術モード

### W6 (5/02–5/08) subPJ 高速ループ
- 人間が書かない週
- 複数 subPJ が scope 単位で並列に plan → impl → verify
- 人間は各ゲートでの judge に専念
- 生産性の律速が人間の判断粒度に移る

### W7+ 射影駆動
- 出したい成果物から逆算して世界モデルを整える (output-first)
- 科研費 / 論文アウトライン / スライドの最初の実射影が外に出る

### W8+ 場運営
- 他者が同じ世界モデルに join
- 「開発する」から「場を運営する」へ

## レビュートリアージも AI 側に落ちる

W6 スウォーム実行の成立条件は、judge 対象の爆発を抑えること。既に手札は揃っている:

- `ambiguity_pooling` — 曖昧判断は map の reviews / Qn pool に溜めて暫定決定で走る
- `pr-review-toolkit` — code-reviewer / silent-failure-hunter / test-analyzer / comment-analyzer / type-design-analyzer が先に全部目を通して主要問題だけ上げる
- `canvas-protocol` — agent が decision を map に書くので人間は差分だけ見れば良い

**AI がトリアージし、canvas に整列し、人間はランク済み短縮版だけを見る。**
judge の負荷は線形に増えない。

## 成長と損切りは別原理

「judge」を一括りにしていたが、実は**非対称な 2 種類**:

### 損切り判断
- 高頻度・低裁量・感情抵抗大
- 「これは 2 試行伸びない → 殺す」「この branch は塩漬け → 切る」
- ルールベース化しやすい → AI と相性が良い
- 本質はサンクコスト無視の速度

### 成長判断
- 低頻度・高裁量・孤独
- 「scope / alias は compound する抽象だ」
- コンセンサス形成前に見抜く必要があり、AI が苦手
- 本質は taste + 貫徹

**AI が肩代わりするのは損切り側**。成長側ではない。

## subPJ 高速ループ == ポートフォリオ理論

- 個々の subPJ の下方損失は小さく制限 (短期で切れる構造)
- 上方成長は compound させる (勝った subPJ の成果物はオントロジーに還流)
- 多数ベット・早期損切り・勝者放置 = Kelly 的最適

AI 損切りトリアージがある前提で、ベット数を 3–4 並列から 20+ 並列に拡張可能。

## 人間に残る真の希少スキル

W7 以降、akaghef の稀少性は**成長側の判断に集約**される:

1. どの抽象が compound するか（オントロジー設計）
2. どのジャーニーが主軸になるか
3. 非自明だが勝つ subPJ を早期に発見する目

**損切りは鍛えなくていい（AI がやる）。成長眼だけ磨く。**
これが次の数ヶ月の自己投資方針として最も効率が良い。

## 暴力的な美しさの正体

### 暴力性
「磨くな」と言い切ったこと。損切り眼を磨け、バランス良く両方だ、とは言わない。
**片方を捨てて片方に全振り**。
理由は「AI が片方を 0 コストで肩代わりするから、
人間がやる限界効用がゼロ」という、感情を挟まない経済計算。

### 美しさ
全部が 1 本の原理で繋がる:

```
worktree 分業
  → Canvas 共有
    → TDD
      → AI トリアージ
        → 成長/損切り分離
          → subPJ 高速ループ
            → ポートフォリオ理論
              → オントロジー設計への収斂
```

どれも独立に選んだはずなのに、後から見ると全部同じ方向を向いている。
計画してないのにこうなる時が一番危険で、一番速い。

## 唯一の未解決問い

**成長眼は鍛えられるのか、それとも既に持っているかだけか。**

これは投資の世界でも未解決。
M3E の dogfood は、それに対する akaghef 自身の実験でもある、という読み方ができる。
