# メタ機能 — M3E 自身のための M3E

M3E が **自分自身** を題材に扱うための機能群。
ドッグフード、自己改善ループ、フィードバック収集、テレメトリ、
プラグイン拡張、fork 容易化、までを横断するブレスト。

> 出典: `idea/00_topic_pool.md` カテゴリ O（O1〜O6）+ 既存メモリ
> `project_alglibmove_dogfood` `project_projection_vision`。

## 方針

- 採用判断はしない（複数案を並べる）
- 実装詳細は最後の MVP ファイルに参考程度のみ
- 「個人ツール vs 公開プロダクト」の二面性を常に意識する
  - akaghef 個人の生産性最大化が第一
  - その上で他者が使える/拡張できる土台を残せるか
- 既存の DEV map / project_alglibmove_dogfood / policy_privacy と接続する
- 「メタ」=自己言及領域なので、再帰・自己観察・自己改変の罠（infinite loop, 副作用）にも触れる

## 論点一覧

ID 付きの論点プール。後でユーザが「Mt-A2 を採用」のように1ID指定できる。

### A. メタ機能の存在意義
- Mt-A1. メタ機能は誰のため？（akaghef専用 / OSS化前提 / 両立）
- Mt-A2. メタ機能の優先度は本体機能とどう競合するか
- Mt-A3. 「メタ過剰」（メタばかりで本業が進まない）の回避線

### B. ドッグフード深化（O1）
- Mt-B1. 開発専用 workspace の独立度（既存 DEV map vs 専用 vault）
- Mt-B2. PR/Issue/レビュー履歴をマップ化する範囲
- Mt-B3. agent の自律タスクとマップ表現の対応
- Mt-B4. ドッグフード度合いを示す指標（自己定量化）

### C. 自己改善ループ（O2）
- Mt-C1. 改善提案の入口（scratch / 専用ノードタイプ / GitHub Issue）
- Mt-C2. 提案 → 設計 → 実装 → 検証 のどこまで自動化するか
- Mt-C3. AI による提案を「採用判定」する人間ゲートの位置
- Mt-C4. 改善提案ノード自体のメタデータ（impact, effort, risk）

### D. フィードバック収集 UI（O3）
- Mt-D1. 収集チャネル（in-app / GitHub / Discord / メール）
- Mt-D2. 収集トリガ（明示送信 / 操作失敗時 / 定期）

### F. プラグインマーケット（O5）
- Mt-F1. 拡張ポイントの形式（hook / iframe / sandbox JS / WASM）

- Mt-F3. 安全モデル（権限スコープ / 署名 / レビュー）

### G. fork 容易化（O6）
- Mt-G1. config と core の分離度

- Mt-G4. fork から upstream に貢献戻す経路

### H. メタ機能の副作用
- Mt-H1. 自己観察が行動を歪める（Hawthorne 効果）
- Mt-H2. 自己改変ループの暴走（無限提案）
- Mt-H3. メタ機能が本体 UX を圧迫

## キーメッセージ

1. メタ機能は **「akaghef の生産性最大化が第一、公開化はおまけ」** という個人ツール側の重力を前提にすると、O3/O4/O5 は一旦保留して O1/O2/O6 を先行する道筋が自然に見える。
2. **「M3E が M3E を改善する」自己改善ループ** は、project_projection_vision（半年で実用化）への近道にもなりうる（自分の改善要望をマップに集約 → AI が設計 → 自分が承認 → 実装のサイクル）。
3. ただしメタ過剰の罠（自己観察ばかりで本業が止まる）が常に隣にある。メタ機能には「予算上限」を最初から組み込むべき。
