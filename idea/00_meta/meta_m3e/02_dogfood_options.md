# 02. ドッグフード深化と自己改善ループ

サブトピック O1 / O2 を担当。
「M3E の開発を M3E でやる」+「M3E の改善提案を M3E に投稿する」の選択肢を並べる。

## 前提

- 既に DEV map で部分ドッグフード中（`feedback_agent_status_routine`）
- agent と人間が canvas 共有（`canvas-protocol` skill）
- triage / scratch / Todo_Pool が稼働中
- 既存メモリ `project_alglibmove_dogfood`: 21+ feature gaps が観測済み

## B. ドッグフード深化（O1）

### Mt-B1. 開発専用 workspace の独立度

| 案 | 概要 | 良い点 | 悪い点 | 推し度 |
|---|---|---|---|---|
| B1-a. 既存 DEV map 続行 | 現状維持、研究マップと同居 | 慣性ゼロ、追加コストなし | 「研究」と「開発」が混在しノイズ化 | 中 |
| B1-b. 専用 workspace 完全分離 | `/dev` workspace を別 vault | 関心の分離、研究マップが綺麗 | 切替コスト、メタ過剰の温床 | 中 |
| B1-c. 研究マップ内 subtree | 研究マップ配下に「dev/」サブツリー | 切替不要、横断検索可 | サブツリー肥大化 | 中高 |
| B1-d. ハイブリッド（symlink ノード） | 同一ノードを両 workspace から参照 | 重複なし | 実装複雑、conflict 危険 | 低 |

### Mt-B2. PR/Issue/レビュー履歴をマップ化する範囲

| 案 | 取り込み対象 | 入口 | 推し度 |
|---|---|---|---|
| B2-a. 何も取り込まない（現状） | なし | git log を直接読む | 中 |
| B2-b. PR タイトル + 状態のみ | `gh pr list` を定期 fetch | 軽量 | 中高 |
| B2-c. Issue/PR + コメント本体 | webhook / GitHub API 取り込み | リッチ | 中 |
| B2-d. レビュー指摘もノード化 | コメント1件1ノード | 細かすぎてノイズ | 低 |
| B2-e. 「決定」だけ手動でノード化 | レビューで決まったことを手動転記 | 高品質低量 | 高 |

> 推し: B2-b + B2-e の組み合わせ（自動の輪郭 + 手動の中身）

### Mt-B3. agent の自律タスクとマップ表現

| 案 | マップ表現 | agent ↔ map の方向 | 推し度 |
|---|---|---|---|
| B3-a. ノード = タスク、status で進捗 | 既存 status 属性活用 | agent → map (write) | 高 |
| B3-b. agent ごとにレーン（カンバン） | 縦軸=agent, 横軸=status | 視覚化強い | 中 |
| B3-c. 「Agent Status」ノード集約 | 全 agent の現状を1ノードに | 既存パターン | 高 |
| B3-d. agent の思考過程もノード化 | 各判断 → 子ノード | 過剰、ノイズ | 低 |
| B3-e. agent ログを attribute で添付 | 簡易、本文を汚さない | 後追い容易 | 中高 |

### Mt-B4. ドッグフード度合いを示す指標（自己定量化）

候補メトリクス:
- Mt-B4-1. **DEV map のノード更新頻度**（週次/月次）
- Mt-B4-2. **「M3E で発見 → M3E で対処」の循環本数**（自己改善ループ完了数）
- Mt-B4-3. **agent タスク完了率**
- Mt-B4-4. **「M3E を使った時間 / 開発した時間」比**
- Mt-B4-5. **メタ機能が本体機能を侵食していない比率**（本体 PR : メタ PR）

> どれも測れないと意味がないので、Mt-E1（何を測るか）と接続。

## C. 自己改善ループ（O2）

「M3E のここが不便」「こうしたい」を **M3E 自身に投稿** する仕組み。

### Mt-C1. 改善提案の入口

| 案 | 経路 | 摩擦 | 既存資産 |
|---|---|---|---|
| C1-a. scratch ノードに直接書く | 既存 m3e-scratch skill | 最小 | あり |
| C1-b. 専用ノードタイプ「Idea/Improvement」 | 専用 type で分類 | 中 | なし（新設） |
| C1-c. キーバインドで一発投稿（`I` キー等） | inbox mode 拡張 | 最小 | keyboard_modes |
| C1-d. ブラウザ拡張で M3E 操作中の任意箇所から | 任意ページ → 提案 | 中 | なし |
| C1-e. GitHub Issue として外出し | 公開向け | 中 | あり |
| C1-f. 音声「あ、ここ不便」即録音 | A1 音声入力と接続 | 最小 | 構想のみ |

> 個人ツール側なら C1-a / C1-c が最適。公開側なら C1-e。

### Mt-C2. 提案 → 設計 → 実装 → 検証 のどこまで自動化するか

| 案 | 自動化範囲 | 人間の関与 | 推し度 |
|---|---|---|---|
| C2-a. 提案収集のみ自動 | 入力 | 設計以降は全て人間 | 高 |
| C2-b. 提案 → 仕様化まで自動 | 入力 + 仕様 | 実装は人間 | 中高 |
| C2-c. 提案 → 実装ドラフト PR まで自動 | 入力 + 仕様 + コード | レビュー＋マージのみ | 中 |
| C2-d. 提案 → マージまで完全自動 | 全て | 異常時のみ介入 | 低（リスク） |

> akaghef の現状: dev-beta-data, dev-beta-visual, codex 等の subagent が既に C2-c に近い動き。
> ただしマップ起点ではなく Todo_Pool / 直接指示が起点。

### Mt-C3. AI による提案を「採用判定」する人間ゲート

| 案 | ゲート位置 | 自動却下条件 | 推し度 |
|---|---|---|---|
| C3-a. 提案受付段階で人間トリアージ | 入口 | なし | 高 |
| C3-b. 設計案の段階でレビュー | 中間 | 影響範囲広→自動保留 | 中高 |
| C3-c. PR レビュー段階のみ | 出口 | テスト失敗→自動却下 | 中 |
| C3-d. ゲートなし、commit 後事後確認 | なし | revert 容易性が前提 | 低 |

### Mt-C4. 改善提案ノード自体のメタデータ

属性候補:
- `impact`: 影響度（small/medium/large）
- `effort`: 推定工数（〜2h / 〜1日 / 〜1週）
- `risk`: 失敗時のコスト（reversible / partial / unrecoverable）
- `area`: 対象（viewer / data / agent / docs）
- `proposed_by`: 提案者（self / agent / external）
- `pool_status`: スレ状態（new / scoped / in-progress / done / dropped）
- `linked_pr`: 関連 PR 番号

> 既存 `Todo_Pool.md` のスキーマと近い。統合 vs 分離は Mt-B1 と連動。

## ドッグフード × 自己改善のシナジー

| 組み合わせ | 効果 |
|---|---|
| B1-c + C1-a | 研究マップ内 dev/ 配下に scratch で提案蓄積 → 単一マップで全完結 |
| B3-c + C2-b | Agent Status ノードに「次に手をつけるべき提案」を AI が表示 |
| B2-e + C4 | 「決定」だけ手動転記、メタデータで impact/effort 必須化 → 高品質ログ |
| B4-2 + C2-a | 「自己改善ループ完了数」をメトリクス化、達成感を維持 |

## 横断的気づき（このファイル限定）

- 既存パターンの **再ラベリング** で半分以上は実装済みになる可能性
  - DEV map ＋ Todo_Pool ＋ Agent Status ＝ 既にドッグフード基盤
- 不足しているのは **「改善提案の専用入口」と「メタデータ標準化」** の2点が大きい
- 「完全自動化（C2-d）」は誘惑的だが、akaghef が研究に使う道具を agent が勝手に変えるのは危険
- 推しの初期構成: **B1-c + B2-b/e + B3-a/c + C1-a/c + C2-b + C3-a + C4 全属性**
