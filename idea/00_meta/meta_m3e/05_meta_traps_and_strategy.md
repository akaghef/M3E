# 05. メタ機能特有の罠 / MVP 道筋 / 未決質問

横断的観察と、akaghef フェーズで実際に動かすときの優先順を扱う。
コードの参考スケッチも一部含める（ブレスト鉄則 2 に沿い軽量）。

## H. メタ機能特有の罠

### Mt-H1. 自己観察が行動を歪める（Hawthorne 効果）

「観察されている」と知ると行動が変わる現象。
M3E の場合、自己観察ダッシュボード（E4-b）を作ると、
「数字を上げるためにマップを触る」が起きうる。

対策:
- 観察結果を **暗黙にしておく**（ノルマ化しない）
- 主要メトリクスは「快適さ」など主観値も並べる
- 「全く触らない週」を肯定的に扱う UI

### Mt-H2. 自己改変ループの暴走（無限提案）

AI が「改善提案」を量産すると、人間の処理が追いつかず堆積する。
既に Todo_Pool で起きやすいパターン。

対策:
- 改善提案ノードに **TTL（time-to-live）** を持たせる
- 一定期間放置の提案は自動アーカイブ
- 「未処理上限」を設定し、超過時は新規提案をブロック
- AI 提案前に「同種の既存提案ある？」チェック必須

### Mt-H3. メタ機能が本体 UX を圧迫

設定画面・ダッシュボード・プラグイン管理が UI を埋め尽くす。

対策:
- メタ機能は **デフォルト非表示**、設定画面の一階層下
- M3E のメイン UI（マップ）はメタ機能を見せない
- 自己観察は「専用 workspace」に隔離

### Mt-H4. 自己定量化が研究の質と乖離

「ノード数が多い = 良い」「触った日数が多い = 良い」が成立しない。
研究の本質的な質と相関しないメトリクスを追ってしまう。

対策:
- メトリクスを目標化しない（観測のみ）
- 質的評価（半年に1回の振り返り）を併設
- 「最も価値ある10ノード」のような質的ラベル

### Mt-H5. メタ機能が「メタ過剰」を発生させる（自己言及罠）

メタ機能を作ること自体に没頭、本体機能と研究が止まる。
最大の罠。

対策:
- メタ機能の **PR 比率** を観測（メタ PR : 本体 PR ≤ 1:3）
- メタ機能着手前に「これで救う本体作業時間」を見積もる
- 1メタ機能 ≤ 1スプリント の上限

## MVP 道筋（akaghef フェーズ向け）

### Phase 0（既存 = 暗黙の MVP）
既に存在するメタ機能:
- DEV map によるドッグフード（B1-a）
- Agent Status ノード（B3-c）
- Todo_Pool による提案蓄積（C1-a 相当）
- canvas-protocol skill による agent ↔ map 連携
- triage / scratch による生インプット捕捉

### Phase 1（軽量メタ補強, 〜2週）
- **M1-1**: 改善提案専用の attribute セット導入（C4 全項目）
- **M1-2**: scratch 内に「improvement」タグで分類（C1-a 拡張）
- **M1-3**: GitHub Issue リンク UI（D1-b、ボタン1つ）
- **M1-4**: 軽量自己観察ノード（手動更新、E4-c）

### Phase 2（自己改善ループの最小実装, 〜1ヶ月）

### Phase 3（共同利用展開時, 半年〜）

- **M3-2**: クラッシュ時のオプトイン送信（D2-b + E2-e）
- **M3-3**: 設定ファイル分離（G1-b）— fork 準備

## 横断的気づき（全ファイル横断）

- メタ機能は **既に多くが実装済み**（DEV map, Agent Status, Todo_Pool 等）
  - 不足は「専用入口」「メタデータ標準化」「予算管理」の 3 つ
- O3/O4/O5（公開向け）は akaghef フェーズでは **意図的に保留** が正解
  - ただし O6（fork 容易化）の最低限（G1-b）は早期投資の価値あり
- メタ機能群はそれぞれ **「個人ツール 7 : 公開 3」のフィルタ** で優先度が逆転する
- 自己改善ループ（O2）は project_projection_vision の **加速装置** になりうるが
  - 同時に最大の罠（Mt-H5 メタ過剰）でもある
  - 「ループ完了数」をメトリクス化することで両義性を制御できる可能性
- **再帰美**（M3E が M3E を観察する workspace）は技術的に可能だが、Hawthorne 効果（Mt-H1）に注意
