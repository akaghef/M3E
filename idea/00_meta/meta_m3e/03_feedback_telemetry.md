# 03. ユーザフィードバック UI とテレメトリ

サブトピック O3 / O4 を担当。
「他者からのフィードバック収集」と「使用状況の自動計測」の選択肢を並べる。

両者ともに **「個人ツール 7 : 公開 3」の重み** を踏まえると優先度は中以下。
ただし O4 の自己観察ダッシュボード（akaghef 自身向け）は個人ツールでも価値あり。

## 前提

- 現状の M3E ユーザは事実上 akaghef 1名
- 共同研究者・指導学生に展開する可能性は中期的にあり
- policy_privacy: 機微情報は plaintext で扱わない、LLM 送信時の検閲も視野
- 既存の telemetry 機構は **無し**（白紙）

## D. ユーザフィードバック収集 UI（O3）

### Mt-D1. 収集チャネル

| 案 | 経路 | 摩擦 | 公開度 | 推し度 |
|---|---|---|---|---|
| D1-a. in-app フィードバックボタン | viewer.ts に常駐 UI | 最小 | クローズ | 高 |
| D1-b. GitHub Issue リンクのみ | リンクで外出し | 中 | フルオープン | 中高 |
| D1-c. Discord / Slack コミュニティ | 別アプリ | 高 | 半オープン | 低 |
| D1-d. メール（mailto:） | mailto: リンク | 中 | 1対1 | 中 |
| D1-e. 専用フォーム（Tally / Google Form） | 別タブ | 中 | クローズ | 低 |
| D1-f. M3E マップそのものを共有送信 | フィードバック用マップ | 高 | 段階共有 | 低（面白い） |

### Mt-D2. 収集トリガ

| 案 | きっかけ | 押し付け度 | 推し度 |
|---|---|---|---|
| D2-a. ユーザの明示送信のみ | ボタンを押した時 | 低 | 高 |
| D2-b. クラッシュ・エラー検出時にダイアログ | exception 検知 | 中 | 中高 |
| D2-c. 連続失敗（同じ操作 3 回失敗） | 検知 | 中 | 中 |
| D2-d. 起動 N 回ごとに「最近どう？」ダイアログ | 定期 | 高（うざい） | 低 |
| D2-e. 「このノードに対するフィードバック」コンテキスト | 右クリック | 低 | 中高 |

### Mt-D3. 収集粒度

| 案 | 含まれるデータ | プライバシー懸念 | 推し度 |
|---|---|---|---|
| D3-a. テキストのみ | 自由記述 | 最小 | 高 |
| D3-b. + 操作直前の URL/scope | コンテキスト | 低 | 高 |
| D3-c. + マップの構造 snapshot（テキストはマスク） | 構造 | 中 | 中 |
| D3-d. + マップ全体（暗号化なし） | 全データ | 高 | 低 |
| D3-e. + セッション録画（rrweb 等） | UI 操作再現 | 高 | 低 |
| D3-f. + コンソールログ | 技術情報 | 中 | 中高 |

> policy_privacy 的に D3-d / D3-e は地雷。D3-a + D3-b + D3-f が無難。

### Mt-D4. 公開ロードマップとの双方向リンク

- D4-a. ロードマップは無し（個人で管理）
- D4-b. GitHub Project / Milestones に紐付け
- D4-c. M3E の特定 workspace を「公開ロードマップ」として export（B5 関連）
- D4-d. フィードバック → 自動で「投票」できる UI（賛成/反対）
- D4-e. ロードマップに採否理由を返す（フィードバック送信者へ通知）

> akaghef フェーズ的には D4-a で十分。展開期に D4-b。

## E. テレメトリ（O4, オプトイン）

### Mt-E1. 何を測るか

| カテゴリ | 候補メトリクス | プライバシー懸念 | 価値 |
|---|---|---|---|
| 利用 | 起動回数、セッション長、アクティブ日数 | 低 | 中 |
| 機能 | 各 mode 利用回数、shortcut 使用頻度 | 低 | 高 |
| 規模 | ノード数、リンク数、workspace 数 | 中 | 高 |
| 性能 | 起動時間、render fps、保存時間 | 低 | 高 |
| エラー | 例外発生数、回復不能エラー | 低 | 高 |
| ノード内容 | テキスト長分布、attribute 種類 | 低 | 中 |
| **本文テキスト** | （絶対送らない） | 最大 | — |
| AI 利用 | LLM 呼び出し回数、token 数 | 中 | 中 |

### Mt-E2. どこに送るか

| 案 | 送信先 | コスト | 信頼 | 推し度 |
|---|---|---|---|---|
| E2-a. 何処にも送らない（local only） | local file | 0 | 最高 | 高（初期） |
| E2-b. 自分の GitHub Repo に PR で投稿 | repo | 低 | 中 | 中（面白い） |
| E2-c. 自前サーバ（Vercel + Postgres 等） | $ | 中 | 自管理 | 中 |
| E2-d. SaaS（PostHog / Plausible 等） | $ | 低 | 第三者依存 | 中 |
| E2-e. ユーザ自身が手動 export して送る | local + 手動 | 0 | 最高 | 中高 |

> E2-a + E2-e のハイブリッドが個人ツール期に最適。
> 公開後に E2-c か E2-d。

### Mt-E3. プライバシー設計（policy_privacy 連動）

設計原則:
- E3-1. 既定値は「送信しない」（オプトイン）
- E3-2. 送信内容を **送信前に画面表示** してユーザが確認できる
- E3-3. テキスト本文は決して送らない（構造・統計値のみ）
- E3-4. 送信ログを local に残し、ユーザが過去送信を見直せる
- E3-5. ユーザ ID は永続乱数 UUID（メアド等は使わない）
- E3-6. 削除リクエスト経路を用意

> L5（エージェント送信検閲）と設計が共通化できる。

### Mt-E4. 計測結果のユーザ自身への還元（自己観察ダッシュボード）

これは公開機能というより **個人ツール側の機能**。

| 案 | ダッシュボード形態 | データ源 | 推し度 |
|---|---|---|---|
| E4-a. 専用 viewer ページ（`/stats`） | 別ページ | local DB | 中高 |
| E4-b. M3E マップ内の専用 workspace「Self-Observation」 | マップ内 | local DB | 高（再帰的に M3E らしい） |
| E4-c. 起動時に「今週の自己観察」ノード | マップ内 | local DB | 中 |
| E4-d. CLI で stats 出力 | terminal | local DB | 低 |
| E4-e. 月次 PDF レポート自動生成 | PDF | local DB | 低 |

> 推し: E4-b。M3E のマップ自体が自己観察の場になる。再帰美。
> ただし「自己観察ばかりして本業止まる」リスク（Mt-H1, Mt-A3）。

## D × E の組み合わせ

| 組み合わせ | 効果 |
|---|---|
| D1-a + D3-a/b/f | in-app ボタン1つで送信、テキスト+context+log のみ |
| D2-b + E1（エラー） | クラッシュ時に自動で「送信しますか？」ダイアログ |
| E4-b + B4 | ドッグフード度合い + 利用状況を同じ workspace で観察 |
| E2-a + E4-b | 送信せず、自分のマップに自分のデータを蓄積 |

## 横断的気づき

- フィードバック UI は **「誰に対する設計か」が定まらない限り過剰投資になる**
  - akaghef フェーズ: D1-b（GitHub link）と D2-a（明示送信）で十分
  - 共同研究者展開期: D1-a + D2-b + D3-a/b
  - 公開期: D4 系も視野
- テレメトリは **「自分自身を観察するため」の機能** と捉え直すと、E4-b（マップ化）が個人ツール期から価値を持つ
- E1 で「何を測るか」決めずに先にダッシュボード作るのは典型的失敗パターン
- D3-e（セッション録画）は機能としては魅力的だが privacy 的に地雷、最後に検討
