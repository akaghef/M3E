# 04. プラグイン拡張と fork 容易化

サブトピック O5 / O6 を担当。
「他者が拡張する」「他者が自分用にコピーして改造する」の選択肢を並べる。

両者とも **akaghef フェーズでは優先度低**（公開期の課題）。
ただし「最初から拡張ポイントを意識した設計」は後で破綻を防ぐので、形だけは見ておく。

## F. プラグインマーケット（O5）

### Mt-F1. 拡張ポイントの形式

| 案 | 形式 | 安全性 | 表現力 | 実装容易さ | 推し度 |
|---|---|---|---|---|---|
| F1-a. 設定 JSON（宣言型） | キーバインドや色設定 | 最高 | 低 | 易 | 高（最初は） |
| F1-b. テンプレート文字列 | ノードテンプレ等 | 高 | 中 | 易 | 高 |
| F1-c. JS hook（同一プロセス） | 関数登録 | 低（任意コード） | 高 | 中 | 中 |
| F1-d. iframe 埋め込み | 別オリジン UI | 中 | 高 | 中 | 中 |
| F1-e. Web Worker / Sandbox JS | 隔離実行 | 中高 | 中高 | 難 | 中 |
| F1-f. WASM プラグイン | 言語自由 | 高 | 高 | 難 | 低 |
| F1-g. 外部プロセス（LSP 風） | サブプロセス | 高 | 高 | 難 | 低 |

> 段階導入: F1-a → F1-b → F1-c/d → F1-e の順が現実的。

### Mt-F2. 配布形態

| 案 | 配布経路 | インストール手順 | キュレーション | 推し度 |
|---|---|---|---|---|
| F2-a. コピペ（README に「ここに貼れ」） | README | 1ステップ | なし | 高（初期） |
| F2-b. git clone + シンボリックリンク | git | 中 | なし | 中 |
| F2-c. npm package | npm | 中 | npm 任せ | 中 |
| F2-d. 公式 GitHub Discussions / Wiki | GitHub | 中 | 緩い | 中高 |
| F2-e. 公式 store（独自） | store API | 易 | 強い | 低（重い） |
| F2-f. URL 指定で動的 fetch | URL | 易 | なし | 低（危険） |

### Mt-F3. 安全モデル

権限スコープの粒度候補:
- F3-1. **権限なし**（純宣言型のみ）
- F3-2. **読み取り権**（マップ内容を読める）
- F3-3. **書き込み権**（マップを変更できる）
- F3-4. **ネットワーク権**（外部 fetch 可）
- F3-5. **LLM 呼出権**（Anthropic API 等を叩ける）
- F3-6. **shell 実行権**（local script 実行）

| 案 | 権限モデル | UI | 推し度 |
|---|---|---|---|
| F3-a. 権限なし固定（純宣言型） | F3-1 のみ | 不要 | 高（初期） |
| F3-b. インストール時一括許可 | 全権限 | 1ダイアログ | 中 |
| F3-c. 権限細分化 + 都度確認 | F3-2〜6 個別 | 都度 | 中高 |
| F3-d. 公式署名のみ実行可 | 署名検証 | 自動 | 中 |
| F3-e. レビュー必須 store | 人手レビュー | 強キュレ | 低（重い） |

### Mt-F4. 「公式 vs コミュニティ」境界

| 案 | 境界 | コミュニティ参入障壁 | 推し度 |
|---|---|---|---|
| F4-a. すべて公式（プラグイン無し） | コア = 全機能 | — | 中（akaghef フェーズ） |
| F4-b. プラグイン仕様だけ公開、自作は自己責任 | 仕様のみ | 低 | 中高 |
| F4-c. 公式プラグイン + コミュニティ自由 | 二層 | 中 | 中 |
| F4-d. 全プラグインを公式レビュー | レビュー | 高 | 低 |

## G. 自分用 fork の容易化（O6）

「他者（または自分の別マシン）が M3E をコピーして自分専用にカスタマイズする」コスト最小化。

### Mt-G1. config と core の分離度

| 案 | 分離度 | カスタマイズ性 | 実装コスト | 推し度 |
|---|---|---|---|---|
| G1-a. 現状（混在） | 低 | 低 | 0 | — |
| G1-b. 設定ファイル分離（`config.json`） | 中 | 中 | 低 | 高 |
| G1-c. テーマ層 + 設定層 + core 層 | 高 | 高 | 中 | 中高 |
| G1-d. 完全 DI（依存注入） | 最高 | 最高 | 高 | 低（過剰） |

### Mt-G2. ブランド/UI の差し替え経路

| 案 | 経路 | 推し度 |
|---|---|---|
| G2-a. CSS 変数だけ書き換え | 高 |
| G2-b. テーマ JSON | 中高 |
| G2-c. ロゴ・タイトル設定可能 | 高 |
| G2-d. 完全な UI 構造書換（template engine） | 低 |

### Mt-G3. データ互換性の保証範囲

| 案 | 保証範囲 | 推し度 |
|---|---|---|
| G3-a. fork 同士でも import/export 可能 | 高 |
| G3-b. core schema は固定、拡張属性は自由 | 高 |
| G3-c. version 移行ツール提供 | 中 |
| G3-d. 完全な後方互換は無保証（破壊変更可） | 中（個人ツール期は OK） |

### Mt-G4. fork から upstream に貢献戻す経路

| 案 | 経路 | コミュニティ可能性 | 推し度 |
|---|---|---|---|
| G4-a. 通常の GitHub PR | 標準 | 中 | 高 |
| G4-b. CONTRIBUTING.md と Issue テンプレ整備 | 中 | 中高 | 中高 |
| G4-c. fork 統計ダッシュボード | 観測 | 中 | 低 |
| G4-d. fork 専用カンファレンス（架空） | コミュ醸成 | 高 | — |

## F × G の組み合わせ

| 組み合わせ | 効果 |
|---|---|
| F1-a + G1-b | 設定ファイル拡張だけで「拡張」になる、最も安全 |
| F1-c + F3-c | hook + 権限細分化、Obsidian 風の自由度 |
| F2-a + G4-b | 摩擦最小、コントリビュート経路だけ整備 |
| G1-c + G2-a/b/c | 教育用 fork 等への展開がしやすい |

## 公開 vs 個人ツールの優先

- **akaghef 個人ツール期（〜半年）**:
  - F: 何もしない（コア機能優先）
  - G: G1-b（config 分離）と G3-b（schema 固定）だけは「将来のため」に設計
- **共同研究者展開期**:
  - F1-a + F2-a で「設定共有」程度の軽量プラグイン
  - G1-c + G2-a/b で簡易ブランド差替
- **公開期**:
  - F1-c/d + F3-c + F4-c で本格プラグイン
  - G4-b で貢献経路整備

## 横断的気づき

- F/G ともに **「今やるべきこと」と「将来のために形だけ準備」の線引き** が重要
- 最も罠なのは F1-c〜f / F3-c〜e の「本格プラグイン基盤」を最初に作ること
  - 1ユーザのために 1ヶ月〜の投資、しかも誰も書かない
- 一方で **G1-b（config 分離）は最初から意識** しないと後で技術的負債になる
- 「fork 容易化」自体は FOSS 文化の規範であり、個人ツールでも README 整備くらいは早めに
- O5/O6 は「ブレストとしてアイデアを並べる」が、akaghef フェーズでは **意図的に保留** が正解
