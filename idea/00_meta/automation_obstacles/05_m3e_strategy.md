# 05. M3E の介入戦略

40+の障害と20の解決パターンの中で、**M3E が直接的に効ける領域** はどこか。
逆に **M3E が手を出すべきでない領域** はどこか。

ツールとしての立ち位置を明確にする。

## M3E の構造的強み

M3E は以下の特性を持つ:
- **マップ構造** = ノード + リンク + 階層
- **属性付与可能** = 任意の attribute をノードに乗せられる
- **エージェント連携可能** = ai_subagent を持つ
- **ローカルファースト** = vault 同期、プライバシー対応
- **マルチビュー化可能** = map_views で展開した通り
- **キーボード駆動可能** = keyboard_modes で展開した通り
- **スクリプタブル** = REST API / MCP server

これを踏まえると、M3E が **得意な役割** は:

### 役割1. 文脈ハブ（Context Hub）
- 散らばった情報の「集約点」になる
- 各ツールから M3E ノードへリンクを貼る
- ノードに「今やっていること」「決めたこと」「保留中」を集約
- 効く障害: C1 / C2 / D3 / D4

### 役割2. 思考の外部化装置
- 暗黙知を構造化されたノードに落とす場所
- 「思考の半製品」を捨てる場所
- 効く障害: A1 / A6 / E2

### 役割3. レビューキューの基盤
- AI 出力を一旦キューに積む場所
- キーボードモードで高速処理
- 効く障害: B1 / B7 / G2

### 役割4. プロンプト・テンプレ資産庫
- マップ自体が「再利用可能なプロンプト」のレジストリ
- 効く障害: D1 / B5 / I1

### 役割5. 痕跡・履歴の保存場所
- 既存 audit_log / backup を活かして provenance を残す
- 効く障害: E2 / G3 / J2

### 役割6. 観察ダッシュボード
- マップ操作・状態変化のメタ可視化
- insights skill との連携
- 効く障害: A4 / D5 / J1

### 役割7. 儀式の場
- 朝の確認 / 週次レビュー / 月次振り返りの **物理的な場所**
- 効く障害: A3 / D5 / J2

## M3E が手を出すべきでない領域

### NM1. 学外コミュニケーション
- メール、SNS、Slack の本質的な置換
- 既存ツールが優れている、戦って勝てない

### NM2. 文書最終化
- LaTeX 編集、Word 編集、図表作成
- 専用ツール（Overleaf / Word / Inkscape）に任せる

### NM3. データ計算・分析
- 統計、機械学習、シミュレーション
- Python/R/Jupyter に任せる

### NM4. リアルタイム協調エディタ
- Google Docs 級の同期編集
- 別領域、戦線拡大しすぎる

### NM5. プロジェクト管理 SaaS
- Linear/Asana/Jira の置換
- 機能を絞り「ハブ」として連携する方が現実的

### NM6. 文献データベース
- Zotero / EndNote の置換
- 連携で十分

## M3E 戦略候補

### Strat-1. ハブ型戦略（最小侵略）
- M3E は「すべての情報の集約地点」
- 他ツールに置換せず、リンクと文脈で繋ぐ
- 利点: 既存資産を捨てなくていい、習慣を壊さない
- 欠点: 統合のためのプラグイン開発が永遠
- 似た事例: Obsidian、Logseq、Roam の方向性

### Strat-2. 思考専用戦略
- 「思考の半製品を扱う」場所と割り切る
- 他ツールには進出しない
- 利点: スコープが明確、品質を上げやすい
- 欠点: 単独では完結しない、他ツールが必要

### Strat-3. AI 第一戦略
- AI 連携を最重視、AI と対話するインターフェースとして特化
- マップは AI 入出力の構造化された場
- 利点: 差別化、新世代ツールとしての立ち位置
- 欠点: AI ベンダーロックイン、コスト

### Strat-4. ライフ OS 戦略
- 研究も生活も含めて全部 M3E に集約
- 健康・財務・予定もノード
- 利点: 統合体験
- 欠点: 全部中途半端になりがち

### Strat-5. 研究プロセス特化戦略
- 研究の「思考」「決定」「振り返り」だけに特化
- 文献管理・執筆・データ分析は連携で
- 利点: 既存メモリ project_projection_vision と整合
- 欠点: 一般市場が小さい

**個人的見立て**: Strat-2 + Strat-5 のハイブリッドが現実解。
M3E = 研究思考の場。他は他に任せる。AI は補助役として深く統合。

## 優先実装の示唆

M3E が ◎ で **未実装または弱い** 機能を優先候補とする:

1. **A1/B2 構造化外部化のための「ノード型テンプレ」** — ノード作成時に型を選ぶと埋めるべき欄が出る
2. **B1 AI レビューキュー** — エージェント出力を一旦キュー化、keyboard_modes で高速処理
3. **C1/D1 文脈ハブとしての強化** — 外部ツールへのリンク強化、「これ何の話だっけ」コマンド
4. **D4 中断再開支援** — 「最後に開いてたノード」「最後の操作」自動保存
5. **D5 進捗可視化** — 既存 insights を発展、マップ操作の自動分析
6. **E2 出典管理** — ノード作成時に出典強制、自動メタデータ抽出
7. **G3 履歴ブラウジング** — backup を時系列で「触れる」UI（idea I1 と整合）

## 戦略的な「やらないことリスト」

明示的に M3E が **やらない** と決めることで、設計が明確になる:

- ❌ メールクライアント機能
- ❌ 文書編集（Word 級）
- ❌ 統計計算・データ分析
- ❌ 認証・権限管理 SaaS
- ❌ 共著者向けの協調エディタ
- ❌ プロジェクト管理 SaaS
- ❌ 完全自動論文生成（思考の核心は人間）
- ❌ 完全自動意思決定（責任は人間）

代わりに、M3E は **「思考の場・記憶の場・振り返りの場」** に集中する。

## 結論的に言えること

- M3E は **障害解決の万能薬ではない**。20+の障害の半数程度に効く
- 効くべき領域に集中投資し、効かない領域は外部ツール／儀式／諦めで対応
- 特に **「思考の外部化」「文脈ハブ」「レビューキュー」** が三本柱
- AI 統合は M3E の差別化要因だが、AI 中心ではなく **AI を使って人間を助ける** 設計を維持
- 「自動化しないこと」を尊重するツール設計（04_what_not_to_automate）が長期的には強い
