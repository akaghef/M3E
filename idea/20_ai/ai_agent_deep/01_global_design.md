# 01. 共通基盤 — 人格・自動化レベル・検閲・ベンダ切替

10 個のサブトピック（C1〜C10）すべてに横断する設計論点をまとめる。
個別機能の検討（02〜04）に進む前提条件のプール。

## G1. AI 人格モデル

「誰が話しかけてくるのか」を決める枠組み。

| 案 | 概要 | 良い点 | 悪い点 | 推し度 |
|---|---|---|---|---|
| G1-a 単一汎用エージェント | 1 体の「M3E アシスタント」 | UI 単純、学習コスト小 | 視点が単調、ペルソナ機能（C9）と矛盾 | △ |
| G1-b 固定ペルソナセット | 指導教員/査読者/学生/批評家を固定提供 | C3/C9 と整合、視点多様性 | 人格の使い分け学習が必要 | ◎ |
| G1-c ユーザ定義ペルソナ | プロンプト＋少量プロファイルを自分で作る | 個別最適、研究分野依存に対応 | 設定コスト高、品質ばらつき | ○ |
| G1-d 動的人格（マップ依存） | 開いている subtree から人格を推定 | 文脈密着 | 予測不能、デバッグ困難 | △ |
| G1-e ハイブリッド（b+c） | 既定セット+カスタム可 | 段階導入できる | 命名・管理に規約必要 | ◎ |

メモ: ペルソナはマップ内 **プロンプトノード**（P4 プロンプト資産化）として保存できると
バージョン管理・共有・引き継ぎが効く。

## G2. 自動化レベル設計

`automation_obstacles/02_solution_patterns.md` P2 の L0〜L4 を本ブレスト機能群に当てる仮置き。

| サブトピック | L0 手動 | L1 提案 | L2 複数案 | L3 自動＋監査 | L4 完全自動 |
|---|---|---|---|---|---|
| C1 Devil's Advocate | 旧来の自分批判 | ノードで「反論ください」ボタン | 反論案 N 件提示 | 反論を子ノードで自動付与 | （N1 違反のため不可） |
| C2 質問生成 | 自分で問う | 「次の問い候補」表示 | 候補を 5 件提示 | 質問ノードを自動生成 | （N7 違反のため不可） |
| C3 多視点 | 自分で複数視点 | 1 視点ずつ提案 | 全視点同時表示 | コメントを自動付与 | 不可 |
| C4 link 提案 | 手動リンク | リンク候補表示 | 候補リスト＋スコア | 仮リンク自動生成、人間確認 | 信頼閾値超で全自動 |
| C5 insight 抽出 | 振り返り | daily insight 提案 | テーマ別複数 insight | 自動でインサイトノード生成 | 不可（N9 内省領域） |
| C6 research assistant | 手動検索 | クエリ補助 | 文献候補 N 件 | inbox 自動投入 | 自動投入＋自動引用 |
| C7 要約階層 | 手動要約 | 要約候補表示 | 1 行/3 行/段落から選ぶ | 自動生成・キャッシュ | 自動更新 |
| C8 会話型クエリ | 手動 fuzzy 検索 | 質問→該当ノードリスト | 複数解釈の候補 | 自動でフォーカス移動 | （クエリ実行は L4 OK） |
| C9 ペルソナ | 自分で他人視点 | ペルソナ提案 | 複数ペルソナ同時 | コメント自動付与 | 不可 |
| C10 自走 AI | 手動掃除 | 朝に「掃除候補」提示 | 複数掃除案 | 夜中に proposal キュー化 | （N1 違反のため不可） |

**観察**: ほぼすべて L4 に到達不可能 / 望ましくない。
M3E は **L1〜L3 のグラデーション** に集中するのが現実解。

## G3. 送信前検閲（L5）

`policy_privacy.md` および topic_pool L5 の「機微情報を LLM に送らない」を
全エージェント呼び出しの前段に置く。

| 案 | 検閲方式 | 良い点 | 悪い点 |
|---|---|---|---|
| G3-a 正規表現＋辞書 | メアド/電話/患者 ID 等の固定パターン | 軽量・確定的 | 文脈型情報を取りこぼす |
| G3-b 軽量 local LLM 検閲 | 別 LLM が「送って良いか」判定 | 文脈考慮可 | 検閲器も誤判定する |
| G3-c ノード属性ベース | `private:true` 属性のノードは送らない | 明示的・監査容易 | 属性付与漏れリスク |
| G3-d Vault 分離（L3） | プライベート vault は AI 機能オフ | 確定的 | 横断検索が不可 |
| G3-e ハイブリッド（a+c+d） | 多層防御 | 漏れにくい | 構成複雑 |

**前提**: どの案でも、**検閲の判定ログ**（何を弾いたか）は audit_log に残す。

## G4. ベンダ切替アーキテクチャ

既存 `ai_infra.ts` には既に `AiProviderConfig` / `modelRegistry` がある。これを起点に。

| 切替軸 | 例 | 適用論点 |
|---|---|---|
| データ機微度 | 公開可→Claude / 機関内→自社 LLM / 機密→ローカル | P10 と直結 |
| 機能特性 | 長文要約→Claude / 高速分類→Haiku / 構造化抽出→GPT | コスト最適 |
| コスト感度 | バッチは安い batch API、対話は通常 | G5 と連動 |
| 可用性 | プライマリ落ちたら fallback chain | 既存 `fallbackChain` を活用 |
| 法域 | EU データは EU リージョン LLM | 規制対応 |

**観察**: 「機能ごとに最適 LLM を選ぶ」のは設定が爆発する。
**現実解は「データ機微度 3 段階 × 用途 2〜3 種」のマトリクス**。

## G5. コスト管理

| 戦略 | 概要 | 効きそうな機能 |
|---|---|---|
| G5-a プロンプトキャッシュ | システムプロンプト＋マップ要約をキャッシュ | C5/C8 のような「マップ全体読み」 |
| G5-b バッチ API | 自走 AI（C10）の夜間タスクを batch で | C10 |
| G5-c 段階的 LLM | 軽量で粗振り→重い LLM で精査 | C4 link 候補、C6 文献分類 |
| G5-d 結果キャッシュ | 同入力→同出力をローカル保存 | C7 要約 |
| G5-e ユーザ予算 | 月額トークン上限を設定、超えたら手動承認 | 全機能 |
| G5-f「無料モデル」レーン | local LLM フォールバック | 全機能 |

## G6. 出力検証スキャフォールド（P5 適用）

AI 出力を **そのまま採用しない** 仕組み。

- **確信度表示**: AI が出した提案に 1〜5 の確信度を付与
- **出典必須**: research assistant は引用なしの主張を禁止
- **diff プレビュー**: 自走 AI の変更案は必ず diff で見せる
- **「なぜそう言ったか」展開**: 1 クリックで根拠ノード/プロンプトを参照
- **チェックリスト埋め込み**: 反論コメントには「事実 / 推論 / 意見」のラベル必須

## G7. プロンプト資産化（P4 適用）

- マップ内に「prompt:」プレフィックスのノード or 専用 type
- 各エージェント呼び出しがどのプロンプトを使ったかを痕跡化
- プロンプトの履歴 / 派生 / 失敗例を同じサブツリーで管理
- 「このプロンプト使ったときに 5 回外した」がメタ認知できる（P19）

## G8. アンドゥ徹底（P15 適用）

- AI による全変更は **1 アクション = 1 アンドゥ単位** とする
- バッチ変更（C10 自走 AI）は **1 セッション全体を 1 ロールバック** で戻せる
- アンドゥ可能であることが、L3 への踏み込み心理障壁を下げる

## G9. 「やらない」基準（N1〜N15 運用）

各機能設計時に必ずチェックすべき自動化禁止領域:

| 自動化禁止領域 | 影響する候補 | 設計上の対応 |
|---|---|---|
| N1 思考の核心 | C5 insight, C10 自走 | insight は「気づきの種」のみ、結論は出さない |
| N2 重要な判断 | C1, C8 | go/no-go は人間が押す |
| N7 創発の場 | C2 質問, C5 insight | 質問は閉じず、開く |
| N9 自己観察・内省 | C5, C9 | 「あなたはこう感じている」とは言わない |
| N11 創造的逸脱 | C7 要約, C10 自走 | 「変わったノード」を勝手に正規化しない |
| N12 下手に書く体験 | C7 要約 | 下書きを勝手に「綺麗に」しない |

## G10. AI 出力の痕跡保存（P9 適用）

ノードに以下を自動記録（必須メタ）:
- `ai_origin`: どのエージェント／プロンプトで生成されたか
- `ai_model`: 使用 LLM とバージョン
- `ai_confidence`: 自己申告確信度
- `ai_human_reviewed`: 人間が確認済みか

これにより 6 ヶ月後に「これ AI が書いたんだっけ？」を確実に判別できる。

## 共通設計の組み合わせ規則

- **G1-e + G7**: ペルソナ自体をプロンプトノードで管理 → カスタム＋共有が両立
- **G3 + G4**: 検閲層→機微度判定→適切なベンダへルーティング、を 1 パイプラインに
- **G2 L3 + G6 + G8**: 自動付与＋検証＋アンドゥの 3 点セットなら自動化を許せる
- **G9 + G2**: N1〜N15 該当機能は L1 を上限とする内部ルール
