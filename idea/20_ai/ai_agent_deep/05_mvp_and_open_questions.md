# 05. MVP 候補・横断観察・未決質問

01〜04 を踏まえた **横断観察 / 最小実装案 / 未決質問**。
採否は決めない、が「最初に手を付けるなら」の見立てを置く。

## 横断観察

### Obs-1. 自動化レベルの実用域は L1〜L3

10 候補すべてで L4（完全自動）は不可 or 非推奨だった（02 G2 表）。
M3E の AI 機能は **L1 提案 / L2 候補列挙 / L3 自動＋アンドゥ** のグラデーションに集中する。
「全部自動でやってくれる AI」は最初から目指さない方が誠実。

### Obs-2. 「報告書を出す秘書」モデルが安全

特に C10 自走 AI で顕著。**夜中に書き換える小人** より **朝に diff を見せる秘書** の方が、
心理的にも技術的にも安全。L8-2（提案のみ）+ L8-4（diff レビュー）の組み合わせが現実解。

### Obs-3. 要約レイヤ（C7）がインフラ的に効く

C5 insight、C8 会話型クエリ、C10 自走 AI のすべてが
「マップ全体を LLM に読ませる」を前提とする。要約があると:
- コストが激減（プロンプトキャッシュ G5-a と相性良）
- 速度が上がる
- LLM の文脈窓を圧迫しない

→ **C7 要約は単独機能ではなく、他の AI 機能の基礎レイヤ** として位置付けると価値が倍増。

### Obs-4. ペルソナ＝プロンプトノード

C9 ペルソナ、C3 多視点、C1 反論スタイルは、すべて **プロンプトノードの育成問題** に還元できる。
G7（プロンプト資産化）を整備すれば、これら 3 機能の中身は **同じインフラ** で動く。
→ ペルソナ管理 UI（M27 Assign mode 拡張）が共通基盤になる。

### Obs-5. 検閲（G3）は MVP からスタブで入れる

L5（送信前検閲）を後付けすると壊滅する。
たとえ regex 1 行でも「LLM 呼び出し関数」に必ず通す層を置く。
MVP は G3-a（regex+辞書）でよい。後で G3-b/c/e に進化させる。

### Obs-6. ベンダ切替は機能単位ではなくデータ機微度単位

機能毎に最適 LLM を選ぶと設定が爆発する。
**「公開可 / 機関内 / 機密」の 3 機微度 × 「対話 / バッチ」の 2 用途** で 6 マスのマトリクスにする。
これで設定数を爆発させずに G4 の利点を取れる。

### Obs-7. 「やらない」境界が AI 機能設計の主要論点

N1（思考の核心）/ N7（創発の場）/ N9（内省）/ N11（創造的逸脱）/ N12（下手に書く体験）
を侵食しない設計が、すべての AI 機能で問われる。
**「AI に何をさせるか」より「させないか」のリスト** が先に必要。

### Obs-8. 検索と生成の二段構成

C8 会話型クエリと C6 research assistant の両方で、
**検索 first（既存ノード or 既存文献）→ 生成 fallback（推論・要約）** の順序が安全。
最初から生成すると幻覚（B1）と捏造の温床。

### Obs-9. AI 出力の痕跡保存はメタ認知の基盤

G10 で `ai_origin` / `ai_model` / `ai_confidence` / `ai_human_reviewed` を必須化すると、
6 ヶ月後に「これ AI が書いた / 自分が書いた」が確実に判別できる。
これは P19（失敗の許容と学習）に直結し、プロンプト改善のループを回す基盤になる。

### Obs-10. C2 質問生成は「日常使い」、C1 反論は「節目使い」

頻度設計の観点から:
- 高頻度（日次）: C2 質問生成、C8 会話型クエリ、C7 要約
- 中頻度（週次）: C4 link 提案、C5 insight、C6 文献調査
- 低頻度（月次）: C1 反論、C3 多視点、C9 ペルソナ、C10 自走の重い処理

→ MVP の優先順位はこの「期待頻度」で決めると自然。

## MVP 候補

### M1. 最小実装 3 候補

優先度の高い 3 つを最小スケッチ:

#### M1-A. Devil's Advocate MVP（C1）

| 項目 | 内容 |
|---|---|
| トリガ | ノード選択中に `D` キー（または右クリック→反論） |
| 入力 | 選択ノード text + 親 path（任意で 1 階層子孫） |
| 出力 | 反論 3 件を子ノードとして追加（ai_origin 属性付き） |
| ベンダ | 既存 `ai_subagent.ts` のパターン踏襲、新サブエージェント `devil-advocate` 追加 |
| 検閲 | regex 検閲（G3-a）必須 |
| 規模感 | 既存 topic-suggest と同程度（〜200 行） |
| アンドゥ | 1 操作で 3 子ノード全削除 |

**価値**: 研究の質に直接効く、効果が見える、ユーザに「AI が答えを出さない」原則を体験させる第一歩。

#### M1-B. 自動 link 提案 MVP（C4）

| 項目 | 内容 |
|---|---|
| トリガ | 手動キック（「リンク候補スキャン」コマンド） |
| 入力 | workspace 全ノード text |
| 出力 | scratch 配下に「link 候補」ノード（H2-a 案） |
| ベンダ | embedding model（軽量・ローカル可） |
| 規模感 | 〜300 行 + embedding キャッシュ |
| アンドゥ | 候補ノード一括削除 |

**価値**: 重複検出（D1）の入口にもなり、マップ衛生の基盤。

#### M1-C. 会話型クエリ MVP（C8）

| 項目 | 内容 |
|---|---|
| トリガ | コマンドパレット `Ctrl+K`（R4-b） |
| 入力 | 自然言語クエリ |
| 出力 | 該当ノード 3〜5 件をリスト＋クリックでフォーカス（R5-b + R5-a） |
| ベンダ | embedding 検索（fast）+ Haiku 等（解釈） |
| 検閲 | クエリ＋結果両方を G3 経由 |
| 規模感 | 〜400 行 + embedding index |
| アンドゥ | 不要（読み取り専用） |

**価値**: 既存マップの **発見可能性** が劇的に向上。書き込み無しなので心理的障壁低い。

### M2. ベンダ抽象化の最低要件

M1-A〜C を共通インフラで動かすための最低限:
- `callLLM(prompt, model, options)` 単一関数（既存 ai_infra.ts 拡張）
- 機微度タグ（`public` / `internal` / `private`）を引数化
- フォールバックチェーン使用（既存 `fallbackChain`）
- `audit_log.ts` への呼び出し記録（G10）
- コスト記録（token / 円換算）

### M3. 検閲層 MVP（G3-a）

最小スタブ:
- 正規表現リスト（メアド / 電話 / クレカ / マイナンバー / 既知の患者 ID 形式）
- 辞書（プロジェクト固有の禁止語句、ユーザ編集可）
- 検閲ヒット時はログに記録、API 呼び出しを中止 or マスク
- マスク方針は `***` か `[REDACTED]` を選択

→ これだけで完璧ではないが「全 LLM 呼び出しが必ずこの関数を通る」アーキテクチャが入る。

## 未決質問（最重要 5 件）

### Q1. ペルソナ管理の UI 場所

C9 ペルソナと C3 多視点、C1 反論スタイルの **切替 UI** をどこに置くか。
- pill バー / コマンドパレット / モード（M27 拡張） / ノード属性
- 候補は多いが、**1 つに決めないと UI が散らかる**

### Q2. 自走 AI の「どこまで書いて良いか」境界

C10 自走の権限レベル（L8-1〜L8-5）のうち **デフォルトをどこに置くか**。
- L8-2（提案のみ）安全だが価値が薄い
- L8-4（diff レビュー）価値高いがレビュー疲れ
- ユーザがまだ AI を信頼していない初期段階のデフォルトを決める必要

### Q3. 検閲（G3）の精度 vs 摩擦

G3-a regex は **誤検出 / 漏れ** があり、ユーザ体験を壊す可能性。
- 漏れ寄せ（過保護）: 「個人情報を含む可能性があるため送れません」が頻発 → AI 機能が使えない
- 摩擦寄せ（緩め）: 個人情報が漏れる事故
- どちらに倒すか、誰が決めるか（ユーザ設定 / プロジェクト設定 / Vault 設定）

### Q4. ベンダコスト透明化のあり方

G5 コスト管理で、ユーザにどう **コスト感** を見せるか。
- 操作ごとに「これは $0.003 です」表示は鬱陶しい
- 月次ダッシュボードでは事後すぎ
- 「予算 80% 消費」アラートのバランスは？

### Q5. 「AI に学ばせる vs 学ばせない」の境界

C9 ペルソナで実在の指導教員を再現するなら、**過去の発言ログを学習させる** ことになる。
- これはプライバシー的に正しいか
- 退職した同僚のペルソナを使い続けて良いか
- ペルソナの「同意」概念をどう扱うか

→ Q5 は技術問題というより **倫理問題**。policy_privacy 拡張が必要かもしれない。

## このブレストから次に派生しうるテーマ

idea/ で別フォルダを切るべき副産物:
- **idea/prompt_nodes/** — G7 プロンプト資産化を独立テーマで深掘り
- **idea/ai_audit/** — G10 痕跡保存と P19 失敗ログを独立テーマで
- **idea/embedding_layer/** — C4 / C8 共通の embedding index 設計
- **idea/persona_management/** — C3 / C9 のペルソナ UI を独立で
- **idea/sensitive_filter/** — G3 検閲を独立テーマで（policy_privacy と接続）

→ いずれも本ブレストの境界を超えるので、別途独立。

## 最後に: このブレストの位置づけ

C カテゴリ（AI/エージェント深化）は M3E のビジョン
（**世界モデル → 射影 → 科研費等の出力**）の **思考インフラ部分** に当たる。
- B（出力・公開）= 射影
- C（本ブレスト）= 思考インフラ
- A（入力・取り込み）= 世界モデルの素材

C を厚くすると、A → B のパイプラインが質的に強化される。
ただし C はそれ自体が複雑で、**「AI を入れすぎて思考の核心を侵食する」** リスクがある。
N1〜N15 の境界線は本ブレスト全体の通奏低音であり、
個別機能設計時に毎回チェックリストとして機能させるべき。
