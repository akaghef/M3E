# 05. MVP・戦略・横断観察・未決質問

協調機能の **戦略選択肢**、**MVP 候補**、**横断観察**、**未決質問** を集約する。
採否は決めない。S1〜S5 の戦略候補と M1〜M6 の MVP 候補を並べる。

## S. 戦略候補

### S1. ハブ型戦略（最小侵略）
- M3E は協調 SaaS にならず、**外部協調ツールと連携の中心** に座る
- Slack/メール/GitHub の議論を「このノードに紐付け」で集約
- 利点: 戦線拡大なし、既存習慣を壊さない
- 欠点: 統合先が無限に増える
- automation_obstacles/05 の Strat-1 と整合
- 推し度: ◎

### S2. 非同期レビュー特化戦略
- Google Docs と同期同編集で勝負しない
- **コメント・レビュー依頼・ハンドオフ trail** に集中投資
- 利点: 差別化明確、NM4 と整合
- 欠点: 同期が必要な場面で物足りなさ
- 推し度: ◎

### S3. AI 中継特化戦略
- 人 ↔ AI / AI ↔ AI のハンドオフ機構として極める
- 人 ↔ 人 はオマケ
- 利点: project_alglibmove_dogfood の認知層強化に直結
- 欠点: 共著者ニーズに弱い
- 推し度: ○

### S4. 軽量同期戦略
- プレゼンス + ロック + 編集ハイライト だけ
- 編集は逐次反映（CRDT 不要、後勝ち + バックアップ）
- 利点: 実装軽量、十分実用
- 欠点: 「同編集してる感」は薄い
- 推し度: ○

### S5. プレゼンテーション特化
- 発言権モード + フォロー + 共有スナップショット
- 研究ミーティング・指導・査読プレゼンに最適化
- 利点: 独自性高い
- 欠点: 範囲が狭い
- 推し度: △

### 戦略の組み合わせ案

| 組合せ | 性格 | 推し度 |
|----|----|----|
| S1 + S2 | ハブ役 + 非同期レビュー | ◎ |
| S1 + S2 + S3 | 上記 + AI 中継 | ◎（dogfood 価値最大） |
| S2 + S4 | 非同期 + 軽量同期 | ○ |
| S5 単独 | プレゼン特化 | △ |
| S1 + S5 | ハブ + プレゼン | ○ |

**個人的見立て**: **S1 + S2 + S3** のトリプル。
M3E の差別化が「思考の文脈に紐付いた非同期レビュー + AI 中継 + 外部ハブ」になる。

## M. MVP 候補

### M1. プレゼンス + ロック表示の最小実装
- UI: U1（ノード端ドット）+ U15-A（鍵アイコン）
- 既存 collab.ts / presence.ts 活用、UX 層のみ追加
- 工数感: 小（〜200行 viewer.ts + α）
- 推し度: ◎（即効性）

### M2. ノード添付コメント + 解決
- スキーマ D1-D を新設、SSE で配布
- UI: U6 + U9
- 工数感: 中（バックエンド + UI 両方）
- 推し度: ◎

### M3. レビュー依頼 → 受信トレイ
- スキーマ D2、UI: U10 + U11-A + U11-C
- AI エージェントを assignee にできる
- 工数感: 中
- 推し度: ◎

### M4. ハンドオフ trail（agent ↔ human）
- スキーマ D3
- 既存の Agent Status 運用を構造化
- UI: U13-A + U14-A + U14-D
- 工数感: 中
- 推し度: ◎（dogfood 即効）

### M5. 招待・共有リンク（read-only から）
- P4-A から、P5-B（トークン直配り）で開始
- cloud_sync.ts 拡張
- 工数感: 中
- 推し度: ○

### M6. 既存 subagent ワークフロー（visual/data/team）への適用
- M4 のハンドオフ trail を visual/data/team に強制適用
- 「Agent Status を書く」を「ハンドオフ作成」に置換
- 工数感: 小（規約 + skill 更新）
- 推し度: ◎

## 横断観察（Cross-File Observations）

### Obs1. 既存実装の「半分」を活かせる
collab.ts / presence.ts / conflict_backup.ts はすでに **協調の骨格** を持つ。
不足は **UX 層 + コメント/レビュー/ハンドオフのスキーマ** であり、
ゼロからの設計ではない。

### Obs2. 3軸を統一スキーマで扱える可能性
コメント・レビュー・ハンドオフは「entity → entity への意図伝達」という共通形を持つ。
人↔人 / 人↔AI / AI↔AI を **同じデータモデル + 異なる UI** で扱える。
collab.ts の entity モデルがすでにこの抽象化を達成している。

### Obs3. ハンドオフ trail こそ M3E 独自価値
他ツールにない: **思考の文脈ノードに紐付いた、人と AI を区別しない引き継ぎ trail**。
Slack/Linear/Jira は文脈と切れたチケット、Google Docs は文脈と紐付くがコメント止まり。
M3E は「文脈 + 構造化議論 + AI 統合 + 出典」を一体化できる。

### Obs4. 「やらないこと」を明示する効果
NM4 を尊重して同編集を諦めることで、設計が大幅にシンプルになる。
ノード単位ロック（楽観的または悲観的）で十分実用、CRDT 不要。
**「諦める」が最大の設計選択** になる典型例。

### Obs5. プレゼンス・通知は「ノイズ vs 気づき」のトレードオフ
全機能 ON だと邪魔、全 OFF だと協調にならない。
**default は控えめ + opt-in で強化** が筋。

### Obs6. dogfood が高速に効く領域
visual/data/team の subagent 連携に M4 ハンドオフ trail を導入すると、
**自分自身が即ベネフィシャリ** になる。dogfood 加速の中核候補。

### Obs7. プライバシーとの直交性
協調は **共有を増やす方向**、policy_privacy は **共有を絞る方向**。
両立には P9（マスキング）と P2-D（属性単位 ACL）が鍵。
LLM 送信検閲（既存 L5 方向）と同じ思想で。

### Obs8. キーボード優先設計の維持
keyboard_modes の哲学（マウス最小化）と整合させるには、
コメント・レビュー・ハンドオフの各操作を **モード化** する必要。
「Comment mode（C）」「Review mode（R）」「Handoff mode（H）」候補。

## 未決質問（Open Questions）

### Q1. 共著者の想定人数
- 1人〜3人（指導教員 + 共著者2名程度）か
- 5人〜10人（小規模研究班）か
- 11人以上（大規模プロジェクト、現実的か？）

人数が変わると ACL 設計・通知設計が大きく変わる。

### Q2. SaaS にするかローカル運用に留めるか
- ローカルファースト（vault 同期で間接共有）
- 専用サーバ運用（小規模 self-hosted）
- 完全 SaaS（ユーザー登録・課金）
- ハイブリッド

これが S1〜S5 の選択にも影響。

### Q3. AI エージェントを「人」として presence に表示するか
- Yes: 統一感、抽象化
- No: 人と AI を視覚的に明示分離
- Cn8 表に基づくと「軸ごとに UI を分ける」のが筋なので、表示も分離が筋か

### Q4. コメント解決は「誰でも」できるか「依頼者だけ」か
- 誰でも resolve OK: 軽い
- 依頼者のみ: 厳格
- AI コメントは AI が自分で resolve できるか

### Q5. ハンドオフが拒否された場合のフロー
- 自動再振り（次の候補へ）
- 依頼者に差し戻し
- 「保留プール」に入れて後で人が振り直す

### Q6. プレゼンス情報の永続化／ログ化はプライバシー的に許容されるか
- 「いつ誰がどこを見たか」は機微情報になりうる
- 完全揮発が安全側、だが履歴が活きる場面もある

### Q7. オフラインからの再接続時、どこまで「自動マージ」を信じるか
- 後勝ち + 自動バックアップ（楽）
- 必ず人が確認（厳格）
- AI に競合解決を委任

### Q8. 発言権モード（U12）の「権限剥奪」は心理的に許されるか
- プレゼンタが他者を read-only に強制する操作
- 学術的なヒエラルキー（指導教員 vs 学生）と組み合わさると微妙な問題

### Q9. 既存 subagent の Agent Status 運用を「ハンドオフ trail」に置換するか共存させるか
- 全置換: スッキリ、移行コスト
- 共存: 二重管理、混乱
- 段階移行: 安全、長期化

### Q10. レビュー依頼の「期限切れ」UI
- 自動でリマインド
- 期限切れで「失効」
- 自動で他のエージェントに振り直し

## 「やらないこと」リスト（automation_obstacles/05 整合）

このブレストでも以下は **明示的にやらない**:

- ❌ Google Docs 級の文字単位同期同編集（NM4）
- ❌ Slack 置換のチャットツール
- ❌ プロジェクト管理 SaaS（Linear/Jira/Asana 置換）
- ❌ 認証・ID 管理サービス（OAuth プロバイダ等）
- ❌ ビデオ通話・音声通話
- ❌ ファイル共有ストレージ（Dropbox/Drive 置換）
- ❌ 「全文検索 SaaS」「分析 SaaS」など独自 SaaS 機能

代わりに M3E は **「思考の文脈に紐付いた、人と AI を区別しない非同期協働の場」** に集中する。

## 結論的に言えること（仮置き）

- M3E の協調は **「ハブ役・非同期レビュー・AI ハンドオフ」の3点集中** が筋
- **既存 collab.ts / presence.ts は骨格としてかなり完成度が高い**、UX 層と
  コメント/レビュー/ハンドオフ trail の追加が現実的
- **NM4 を守ることで設計が大幅に簡略化** される。CRDT 不要、ノード単位ロックで十分
- dogfood 価値が最大なのは **M4 ハンドオフ trail** + **M6 既存 subagent 適用**
- 戦線拡大の誘惑（プロジェクト管理 SaaS 化等）に注意。**やらないリストを定期的に再確認**
