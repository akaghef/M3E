# 03. Runtime UX — UI / 操作系のバリエーション

協調の各モードを具体的にどう **見せる・触らせる** かを並べる。
採否は決めない。U1〜U18 を ID 付きで列挙、比較表で検討する。

## プレゼンス系（U1-U5）

### U1. プレゼンス点表示（ノード端ドット）
ノード右上に小さい色付きドット（最大3〜5人分）。
- 良い点: ノイズ少、構造視認を邪魔しない
- 悪い点: 画面外のプレゼンスは見えない
- 推し度: ◎（最小コストで効果大）

### U2. プレゼンスリスト（サイドバー）
画面右に「いまアクティブ: akaghef, 共著者A, agent-visual」のリスト。
- 良い点: 全体像が常に見える
- 悪い点: サイドバー消費
- 推し度: ○

### U3. ライブカーソル / フォーカスハイライト
他者がフォーカス中のノードをハイライト（点滅 or 枠線色）。
- 良い点: いまどこを見ているか即わかる
- 悪い点: ノイズ（複数人いると点滅多発）
- 推し度: ○（人数制限が必要）

### U4. フォロー（他人の視点に追従）
「akaghef を follow」ボタン → 自分の viewport が akaghef に追従。
- 良い点: プレゼンや指導で強力
- 悪い点: 不意のジャンプで酔う
- 推し度: ○（明示的 ON/OFF が必須）

### U5. 編集ハイライト
最近 N 時間以内に他者が編集したノードを淡い色で残す。
- 良い点: 「あ、ここ変わってる」即気づける
- 悪い点: 古い編集の色が残る判定が難しい
- 推し度: ◎

| ID | 主用途 | 視覚密度 | 推し度 |
|----|----|----|----|
| U1 | ノード単位の存在確認 | 低 | ◎ |
| U2 | 全体把握 | 中 | ○ |
| U3 | 集中作業中の他者位置 | 中 | ○ |
| U4 | プレゼン・指導 | 高 | ○ |
| U5 | 非同期での「変化察知」 | 低 | ◎ |

## コメント・注釈系（U6-U9）

### U6. コメント（ノード添付）
ノードに紐付くスレッド型議論。Google Docs 風。
- 良い点: 議論が文脈に乗る
- 悪い点: 数が増えるとノイズ
- 推し度: ◎

### U7. 注釈レイヤー（マップ上のフリー注釈）
ノードに紐付かないキャンバス上の付箋・矢印。
- 良い点: 構造を超えた指摘ができる
- 悪い点: ノードと紐付かないので所在が不明確になる
- 推し度: △

### U8. スレッド議論
コメントへの返信、ネスト1〜2階層。
- 良い点: 議論の流れがわかる
- 悪い点: 深いネストは可読性下がる
- 推し度: ○

### U9. 解決（resolve）/ 再オープン
「解決」ボタンでコメントを畳む、必要なら再オープン。
- 良い点: 完了感、ノイズ削減
- 悪い点: 解決済みを見つけにくくなる（フィルタが必要）
- 推し度: ◎

## レビュー系（U10-U11）

### U10. レビュー依頼ボタン
ノードコンテキストメニュー → 「レビュー依頼」→ 受領者選択（人 or エージェント）。
- 良い点: 1クリックで非同期依頼成立
- 悪い点: 受領者リストの管理 UX
- 推し度: ◎

### U11. レビュー集約ビュー（受信トレイ）
画面どこかに「自分宛レビュー: 3件」「自分発: 5件」を表示。
専用ビューで一覧化、状態フィルタ（pending/done）。
- 良い点: 漏れ防止
- 悪い点: 別ビューを設計する必要
- 推し度: ◎

### U10/U11 の組み合わせ案

| 案 | UI | 良い点 | 悪い点 |
|----|----|----|----|
| U10/11-A | サイドバーに小さく件数バッジ | 軽い | 詳細は別ビュー |
| U10/11-B | top-bar に通知ベル | 慣れた UI | スペース消費 |
| U10/11-C | 専用キーボードモード（Review mode） | 高速処理 | 学習コスト |
| U10/11-D | スライドショー風に1件ずつ | 集中処理 | 一覧性低い |

推し: A + C の組み合わせ（バッジ + Review mode）。

## 発言権 / プレゼン系（U12）

### U12. 発言権モード（プレゼンタ駆動）
- プレゼンタが「発言権を取る」→ 他者は閲覧のみ
- 自動でフォローされる（U4 強制 ON）
- プレゼン終了で全員に編集権が戻る

| 案 | 仕様 | 良い点 | 悪い点 |
|----|----|----|----|
| U12-A | 完全ロック（他は read-only） | 集中度高い | 厳しすぎる |
| U12-B | プレゼンタ追従＋コメントは可 | 議論しながらプレゼン | 注意散漫の可能性 |
| U12-C | 発言権の譲渡（マイク回し） | 双方向プレゼンに対応 | UI 複雑 |
| U12-D | 個人作業用「集中モード」流用 | 1人のときも有用 | 概念混乱の懸念 |

推し: U12-B 基本、U12-C オプション。

## ハンドオフ系（U13-U14）

### U13. ハンドオフ「投げる」UI
ノードコンテキストメニュー → 「Hand off to ...」→ 受領 entity 選択 + メモ入力。
- 良い点: 引き継ぎが明示的に残る
- 悪い点: 既存の「Agent Status を手動で書く」習慣との両立

| 案 | 仕様 | 推し度 |
|----|----|----|
| U13-A | コンテキストメニューから | ◎ |
| U13-B | キーボードショートカット（H など） | ○ |
| U13-C | ノード attribute「assignedTo」を変えるだけ | ○ |
| U13-D | 専用ハンドオフモード | △ |

### U14. ハンドオフ「受ける」UI（受信トレイ）
- 自分宛ハンドオフ一覧
- 「承認」「差し戻し」「保留」アクション
- ハンドオフ trail（誰→誰→誰）の可視化

| 案 | 仕様 | 推し度 |
|----|----|----|
| U14-A | 受信トレイビュー（DEV map に組込） | ◎ |
| U14-B | 別パネル（独立 UI） | ○ |
| U14-C | 通知ベルから一覧へ | ○ |
| U14-D | 朝一ダイジェスト（X2A-5 と統合） | ◎ |

推し: U14-A + U14-D 併用。

## ロック系（U15）

### U15. ロック表示
collab.ts の ScopeLock を可視化。
- ノード端に鍵アイコン、誰がロック中か
- 自分が編集を試みると「他者編集中、待機？」ダイアログ
- リース時間の残量表示

| 案 | 表示 | 推し度 |
|----|----|----|
| U15-A | 鍵アイコン + tooltip で詳細 | ◎ |
| U15-B | ノード自体を半透明グレーアウト | ○（強い） |
| U15-C | ロック取り合い UI（待機列） | △ |
| U15-D | ロックなし運用（後勝ち + 競合バックアップ） | ○（簡素） |

## 履歴 / 差分系（U16）

### U16. 過去差分の overlay
「先週のこのノードはどうだった？」を1クリックで表示。
- 良い点: 共同研究の履歴追跡
- 悪い点: UI が複雑化
- 推し度: ○（idea/I1 time travel と統合可能）

| 案 | 仕様 |
|----|----|
| U16-A | ノードに「履歴を見る」ボタン |
| U16-B | スライダーで時間軸操作 |
| U16-C | 差分アニメ再生 |

## 通知系（U17）

### U17. 通知
コメント・レビュー・ハンドオフの通知。

| 案 | チャネル | 良い点 | 悪い点 |
|----|----|----|----|
| U17-A | マップ内バッジのみ | 集中阻害なし | 見逃しやすい |
| U17-B | デスクトップ通知 | 即気づく | 邪魔 |
| U17-C | メール通知（重要のみ） | 確実 | 遅延 |
| U17-D | Slack/Discord 連携 | 既存ワークフロー | 依存増 |
| U17-E | DM（マップ内チャット） | 文脈直結 | スコープ拡大 |

推し: U17-A + U17-C を主軸、必要なら B を opt-in。

## オフライン・競合系（U18）

### U18. オフライン挙動と再接続時の競合 UI
- オフライン時の編集はキューに積む
- 再接続時に競合検出 → conflict_backup.ts を活かして「触れる」UI

| 案 | 仕様 | 推し度 |
|----|----|----|
| U18-A | 競合時に diff 表示してマージ | ◎ |
| U18-B | 自分の編集を一旦バックアップして、相手の最新を採用 | ○ |
| U18-C | 競合は別ノードに「枝分かれ」 | △ |

## 統合観察

### 観察1. キーボードモードとの相性
keyboard_modes に「Review mode」「Comment mode」「Handoff mode」を追加できる余地。
既存の Capture mode / Status mode と同列に並ぶ。

### 観察2. 「ノイズ」と「気づき」のトレードオフ
プレゼンス・編集ハイライト・通知すべて「気づきが増える＝邪魔も増える」。
**ON/OFF を簡単に切り替えられる** ことが必須。

### 観察3. ハンドオフ trail こそ M3E の独自価値
人 ↔ 人 / 人 ↔ AI / AI ↔ AI を **同じノード上に時系列で残せる** のは
他ツールにない特性。プロビナンス（誰が何をいつ）が思考の文脈に直結する。

### 観察4. プレゼン用途は「ハブ役」を超える可能性
発言権モード（U12）は研究ミーティングで重宝するが、それを支える機構は
通常の協調機能の延長で作れる。M3E がプレゼンツールに化ける入り口。

## UX 全体の論点

- Ux1. 協調 UI は default OFF で個人モードを邪魔しないか
- Ux2. キーボード操作優先設計は維持できるか（マウス中心になっていないか）
- Ux3. AI エージェントを「人っぽく」見せるか「ツールっぽく」見せるか
- Ux4. プレゼンス色の割り当て（誰が何色）の管理 UX
- Ux5. 通知の集中阻害をどう抑えるか（Do Not Disturb 連携）
