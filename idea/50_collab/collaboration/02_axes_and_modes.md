# 02. 3軸 × 同期/非同期 のモード列挙

協調を「人 ↔ 人」「人 ↔ AI」「AI ↔ AI」の **3軸** で整理し、
さらに **同期 / 非同期 / 半同期** で切る。
各セルにモード候補を並べる。採否は決めない。

## X1. 人 ↔ 人

### X1-Sync. 同期モード

| ID | モード | 内容 | 推し度 |
|----|----|----|----|
| X1S-1 | プレゼンスのみ表示 | 誰がどのノードを見ているか可視化、編集は逐次 | ◎ |
| X1S-2 | フォローモード | 他者の視点に追従、プレゼン用 | ○ |
| X1S-3 | 発言権モード | 1人だけ操作、他は閲覧 | ○ |
| X1S-4 | ノード単位ロック | 編集者がノードを「占有」、他者は読み取り | ◎ |
| X1S-5 | 軽量同編集 | text フィールドだけ文字単位同期（CRDT） | △ |
| X1S-6 | 完全同編集（Google Docs 級） | 全フィールド文字単位 | × NM4 |

### X1-Async. 非同期モード

| ID | モード | 内容 | 推し度 |
|----|----|----|----|
| X1A-1 | コメント添付 | ノードに議論スレッド | ◎ |
| X1A-2 | レビュー依頼 | ノードを「これ見て」と他者に投げる | ◎ |
| X1A-3 | 編集後ハイライト | 他者の最終編集を色で残す（24h） | ○ |
| X1A-4 | 差分ダイジェスト | 「あなたが寝ている間にこれが変わった」 | ○ |
| X1A-5 | 投票・スタンプ | コメントに賛同／反対を1クリック | △ |
| X1A-6 | 共有スナップショット | 特定時点のマップを read-only で配布 | ○ |

### X1-Semi. 半同期（ニアタイム）

| ID | モード | 内容 | 推し度 |
|----|----|----|----|
| X1H-1 | 通知付きコメント | 添付したら相手にデスクトップ通知 | ○ |
| X1H-2 | 「いま見てる？」ピング | 相手のプレゼンスを誘発 | △ |
| X1H-3 | レビュー期限タイマー | 24h 以内応答、過ぎたら別エージェント | ○ |

| ID | モード | 内容 | 推し度 |
|----|----|----|----|
| X2S-1 | エージェント実行中の進捗ストリーム | スピナー + 中間出力ノード | ◎ |
| X2S-2 | エージェントへの割込み | 「中止」「指示変更」を即送信 | ○ |
| X2S-3 | 共同作業（pair） | エージェントと akaghef が同じノードを交互に編集 | △ |
| X2S-4 | 観覧モード | エージェントが何を読んでいるか表示 | △ |

### X2-Async. 非同期モード（ハンドオフ系）

| ID | モード | 内容 | 推し度 |
|----|----|----|----|
| X2A-1 | dispatch（人→エージェント） | 「このノードを elaborate して」 | ◎ |

| X2A-3 | 完了通知（エージェント→人） | 受信トレイに表示 | ◎ |
| X2A-4 | 差し戻し（人→エージェント） | 受信トレイから「やり直し」 | ○ |
| X2A-5 | 自走報告（夜間 AI から朝） | 1日分まとめてダイジェスト | ◎ |
| X2A-6 | 質問エスカレーション（AI→人） | 「これは判断できない」を保留タグで投げ返す | ○ |

### X2-Semi. 半同期

| ID | モード | 内容 |
|----|----|----|
| X2H-1 | バックグラウンド継続 + 完了通知 | dispatch して別作業、完了で戻る |
| X2H-2 | 進捗マイルストーン通知 | 33%/66%/100% で通知 |
| X2H-3 | 信頼度低下時のみ介入要求 | confidence < 閾値で人を呼ぶ |

## モード組み合わせの観察

### 観察1. 同じデータモデルに乗る
コメント／レビュー／ハンドオフは **3軸とも同じスキーマ** で扱える可能性が高い。
人間 entity と AI entity を区別するのは表示層だけ、というのが既存 collab.ts の方針。

### 観察2. 同期度の差は「通知タイミング」
- 同期: 即 SSE push
- 半同期: 数秒バッファ後に通知
- 非同期: 受信トレイで自分のタイミングで開く

通知ポリシーを切り替えるだけで同じ機構を 3モードで使い回せる。

### 観察3. AI ↔ AI は既に運用中
DEV/Agent Status / tomd / backlog による非同期共有は既に動いている。
新規実装より「明示的な機構として整理する」方が筋。

### 観察4. プレゼン用途の特殊性
X1S-2 フォロー + X1S-3 発言権はプレゼン専用。
個人作業時の「集中ファシリ」モード（自分の視点を未来の自分に追従させる）にも転用可能。

## モード × 既存実装マトリクス

| モード | 既存実装 | 不足 |
|----|----|----|
| プレゼンス | `presence.ts` あり | UI 表示・揮発以外の保存 |
| ロック | `collab.ts` ScopeLock あり | UI 可視化、リース更新 UX |
| コメント | なし | スキーマ・UI・通知すべて |
| レビュー依頼 | なし | スキーマ・受信トレイ UI |
| ハンドオフ | DEV map で運用 | 形式化・履歴の構造化 |
| 完了通知 | なし | プッシュ機構 |
| 共有リンク | `cloud_sync.ts` 部分 | read-only / comment-only モード |
| 競合バックアップ | `conflict_backup.ts` あり | UI で「触れる」化 |

## 開発・運用上の論点

- Md1. プレゼンスを永続化するか揮発のままか（プライバシー観点）
- Md2. ロックの粒度（ノード単位 / subtree 単位 / フィールド単位）
- Md3. SSE 接続切断時のリカバリ
- Md4. AI エージェントを presence に出すか、別カテゴリで出すか
- Md5. 同期/非同期の境界 UX（「いま誰かいます」表示の閾値）
- Md6. モバイルでの協調は範囲外か、最小限だけ対応か
