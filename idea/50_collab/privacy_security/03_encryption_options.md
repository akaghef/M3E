# 03. 暗号化オプション — 何を、どの粒度で、どう暗号化するか

policy_privacy は **同期データの E2E 暗号化**（AES-256-GCM、`~/.m3e/sync.key` 1本）を既決。
本ファイルは **その上に** 追加する選択肢を並べる。
At-Rest（ローカル保存）と粒度の議論が中心。

## E1-E5. 暗号化の粒度

| ID | 粒度 | 何を暗号化 | 良い点 | 悪い点 |
|---|---|---|---|---|
| **E1** | ファイル全体 | `m3e-map.json` 全体を暗号化 | 実装単純、全部守れる | 起動時復号必須、検索しづらい、頻繁書込で I/O 重 |
| **E2** | vault 単位 | 特定 vault ファイルだけ暗号化 | 公開部分は普通に扱える | vault 分離設計と一体 |
| **E3** | subtree 単位 | 特定ノード以下を暗号化 | 既存 workspace 内で混在可 | 検索・list で穴あき |
| **E4** | ノード単位 | 各ノードを個別暗号化 | 細かい制御 | キー管理複雑、メタデータ漏洩 |
| **E5** | フィールド単位 | text / details / attribute だけ暗号化 | tags / position は検索可能維持 | 実装複雑、漏洩面増 |

### 比較表

| / | E1 | E2 | E3 | E4 | E5 |
|---|---|---|---|---|---|
| 実装コスト | 低 | 中 | 中 | 高 | 最高 |
| 検索 | × | △ | △ | △ | ○ |
| 部分共有 | × | ○ | ○ | ◎ | ◎ |
| 性能 | △ | ○ | ○ | △ | △ |
| メタ漏洩 | なし | 構造のみ | tags のみ | tags のみ | 多数 |

組合せ案:
- **E1 + E2**: 全体ベース暗号 + 機密 vault は別暗号
- **E2 + E5**: vault 単位 + 機密フィールドのみ追加暗号
- **E3 + E5**: subtree + 機密フィールド

## E6-E8. キー管理の選択肢

policy_privacy は「ローカルキー1本（`~/.m3e/sync.key`）」を既決だが、
At-Rest 拡張では複数キー案も検討候補:

| ID | キー方式 | 内容 | 良い点 | 悪い点 |
|---|---|---|---|---|
| **E6** | 単一キー | 1本のキーで全暗号 | シンプル、policy_privacy 整合 | キー紛失で全壊滅 |
| **E7** | vault 別キー | vault ごとに別キー | 部分漏洩で被害局所化 | キー管理 UI 必要、policy_privacy 拡張 |
| **E8** | per-node DEK | 各ノードに個別 DEK、マスターキーで包む | 最細粒度、共有しやすい | 実装複雑、keyring 必要 |

policy_privacy の精神（「複雑化なし」）を尊重するなら **E6 単一キー** がベース。
ただし **機密 vault は別キー** を許容する余地は残しておきたい（E7）。

### キー管理 UX 論点

- **K1. パスフレーズ** — キーファイルを開く時に毎回 / セッション初回 / 機密アクセス時のみ
- **K2. キーファイル保管場所** — ホームディレクトリ / external drive / hardware key
- **K3. キー復旧** — リカバリーフレーズを生成するか、しないか（しないと紛失で全壊）
- **K4. キーローテーション** — 定期入替を支援するか、しないか
- **K5. 複数端末配布** — どう安全に持ち運ぶか（QR / NFC / 紙 / 暗号化 USB）

| 案 | 良い点 | 悪い点 |
|---|---|---|
| **K-a パスフレーズなし** | 楽 | 端末紛失で即漏洩 |
| **K-b 起動時1回** | バランス | パスフレーズ忘れで詰む |
| **K-c S4 以上のみ** | 普段楽、機密だけ厳重 | 切替コスト |
| **K-d 操作ごと** | 最高安全 | 使いものにならない |

推し: **K-c S4 以上のみパスフレーズ**

## E9-E10. アルゴリズム選択

policy_privacy は **AES-256-GCM 既決**。
代替案を並べる:

| ID | アルゴリズム | 良い点 | 悪い点 | M3E 採用可否 |
|---|---|---|---|---|
| **E9** | AES-256-GCM | Node.js crypto 標準、高速、既決 | nonce 再利用に注意 | **既決** |
| **E10** | XChaCha20-Poly1305 | nonce 大きい、偶発再利用に強い | Node.js 標準外（追加依存） | policy_privacy 違反（外部依存） |
| **E10-b** | age (rage) | モダン、シンプル、ファイル指向 | 外部バイナリ必要 | policy_privacy 違反 |
| **E10-c** | libsodium | 包括的、安全 | 外部依存 | policy_privacy 違反 |
| **E10-d** | OpenPGP | 既存ツール互換 | 重い、複雑 | 採用しない |

policy_privacy の **外部依存なし** 制約を守るなら **E9 AES-256-GCM 一択**。
nonce 管理（再利用回避）の運用設計だけが追加論点。

## E11-E15. 暗号化を「どこで」やるか（実装 layer）

| ID | レイヤ | 内容 | 良い点 | 悪い点 |
|---|---|---|---|---|
| **E11** | sync API 層 | サーバー送信前に暗号化（既決の延長） | 既存設計と整合 | At-Rest は守れない |
| **E12** | save API 層 | ローカル保存時に暗号化 | At-Rest 守れる | 起動時復号、メモリ平文 |
| **E13** | UI 層 | 表示時に復号、操作時暗号化 | 細粒度、メモリ被曝最小 | 実装複雑、性能落ちる |
| **E14** | DB 層 | SQLite encryption 等 | 透過的 | 鍵管理 SQLite 任せ |
| **E15** | OS 層 | フルディスク暗号化に任せる | 実装ゼロ | OS 起動後は無防備 |

組合せ:
- 既決: **E11 sync 層** のみ
- 拡張案: **E11 + E12** で At-Rest も暗号
- 重装: **E11 + E12 + E13** でメモリ被曝も最小

## E16-E20. メタデータと暗号化

ノード本文を暗号化しても、**メタデータ漏洩** で攻撃者は多くを知れる:

| メタ | 漏洩で何がわかるか | 防御選択肢 |
|---|---|---|
| **E16** ノード ID | 構造、規模 | ID 乱数化（既存？） |
| **E17** タイムスタンプ | 活動パターン、不在期間 | timestamp 丸め / 暗号化 |
| **E18** タグ | テーマ、文脈 | 機密ノードはタグも暗号化 |
| **E19** リンク構造 | 関連性、思考マップ | リンクメタも暗号化 / 構造のみ漏洩許容 |
| **E20** ファイルサイズ | 内容量 | padding 注入（過剰） |

実用的には **E18 タグ暗号化** までが現実的、E19-E20 は諦めるラインか。

## E21-E25. 復号データのライフサイクル

復号後のデータがメモリに残る期間も論点:

| ID | 戦略 | 内容 | 良い点 | 悪い点 |
|---|---|---|---|---|
| **E21** | 永続復号 | 起動時に全復号、終了まで平文 | 性能◎ | メモリダンプで全壊 |
| **E22** | LRU 復号 | 最近触ったノードだけ平文、他は再暗号 | バランス | 切替コスト |
| **E23** | アクセス時復号 | ノード開いた時だけ復号、閉じたら破棄 | メモリ最小 | 性能△ |
| **E24** | 機微度別 | S1-S2 は永続、S3+ はアクセス時 | 合理的 | 実装複雑 |
| **E25** | session 終了時クリア | ブラウザタブ閉じたら全消去 | 自然 | リロードで再認証 |

推し: **E24 機微度別**（S1-S2 永続、S3+ アクセス時）

## E26-E30. 暗号化失敗時の挙動

意外と設計されない領域:

| ID | 状況 | 選択肢 |
|---|---|---|
| **E26** | キーファイルなし | 起動拒否 / 平文モードに降格 / 新規生成 |
| **E27** | 復号失敗（破損） | 該当ノード非表示 / 警告表示 / バックアップ復元提案 |
| **E28** | 暗号化失敗（書込時） | 保存中止 / 平文保存 / リトライ |
| **E29** | キー不一致 | 別 vault と判断 / エラー / 鍵入替 UI |
| **E30** | nonce 重複検出 | 警告 / キー再生成 / 無視 |

「**平文に降格しない**」が policy_privacy の精神。E28 は **絶対に保存中止 + 警告**。

## ストレージ別の暗号化考察

M3E が使う / 使う可能性があるストレージごとの扱い:

| ストレージ | 場所 | 暗号化方針候補 |
|---|---|---|
| `m3e-map.json` ローカル | ファイル | E1 全体 / E12 save 層 |
| `~/.m3e/sync.key` | ファイル | OS パーミッション + パスフレーズ（K2-K3） |
| IndexedDB (browser) | ブラウザ | E13 UI 層 / E14 DB 層 |
| sync server 上 | リモート | **既決 E11 + AES-256-GCM** |
| backup (.bak) | ファイル | 同期と同等暗号 |
| audit_log | ファイル | 機微情報を含めば暗号化 |
| search index | ファイル | E15 OS 層 / 機微 opt-out |

## キーポイント

- policy_privacy（外部依存なし、AES-256-GCM、ローカルキー1本）が上限制約
- 粒度は **E2 vault + E3 subtree** の組合せが現実的バランス
- キー管理は **K-c S4 以上のみパスフレーズ** が UX/安全のスイートスポット
- 復号データのライフサイクル（E21-E25）は議論されにくいが事故源
- **暗号化失敗時に絶対に平文降格しない** が policy_privacy の精神
- メタデータ漏洩は完全には防げない、E18 タグまでが実用ライン
