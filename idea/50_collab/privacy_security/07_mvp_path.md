# 07. MVP パス・横断観察・未決質問

01-06 で並べた選択肢から「最小限どこから始めるか」「横断的気づき」「未決の重要質問」を整理。
**実装の確定ではなく、優先順位の候補と未解決論点の棚卸し** が目的。

## Mv1-Mv6. MVP の組合せ候補

最小実装パッケージの選択肢:

### Mv1. 表示だけ MVP（最低限）

「機微度の見える化だけ、暗号化なし」スタート:

| 要素 | 内容 |
|---|---|
| 機微度 attribute | S1-S5 を `node.attrs.sensitivity` で持つ |
| 色帯表示 | U2 で表示 |
| 警告表示 | AI 送信時にラベル付ノードを警告 |
| 暗号化 | なし（policy_privacy 既決の sync 暗号のみ） |

**良い点**: 1日で実装可、ユーザーが慣れる
**悪い点**: 実質防御なし、誤って S5 を扱うと漏れる

### Mv2. 検出ベース MVP

「自動検出 + マスキングだけ、暗号化なし」:

| 要素 | 内容 |
|---|---|
| 検出 | M1 regex（メール / 電話 / API キー） |
| マスキング | Mk2 トークン化 + Mk4 restore |
| AI 送信ガード | Mk10 警告 + Mk12 dry-run |
| ローカル LLM | なし |

**良い点**: project_projection_vision に直結（科研費送信の基本ガード）
**悪い点**: At-Rest 無防備

### Mv3. Vault 分離 MVP

「workspace 拡張で物理分離」:

| 要素 | 内容 |
|---|---|
| Vault | W1 で既存 workspace を vault 化 |
| 切替 UI | VS6 常時表示 + VS3 ショートカット |
| 暗号化 | secret vault のみ E1 全体暗号 |
| 検出 | なし（ユーザー手動投入） |

**良い点**: policy_privacy 整合、シンプル
**悪い点**: 検出なしで誤投入リスク

### Mv4. 統合 MVP（理想）

| 要素 | 内容 |
|---|---|
| 機微度 | S1-S5、デフォルト S2 |
| Vault | 3 vault（public/work/secret） |
| 暗号化 | secret vault は E1、work は E2 |
| 検出 | M1 + M2 ユーザー辞書 |
| マスキング | Mk2 + Mk4 |
| AI ガード | Mk10 + Mk12 + Mk15 |
| UX | U2 色帯 + U13 プレビュー + U25 vault バー |
| ローカル LLM | LL3-c 明示切替（手動） |

**良い点**: 全層カバー
**悪い点**: 実装重い

### Mv5. ローカル LLM 重視 MVP

| 要素 | 内容 |
|---|---|
| 検出は最小 | M1 regex のみ |
| マスキングなし | 全部ローカル LLM 経由 |
| Ollama 連携 | LL3-a 全体モード |
| AI ガード | Mk13 ローカル退避自動 |

**良い点**: 機密問題を「送らない」で解決、シンプル
**悪い点**: 性能トレードオフ、ローカル LLM セットアップ要

### Mv6. 段階導入

| Phase | 期間 | 内容 |
|---|---|---|
| **P1** | 1週 | Mv1 表示のみ |
| **P2** | 2週 | + Mv2 検出 + Mk10 警告 |
| **P3** | 1ヶ月 | + Mv3 vault 分離 |
| **P4** | 2ヶ月 | + Mv4 統合 |
| **P5** | 3ヶ月 | + Mv5 ローカル LLM オプション |

推し: **Mv6 段階導入、P1-P2 を半年以内に**（project_projection_vision の科研費出力に間に合わせる）

## 横断観察 — ファイルを跨ぐ気づき

### O1. 暗号化と AI ガードは独立に設計

01_concept.md の C5（3層モデル）、03_encryption_options.md（Layer 1/2）、
04_masking_detection.md（Layer 3）が示す通り、**At-Rest 暗号化**と**AI 送信ガード**は
完全に独立。一方を持つだけでは他方の防御にならない。

混同事故例:
- 「暗号化してるから安全」→ 復号して AI に送ったら漏洩
- 「AI に送ってないから安全」→ ローカルファイルが OS 漏洩で漏れる

両方を別個の防御として設計する必要。

### O2. UX が物理層の最終防衛線

03 で暗号化、04 で AI ガード、05 で vault 分離を扱っても、
**画面に映ってる文字** は撮られたら防げない。
06 の U25-U30（視認性）が暗号より重要なケースが多い。

「どこに何が映ってる」を一秒で把握できる UI = 最大の物理防御。

### O3. 検出は「漏れ抑制」目的、完全防御は不可能

04 の検出（M1-M10）は誤検出 / 見落とし両方が出る。
**完全防御を目指すと使えない、UX 摩擦最小化を目指すと漏れる**。
だから「検出 + 警告 + ユーザー判断」（MS-c 段階信頼）が現実解。

### O4. policy_privacy の「複雑化なし」原則は強い制約

E10（外部依存禁止）、E6 単一キー推奨、K1 パスフレーズ慎重。
これを守ると **シンプルだが完璧ではない** 設計になる。
逆にこの制約があるから設計が定まる。

完璧主義に走らず、policy_privacy の精神を守る。

### O5. ローカル LLM が「機密 × AI」の唯一の正解

04 LL2 が示す通り、機密情報整理用途では **ローカル LLM が必須**。
推敲・アイデアはクラウドで補完。
「全部ローカル」「全部クラウド」の二択でなく、機微度別ルーティング（C4-c）が筋。

### O6. 段階分け（S1-S5）が他全部の軸になる

機微度分類（02 S1-S5）が:
- 暗号化粒度の決定（03 E2/E3 の選択基準）
- マスキング決定木（04 の決定木）
- vault 配置（05 V1-V5 の使い分け）
- 表示色（06 U2 の色帯）
- LLM 切替（04 LL3-b）

**全部の軸**になる。S1-S5 をユーザー自然に運用できる UX 設計が
最も投資効果が高い。

### O7. 内部脅威（誤操作）が最大被害

02 T13-T16 が示す通り、外部攻撃より自分のミスが最大事故源。
- 誤って S5 ノードに公開タグを付ける
- export で機密混入
- 過去 backup に残る
- AI に間違って送る

技術的暗号より **誤操作防止 UX** に投資すべき。

### O8. capture_ingest との一体設計

入口（capture）で機微度ラベル付与しないと、後から仕分けは困難。
capture_ingest/05_data_privacy.md の L1-L4 と本ブレストは
**入口と出口で一貫した privacy 設計** を持つ必要。

### O9. 監査ログは将来的に「研究倫理対応」で必須化の可能性

被験者データの送信履歴、共著者情報の AI 送信履歴、
これらが将来的に倫理委員会・所属機関から要求される可能性。
今から監査ログ（Mk15）の基盤を作っておく価値はある。

### O10. 切替 UX が一番難しい

vault 切替、ローカル / クラウド LLM 切替、機微度変更...
**切替が多い = ミスが多い**。
「いま何モード」を常時可視化（U17, U25-U28）するのが切替設計の鍵。

## 組み合わせ高効果ペア

### CO1. S1-S5 + Vault + 色帯
機微度 → vault 自動配置 → 色帯自動表示の連鎖。
ユーザーは「機微度ラベル付けるだけ」で残り全部自動。

### CO2. Mk2 トークン化 + Mk4 restore + 監査ログ
AI 推敲フローが文意保持で動き、何を送ったかも追跡可能。
project_projection_vision の科研費推敲の本命候補。

### CO3. ローカル LLM 自動切替 + 機微度ルーティング + 警告
LL3-b + C4-c + Mk10。ユーザーは普段意識せず、機密時だけ自動退避。

### CO4. Cleanup mode + 機密 hide + プレゼン
keyboard_modes の Cleanup と本ブレストの U11/U12 を組合せ、
学会発表時の事故防止。

### CO5. 視認性 + panic + readonly
U25-U30 + U9 + U10 で「同僚が来た瞬間に画面安全化」のフロー。

## 未決質問（最重要）

ユーザー（akaghef）に確認 / 判断を委ねたい論点:

### Q1. 機微度の段階数は何段階？
- 3段階（public/private/secret）か 5段階（S1-S5）か
- 段階多いと運用負荷、少ないと表現不足
- **判断材料**: 実際の研究情報をどれだけ細かく分けたいか

### Q2. ローカル LLM の前提性能はどれくらい？
- 8B 量子化で足りるか、70B が必要か
- 性能不足だと「結局クラウド使う」で機能しない
- **判断材料**: akaghef の現端末スペック、用途の許容性能

### Q3. policy_privacy の「キー1本」を拡張する是非
- 現 policy: `~/.m3e/sync.key` 1本
- 機密 vault に別キー（E7）を追加して良いか
- **判断材料**: 「複雑化なし」精神を破る価値ある拡張か

### Q4. 検出の精度 vs UX 摩擦のバランス
- 過剰検出（MS-d）で安全寄り vs 段階信頼（MS-c）で実用寄り
- **判断材料**: akaghef がどちらの誤りを許せないか

### Q5. 監査ログの保存範囲
- AI 送信プロンプト全文を保存するか、統計のみか
- 後の倫理対応 vs ストレージ膨張
- **判断材料**: 研究倫理委員会の現時点要求と将来予測

### Q6. デフォルト機微度は S1（公開）か S2（準公開）か
- デフォルト S1 = 使いやすいが事故リスク
- デフォルト S2 = 安全だが運用面倒
- **判断材料**: 研究者の典型的入力内容

### Q7. 既存ノードの遡及対応
- 過去 800+ ノードに後から機微度を付けるか
- 一括スキャン（DT4）で自動 + ユーザー確認 か放置か
- **判断材料**: 過去ノードに機密が紛れている確率

### Q8. vault 分離の物理性をどこまで上げるか
- V5 別 workspace のみで足りるか、V6 別 device まで必要か
- **判断材料**: 想定脅威（T1-T16）のうちどれを真剣に防ぐか

### Q9. AI 送信時の「最終確認」を毎回出すか
- 毎回出す = 安全 / 確認疲れ
- ノード機微度別 = バランス / 設計複雑
- **判断材料**: akaghef の集中作業頻度

### Q10. M3E が「主たる credential 保管庫」になるか
- API キー等を M3E に置く = 集約のメリット vs 1Password 等の専用ツールに任せる
- **判断材料**: 既存ツールへの満足度、集約価値

## キーメッセージ（横断）

1. **暗号化と AI ガードは別レイヤ、両方必要**
2. **UX 視認性が物理層の最終防衛線**
3. **誤操作対策が技術防御より重要**
4. **機微度分類（S1-S5）が他すべての設計軸**
5. **policy_privacy の「複雑化なし」精神を守ると設計が定まる**
6. **ローカル LLM が「機密 × AI」の必需品**
7. **段階導入（Mv6）で半年以内に P1-P2 着地が project_projection_vision に間に合う**
