# 04. マスキング・検出 — AI 送信前の機微情報フィルタ

policy_privacy が扱っていない **Layer 3（AI Pipeline）** の防御を扱う。
M3E 最大の未開拓領域、かつ project_projection_vision（科研費出力）に直結。

## M1-M10. 検出方式の選択肢

機微情報を **どう見つけるか**。ルール系と統計系を並べる。

### ルールベース系

| ID | 方式 | 検出例 | 良い点 | 悪い点 |
|---|---|---|---|---|
| **M1** | 正規表現 | メール / 電話 / 郵便番号 / クレカ / API キー | 単純、確定的、ローカル完結 | 誤検出多い、未知形式漏れ |
| **M2** | 辞書マッチ | 人名リスト / 機関名 / 既知の被験者 ID | 確定的、ユーザー追加可 | 辞書メンテ負荷 |
| **M3** | テンプレート | 「予算: XXX 万円」「氏名: XXX」 | 構造的、低誤検出 | 形式が崩れると外れる |
| **M4** | キーワード周辺 | 「パスワード」近傍の文字列 | 文脈考慮 | 偽陽性高い |
| **M5** | ファイルパス / URL | `\\server\path` / `localhost:` | 簡単 | 本質的にはマスク不要なものも |

### 統計 / ML 系

| ID | 方式 | 検出例 | 良い点 | 悪い点 |
|---|---|---|---|---|
| **M6** | NER（固有表現） | 人名・地名・組織 | 汎用 | ローカル動作要、精度ばらつき |
| **M7** | LLM 自己検出 | 「この文に機微情報ある？」を別 LLM に問う | 文脈考慮 | コスト・遅延・自己矛盾 |
| **M8** | embedding 類似 | 機密例との距離 | 学習可能 | 精度不明、説明性低い |
| **M9** | 言語モデル perplexity | 異常文字列（API キー等） | キー検出に強い | 適用領域狭い |
| **M10** | ユーザー学習 | 過去のマスク履歴を学習 | パーソナル化 | コールドスタート |

### 比較表

| / | M1 regex | M2 辞書 | M6 NER | M7 LLM | M10 学習 |
|---|---|---|---|---|---|
| 精度（個人情報） | 中 | 高（既知） | 高 | 最高 | 上昇 |
| 精度（研究特有 R1-R5） | 低 | 高（手動辞書） | 中 | 高 | 高 |
| 速度 | 最速 | 最速 | 中 | 遅 | 速 |
| ローカル完結 | ○ | ○ | △（モデル依存） | ×（クラウド LLM 矛盾） | ○ |
| 誤検出 | 多 | 少 | 中 | 少 | 中 |
| 説明性 | 高 | 高 | 低 | 中 | 低 |

組合せ案:
- **基本**: M1 + M2 + M3（ルール3点セット、ローカル完結）
- **追加**: M6 NER（ローカル小型モデル）
- **慎重**: M7 LLM 自己検出（高機微度時のみ、ローカル LLM）
- **進化**: M10 ユーザー学習（マスク確認履歴から辞書育成）

## Mk1-Mk8. マスキング処理の選択肢

検出した機微情報を **どう加工するか**。

### マスキング方式

| ID | 方式 | 例 | 良い点 | 悪い点 |
|---|---|---|---|---|
| **Mk1** | 全置換 | `田中太郎` → `[REDACTED]` | 完全に隠す | 文意が壊れる |
| **Mk2** | トークン化 | `田中太郎` → `[PERSON_001]`、対応表をローカル保持 | 文意保持、復元可能 | 対応表管理、トークン長による位置情報漏洩 |
| **Mk3** | 部分マスク | `田中太郎` → `田**` | 部分情報残し | 部分でも特定可能なケース |
| **Mk4** | placeholder + restore | 送信時 `[PERSON_001]`、結果受領後復元 | 文意保持、復元自動 | LLM が placeholder を変えると失敗 |
| **Mk5** | 構造化伏字 | `田中太郎`→`{name:1}`、ハッシュで復元 | 表現力高い | プロンプトインジェクション余地 |
| **Mk6** | 一般化 | `30万円` → `〜数十万円規模` | 自然 | 元の精度失う |
| **Mk7** | 削除 + 注記 | 該当行ごと削除し「機密削除」注記 | 安全 | 文意大破壊 |
| **Mk8** | ダミー差し替え | `田中太郎` → `山田次郎`（架空） | 完全に自然 | 復元時の対応保持必要、誤読リスク |

推し: **Mk2 トークン化 + Mk4 restore** の組合せ（文意保持＋復元可）

### マスキング決定木

```
検出された機微情報あり
├─ 機微度 S5: Mk7 削除 or 送信中止
├─ 機微度 S4: Mk2 トークン化 + Mk4 restore
├─ 機微度 S3: Mk3 部分マスク or Mk6 一般化
└─ 機微度 S2: 警告のみ、置換なし（ユーザー判断）
```

## Mk9-Mk16. AI 送信前ゲートの選択肢

検出 → マスキング後の **送信判断** をどうするか:

| ID | ゲート方式 | 内容 | 良い点 | 悪い点 |
|---|---|---|---|---|
| **Mk9** | 自動却下 | 機微あれば即送信中止 | 安全 | 使えない |
| **Mk10** | 警告 + 確認 | プレビュー表示、ユーザーが OK | バランス | 確認疲れ |
| **Mk11** | サイレント置換 | マスクして送信、警告なし | 摩擦最小 | 透明性低い |
| **Mk12** | dry-run | 「送ったらこうなる」事前表示 | 教育的 | 操作1段増 |
| **Mk13** | ローカル LLM 退避 | 機微検出時はローカル LLM 経路へ自動切替 | 摩擦最小 | 性能トレードオフ |
| **Mk14** | 段階送信 | 最小情報で試し、不足なら追加送信 | 漏洩最小化 | 性能落ちる |
| **Mk15** | 監査ログ | 送信内容を全部ローカル保存 | 事後追跡 | ストレージ増 |
| **Mk16** | 送信ホワイトリスト | 「このノードは送信可」を明示しないと送らない | 最高安全 | 操作負荷大 |

組合せ:
- **デフォルト UX**: Mk10 警告 + Mk12 dry-run + Mk15 監査ログ
- **高機微度時**: Mk13 ローカル LLM 退避 自動切替
- **S5 ノード**: Mk16 送信ホワイトリスト（明示許可なき限り送らない）

## L5. ローカル LLM 限定モード

00_topic_pool L5 を深掘り。研究機微情報と AI 統合の唯一の正解候補。

### LL1. ローカル LLM の選択肢

| 候補 | 種類 | 性能 | 機密性 | 実装コスト |
|---|---|---|---|---|
| **Ollama** | wrapper | モデル次第 | ◎（完全ローカル） | 低（プロセス連携） |
| **llama.cpp** | C++ | モデル次第 | ◎ | 中 |
| **MLX (Mac)** | Apple Silicon 最適 | 高（M2/M3） | ◎ | 中 |
| **WebLLM (in-browser)** | WebGPU | 中 | ◎（ブラウザ内） | 高 |
| **LM Studio** | GUI app | モデル次第 | ◎ | 低（local API） |

### LL2. ローカル LLM のユースケース

| ユース | クラウド LLM 必要性 | ローカルで足りるか |
|---|---|---|
| 文章推敲 | △（クラウドが質高） | ○（中規模で足りる） |
| 要約 | ○ | ○ |
| 翻訳 | ◎（クラウド優勢） | △ |
| アイデア出し | ◎ | △ |
| コード生成 | ◎ | △ |
| 質問応答 | ◎ | △ |
| 機密情報整理 | × | **◎（必須）** |
| 感情分析 / 整理 | ○ | ○ |

ローカル LLM は **「ハイエンドな知性は要らないが機密」** な用途で活きる。

### LL3. モード切替 UX

| ID | 切替方式 | 良い点 | 悪い点 |
|---|---|---|---|
| **LL3-a 全体モード** | アプリ全体を「ローカル LLM 限定」に | 単純、安全 | 全機能性能低下 |
| **LL3-b ノード機微度連動** | S4-S5 ノード操作時はローカル | 自動、合理的 | 切替が見えにくい |
| **LL3-c ユーザー明示** | 「いまローカル LLM」を選ぶボタン | 透明 | 操作負荷 |
| **LL3-d 機能別** | 「推敲はローカル、翻訳はクラウド」 | 細かい | 設定多数 |
| **LL3-e フォールバック** | クラウド失敗時にローカル | 可用性向上 | 機密性とは別問題 |

推し: **LL3-b ノード機微度連動 + LL3-c 明示インジケータ**

### LL4. ローカル LLM が現実に使えるか

研究用途で本当に使えるかの正直な評価:

| タスク | 現状（2026年）期待 |
|---|---|
| 8B 量子化モデルの推敲 | △ akaghef の文章レベルに届かない場合あり |
| 70B モデルの推敲 | ○ 16GB+ VRAM 必要、ノート PC 厳しい |
| エンベディング生成 | ◎ 軽量、機微情報の类似検索に有効 |
| 構造化抽出 | ○ プロンプト工夫で実用的 |
| 長文要約 | △ コンテキスト長制約 |

**現実的な落とし所**:
- ローカル LLM = 「最低限の整理・要約・抽出」
- クラウド LLM = 「高品質な推敲・アイデア」
- マスキング層 = 「クラウド使用時の機密ガード」
- の3点でカバー

## L3 系: 機微情報の自動検出（人名・住所・パスワード）

| カテゴリ | 検出方式 | 優先度 | 誤検出リスク |
|---|---|---|---|
| **D1. メールアドレス** | M1 regex | 高 | 低 |
| **D2. 電話番号** | M1 regex | 高 | 中 |

### 検出 UX

| ID | 表示方法 | 良い点 | 悪い点 |
|---|---|---|---|
| **DT1** | リアルタイム下線 | 入力中に見える | 集中阻害 |
| **DT2** | 保存時警告 | 集中保持 | 後で気づく |
| **DT3** | AI 送信時のみ | 必要時に集中 | 普段は無防備 |
| **DT4** | 定期スキャン | 過去ノード対応 | 重い |

## マスキング・検出の不確実性に対するスタンス

| 案 | スタンス | 良い点 | 悪い点 |
|---|---|---|---|
| **MS-a 自動信頼** | 検出に任せて自動マスク、警告なし | UX 摩擦最小 | 漏洩責任不明確 |
| **MS-b ユーザー最終判断** | 検出は提案、決定はユーザー | 透明、責任明確 | 確認疲れ |
| **MS-c 段階信頼** | 既知パターン（API key 等）は自動、曖昧（人名）はユーザー | バランス | 設計複雑 |
| **MS-d 過剰検出** | 疑わしきは罰、ユーザーが解除 | 安全寄り | 偽陽性多 |

推し: **MS-c 段階信頼**（M9 で確実な API キーは自動、M6 NER の人名は確認）

## 監査ログ（Mk15）の設計論点

| 論点 | 選択肢 |
|---|---|
| **AL1. 何を記録するか** | 送信プロンプト全文 / マスク後 / マスク前 / 統計のみ |
| **AL2. 保存形式** | 平文 / 暗号化 / 別 vault |
| **AL3. 保存期間** | 永久 / N日 / セッションのみ |
| **AL4. 検索 UI** | 提供する / しない |
| **AL5. 自分以外の閲覧** | 不可 / 監査者ロール |

研究倫理上、**「いつ何を AI に送ったか」のログ** は将来的に必須になる可能性。
（被験者データを送ってしまった場合の追跡など）

## キーポイント

- **ルール（M1-M5）+ NER（M6）+ ユーザー学習（M10）** の3層で検出が現実的
- マスキングは **Mk2 + Mk4 トークン化 + restore** が文意保持と安全性のバランス
- ゲートは **Mk10 警告 + Mk12 dry-run + Mk15 監査ログ** が標準 UX 候補
- ローカル LLM（L5）は「機密情報整理」用途で **本物の必需品**、推敲はクラウドで補完
- 検出 UI は **入力・送信・出力の多層防御**（DT2 + DT3 + DT5）
- 自動 vs 手動は **MS-c 段階信頼**（確実なものは自動、曖昧なものは確認）
- 監査ログは将来的に研究倫理対応で必須になりうる
