# 01. コンセプト — 何を守り、誰のために、どう違うか

M3E のプライバシー / セキュリティ機能は **「研究者の思考母艦に機微情報を同居させる」** ための設計。
追い出すのではなく、安全に同居させる方向で考える。

## C1. 「機微情報を M3E に入れる」を許容するか、追い出すか

| 選択肢 | 内容 | 良い点 | 悪い点 |
|---|---|---|---|
| **C1-a 全部追い出す** | 機微情報は別ツール（1Password / Notion E2E / 紙）に隔離 | M3E は安全。実装コスト最小 | 「思考の一元化」が崩れる。コピペ事故が増える |
| **C1-b 全部入れる** | あらゆる機微情報を M3E に集約、暗号化で守る | 思考母艦として完結 | 漏洩時の被害が最大。実装責任が重い |
| **C1-c レーン分離** | 公開 / 準公開 / 機密のレーンを明示分離して同居 | 入口ベースで事故抑制 | 設計複雑。誤レーン投入のリスク |
| **C1-d ハイブリッド** | テキストは入れる、認証情報・実名 DB は外部参照 | バランス良 | 線引きが曖昧化しやすい |

推し: **C1-c**（policy_privacy のクライアント暗号化前提なら同居可、ただし分離線が必要）

## C2. 既存ツール差分

| ツール | 暗号化 | AI 統合 | グラフ | M3E との差 |
|---|---|---|---|---|
| **Obsidian + Encrypt plugin** | ノード単位 | プラグイン依存 | グラフあり | M3E は AI 統合と E2E sync をネイティブ前提 |
| **Standard Notes** | E2E 標準 | 弱い | なし | M3E はグラフと AI 連携が固有強み |
| **Cryptee** | E2E 標準 | なし | なし | M3E は思考構造化に特化 |
| **Logseq + age** | ファイル単位 | 弱い | グラフあり | M3E は workspace 分離と AI ガードが先行 |
| **Notion** | サーバー側暗号 | 強い | 弱い | M3E はローカルファーストで主権あり |
| **1Password** | 認証情報特化 | なし | なし | 役割が違う、連携先候補 |

差別化軸:
- **A. グラフ + 暗号化 + AI ガード の3点セット** — 単独で揃う製品は少ない
- **B. ローカルファースト × E2E 同期** — クラウド主権の対立軸
- **C. 研究者ワークフロー特化** — 科研費・査読・実験データの実情を反映

## C3. 「漏洩しない確信」をどう作るか

技術的に暗号化されているだけでは確信にならない。
ユーザーが日々使う中で「大丈夫」と感じる UX 要素:

- **C3-a 視覚的フィードバック** — 暗号化済ノードに鍵アイコン、機微レーンに色帯
- **C3-b 操作前確認** — AI 送信前に「この内容を送ります」プレビュー
- **C3-c 監査ログ** — 「いつ何を送信したか」の閲覧
- **C3-d ドライラン** — 「もし送信したら何がマスクされるか」シミュレーション
- **C3-e 暗号化失敗の明示** — 暗号化できなかった場合「平文で保存しません」と拒否
- **C3-f panic ボタン** — 即時ロック、画面ブラック化（同僚が来た時）

## C4. ローカルファースト原則と AI クラウド送信の両立

M3E は **ローカルファースト** で機微情報を抱える。
しかし AI（Claude API 等）は **クラウド** にある。
この矛盾の解決方針:

| 方針 | 内容 | トレードオフ |
|---|---|---|
| **C4-a 全部クラウド AI** | 暗号化解除 → 送信 → 結果を暗号化 | 性能◎ / 機微情報露出 |
| **C4-b 全部ローカル LLM** | Ollama 等で完結 | 機微情報安全 / 性能△ |
| **C4-c 機微度別ルーティング** | S1-S2 はクラウド、S3-S5 はローカル | バランス◎ / 実装複雑 |
| **C4-d マスク後クラウド** | placeholder 化 → 送信 → restore | 性能○ / マスク漏れリスク |
| **C4-e ユーザー選択** | ノードごとに「送信可」フラグ | 透明 / 操作負荷 |

推し: **C4-c + C4-d の組合せ**（機微度で振り分け、中間レーンはマスク後送信）

## C5. 3層モデル（At-Rest / In-Transit / In-AI-Pipeline）

privacy 設計を3層で分離:

```
Layer 1: At-Rest        — ローカルファイル / IndexedDB / SQLite に保存中
Layer 2: In-Transit     — sync サーバーや別端末への転送中
Layer 3: In-AI-Pipeline — LLM への送信 / 結果の取込 中
```

それぞれ独立した防御:

| 層 | 既存 policy | 本ブレストで広げる範囲 |
|---|---|---|
| Layer 1 | （議論中、ローカルは信頼境界内とみなす説あり） | E1-E5 ノード単位暗号化を選択肢化 |
| Layer 2 | **AES-256-GCM 必須、`~/.m3e/sync.key` 1本** | 既決、変更しない |
| Layer 3 | （未定義） | M1-M10 / Mk1-Mk8 / L5 を新規定義 |

Layer 3 が **M3E の最大未開拓領域**。本ブレストの中心。

## ユースケース（誰が・いつ・どんな研究シーン）

### UC1. 科研費申請書を AI に推敲依頼したい（高頻度）
- akaghef が申請書ドラフトを M3E ノードに書く
- 予算金額・共同研究者氏名・研究機関名・未公開アイデアが含まれる
- AI に「文章を磨いて」と依頼したい
- そのままだと予算情報・人事情報が Anthropic サーバーに渡る
- → S3 機密扱い、Mk4 placeholder、または C4-c でローカル LLM 送り

### UC2. 被験者情報をマップに残したい（中頻度）
- 心理実験の被験者 ID / 個人特性 / 発言録音を扱う
- 倫理委員会承認の範囲を超えた漏洩は研究停止リスク
- M3E に入れたいが AI 経由で漏れたら大問題
- → V5 別 workspace（実験 vault）、L5 ローカル限定

### UC3. 査読中の他人論文の内容（中頻度）
- 査読中論文の議論を M3E でメモしたい（自分の理解を構造化）
- 「査読中」という事実自体が秘密、内容も秘密
- 半年後に出版されたら公開可
- → 時限暗号化（E期間後に自動公開化）が欲しい

### UC4. パスワード・API キー・シークレット（低頻度・高重要度）
- M3E 自身の dev で API キーをノードに書きたくなる時がある
- 1Password 等専用ツールがあるが、文脈と一緒に置きたい欲求がある
- 漏洩したら金銭被害
- → ノード単位 E1 暗号化、AI 送信絶対禁止フラグ

### UC5. 個人の感情・愚痴・人間関係メモ（高頻度）
- 共同研究者への不満、自分の弱さ、家庭の状況
- 漏洩したら人間関係破壊
- AI に相談したい時もある（推敲・整理）
- → S2-S3 中間レーン、Mk2 トークン化で人物名を匿名化

### UC6. 公開可な研究メモ（最高頻度）
- 公知の文献まとめ、自分の考えのうち発表予定のもの
- 暗号化不要、AI 自由送信可
- → S1 公開可、デフォルトレーン

### UC7. プレゼン中の画面共有（中頻度）
- 学会発表でマップを画面共有
- 隣のノードに機密情報が映ったら事故
- → Cleanup mode 拡張、機密ノードは「プレゼンモードで隠す」

## C5-extra. 「研究者特有の機微情報」5分類

一般的な PII（名前・住所・電話）に加え、研究者特有のものをリスト化:

| 分類 | 例 | 漏洩時被害 |
|---|---|---|
| **R1. 未発表アイデア** | 仮説、実験計画、論文ドラフト | 先取権喪失 |
| **R2. 査読・選考情報** | 査読対象、審査委員、選考過程 | 利益相反、関係破壊 |
| **R3. 被験者・被検体情報** | 個人識別子、特性、発言 | 倫理違反、研究停止 |
| **R4. 予算・財務情報** | 科研費額、人件費、機関間配分 | 競争劣位、政治問題 |
| **R5. 共同研究者の私的言動** | メール、雑談、内部不満 | 関係破壊、信頼喪失 |

これらは generic な NER（人名・住所検出）では捕れない領域がある。
M4-M10 の検出設計で **研究ドメイン辞書** を組み込む必要が出る可能性。

## キーポイント

- 「追い出す」のではなく「同居させる」設計が研究思考母艦の本質
- 既存ツールに対する差は **グラフ + 暗号化 + AI ガード** の3点セット
- ユーザーの確信は技術ではなく UX で作る（C3）
- AI 統合と機微情報の交点（Layer 3）が最大未開拓領域
- 研究者特有の機微情報（R1-R5）は generic 検出で捕れない
