# 01 Concept — なぜクロスデバイス・モバイルか

M3E を **PC 以外でも触る** ことの意味と前提を整理する。
研究者 akaghef の生活と研究シーンに即した「いつ／どこで／何故」を網羅。

---

## Cn1. PC 体験を 100% 再現すべきか、別物として割り切るか

| 案 | 良い点 | 悪い点 | 推し度 |
|---|---|---|---|
| **完全互換**（PC と同じ操作） | 学習コスト 0、機能差なし | タッチで使えない、画面狭すぎ | △ |
| **縮約互換**（同じ概念だが UI 縮小） | 移行スムーズ、レスポンシブで済む | 何もかも中途半端、両方使いにくい | ○ |
| **別物**（モバイル専用 UI 設計） | タッチ・移動シーンに最適化 | 学習コスト、開発コスト2倍 | ◎ |
| **モード別**（閲覧は互換、編集は別物） | 用途別最適 | モード切替の認知負荷 | ◎ |

**観察**: モバイルで PC 操作を再現してもキーボードショートカットが死ぬ。
M3E は keyboard_modes に依存度が高い → モバイルは別パラダイム必須。

---

## Cn2. 「いつ／どこで」触りたいか — シーンの優先度

研究者 akaghef のモバイル使用シーン候補（重要度の見立て）:

| シーン | 何をしたいか | 優先度 | 関連モード |
|---|---|---|---|
| 通勤電車 | 昨日のマップ確認、思いつきメモ | ⭐⭐⭐ | Md1 閲覧 / Md3 capture |
| 散歩・移動中 | 音声でマップを聞く、口頭メモ | ⭐⭐⭐ | Md4 読み上げ / Md10 音声会話 |
| ベッド・寝る前 | 軽く眺めて整理 | ⭐⭐ | Md1 閲覧 |
| 実験室・現場 | 実験結果を即捕捉、写真添付 | ⭐⭐⭐ | Md3 capture |
| カフェ作業 | タブレットで全機能 | ⭐⭐ | Md2 編集 |
| 学会・打ち合わせ | スマホで自分のマップを見せる | ⭐ | Md12 プレゼン補助 |
| 朝のチェックイン | 通知から「今日のノード」 | ⭐⭐ | Md5 通知 / Md9 今日のノード |
| トイレ・隙間時間 | 1ノードだけ見る、SRS 1問 | ⭐ | Md11 コーチ通知 |
| 出張・旅行 | オフラインで作業 | ⭐⭐ | Sy1 オフライン編集 |
| 家事中 | 耳で聞きながら考える | ⭐⭐ | Md4 読み上げ |

**観察**: 上位 3 シーン（通勤・散歩・実験室）は **入力・読み上げ・捕捉** が主。
編集の優先度は意外と低く、ベッドや散歩で深い編集はしたくない。

## Cn3. capture_ingest（A9）との分担

`idea/capture_ingest/` の A9（モバイル共有シート）は本ブレストと重複しうる。
明確に切り分けると:

| 領域 | capture_ingest | cross_device |
|---|---|---|
| **入力** | 全チャネル網羅（A1-A20） | モバイル UI 全般の中の Md3 |
| **モバイル特有** | A9（共有シート）のみ深掘り | 共有シート + 閲覧 + 編集 + 通知 + 読み上げ + 同期 全部 |
| **デバイス比較** | チャネル比較が主 | プラットフォーム比較が主（PWA/ネイティブ等） |
| **同期** | 触れない | Sy1-Sy10 で深掘り |
| **読み上げ** | 触れない | Md4 で深掘り |

**結論**: `capture_ingest/A9` は「**入口の 1 チャネル**」、
本ブレストは「**モバイル全体験**」と階層が違う。両立可能。

## Cn4. クロスデバイス同期はそもそもユーザー要件か

研究者 akaghef は **1 名運用**。複数人協調は当面不要（idea/ にも E. 協調がまだ 🌱）。
だが「同じ自分」が PC とスマホで触るので、**シングルユーザー・マルチデバイス同期** は必要。

| 案 | 良い点 | 悪い点 | 推し度 |
|---|---|---|---|
| **同期なし**（モバイルは閲覧専用） | 実装簡単、競合なし | モバイルで書いたものが消える | ○ |
| **片方向 sync**（PC → mobile スナップショット） | 競合ゼロ、読み専用なら安全 | モバイル編集できない | ◎ |
| **双方向 sync**（vault 連動） | 完全な体験 | 競合解決が必要、conflict_backup 拡張 | ○ |
| **モバイル専用 vault** | 分離で安全 | 知識が二重化、統合が手作業 | △ |

**推し**: 段階的に「読み専用 → capture のみ書込 → 全編集」と進めるのが筋。

## Cn5. PC 主・モバイル従の比率

設計判断の前提として、PC とモバイルの利用バランスを仮定する:

- **90:10 案** — モバイルは「補助・捕捉」だけ。投資は最小。
- **70:30 案** — モバイルで閲覧・読み上げも積極。MVP 推し。
- **50:50 案** — タブレットで本格作業。手書き含む。
- **30:70 案** — モバイルファースト（akaghef のライフスタイルに依存）

**推し**: 70:30。akaghef の現状（PC 中心の研究者）から、モバイルは
「**捕捉 + 閲覧 + 読み上げ + 通知**」の 4 機能で 30% を占める設計。

## ユースケース具体化（5シーン以上）

### UC1. 通勤電車での「昨日の振り返り」
状況: 7:30 朝の電車、座れてスマホ。
動作: アプリ起動 → 昨日触ったノード一覧 → 1 つずつ眺めて軽く編集 / コメント追記。
鍵: ロード速度、片手操作、戻る/次へが大きいボタン。

### UC2. 散歩しながらの「マップを聞く」
状況: 18:00 散歩中、Bluetooth イヤホン。
動作: 「今週の done を読み上げて」と音声起動 → TTS で連続再生。
気になったら「これコメントして」と口頭で追記。
鍵: hands-free、音声 UI、復帰時の差分マージ。

### UC3. 実験室での「結果即捕捉」
状況: 14:00 実験中、片手は装置、片手はスマホ。
動作: 写真撮影 → 即 scratch ノード化（OCR で値を抽出） → 数式や数値を音声で追加。
鍵: 1 タップ起動、写真→ノード自動化、後で triage。

### UC4. ベッドで「明日の準備」
状況: 23:00 寝る前、横になってスマホ。
動作: 「明日の予定」ビュー表示 → 優先度確認 → 軽く並び替え。
鍵: 暗いテーマ、文字大きめ、誤タップ防止。

### UC5. 学会会場での「マップを見せる」
状況: 13:00 学会の懇親会、立ち話。
動作: 自分のマップの研究テーマ部分を相手のスマホに「見せる」。
QR/URL/AirDrop でスナップショット共有。
鍵: 一時公開リンク、機微情報マスク、見せる用ビュー。

### UC6. オフライン出張先での作業
状況: 飛行機内、機内 Wi-Fi なし。
動作: 事前 sync しておいたマップをオフライン編集 → 着陸後 vault に反映。
鍵: 完全オフライン動作、競合解決 UX。

### UC7. ウォッチでの「リマインド受け」
状況: 15:00 デスクワーク中、Apple Watch 振動。
動作: 「今日触っていないノード: 〇〇」通知 → 1 タップで「後で」or「今見る」。
鍵: 通知設計、タップ範囲、振動の意味付け。

## 既存ツール差分

| ツール | モバイル対応 | M3E が学べる点 / 差別化点 |
|---|---|---|
| **Obsidian Mobile** | ネイティブ、vault 同期あり | vault 連動の設計参考、ただし graph 表示が遅い |
| **Logseq Mobile** | PWA + ネイティブ | PWA でどこまでやれるかの実例 |
| **Roam Research** | web のみ、モバイル弱い | 反面教師、Graph UI のモバイル難 |
| **Notion** | ネイティブ、よくできてる | 編集 UX の参考、ただし graph なし |
| **Apple Notes / Drafts** | ネイティブ、捕捉特化 | F2 capture モード参考 |
| **Bear** | ネイティブ、軽量 | UX 軽快さの参考 |
| **Heptabase** | iPad アプリあり、white board 風 | F6 手書き / タブレット参考 |
| **Voice Notes / Otter** | TTS / STT 特化 | Md4 読み上げ・音声会話の参考 |
| **MindNode / iThoughts** | iOS/Mac、マインドマップ | グラフのタッチ操作 UI 参考 |

**M3E の差別化軸**:
- **思考の継続性**: PC で書いたものを散歩中に耳で聞き直せる（Md4）
- **研究文脈統合**: 実験データ → 即マップ化 → 後で論文化（B1 と接続）
- **AI 思考パートナー**: モバイルでも Devil's Advocate / Coach（Md10/Md11）

## 横断的観察

- **モバイルは「機能拡張」ではなく「シーン拡張」** — できることを増やすのではなく、
  使える時間と場所を増やす設計が筋。
- **完全互換を目指すと PC 体験が劣化する** — モバイル都合で PC UI を妥協する罠あり。
  モード切替で **別物** にする方が安全。
- **読み上げと音声入力は研究者の「移動時間」を労働化する** — 競争優位の源泉になりうる。
- **オフラインは飛行機・地下鉄の研究者にとって死活問題** — 海外学会・出張多いと特に。
- **手書き（F6）はタブレット限定で別ブレスト推奨** — スマホでは難しい。
