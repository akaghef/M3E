# 05 MVP Path — 最小実装案・横断観察・未決質問

これまでの 4 ファイルを横断し、「最小投資で最大価値」の MVP 候補を絞る。
採否決めず複数案、ただし推し度は明示。

---

## Mv1. どの 1-2 シーンから始めるか

Cn2 の優先度上位（通勤・散歩・実験室）を改めて MVP 候補に並べる。

| 案 | シーン | 必要モード | 必要プラットフォーム | 工数 | 推し度 |
|---|---|---|---|---|---|
| **MVP-A: 通勤閲覧** | 電車で昨日のマップ確認 | Md1 + Md9 | Pf1 PWA | ⭐ | ◎ |
| **MVP-B: 散歩 capture** | 思いつき即捕捉 | Md3 | Pf1 PWA + Pf6 Shortcut | ⭐⭐ | ◎◎ |
| **MVP-C: 散歩読み上げ** | マップを耳で聞く | Md4 | Pf1 PWA + Web Speech | ⭐⭐ | ◎ |
| **MVP-D: 実験室 capture** | 写真+音声 scratch | Md3（写真+音声） | Pf1 PWA | ⭐⭐⭐ | ○ |
| **MVP-E: 朝の通知** | 7:00 今日のノード | Md5 + Md9 | Pf1 PWA + 通知 | ⭐⭐ | ○ |
| **MVP-F: 学会プレゼン** | スマホで他人に見せる | Md12 | Pf1 PWA + 一時 URL | ⭐⭐ | △ |

**推し**: **MVP-B（散歩 capture）+ MVP-A（通勤閲覧）** の 2 本立て。
合計 1-2 週で形になる、akaghef のシーン上位 2 つをカバー。

## Mv2. PWA か、レスポンシブ web で済むか

| 案 | 内容 | 工数 | 価値 |
|---|---|---|---|
| **Mv2.a レスポンシブのみ** | CSS でタッチ最適化 | ⭐ | ホーム画面追加でアプリ感 |
| **Mv2.b PWA 化** | Service Worker + manifest | ⭐⭐ | オフライン + 「インストール」 |
| **Mv2.c PWA + 通知** | Push 通知も実装 | ⭐⭐⭐ | Md5 通知モード可能 |
| **Mv2.d Capacitor で WebView 化** | ストア配布 | ⭐⭐⭐⭐ | iOS 制約解除 |

**推し**: **Mv2.b PWA**。レスポンシブだけだと体験が弱い、Capacitor は MVP 過剰。

## Mv3. capture vs read どちらが先か

| 案 | 順序 | 理由 |
|---|---|---|
| **capture 先**（書込先） | mobile → PC のみで済むので競合最小 | ⭐⭐ akaghef の最頻シーン |
| **read 先**（PC → mobile スナップショット） | 競合ゼロ、データ整合性問題なし | ⭐⭐⭐ 安全側 |
| **同時** | MVP-A + MVP-B 並行 | 1-2 週で両方可能 |

**推し**: **同時**（MVP-A + MVP-B）。両方 PWA で実装可能、競合なし。

---

## Mv4. オフラインを最初から入れるか

| 案 | 良い点 | 悪い点 |
|---|---|---|
| **MVP からオフライン対応** | Service Worker で簡単、本物感 | 同期ロジック必要 |
| **オフラインなし MVP** | 実装速い | 飛行機・地下鉄使えない |
| **読み専用オフラインのみ MVP** | キャッシュだけ、競合なし | 中途半端 |

**推し**: **読み専用オフラインのみ MVP**。Service Worker でキャッシュ、書込は online 必須。

---

## Mv5. ウォッチ・手書きは MVP 外でよいか

| 機能 | MVP 含める？ | 理由 |
|---|---|---|
| **ウォッチ通知（Md5）** | △ Phase 1 後 | iOS PWA の通知制約、Apple Watch は別工数 |
| **手書き（Md6）** | × Phase 3+ | 別ブレスト推奨、タブレット限定 |
| **音声会話（Md10）** | △ Phase 2 | LLM 統合が別案件 |
| **読み上げ（Md4）** | ◎ MVP 候補 | Web Speech API で容易 |
| **検索（Md7）** | ○ MVP 含む | 既存 viewer の拡張 |

**推し**: MVP は **capture + 読み上げ + 閲覧 + 検索** の 4 機能、ウォッチ・手書きは将来。

## 推奨 MVP 構成（2 案）

### 案 1: ライト MVP（1-2 週）

```
Pf1 PWA
├── Md1 閲覧モード（リスト + ピンチでマップ）
├── Md3 capture モード（テキスト爆速版）
├── Md7 検索モード
└── Sy9 読み専用 JSON snapshot（オフライン用）
```

**目的**: 「歩きながら捕捉」と「電車で閲覧」を最小工数で。
**未対応**: 通知、双方向同期、読み上げ、手書き、ウォッチ。

### 案 2: ミディアム MVP（1 月）

```
Pf1 PWA + Pf6 Shortcut
├── Md1 閲覧（リスト + マップ全景）
├── Md3 capture（テキスト + 音声録音）
├── Md4 読み上げ（ポッドキャスト版）
├── Md7 検索
├── Md9 今日のノード
├── Sy1 オフライン読み + キュー書込
├── Sy4 conflict_backup 流用
└── Sy7 mode 別同期方向
```

**目的**: 主要 4 シーン（通勤・散歩・実験室・ベッド）カバー。
**未対応**: 通知、ウォッチ、手書き、音声会話。

## 横断的観察

### 観察 1: モバイル化の本質は「シーン拡張」
PC の機能をスマホに詰め込むのではなく、**PC では発生しないシーン**（散歩、移動、実験室）
で **新しい価値** を作る。MVP-B（散歩 capture）と MVP-C（読み上げ）が代表。

### 観察 2: capture と閲覧の双子設計
モバイルの 80% は **capture（書く）と閲覧（読む）** で済む。編集はタブレットか PC で。
この双子に集中することで Md2 編集モードの複雑さを回避できる。

### 観察 3: PWA で iOS は 80% 解決
2026 年現在、iOS PWA も push 通知が使える。バックグラウンド同期は弱いが、
「アプリ起動時に同期」で十分なケースが多い。WebView ラッパーは MVP では過剰。

### 観察 4: conflict_backup.ts が既にあることが大きい
他ツールが CRDT で苦労する競合解決を、M3E は **既存の conflict_backup** で
妥協 OK。MVP では LWW + backup で 90% カバー、CRDT は将来検討。

### 観察 5: 音声系（Md4 + Md10）は M3E 独自性の源泉
他のメモアプリが提供しない「マップを聞く」「マップと話す」は、研究者の
**移動時間を労働化** する。AlgLibMove 認知層との接続も自然。

### 観察 6: 通知は最後に検討
iOS 通知の制約と、akaghef の通知過多嫌い（推測）から、Md5 は Phase 3 まで保留。
Md8 Widget の方が先かもしれない。

### 観察 7: capture_ingest と本ブレストは「軸違い」
capture_ingest は **チャネル網羅**（A1-A20）、本ブレストは **デバイス全体験**。
両者を統合するメタブレストが将来必要かも（idea/cross_channel_meta/ など）。

### 観察 8: 手書きとウォッチは別世界
Md6（手書き）と Md8/Md11（ウォッチ）はそれぞれ独立した深いトピック。
本ブレストでは触りだけ、深掘りは将来 `idea/handwriting/`、`idea/wearable/` に分離推奨。

### 観察 9: 同期の「不要さ」を疑う
「同期は当然必要」と思いがちだが、Sy 全体を通じて、**read-only スナップショットで十分**
なシーンが多い。最小から始める姿勢が大事。
