# 02 Platform Options — 配布形態の選択肢

「どの技術スタックで、どう配るか」を決めずに並べる。
PWA / ネイティブ / WebView / 別アプリ / レスポンシブ web の 5 択 + 派生。
これは指示「**PWA / ネイティブ / WebView / 別アプリ の選択肢比較を必ず触れる**」の本体。

---

## Pf1. PWA（Progressive Web App）

既存 React+TS+SVG をそのまま PWA 化（Service Worker + manifest.json）。

| 観点 | 内容 |
|---|---|
| **配布** | URL 共有、ブラウザから「ホーム画面に追加」 |
| **開発コスト** | ⭐ 低（既存資産の流用） |
| **iOS 対応** | △ Safari の制約（push 通知 16.4+、バックグラウンド弱い） |
| **Android 対応** | ◎ Chrome で完全動作 |
| **オフライン** | ◎ Service Worker で対応可能 |
| **ストア審査** | 不要 |
| **アップデート** | 自動（Service Worker 更新） |
| **ハードウェアアクセス** | △ カメラ・マイク・GPS は OK、Bluetooth/NFC 限定 |
| **TTS / STT** | ◎ Web Speech API |
| **手書き（Pencil）** | △ Pointer Events 経由、筆圧は OK だが Apple Pencil 機能フル活用は難 |
| **akaghef 親和性** | ◎ 単独研究者なら配布審査不要が大きい |

**良い点**: 既存資産流用、即配布、無料、アップデート楽。
**悪い点**: iOS 制約、ハードウェア機能フル活用できない、通知弱い。

---

## Pf2. ネイティブ iOS / Android

Swift / Kotlin、または React Native / Flutter で書き直し。

| 観点 | 内容 |
|---|---|
| **配布** | App Store / Google Play |
| **開発コスト** | ⭐⭐⭐⭐ 高（実質ゼロから書き直し） |
| **iOS 対応** | ◎ |
| **Android 対応** | ◎ |
| **オフライン** | ◎ |
| **ストア審査** | 必要（Apple Developer $99/年） |
| **アップデート** | ストア経由 |
| **ハードウェアアクセス** | ◎ 全 API |
| **TTS / STT** | ◎ ネイティブ API |
| **手書き** | ◎ Apple Pencil API フル活用 |
| **akaghef 親和性** | △ 個人研究者がストア申請するコスト高 |

**良い点**: 性能、ハードウェア、UX のフル活用。
**悪い点**: 開発コスト極大、ストア審査、コード二重化（PC と分離）。

### Pf2a. React Native
- React 知見流用、コードの一部共有可能
- ただし SVG 描画は web ほど自由ではない

### Pf2b. Flutter
- 性能高い、UI 自由
- だが Dart の学習必要、既存資産活用度低

### Pf2c. Swift / Kotlin（純ネイティブ）
- 最高性能、最高 UX
- 開発コスト最大

---

## Pf3. WebView ラッパー

既存 web コードを Capacitor / Tauri Mobile / Cordova で包んで配る。

| 観点 | 内容 |
|---|---|
| **配布** | App Store / Google Play / 自前配布 |
| **開発コスト** | ⭐⭐ 中（既存資産流用 + ラッパー設定） |
| **iOS 対応** | ◎ |
| **Android 対応** | ◎ |
| **オフライン** | ◎ |
| **ストア審査** | 必要（PWA より体験は良くなる） |
| **アップデート** | ストア + 内部更新可 |
| **ハードウェアアクセス** | ○ プラグイン経由 |
| **TTS / STT** | ◎ プラグインで OK |
| **手書き** | ○ Pointer Events |
| **akaghef 親和性** | ○ 既存資産流用、ストア審査がネック |

**良い点**: PWA より良い体験、既存資産流用、ハードウェアアクセスはネイティブ並。
**悪い点**: ストア審査、ラッパーのバグ、PWA の上位互換だが手間は倍。

### Pf3a. Capacitor（Ionic 系）
- 現代的、活発、TypeScript 親和
- iOS/Android 同時、プラグイン豊富

### Pf3b. Tauri Mobile
- Rust ベース、軽量
- まだ若い、iOS は alpha 段階

### Pf3c. Cordova
- 古い、保守モード
- 推奨しない

## Pf4. 別アプリ（capture / read 専用の薄いネイティブ）

M3E 全体ではなく、**特定機能だけ** をネイティブで作る。

| 観点 | 内容 |
|---|---|
| **配布** | ストア or PWA、用途特化 |
| **開発コスト** | ⭐⭐ 中（範囲が小さいので） |
| **メイン M3E との関係** | API/vault 経由で連携 |
| **強み** | 1 機能特化で UX 極めて良い |
| **弱み** | アプリが増える、認知負荷 |

**例**:
- **M3E Capture** — 1 タップで scratch 投入のみ。
- **M3E Reader** — 読み上げ専用、移動中体験。
- **M3E Today** — 「今日のノード」通知のみ。
- **M3E Watch** — Apple Watch / Wear OS のリマインド。

**良い点**: 各アプリが小さく軽く、目的明快。
**悪い点**: アプリが分散、それぞれ保守コスト、ユーザーは 1 アプリで済ませたい。

---

## Pf5. レスポンシブ web のまま

何もせず、現 web をモバイルブラウザで開く。CSS だけタッチ最適化。

| 観点 | 内容 |
|---|---|
| **配布** | URL のみ |
| **開発コスト** | ⭐ 最小（CSS とタッチ対応のみ） |
| **オフライン** | × |
| **ハードウェアアクセス** | △ ブラウザ依存 |
| **「アプリ感」** | × |
| **akaghef 親和性** | ○ ゼロコスト、ただし体験は劣る |

**良い点**: タダ、即動く、URL 共有。
**悪い点**: オフライン不可、通知不可、ホーム画面追加してもアプリ感薄い。

---

## Pf6. ショートカット / Siri Shortcuts / Tasker

OS 標準のオートメーションツールから M3E を呼ぶ（HTTP / URL scheme）。

| 観点 | 内容 |
|---|---|
| **配布** | レシピ配布 |
| **開発コスト** | ⭐ 極小（API 提供のみ） |
| **強み** | ゼロアプリ、ユーザー自由度高 |
| **弱み** | 機能限定、設定が技術的 |

**例**:
- iOS Shortcut「マップに追加」 → POST /api/scratch
- Tasker「位置情報変化」 → 自動 capture

**良い点**: 軽量、ハック可能、研究者好み。
**悪い点**: 一般ユーザー向きではない、UI 設計が OS 任せ。

---

## Pf7. メール・iMessage・LINE 経由（ゼロアプリ）

既存メッセンジャーアプリで完結、特別なクライアント不要。

| 観点 | 内容 |
|---|---|
| **配布** | アドレス共有のみ |
| **開発コスト** | ⭐ 極小（メール受信サーバー） |
| **強み** | アプリ不要、誰でも使える |
| **弱み** | 閲覧不可、入力のみ、リアルタイム性低 |

`capture_ingest/A5` と完全重複するが、**モバイル文脈** では特に強い。

---

## Pf8. 電子書籍ビューア化

マップを epub/PDF にスナップショット化、Kindle 等で読む。

| 観点 | 内容 |
|---|---|
| **配布** | epub ファイル / Kindle 送信 |
| **開発コスト** | ⭐⭐ 中（書き出しパイプライン） |
| **強み** | E-Ink 端末で目に優しい、読書体験 |
| **弱み** | 読み専用、編集不可、構造失われる |

意外と「移動中マップを読む」シーンに刺さる可能性あり。

## 組み合わせパターン

| パターン | 構成 | 強み | 弱み |
|---|---|---|---|
| **A: PWA only** | Pf1 のみ | 最小コスト | iOS 通知弱い |
| **B: PWA + Shortcut** | Pf1 + Pf6 | 軽量補完 | UI バラつき |
| **C: PWA + 別アプリ** | Pf1 + Pf4 capture 専用 | 主役+特化 | アプリ複数 |
| **D: WebView 全部入り** | Pf3a | フル機能 | ストア審査 |
| **E: 何もしない+メール** | Pf5 + Pf7 | 0 開発 | 体験劣 |
| **F: ネイティブ書き直し** | Pf2 | 最高 UX | コスト最大 |

**推し**: **B → C → D** の順で段階的に拡張。

## 選択軸の整理

プラットフォーム選びの判断軸:

1. **配布の容易さ**: PWA = レスポンシブ > Shortcut > WebView > ネイティブ
2. **開発コスト**: レスポンシブ < PWA < Shortcut < WebView < RN < Flutter < ネイティブ
3. **iOS 体験品質**: ネイティブ > WebView > PWA > レスポンシブ
4. **オフライン**: ネイティブ = WebView = PWA > Shortcut > レスポンシブ
5. **ハードウェア**: ネイティブ > WebView > PWA > レスポンシブ
6. **保守性**: PWA = レスポンシブ（コード一本） > WebView > ネイティブ（コード二本）

---

## akaghef 文脈での見立て

- 単独研究者・個人開発 → **配布の容易さ最優先**
- 既存 React+TS 資産 → **PWA / WebView 系最有力**
- iOS / Android 両対応 → **PWA で 80% カバー、残りは Shortcut / メール**
- Apple Developer 費用 → **必要なら出すが、後回し**
- ストア審査 → **避けたい（時間と心理コスト）**

→ **MVP は Pf1 PWA + Pf6 Shortcut + Pf7 メール の 3 段構え**

---

## 横断的観察

- **PWA の iOS 制約は年々緩和** — 2026 年現在、push 通知も使える。残るは bg 同期とハード機能。
- **Capacitor は PWA の上位互換** — PWA で詰まったら Capacitor に移すパスが自然。
- **「アプリじゃない」抵抗感** — ホーム画面追加で 99% 解消、教育コストのみ。
- **複数プラットフォーム並走は罠** — UI 整合性、保守、すべてが倍。1 つに絞るのが鉄則。
- **Shortcut / メール は意外と強い** — 「アプリを開く前に終わる」のは究極の UX。
