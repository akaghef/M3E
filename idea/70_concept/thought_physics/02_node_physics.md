# 02. ノード物性の選択肢

ノードに持たせる **物理量** の候補を網羅的に並べる。
各物性は「何を意味するか」「何から算出されるか」「何に影響するか」の3点で整理する。

採否は決めない。複数案を並列で残す。

## 設計フレーム

各物性に対して以下の4要素を埋める:

```
- 意味（メタファでの解釈）
- 入力（どの属性／イベントから算出するか）
- 出力（描画・挙動にどう影響するか）
- リスク（誤解・パフォーマンス）
```

## N1. 重力（Gravity）

### N1.1 重力 = 重要度
- **入力**: priority 属性 / 子孫ノード数 / リンク被参照数
- **出力**: 画面下方向に沈もうとする / 周辺ノードを引き寄せる
- **リスク**: "下に沈む = 重要" は直感に反する人もいる（"上が大事" 文化）

### N1.2 重力 = 古さ（年齢に応じて重くなる）
- **入力**: created_at からの経過日数
- **出力**: 古いノードほどゆっくり沈む（地層イメージ）
- **リスク**: "重要だけど古い" ノードが沈みすぎる

### N1.3 重力 = エディタ集中度（読まれた時間）
- **入力**: 累積エディタ滞在時間 / view 回数
- **出力**: よく見られたノードが画面中央に "落ち着く"
- **リスク**: 計測コスト（滞在時間ロギング）

### N1.4 重力方向の選択肢
| 案 | 方向 | 効果 |
|----|------|------|
| N1.4.a | 画面下方向（地球重力） | 「下に沈む」直感 |
| N1.4.b | マップ中心方向 | 重要が中心へ集まる |
| N1.4.c | 親ノード方向 | 階層を強化する |
| N1.4.d | カスタム重力源（ピン留めノード） | 任意の "中心" を作れる |

### N1.5 推し度メモ
- 単純さ重視: N1.1（重要度）+ N1.4.b（中心方向）
- 既存 freshness と分けたい: N1.3（読まれた時間）

## N2. 磁力（Magnetism）

### N2.1 磁力 = タグ親和性
- **入力**: 共通タグ数
- **出力**: 同タグノード同士が引き寄せ合う（クラスタ形成）
- **リスク**: タグ数が多いと計算量爆発

### N2.2 磁力 = AI 推奨リンク強度
- **入力**: AI が計算した意味類似度
- **出力**: 似たノードが互いに近づく（C4 自動 link 提案 と接続）
- **リスク**: AI コスト、変動が激しいと不安定

### N2.3 磁力 = 同種ノード（同 type）
- **入力**: type 属性（task / note / question / decision 等）
- **出力**: 同種が集まる、異種が反発
- **リスク**: 異種が混ざる場面の方が研究的に重要なことも

### N2.4 N極/S極の概念
- N2.4.a: 単純引力のみ（極なし）
- N2.4.b: 同型引力・異型反発（"対立する立場" のノード関係を表現）
- N2.4.c: ユーザーが極性を手動指定（"この2ノードは反対方向に置きたい"）

## N3. 温度（Temperature）

### N3.1 温度 = 直近活性度
- **入力**: 直近 N 日の編集回数 / view 回数
- **出力**: 色（赤=熱い〜青=冷たい） / 振動アニメ
- **リスク**: 既存 freshness と被る

### N3.2 温度 = 議論の活発さ
- **入力**: 直近のコメント数 / 子ノード生成速度
- **出力**: ホットエリア（赤）として描画
- **リスク**: 単独活動の場合は意味薄い

### N3.3 温度 = 不確実性／揺らぎ
- **入力**: status の変動回数 / 未決マーク
- **出力**: "沸騰" 表現（震えるアニメ）
- **リスク**: メタ認知ログ（J5）と被るが、こちらは挙動として表現

### N3.4 熱伝導
- N3.4.a: 隣接ノードに熱が伝わる（時間で平準化）
- N3.4.b: リンクを通じて熱が伝播（強リンクほど早く）
- N3.4.c: 伝導なし（局所性のみ）

### N3.5 冷却曲線
- N3.5.a: 線形冷却
- N3.5.b: 指数冷却（最初は早く、後はゆっくり）
- N3.5.c: 段階的冷却（hot → warm → cool → cold → frozen）

## N4. 電荷（Charge）

### N4.1 電荷 = 立場・スタンス
- **入力**: 手動付与（賛成 +、反対 −、中立 0）
- **出力**: 同電荷反発、異電荷引力
- **リスク**: 隠喩としては強いが、ノードへの政治色付けにユーザーが抵抗するかも

### N4.2 電荷 = 質問 vs 答え
- **入力**: type が question/answer
- **出力**: 質問ノードと答えノードが引き寄せ合う
- **リスク**: type 二値化が硬すぎる

### N4.3 電荷 = エネルギー源/吸収体
- **入力**: 「アイデアを生む」ノードか「整理するノード」か
- **出力**: 流れの方向を作る（エネルギー伝播）

## N5. 質量（Mass）

### N5.1 質量 = テキスト量
- **入力**: text + details の文字数
- **出力**: 大きく描画 / 動かしにくい / 周囲を押しのける
- **リスク**: 大きさが画面を圧迫しすぎる

### N5.2 質量 = 子孫数
- **入力**: 子孫ノード数
- **出力**: 親が "重く" なり安定する
- **リスク**: ルートが極端に重くなる

### N5.3 質量と慣性
- 慣性大 = ドラッグしてもゆっくり動く
- 慣性小 = すぐ反応する
- → 「重要なノードは動かしにくい」を物理的に表現

## N6. 粘性（Viscosity）

### N6.1 粘性 = ノード周辺の "編集しにくさ"
- **入力**: ロック状態 / 完了状態
- **出力**: ドラッグ時に抵抗感（マウス重く感じる）
- **リスク**: UX 上の不快感を生むかも

### N6.2 粘性 = 思考の流動性
- **入力**: 直近の子ノード生成速度
- **出力**: 流動的領域は "サラサラ"、停滞領域は "ドロドロ" 描画
- **リスク**: 抽象度が高すぎる

## N7. 鮮度（Freshness） — 既存属性との接続

既存 M3E に freshness や last_touched 系がある場合、本物理隠喩はその上に乗る。

### N7.1 鮮度の物理的拡張
- 鮮度高 = 熱い・浮く・色鮮やか
- 鮮度中 = 中性
- 鮮度低 = 冷たい・沈む・退色（古地図色）
- 化石化 = 完全停止・地層の底

### N7.2 老化曲線の選択肢
| 案 | 曲線 | 特徴 |
|----|------|------|
| N7.2.a | 線形 | 単純、予測しやすい |
| N7.2.b | 指数（半減期型） | 直近偏重、忘却曲線に近い |
| N7.2.c | 段階関数 | hot/warm/cool/cold の閾値ベース |
| N7.2.d | カスタム per node | ノードごとに半減期を変える（"忘れたくないノード" を遅く老化） |

### N7.3 触ると若返る量
- N7.3.a: フル回復（触れば 100%）
- N7.3.b: 部分回復（触っても少しずつしか戻らない、再覚醒コスト表現）
- N7.3.c: 触るだけ vs 編集 で回復量を変える

## N8. 物性の手動指定 vs 自動算出

| 軸 | 手動 | 自動 |
|----|------|------|
| ユーザー負担 | 多い | ゼロ |
| 意味の正確さ | 高い | 算出ロジック次第 |
| 学習コスト | 物性UIが必要 | 透過的 |
| デバッグ性 | 説明しやすい | "なぜ熱いのか" が説明困難な可能性 |

### N8.1 ハイブリッド案
- デフォルトは自動算出
- ユーザーが override できる（"このノードは常に熱く見せたい"）
- override は色違いで表現（"操作された物性" を明示）

## 物性まとめ表

| ID | 物性 | 主候補入力 | 主出力 | 既存属性との関係 |
|----|------|------------|--------|------------------|
| N1 | 重力 | priority/子孫数 | 沈む・引き寄せる | priority と接続 |
| N2 | 磁力 | タグ/AI類似度 | クラスタ化 | tag と接続 |
| N3 | 温度 | 直近編集回数 | 色・振動 | freshness と被る |
| N4 | 電荷 | スタンス（手動） | 反発・引力 | 新規属性 |
| N5 | 質量 | テキスト量/子孫数 | 大きさ・慣性 | 派生属性 |
| N6 | 粘性 | ロック/完了 | 操作抵抗 | status と接続 |
| N7 | 鮮度 | last_touched | 色・浮沈 | 既存 freshness 拡張 |

## 横断観察

- **N3 と N7 は被る** — 設計時にどちらを採るか統合する必要あり（M2 で議論）
- **N1, N5 は両方 "重さ" を扱う** — 重力 = 落下方向、質量 = 動かしにくさ で分担可能
- **N4（電荷）は研究シーンで強力** — Devil's Advocate (C1) と接続すると相性良い
- **手動 override は必須機能** — "AI が決めた物性" を完全自動だと不信感を生みやすい
