# Q5. ノードの「死」と「再生」

ノードを **永遠に保存する vs 削除する** の二択ではなく、
**死ぬ → 痕跡が残る → 別形態で再生する** というライフサイクルを設計する。

automation_obstacles N8 失敗の記録 / Q2 忘却（連続体）/ thought_physics P2 物性 と接続。

## なぜ「死」を仕組み化するか

- 死を仕組まないと、マップは "屍" で埋まる（捨てられないノード）
- 削除は **痕跡を消す** → 失敗から学べない
- ノードに「死期」があると、生のノードがより輝く（対比効果）
- "死んだ" 概念から **新しい概念が生まれる** プロセスが研究では頻繁
- 失敗ログの蓄積は研究の **資産**（同じ失敗を繰り返さない）

## ユースケース

- **U1.** 「やめた研究テーマ」が消えず、墓標として残る → 再開時に参照
- **U2.** 「失敗した実験」がアーカイブではなく "学び" として再生される
- **U3.** 5 年前に放棄した仮説が、新しい論文で蘇る瞬間
- **U4.** 学生プロジェクト終了後、構造を残して詳細を "化石化"
- **U5.** "もう答えが出た問い" を死なせて、回答ノードを生かす
- **U6.** 重複ノードの merge（D1）を「片方が死んで合体する」と解釈
- **U7.** 古いリンクが切れた時、ノードが "孤独死" するアラート
- **U8.** マップ全体の「死亡率」を見て、健全性を測る

## サブ論点

### Dr1. 死の段階

| 段階 | 名称案 | 状態 |
|---|---|---|
| 0 | alive | 通常ノード |
| 1 | aging | 触られず古びている（Q2 dormant に近い） |
| 2 | dying | 「もうすぐ死ぬ」と本人/AI が判定 |
| 3 | dead | 死亡、墓標として残る |
| 4 | fossilized | 化石化、構造のみ残し詳細は圧縮 |
| 5 | erased | 完全削除（要明示操作） |

→ Q2 忘却との連続性を保つ。Q2 = 認知層の見え方、Q5 = データ層のライフサイクル。

### Dr2. 死の表現

| 案 | 表現 |
|---|---|
| Dr2.a | 墓標アイコン（dead ノードに小さい "✝" や "🪦"） |
| Dr2.b | 黒枠・モノクロ表示 |
| Dr2.c | 半透明 + ストライプパターン |
| Dr2.d | "RIP" ラベル + 死亡日 |
| Dr2.e | 化石模様（fossilized ノード用） |
| Dr2.f | 別レイヤー "墓場" view に移動 |
| Dr2.g | 死後年数を表示 ("died 2024-01") |

emoji を使わない方針なら Dr2.b/c/d で表現。

### Dr3. 死の方法

| 案 | 死亡トリガー |
|---|---|
| Dr3.a | 手動で「これを死なせる」操作 |
| Dr3.b | aging が一定期間継続したら自動 dying |
| Dr3.c | リンク切れで孤立した時 |
| Dr3.d | 別ノードに merge された時（片方が死ぬ） |
| Dr3.e | 結論に至った時（問いノードが死に、回答ノードが残る） |
| Dr3.f | プロジェクト終了宣言で配下が一斉死 |
| Dr3.g | AI 提案による死（「これ死なせていい？」） |

### Dr4. 死の儀式

死は意味を持つので、軽率に殺さない仕掛け:

- Dr4.a. 死亡時に "死因" を記録必須（学びを残す）
- Dr4.b. 死亡時に "次の問い" を生成可能（再生の種）
- Dr4.c. 死亡 confirm ダイアログ（cmd+shift+死 のような重い操作）
- Dr4.d. 月次の "死亡通知" レポート（今月死んだノード一覧）
- Dr4.e. 死亡時にリンク先ノードに "○○が死にました" 通知

### Dr5. 再生の方法

死んだノードから新しいノードが生まれる仕組み:

| 案 | 再生パターン |
|---|---|
| Dr5.a | 死んだノードを参照する新ノード作成（"継承"） |
| Dr5.b | 死後 N 年で「復活候補」アラート |
| Dr5.c | 似たノードが新規作成された時、過去の死ノードを提示 |
| Dr5.d | "fossilized" → "alive" 復元操作 |
| Dr5.e | 死んだノードの "死因" を読んで、回避策を新ノード化 |
| Dr5.f | AI が「過去の死から新しい組み合わせ」を提案 |

### Dr6. 失敗ログとしての死

automation_obstacles N8 と直結。

- Dr6.a. 死因が "failed" のノードを集約 view → "失敗集"
- Dr6.b. 失敗パターンの分類（仮説誤り / 手法誤り / リソース不足 / etc）
- Dr6.c. 失敗から得た "教訓" を専用属性で
- Dr6.d. 助成申請書の "Lessons learned" セクションに自動引用
- Dr6.e. 同じ失敗を繰り返しそうな新ノードに警告

### Dr7. ノードのライフサイクル属性

| 属性 | 例 |
|---|---|
| `birthed_at` | 作成日（既存 created_at） |
| `aged_since` | aging 入りした日 |
| `died_at` | 死亡日 |
| `cause_of_death` | failed / superseded / merged / completed / forgotten |
| `successor: <node_id>` | 後継ノード（再生先） |
| `predecessor: <node_id>` | 前世ノード（継承元） |
| `lessons` | 死から得た教訓テキスト |

### Dr8. マップの "生死バランス"

- Dr8.a. ヘッダーに "alive: N / dead: M / fossil: K" 表示
- Dr8.b. 健全マップ判定: alive 比率が一定範囲
- Dr8.c. 死亡率が高すぎる時 = 過剰削除傾向の警告
- Dr8.d. 死亡率が低すぎる時 = ため込み傾向の警告
- Dr8.e. 死生比のグラフ（時系列）

### Dr9. 「集団死」の扱い

プロジェクト終了等で大量に死ぬケース:

- Dr9.a. プロジェクトノードを "completed" にしたら配下を一斉 dying
- Dr9.b. 一斉死亡の儀式 view（"このプロジェクトをここで閉じます"）
- Dr9.c. 一斉死後、構造のみ残した "プロジェクト化石" を生成
- Dr9.d. 化石を Read-only モードで保存

### Dr10. 死と "完成" の区別

| 状態 | 性質 | 表現 |
|---|---|---|
| done | 達成された | 通常表示、Q1 では「次の問い」必須 |
| completed | 結論に至った | 灰色、参照可 |
| dead (failed) | 失敗で終わった | 墓標、教訓付き |
| dead (superseded) | 後継に取って代わられた | 墓標、後継リンク付き |
| dead (forgotten) | 自然死 | 墓標、Q2 forgotten から移行 |
| fossilized | 化石化 | 構造のみ、詳細圧縮 |

### Dr11. 「死を見せる」UI モード

- 通常モード: alive のみ表示、墓標は薄く
- 墓場モード: dead/fossil のみ表示、生者は薄く
- 全生死モード: 全部表示

### Dr12. AI と死

- Dr12.a. AI に渡す context は alive のみがデフォルト
- Dr12.b. "墓場 context モード" で過去の失敗を AI に伝える（同じ失敗回避）
- Dr12.c. AI が「これ死んでいいよ」を提案
- Dr12.d. 死亡時に AI が "死因の言語化" を補助

## 反対意見

- **R1.** "死" メタファは重い。日常使いに気が滅入る
- **R2.** ノードに人格を与えるのは過剰
- **R3.** 削除で十分、ライフサイクルは複雑化を招く
- **R4.** "失敗" を可視化するのは精神的負担
- **R5.** 化石化したデータがマップを重くする

→ 答え:
- R1 → "完了/退役/化石" のような穏当ラベルでも代替可
- R2 → 比喩は思想層、UI 表現は穏当に（"retired" でも可）
- R3 → 削除すると失敗ログが消える、それが問題
- R4 → 失敗の "教訓化" がポイント、責めるのではなく学ぶ
- R5 → fossilized は構造のみ → 軽量化される

| ツール | ノードの死の扱い |
|---|---|
| Notion | 削除 / 復元（30 日）、ライフサイクルなし |

## 半年実用化への寄与

- **短期**: Dr1.0-3 + Dr4.a の最小実装で「失敗ログ」機能が完成
- **中期**: Dr6 失敗集が助成申請書の "Lessons learned" 自動生成に直結
- **長期**: Dr5 再生パターンが研究のメタ知識（同じ問いの繰り返し回避）

## キーメッセージ

- **死を仕組み化することで、失敗が資産化する。**
- **Q2 忘却との連続体** として設計する（連続でなく断絶を作ると複雑化）
- **死因記録（Dr4.a）が最重要属性。** これだけでも automation_obstacles N8 が動く
- **比喩は調整可能。** "死" でも "退役" でも仕組みは同じ → 思想押し付け回避
- **再生プロセスが本体。** 死だけでは虚しい、再生があって初めて意味
