# Q1. 「マップは未完成であるべき」原則のシステム化

「完成したマップは死んだマップ」という命題を、UX・データ・運用の各層で
**システムとして表現する** ためのブレスト。

## なぜ未完成原則か

- 研究は本質的に **常に進行中**。完成したという自己宣言は思考停止のトリガー
- Notion/Obsidian が「整理ツール」として収束しているのに対し、
  M3E は **未整理を価値化するツール** として差別化できる
- automation_obstacles N7「創発の場」と接続: **余白がないと新しい結合が起きない**
- project_alglibmove_dogfood 批判（M3E に認知層が無い）への思想的回答: 
  「認知層 = 未完成性を見せる層」と再定義できる

## ユースケース（誰がいつ嬉しいか）

- **U1.** 博士課程 3 年目のリサーチャーが、4 年目の自分に向けて「ここに穴がある」を残す
- **U2.** 共著者が「ここはまだ詰まっていない」を瞬時に把握できる
- **U3.** 助成申請書を書く時、「未解決問題」を即座にエクスポートできる
- **U4.** 1 年後の自分が「ここから手を付けるべき」が分かる
- **U5.** 学生がマップを見て「教授も悩んでるんだ」と知り、参加できる
- **U6.** AI が「ここは未完成宣言があるので深堀って」と判断できる

## サブ論点

### Uf1. 「未完成」をどう表現するか（ノード属性）

| 案 | 内容 | 良い点 | 悪い点 | 推し度 |
|---|---|---|---|---|
| Uf1.a | `incomplete: true` フラグ | 単純、API 変更最小 | 二値で粒度が粗い | 中 |
| Uf1.b | `completion: 0-100` スコア | 進捗を量で表せる | 「進捗」と「完成度」は別物 | 低 |
| Uf1.c | `gaps: ["...", "..."]` 配列 | 何が未完成か明示 | 入力負荷高い | 高 |
| Uf1.d | `next_question: "..."` 単一フィールド | 「次の問い」だけ書く軽さ | 複数の未完成を表現できない | 高 |
| Uf1.e | 専用子ノード `?` プレフィックス | 既存ノード構造で表現可 | 検索/集約しづらい | 中 |
| Uf1.f | リンク種別「question」「assumption」 | 関係として表現 | リンク種別爆発リスク | 中 |

推し: **Uf1.d + Uf1.f の組み合わせ**（軽い属性 + 重要な未完成は専用リンク）。

### Uf2. 「完成宣言」を許すか / 禁じるか

| 案 | 内容 |
|---|---|
| Uf2.a | done ステータスを廃止、「current」「dormant」「archived」の 3 状態に |
| Uf2.b | done を残すが、完成後 N 日で自動「revisit_due」化 |
| Uf2.c | done に "次の問い" 必須化（書かないと done にできない） |
| Uf2.d | 完成宣言は許すが、UI で done ノードを縮小表示（薄く） |
| Uf2.e | 完成は許すが、マップ全体の "未完成率" を常時表示 |

推し: **Uf2.c**（done の代償として "次の問い" を要求）→ Uf1.d と整合。

### Uf3. マップ全体の「未完成度」可視化

| 案 | 表現 |
|---|---|
| Uf3.a | ヘッダーに「未完成 N 個 / 総 M 個」テキスト |
| Uf3.b | マップ全体の "色温度"（未完成多いほど青寄り） |
| Uf3.c | ヒートマップ（subtree ごとの未完成率） |
| Uf3.d | "未完成インデックス" を時系列グラフ化（増減追跡） |
| Uf3.e | エクスポート時に「Open Questions」セクション自動付与 |

推し: **Uf3.a + Uf3.e**（最小コストで効果大）。

### Uf4. 「完成至上主義」アンチパターンの抑止

| 案 | 仕組み |
|---|---|
| Uf4.a | done 比率が 90% を超えたら "warning: マップが硬直しています" |
| Uf4.b | 30 日以上 incomplete 追加ゼロなら「探索停滞」通知 |
| Uf4.c | 全ノード done 状態を達成不能にする（必ず 1 つは ? が残る） |
| Uf4.d | 起動時に「今日 1 つ "未完成" を書きませんか」プロンプト |
| Uf4.e | 完了率を画面に出さない（そもそも測らせない） |

推し: **Uf4.b + Uf4.d**（緩い nudge）。Uf4.c は教条的すぎる。

### Uf5. 未完成 vs TODO の区別

両者は似て非なる。整理の論点。

| 軸 | 未完成（Uf） | TODO |
|---|---|---|
| 性質 | 知的な "穴" | 行動の指示 |
| 完了基準 | 多くは完了しない / し続ける | 完了する |
| 例 | 「この理論の射影が分からない」 | 「明日 10 時に提出」 |
| 見せ方 | 周辺に薄く広がる | 集中リストに |
| 既存接続 | Q4 自己発見 | Todo_Pool.md（feedback_todo_single_file） |

→ 区別する場合 → 別属性 / 別 view。混ぜる場合 → 同じ属性で見え方だけ変える。

### Uf6. 「未完成」を AI に伝える

- Uf6.a. AI prompt の system に「このマップは未完成前提」と明示
- Uf6.b. AI に「埋めて完成させて」とは頼まず「次の問いを増やして」と頼む
- Uf6.c. C1 Devil's Advocate と組み合わせて「未完成箇所への追加質問」
- Uf6.d. C5 insight 抽出を「埋めるべきギャップ列挙」モードに

推し: **Uf6.b** が思想的に一貫。

### Uf7. 既存ノード集合への適用方針

- Uf7.a. 既存 done ノード全部に "次の問い" を後付け（一括で苦しい）
- Uf7.b. 新規ノードからのみ適用、既存は据え置き
- Uf7.c. 既存 done は "completed_legacy" タグで隔離、思想は新規のみ
- Uf7.d. ユーザーが「思想モード ON」した時だけ適用

推し: **Uf7.b + Uf7.d**（漸進）。

## 反対意見（採否を決めない素材として）

- **R1.** 完成感がモチベーションを支える。完成宣言を奪うと続かない
- **R2.** 「常に未完成」は心理的負担。安心して閉じられない
- **R3.** 学生・若手は完成体験で力をつける。未完成原則は熟練者向け
- **R4.** 助成・査読など「完成版」を要求される場面がある
- **R5.** 完成しない概念は科学的検証ができない（反証可能性問題）

→ これらに対する答え:
- R1 → "完成 = 次の問い登録" で代替モチベ
- R2 → モード切替（思想 ON/OFF）
- R3 → 学生 view では完成宣言可、教員 view では未完成強調
- R4 → エクスポート時には未完成を別セクション化
- R5 → 「未完成」は問いを残すこと、答えの未確定ではない

## 既存ツール差分

| ツール | 完成感の扱い | M3E の差別化点 |
|---|---|---|
| Notion | チェックボックス完了で達成感 | M3E は完了しても「次の問い」必須 |
| Obsidian | 永続ノートは "完成" を志向 | M3E は永続「未完成」を志向 |
| Roam | daily note で流動的 | M3E は流動性 + 構造を両立 |
| Anki | 完成 = 覚えた | M3E は覚えた後の再問題化を促す |
| Logseq | TODO 状態切替 | M3E は TODO と未完成の区別 |

## キーメッセージ

- **未完成原則はブランドになる。** 「全部 done にできないツール」というポジションは
  GTD 流ツール群との明確な差別化になる
- **思想だけでなく "1 機能" に落とせる。** Uf1.d（next_question 属性）+ Uf2.c（done 時必須）
  だけで思想が動き始める。半日で実装可能なレベル
- **モード切替が必要。** 思想原理主義を避けるため、ON/OFF 可能にする（Uf7.d）
