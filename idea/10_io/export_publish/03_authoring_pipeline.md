# 03. 著作パイプライン — マップ → 中間ビュー → 出力物

「マップから直接出力」ではなく、**中間ビュー**を挟む二段階変換が現実的。
中間ビューは「同じマップを別角度から見たもの」で、複数の出力先で再利用可能。

## パイプラインの基本形

```
マップ（生データ）
   ↓ ① ビュー定義（どのノードを・どの順序で・どう分類して）
中間ビュー（IMRAD / 章節 / 時系列 / クラスタ / Q&A 等）
   ↓ ② テンプレ展開 + LLM 介入
出力物（LaTeX / PPTX / .apkg / .ttl 等）
   ↓ ③ ポストプロセス（マスキング / フォーマット調整）
公開可能ファイル
```

各段階で複数の選択肢がある。

## 中間ビューの種類

### Vw1. IMRAD ビュー
- Introduction / Methods / Results / Discussion の4軸
- 主に B1（論文）向け
- マップノードに `imrad: intro` 等のタグで分類 or 専用 subtree

### Vw2. 章節ビュー
- 階層的な章 → 節 → 項
- B1, B6（ブログ長文）向け
- マップの subtree 構造を直接流用しやすい

### Vw3. 時系列ビュー
- ノード作成日 / 決定日 でソート
- B6（週次レビュー）, B4（プロジェクト史動画）向け
- backup/snapshot との接続が強い

### Vw4. クラスタビュー
- 似たノードを自動グループ化
- B5（タグ別ページ）, B7（テーマ別カードデッキ）向け
- D1（重複検出）と技術スタック共通

### Vw5. Q&A ビュー
- 質問ノード ↔ 回答ノード のペア抽出
- B7（SRS）, B6（FAQ ブログ）向け
- C2（質問生成器）と接続

### Vw6. 戦略ビュー
- strategy / decision タグ付きノード
- B2（申請書「研究計画」）, B6（決定ログ）向け

### Vw7. 引用ビュー
- 文献ノード + そのノードを引用しているノード
- B1（参考文献リスト）, H4（Zotero 連携）

### Vw8. グラフビュー
- ノード + リンク構造のみ
- B8（RDF）向け、テキスト無視

### Vw9. カスタムクエリビュー
- ユーザ定義クエリ（タグ・属性・テキスト検索の組合せ）
- 全出力先で使える汎用形

## ビュー定義の置き場（論点 Vw2）

| 案 | 置き場 | 良い点 | 悪い点 |
|----|-------|------|------|
| Vw2a | マップ内の "ビュー定義ノード" | マップ自己完結 | マップが汚れる |
| Vw2b | 別 .view.json ファイル | 分離、git 差分見やすい | 二重管理 |
| Vw2c | URL パラメータ (`?view=imrad`) | URL 共有可能 | 永続化弱い |
| Vw2d | workspace 設定 | グローバル | 共有しにくい |
| Vw2e | テンプレ + ユーザ上書き | 柔軟 | 複雑 |

推し: Vw2a + Vw2b のハイブリッド。マップ内に定義を置くが export 時に .view.json として書出可能。

## テンプレート方式（論点 Tp）

### Tp1. 固定テンプレ
- 「論文 IMRAD テンプレ」「科研費基盤C テンプレ」を予め用意
- マップから値を埋めるだけ
- 良い点: 確実、速い
- 悪い点: 多様性が無い

### Tp2. パラメータ可変テンプレ
- セクション数、字数制限、引用形式をパラメータ化
- 良い点: 1つのテンプレで多形式対応
- 悪い点: パラメータ設計が難しい

### Tp3. LLM プロンプト合成
- 「このマップ subtree を IMRAD 論文の introduction として書け」と指示
- 良い点: 自由度高い
- 悪い点: 再現性低い、ハルシネーション

### Tp4. ユーザ DSL
- ユーザが自分のテンプレを定義（Mustache, Liquid 風）
- 良い点: ユーザの暗黙知を形式化
- 悪い点: 学習コスト

### Tp5. Few-shot 例示テンプレ
- 過去の出力サンプル数本を LLM に与えて「同じ調子で書け」
- 良い点: スタイル転写、過去申請書から学習
- 悪い点: サンプル管理コスト

### 比較表

| 案 | 自由度 | 再現性 | 学習コスト | 推し度 |
|----|-------|-------|----------|-------|
| Tp1 | 低 | 高 | 低 | ⭐ |
| Tp2 | 中 | 高 | 中 | ⭐ |
| Tp3 | 高 | 低 | 低 | △ |
| Tp4 | 高 | 高 | 高 | (上級) |
| Tp5 | 高 | 中 | 中 | ⭐⭐ |

推し: Tp1 から始めて、過去出力が溜まったら Tp5 に移行。

## LLM 介入度（論点 LLM）

連続的に切替可能にしたい:

| Lv | 介入内容 | 用途例 | リスク |
|----|---------|-------|------|
| L0 | 介入なし、純テンプレ展開 | RDF, Anki カード | なし |
| L1 | 要約・短縮のみ | 申請書字数調整 | 意味歪み |
| L2 | 接続文・段落導入を生成 | 論文ドラフトの繋ぎ | スタイル不一致 |
| L3 | 各節を再構成 | スライドの説明文 | ハルシネーション |
| L4 | 構成・論理展開も AI | 完全自動執筆 | 倫理問題、捏造 |

### ユーザ選択 UI
- export 時に「介入度スライダー L0〜L4」を見せる
- デフォルトは L1 か L2
- L3 以上は明示的承認

### 倫理ガード
- L4 は研究倫理違反になる可能性
- 出力物に「LLM 介入度」をメタデータとして付与
- 申請書・論文では L3 以下を推奨

## マップ → 中間ビューの変換（① 段階）

### 変換アルゴリズム候補
| 案 | 内容 | 良い点 | 悪い点 |
|----|------|------|------|
| Cv1 | タグ・属性ベースのフィルタ | 明示的、デバッグ容易 | タグ漏れで欠落 |
| Cv2 | グラフ走査（特定ノードから N-hop） | 関連性で抽出 | スコープ膨張 |
| Cv3 | LLM が「IMRAD として何を抽出すべきか」推論 | 賢い | ブラックボックス |
| Cv4 | ユーザが手動で「これを intro にする」マーク | 確実 | 手間 |
| Cv5 | 過去出力からの学習 | 学習データ次第で精度↑ | 訓練データ要 |

推し: Cv1 + Cv4 のハイブリッド。タグ自動 + 人間補正。

## 中間ビュー → 出力物の変換（② 段階）

### 出力プロファイル
- 同じビュー定義 + 異なる出力プロファイル → 異なるファイル形式
- プロファイル例: `journal-nature.profile`, `kakenhi-kibanC.profile`, `slideshow-revealjs.profile`

### プロファイルの中身
- フォーマット仕様（LaTeX クラス、PPTX マスター）
- 制約（字数、図数）
- スタイル（フォーマル/カジュアル、能動/受動）
- 引用形式（APA, IEEE, Vancouver）

### プロファイル管理
| 案 | 管理場所 | 良い点 | 悪い点 |
|----|---------|------|------|
| Pf1 | M3E 同梱（標準セット） | 即使える | 多様性なし |
| Pf2 | コミュニティ共有（GitHub） | 多様 | 品質ばらつき |
| Pf3 | ユーザ自作（マップ内ノード） | 個別最適 | 学習コスト |
| Pf4 | LLM が公募要項 PDF を読んで自動生成 | 最新追従 | 精度依存 |

## ポストプロセス（③ 段階）

### Pp1. マスキング
- 機微情報の自動検知（L5 連動）
- ユーザ確認 → 削除/置換/承認
- 詳細は 05_data_privacy.md

### Pp2. フォーマット調整
- 全角半角統一
- 引用番号採番
- 図番号・表番号の付与
- 改行・余白の調整

### Pp3. リント
- 文字数制限チェック
- 重複文検出
- 引用先存在チェック（リンク切れ）
- スタイル違反検出

### Pp4. 検証
- LaTeX なら pdflatex でビルド試行
- HTML なら lint
- Anki なら .apkg のスキーマ検証

### Pp5. メタデータ付与
- 出力日時、マップ snapshot ID、LLM 介入度
- 出力物のヘッダ or 末尾に埋め込み

## パイプラインの段階制御 UI

### 案 Ui1. ワンショット
- ボタン1つで「マップ → 出力物」一気通貫
- 速いが途中介入できない

### 案 Ui2. 段階確認
- 各段階で human-in-the-loop
- ビュー → OK → テンプレ展開 → OK → マスキング → OK → ファイル保存
- 安全だが遅い

### 案 Ui3. プリセット + ワンショット
- 過去使った設定を保存、次回はワンショット
- 初回 Ui2、2回目以降 Ui1

### 案 Ui4. CLI + GUI 両対応
- バッチ処理は CLI、初回設定は GUI
- 自動化（cron で週次レポート）と相性

推し: Ui3。学習コストと速度のバランス。

## 双方向化の検討（論点 Cy）

「出力 → 修正 → マップに戻す」サイクル:

| 案 | 内容 | 実現性 |
|----|------|------|
| Cy1 | 一方向のみ（出力で完結） | 高（楽） |
| Cy2 | 出力物のコメントをマップに取込み | 中 |
| Cy3 | 出力物を編集 → 差分をマップに反映 | 低（双方向同期難しい） |
| Cy4 | 出力物に "M3E node ID" を埋め込み、後追い可能に | 中 |

推し: Cy1 を基本、Cy4 を併設。Cy2/Cy3 は将来。

## 横断観察

- **二段階パイプライン（マップ → ビュー → 出力物）が安定設計**
- **ビュー定義は再利用される資産** → マップ内に置きたい
- **LLM 介入度は連続スライダーであるべき**（L0〜L4）
- **プロファイルがコア概念**：同じビューで複数フォーマット出す
- **ポストプロセスは軽視されがちだが、品質を決める**
- **Cv1+Cv4（タグ+手動）が現実解**、純LLM自動はリスク

## 論点

- **Vw2. ビュー定義の置き場** — マップ内 / 別ファイル / URL / 設定
- **Tp. テンプレート方式** — 固定 / 可変 / LLM / DSL / few-shot
- **LLM. 介入度** — L0〜L4 連続
- **Cv. 抽出アルゴリズム** — タグ / グラフ / LLM / 手動 / 学習
- **Pf. プロファイル管理** — 同梱 / コミュニティ / 自作 / 自動生成
- **Pp. ポストプロセス** — マスキング / リント / 検証 / メタデータ
- **Ui. UI フロー** — ワンショット / 段階確認 / プリセット / CLI
- **Cy. 双方向化** — 一方向 / コメント取込 / 双方向同期 / ID 埋め込み
