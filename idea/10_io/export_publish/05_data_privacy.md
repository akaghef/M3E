# 05. データモデル・プライバシー・互換性

出力物の **永続化・バージョン管理・プライバシー境界・他形式互換** をブレストする。
特にプライバシーは公開時の最大リスク。

## データモデル

### 出力物のメタデータ
出力するたびに以下を記録すべき:

| フィールド | 型 | 説明 |
|----------|-----|------|
| output_id | string | 出力物の一意 ID |
| source_map_snapshot | string | マップのスナップショット ID (I time travel と接続) |
| view_definition | object | 中間ビュー定義 |
| profile | string | 出力プロファイル名 |
| llm_intervention_level | int | L0〜L4 |
| llm_model | string | "claude-opus-4-6" 等 |
| created_at | timestamp | |
| created_by | string | ユーザ ID |
| target_format | string | "latex", "pptx", "apkg", "ttl" 等 |
| privacy_level | enum | private/limited/public |
| masking_applied | bool | マスキングが施されたか |
| diff_from_previous | object? | 前バージョンとの差分（あれば）|

### 出力物の保存場所候補

| 案 | 保存先 | 良い点 | 悪い点 |
|----|-------|------|------|
| Sl1 | マップ内のノードとして保存 | 自己完結、検索可 | マップ肥大 |
| Sl2 | 別フォルダ `outputs/` に保存 | git 管理しやすい | マップとの紐付け弱い |
| Sl3 | 外部ストレージ（S3等） | スケール | クラウド依存 |
| Sl4 | output_id を持つメタノード + 実体は別ファイル | バランス | 実装複雑 |

推し: Sl4。マップにメタノードだけ、実体は `outputs/<output_id>.pdf` 等。

### 出力物履歴

```
output_v1 (snapshot_2026-04-16) ─┐
                                  ├─ 同じテンプレ、マップ更新で再生成
output_v2 (snapshot_2026-04-20) ─┘
```

- バージョン間の diff を可視化
- 「v1 で投稿、v2 で改訂版」を区別

## バージョニング戦略（論点 Vn）

### Vn1. semver 風（major.minor.patch）
- major: 投稿先変更
- minor: セクション追加・大幅改訂
- patch: typo 修正

### Vn2. snapshot ID ベース
- マップの snapshot ID をそのまま出力物の version とする
- 「2026-04-16T10:00 のマップから生成」が一目瞭然

### Vn3. 投稿先 × バージョン
- nature_v1, jnature_v2, biorxiv_v1 のように投稿先別
- 投稿先間で異なるカスタマイズが履歴化

### Vn4. 用途別タグ
- "draft", "submitted", "revised", "accepted", "published"
- 進捗状態として管理

推し: Vn2 + Vn4 のハイブリッド。snapshot ID で技術的に管理、タグで人間管理。

## アップデート伝播（論点 Up）

マップを更新したら、過去の出力物はどうする？

| 案 | 伝播戦略 | 良い点 | 悪い点 |
|----|---------|------|------|
| Up1. 何もしない | 出力物は throw-away | 楽 | 古い情報が残る |
| Up2. 差分通知のみ | ユーザに「再生成しますか？」 | 制御権あり | 通知疲れ |
| Up3. 自動再生成 | バックグラウンドで再生成 | 常に最新 | 過去版が消える |
| Up4. v1 残しつつ v2 を提案 | 履歴保持 + 新版提案 | 履歴 + 最新 | ストレージ肥大 |
| Up5. 投稿済みは固定、未投稿は自動更新 | 状態別に挙動変える | 賢い | 状態管理必要 |

推し: Up5。submitted 以降は固定、draft は自動更新オプション。

## プライバシー境界（論点 Pv / Ms）

### 公開時の機微情報リスク
公開時にこそ機微情報が漏れる。capture 時より厳重。
（automation_obstacles の L1〜L5 障害群と直結）

### 機微情報の例
- 個人情報（人名、メール、住所）
- 未公開の研究アイデア（先取権リスク）
- 査読中の論文情報
- 共同研究者の発言・意見
- ネガティブ結果・失敗
- 社内専用情報
- 医療情報・被験者情報

### マスキング戦略

#### Ms1. ノード属性ベース
- 各ノードに `privacy: private/limited/public` 属性
- 出力時に privacy で除外/マスキング
- 良い点: 明示的、確実
- 悪い点: 属性付け忘れ

#### Ms2. ルールベース
- 正規表現で「メールアドレス」「電話番号」自動マスク
- 良い点: 機械的、漏らしにくい
- 悪い点: 文脈依存マスク不可

#### Ms3. LLM ベース（L5 連動）
- LLM が「これは公開しない方が良い」と判定
- 良い点: 文脈理解
- 悪い点: 誤判定、外部 LLM への送信自体がリスク

#### Ms4. 別 vault 物理分離
- 公開用 vault と private vault を分離
- 公開用には最初から機微なし
- 良い点: 物理境界、安全
- 悪い点: 二重管理、移動コスト

#### Ms5. 段階承認
- 出力 → ユーザがレビュー → 承認 → 公開
- 良い点: 最終決定権が人間
- 悪い点: 自動化と矛盾

### 推奨組合せ
- 必須: Ms1（属性）+ Ms5（段階承認）
- 補助: Ms2（ルール）+ Ms3（LLM、ローカル LLM 推奨）
- 物理分離（Ms4）は超機微マップ用

### マスキング実装案
| レベル | 内容 |
|-------|-----|
| Mask0 | ノード自体を出力から除外 |
| Mask1 | テキストを `[REDACTED]` に置換 |
| Mask2 | 一般化（"田中太郎" → "ある研究者"）|
| Mask3 | 暗号化（鍵保有者のみ復号）|
| Mask4 | 構造のみ残しテキスト消去（L4 匿名共有風）|

### マスキング UI 案
- 出力プレビュー画面で機微情報がハイライト表示
- ホバーで「なぜマスクされたか」表示
- ワンクリックで解除可能（責任は user）
- マスク履歴をログ保存

## プライバシーレベル別の出力許可

| 公開先 | 許可される privacy レベル |
|-------|-------------------------|
| 自分のみ（Anki, ローカル PDF）| private + limited + public |
| 共著者（限定共有）| limited + public |
| 査読者（投稿）| public のみ + limited（説明付き） |
| 一般公開（Web, ブログ）| public のみ |
| 機械可読（RDF）| public + 構造のみ（L4 風）|

## マルチ vault 切替（L3 連動）

### 想定 vault 構成
| vault | 用途 |
|-------|-----|
| public | 公開可能なノードのみ、git で公開 |
| private | 個人思考、暗号化保存 |
| medical | 医療情報、別暗号化キー |
| commercial | 受託研究、NDA 配下 |
| sandbox | 実験用、いつでも捨てる |

### vault 間移動
- ノードを「private → public」に移動 = 公開判断
- LLM/人間レビューを経て移動
- 移動履歴をログ

### 出力時の vault チェック
- 出力対象に異 vault のリンクがあったら警告
- 例: public 出力に private リンクがあれば「壊れた参照になります」

## 互換性・portability

### 他ツールとの互換性

#### 入力側（capture_ingest と接続）
- Obsidian, Roam, Logseq からの import
- マップ → 各ツールへの export
- 案 In1: Obsidian Markdown 出力（[[link]] 形式）
- 案 In2: Roam JSON
- 案 In3: Logseq EDN

#### 標準形式（B8）
- RDF/Turtle
- JSON-LD (Schema.org, SKOS, FOAF)
- OPML（アウトライン標準）
- GraphML（グラフ標準）

#### 双方向同期 vs 一方向 export
- 一方向 export: 楽、確実
- 双方向同期: 真にユーザの自由、実装難

### バックアップ戦略
- マップ自体: 既存 backup.ts
- 出力物: outputs/ フォルダごと git
- メタデータ: マップ内ノード（バックアップに含まれる）

## 出力物のセキュリティ

### 暗号化
- 出力物が機微情報を含む場合の保管
- ファイル単位暗号化（LUKS, EncFS）
- パスワード付き PDF / ZIP

### アクセスログ（L6）
- いつ・誰が出力物を生成・閲覧したか
- 出力物自体にトラッキング ID 埋込
- 共同研究者間の信頼性向上

### 期限付き共有
- 出力物に「2 週間後に削除」期限
- 限定共有 URL の自動失効

## ライセンスとクレジット

### ライセンス自動付与
- 出力物のヘッダ/フッタに CC ライセンス埋込
- マップノードのライセンスを継承
- ライセンス衝突警告（CC-BY ノードを含む CC0 出力など）

### クレジット
- 出力物に「Generated from M3E」表示
- マップ snapshot ID を引用可能形式で
- LLM 介入度・モデル名を明記（透明性）

## コンプライアンス

### 研究倫理
- 被験者データを公開してないか
- IRB 承認番号の付与
- 二重投稿防止

### 著作権
- 引用範囲の妥当性
- 図の転載許諾
- 共著者の同意

### GDPR / 個人情報保護法
- EU からのアクセスがあれば配慮
- 個人情報を含む場合の扱い

### 学術不正
- LLM 介入度の開示（ジャーナルが要求）
- ChatGPT 等を著者として記載しないルール（多くのジャーナル）

## 横断観察

- **メタデータの自動付与が品質保証の基礎**（snapshot, model, intervention level）
- **マップは「真実源」、出力物は派生物** の哲学を貫く
- **マスキングは Ms1（属性）+ Ms5（承認）が最低ライン**
- **vault 物理分離（Ms4）は超機微情報用の最終手段**
- **LLM 介入度の開示** は学術出版の透明性で必須化しつつある
- **アップデート伝播は submitted 後は固定（Up5）** が現実解
- **出力物のバージョニングは snapshot ID ベース（Vn2）** が技術的に綺麗

## 論点

- **Sl. 出力物保存場所** — マップ内 / 別フォルダ / 外部 / メタ＋実体分離
- **Vn. バージョニング** — semver / snapshot / 投稿先別 / 用途タグ
- **Up. 伝播戦略** — 何もしない / 通知 / 自動再生成 / 状態別
- **Ms. マスキング** — 属性 / ルール / LLM / 物理分離 / 承認
- **Pv. プライバシー境界** — vault 分離、出力時チェック
- **In. 他ツール互換** — Obsidian/Roam/Logseq 入出力
- **Lc. ライセンス** — 自動付与、衝突警告
- **Cm. コンプライアンス** — 倫理・著作権・GDPR・LLM 開示
