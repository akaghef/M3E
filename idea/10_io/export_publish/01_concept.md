# 01. コンセプト — マップを「外に出す」とは何か

## 何を作るか

M3E マップに溜め込んだノード群（思考・知識・引用・意思決定ログ・図）を、
**外部の読者が読める形に変換して書き出す**機能群。

入力（capture）の対称軸。capture が「世界 → マップ」なら、export は「マップ → 世界」。

ただし「ただの dump」ではなく、**読者の目的に合わせて射影面を切り替える**のがコア。
同じマップから論文も申請書もスライドも出る。

## 想定ユースケース（誰が・いつ・何を）

| # | シーン | 出力先 | 期待効果 |
|---|--------|-------|---------|
| 1 | 論文初稿を1日で書き上げたい | B1 LaTeX | IMRAD ビューから雛形が出る、肉付けに集中 |
| 2 | 科研費「研究目的」欄を 800字でまとめたい | B2 申請書テンプレ | マップから抽出 → 字数制限内で要約 |
| 3 | 来週のラボミーティングで進捗発表 | B3 スライド | 戦略 subtree から自動でスライド生成 |
| 4 | 新メンバーへのオンボーディング動画 | B4 動画 + ナレーション | マップツアー＋音声合成で15分の入門動画 |
| 5 | 研究室の Web ページに研究紹介を載せる | B5 静的サイト | 公開タグ付きノードのみ抽出してデプロイ |
| 6 | 月1の研究ブログを書く | B6 Markdown ブログ | publish タグ付きの insight を記事化 |
| 7 | 暗記したい論文用語を Anki に流す | B7 SRS カード | 重要ノード（重要度 5）を q&a 形式で書出 |
| 8 | 共同研究者に "知識グラフだけ" 渡したい | B8 RDF/JSON-LD | テキスト抜き、構造のみ機械可読で共有 |
| 9 | 過去の発表スライドを再利用したい | B3 + I1 time travel | 半年前のスナップショットから再構築 |
| 10 | 自分用に「今月の決定事項」レポート | B6 改 | weekly review の自動レポート化 |

ユースケース毎に求められる射影面が違う:
- (1)(2) → 構造厳密、文字数制約厳しい、文体フォーマル
- (3)(4) → 演出・順序・1スライド1メッセージ、口語可
- (5)(6) → 公開向けマスキング、SEO、画像
- (7) → 短文、Q&A 形式、忘却曲線対応
- (8) → 仕様準拠、テキストは伏せられる
- (9)(10) → 時系列・差分・再現性

## 既存ツール / 競合との差分

### 競合: 汎用エディタ系
- **Notion / Obsidian publish**: マップ構造を活かさず、線形ドキュメントとして出すだけ
- **Roam → 出力**: グラフ → ドキュメント変換は手動でしかできない
- **Logseq → ブログ**: 1ページ = 1記事の前提でグラフは活きない

### 競合: 学術系
- **Overleaf / LaTeX**: ゼロから書く前提。マップ → 雛形のステップが無い
- **Manuscripts / Authorea**: クラウド執筆だが構造化された思考ログとは別レイヤー
- **Scrivener**: アウトライン執筆だが、マップ的双方向リンクは弱い

### 競合: スライド系
- **Reveal.js / Marp**: Markdown → スライド。マップ構造を入力にしない
- **PowerPoint Designer**: AI レイアウトだが構造ソースが無い

### 競合: SRS / 公開系
- **Anki**: 自分でカード書く前提
- **Hugo / Jekyll**: ブログ生成だが M3E 連携無し

### M3E でやる価値
- **マップ＝唯一の真実源**: 出力物は派生物。二重管理しない
- **構造化されたまま射影**: ヘッダ・章節・関係性が落ちない
- **複数射影面**: 同じソースから論文も申請書もスライドも
- **段階的 LLM 介入**: 純テンプレから自由生成まで連続的に選べる
- **プライバシー境界**: マップ自体は private、射影時にだけ公開判断

## 公開行為の心理学（隠れたユースケース）

「出力する」は単なる技術行為ではなく、**書き手の心理状態**と密接:

| 心理状態 | 求められる体験 |
|---------|--------------|
| 白紙の恐怖 | テンプレが既に埋まっている状態でスタート |
| 完璧主義による出版遅延 | "とりあえず投げる" モード（草稿マーク付き） |
| 書きたくないけど書かなきゃ | AI に8割書かせて自分は校正だけ |
| 自分の思考に自信が無い | Devil's Advocate（C1）連動で先に弱点潰す |
| 過去の自分を恥じる | time travel（I1）で旧マップから過去スタイル再現 |
| 共著者に見せるのが怖い | 限定公開・コメント付きレビュー先行 |

→ export は **執筆のメンタルブロック解除** でもある。
　automation_obstacles の「白紙の恐怖」「自己欺瞞」障害群と直接接続する。

## やらないこと（初期スコープ外候補）

- **完全自動論文生成**（捏造リスク・倫理問題、研究倫理違反になる）
- **査読そのものの自動化**（C エージェント深化の領分、export ではない）
- **印刷物デザイン（DTP）**（InDesign 領域、専門ツールに任せる）
- **動画編集 GUI**（B4 は書出しまで、編集は Premiere 等）
- **マップ閲覧者が直接編集できる公開モード**（E コラボの領分）

これらは「あったら嬉しいが本質ではない」「別カテゴリの領分」なので別フェーズ。

## 「公開」と「書き出し」の区別

似て非なる2軸を意識:

| 軸 | 例 | 主目的 |
|----|----|------|
| **書き出し（Export）** | LaTeX ファイル生成、Anki .apkg 出力 | 別ツールに渡す |
| **公開（Publish）** | Web 公開、論文投稿 | 第三者に届ける |

- 書き出しだけして公開しないこともある（自分用 Anki）
- 公開には承認・マスキング・流通・DOI などの追加レイヤー
- Pattern A としては export を基底、publish はその上のオプションとして整理

## 出力物のライフサイクル

「出力したら終わり」ではなく、ループの一部:

```
マップ更新 → 射影 → 出力物 → 読者反応 → コメント・引用・修正依頼 → マップに取り込み → ...
```

このループを **どこまで M3E で扱うか** は論点 Cy（一方向／双方向）で扱う。

## 「読者モデル」を明示する

研究者の出力は読者によって全く違う形にすべき。M3E では出力時に **読者モデル** を選ぶ:

| 読者 | 既知前提 | 期待形式 | 文体 | 評価軸 |
|------|--------|--------|-----|------|
| 査読者（同分野） | 用語OK、先行研究知る | 論文厳密 | フォーマル、能動 | 新規性・厳密性 |
| 査読者（隣接分野） | 用語要解説、背景必要 | 論文+解説 | フォーマル、丁寧 | 妥当性・普遍性 |
| 配分機関審査員 | 専門外も含む | 申請書定型 | フォーマル、簡潔 | 社会的価値 |
| 指導教員 | 研究文脈共有 | スライド・口頭 | カジュアル可 | 進捗・問題発見 |
| 共同研究者 | 共通文脈最大 | 内部メモ | 業界用語OK | 役割分担明確化 |
| 学生・後輩 | 用語ゼロから | 教科書的 | 平易、例多 | 理解しやすさ |
| 一般読者 | ゼロ知識 | ブログ・記事 | 物語的 | 面白さ・分かりやすさ |
| 未来の自分 | 当時の文脈忘却 | 構造ログ | 自由 | 思い出せるか |

→ 読者モデル選択 = 出力プロファイル選択 と直結。

## 出力フォーマットの分類軸

出力先 B1〜B8 を以下の軸で分類できる:

### 軸1. 構造化度
- 高（論文 IMRAD・申請書）: 形式が決まっている
- 中（ブログ・スライド）: 緩い構造
- 低（動画・SNS）: 自由

### 軸2. テキスト/メディア比率
- テキスト中心: 論文・申請書・ブログ
- 視覚中心: スライド・動画・図
- 構造のみ: RDF・SKOS

### 軸3. 同期 vs 非同期
- 同期（presentation）: スライド・動画
- 非同期（published）: 論文・ブログ・申請書

### 軸4. 一回性 vs 反復性
- 一回（論文・申請書）: 一度提出したら確定
- 反復（ブログ・SRS）: 何度も生成

### 軸5. 機械可読 vs 人間可読
- 機械: RDF・JSON-LD
- 人間: それ以外
- 両対応: 構造的 Markdown

これらの軸の組合せで「どの出力先に注力するか」の判断ができる。

## 論点（このファイルから）

- **論点 Ex1. 出力単位** — マップ全体／subtree／タグセット／クエリ／時系列
  - 推し: subtree ベース、タグでフィルタ
- **論点 Vw1. 中間ビュー** — マップに「IMRAD ビュー」「章節ビュー」を乗せる vs 別 file
  - 推し: マップ上に "ビュー定義ノード" を置く（slideshow と同じ哲学）
- **論点 Cy1. 双方向 vs 一方向** — 出力後の編集をマップに戻すか
  - 推し: 一方向を基本、コメント取込みは別フェーズ
- **論点 Vn1. バージョン管理** — 出力物 v1 / v2 をマップ snapshot と紐付けるか
  - 推し: snapshot ID を出力物に埋め込み、time travel と接続
- **論点 Rd. 読者モデル選択** — 上記 8 種から選ぶ／自由定義／プロファイル統合
  - 推し: 同梱 8 種 + ユーザ定義可、プロファイルにバンドル
- **論点 Ax. 分類軸の重視** — 構造化度／メディア比率／同期性／反復性／可読性
  - 推し: 構造化度を最重視（M3E はグラフ構造が強み）
