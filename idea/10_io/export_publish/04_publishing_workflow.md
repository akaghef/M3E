# 04. 出版ワークフロー — 草稿から DOI まで

「書き出した後」の世界。学術出版業界の慣習（査読・引用・DOI・OA）を踏まえ、
M3E がどこまで関与すべきかをブレストする。

## 出版ワークフローの全体像

```
草稿 → 共著者レビュー → 投稿前チェック → ジャーナル投稿
  → 査読 → 改訂（リバイス）→ 受理 → 校正 → 出版
  → DOI 付与 → 流通（プレスリリース・SNS）→ 引用
  → ポストパブリケーション議論（コメント・引用ネットワーク）
```

各段階で M3E が介入できる/すべき/すべきでない箇所がある。

## 段階別の M3E 関与度

| 段階 | M3E 関与度 | 主な機能 | 既存 idea 連携 |
|------|----------|---------|--------------|
| 草稿生成 | 高 | B1, B2 出力 | export_publish 本ブレスト |
| 共著者レビュー | 中 | 限定共有、コメント取込 | E コラボ |
| 投稿前チェック | 中 | リント、マスキング | 03 Pp |
| 投稿 | 低 | API 投稿は将来 | H ツール統合 |
| 査読対応 | 中 | 査読コメントをマップに取込 | C Devil's Advocate |
| 改訂 | 高 | 改訂版生成、差分管理 | I time travel |
| 受理・校正 | 低 | 校正は別ツール | (外) |
| DOI 付与・流通 | 低〜中 | 投稿先案内、SNS 文生成 | B6 ブログ |
| ポスト議論 | 中 | 引用・コメントを取込 | A8 AI 対話取込 |

## A. 草稿〜投稿前

### A1. 草稿の質保証
- 03 のパイプラインで生成した草稿の **自己レビュー**
- 案 A1a: LLM がジャーナル基準で「足りない節」を指摘
- 案 A1b: Devil's Advocate（C1）と連動して批判生成
- 案 A1c: 過去の自分の論文を参照し「いつもの弱点」を提示
- 案 A1d: 共著者ペルソナで予測コメント生成

### A2. 共著者レビュー
- 「マップを共有」vs「出力物を共有」
| 案 | 共有単位 | 良い点 | 悪い点 |
|----|---------|------|------|
| A2a | マップ全体（read-only） | 思考過程まで見える | 公開範囲広すぎ |
| A2b | 出力物（PDF/LaTeX） | 既存ワークフロー | 文脈失われる |
| A2c | 出力物 + コメント可ノード | 中庸 | 実装負荷 |
| A2d | 出力物に M3E node ID 埋め込み（双方向リンク） | 文脈追える | UI 工夫要 |

### A3. 投稿前チェックリスト
- ジャーナル別チェックリスト（abstract 250語、図数 6 まで等）
- COI（利益相反）開示
- 倫理審査番号
- 著者貢献度（CRediT taxonomy）
- 案 A3a: マップ内に「投稿前チェックリスト」テンプレ
- 案 A3b: ジャーナル選択 → 自動でチェックリスト生成

### A4. 投稿フォーム自動入力
- 案 A4a: 手動（M3E 関与なし）
- 案 A4b: ブラウザ拡張 + マップ → フォーム自動入力
- 案 A4c: ジャーナルの API 経由（一部のみ対応）
- 推し: A4a で十分、A4b は楽したい人向け

## B. 査読対応

### B1. 査読コメントの取込
- 査読コメント PDF/メールを M3E マップに取込
- 各コメント = 1ノード、対応する原稿ノードにリンク
- ステータス: pending/in-progress/done/rejected

### B2. 改訂対応プロセス
| 案 | 内容 | 良い点 | 悪い点 |
|----|------|------|------|
| B2a | コメント1つに対し response ノードを作る | 構造化 | ノード数膨張 |
| B2b | 元論文ノードに「改訂履歴」を attribute で | 軽い | 比較しにくい |
| B2c | time travel（I1）で「改訂前/後」スナップショット | 履歴見える | 機能依存 |
| B2d | revision-letter テンプレで一括出力 | フォーマル要件満たす | ボイラープレート |

### B3. 査読者ペルソナでの自己査読
- C エージェントで「査読者役」を演じさせる
- 「nature reviewer / niche journal reviewer / hostile reviewer」プロファイル
- 提出前リハーサル

## C. 受理〜出版

### C1. 校正対応
- 通常は出版社が PDF 校正を送る（M3E 関与少）
- ただし **マップに最終版 PDF を取込んで「決定版」マーク** はやる価値あり
- 案 C1a: 受理ノードに最終 PDF 添付、過去版を archive
- 案 C1b: DOI 取得後、自動でマップに DOI ノード追加

### C2. preprint vs ジャーナル
- preprint（arXiv, bioRxiv）: 早期公開、DOI あり
- 案 C2a: M3E から preprint 投稿用 LaTeX 出力（B1 の派生プロファイル）
- 案 C2b: 投稿先別プロファイル（Nature, Science, Cell, IEEE 等）

## D. ポストパブリケーション

### D1. 流通（promotion）
- Twitter / Mastodon / SNS 投稿用の短文生成
- プレスリリース風文章生成
- 図 1 つを「Twitter カード」に
- 案 D1a: B6 ブログ生成と統合
- 案 D1b: 専用 "promo" プロファイル

### D2. 引用追跡
- Google Scholar / OpenAlex / Semantic Scholar API で被引用追跡
- 新しい引用を M3E にノードとして取込
- → A8（AI 対話取込）と同じ取込パターン

### D3. ポスト議論の取込
- PubPeer のコメント、SNS の議論を M3E に
- 自分の論文ノード配下に「外部反応」subtree
- 次の研究のネタに

## 業界慣習を踏まえた設計考慮

### 査読
- 一般に「double-blind」「single-blind」「open peer review」がある
- マップから著者名・所属を抜く出力プロファイルが必要
- → L4 匿名共有 / L2 マスキング と連動

### 引用形式
| 形式 | 例 | 主分野 |
|------|---|------|
| APA | (Smith, 2020) | 心理学・社会科学 |
| IEEE | [1] | 工学 |
| Vancouver | (1) | 医学 |
| ACS | super-script | 化学 |
| Chicago | footnote | 人文 |

- 案: M3E は中間表現で持ち、出力時に変換（CSL = Citation Style Language 採用）
- Zotero / Mendeley も CSL 採用 → 互換性

### DOI
- DOI は基本的に出版社/preprint サーバが付与
- Zenodo は誰でも DOI 取れる（ソフトウェア・データセット）
- 案: M3E マップ自体に Zenodo DOI を取って引用可能にする？
  - 良い点: マップを論文の supplementary として参照可能
  - 悪い点: 公開リスク、versioning 管理

### OA（オープンアクセス）
- Gold OA: ジャーナルが公開、APC（著者払い）
- Green OA: preprint or リポジトリで著者公開
- 案: Green OA のためのリポジトリ投稿支援（HAL, OSF, 機関リポジトリ）

### ライセンス
- CC-BY, CC-BY-NC, CC0
- 出力物にライセンス自動付与
- マップ内ノードのライセンスを継承

### ORCID
- 著者識別子
- 申請書・投稿フォームで必須化
- M3E ユーザプロファイルに ORCID を保持

## 申請書（B2）の特殊ワークフロー

論文と異なる点:

### 段階
```
公募要項読解 → テンプレ生成 → 草稿 → 内部レビュー（共同研究者）→ 提出 → 採択/不採択 → 報告書
```

### 特殊性
- **公募要項が毎年微妙に変わる** → テンプレを毎年更新必要
- **採択率 20-30%** → 不採択時の改善ループが重要
- **業績欄の自動更新** → 過去業績マップから自動抽出
- **ガントチャート** → マップから生成（M3 タイムライン機能と連携？）
- **予算計画** → 別シートだが、マップ上の "task" ノードと工数連動

### 申請書テンプレの育成
- 案 W1. 過去の自分の申請書（採択・不採択）を学習データに
- 案 W2. 採択者の公開申請書（KAKEN DB）から学習
- 案 W3. 「採択された申請書はこういう構造」をテンプレ化

## 引用ネットワーク管理

### 案 Cn1. マップ内に文献ノード
- 1論文 = 1ノード、メタデータ（DOI, 著者, 年）を attribute
- 自分の論文との関係性をリンクで表現

### 案 Cn2. Zotero 連携
- Zotero ライブラリと M3E マップを双方向同期
- 引用 metadata は Zotero に、思考メモは M3E に

### 案 Cn3. 引用ネットワーク可視化
- map_views（idea/map_views/）に引用ネットワーク view 追加
- 「自分の研究の引用先」「自分を引用している論文」

## 外部システム連携の優先度

| 連携先 | 用途 | 優先度 |
|-------|-----|------|
| Zotero | 引用管理 | 高 |
| Overleaf | 論文執筆 | 中 |
| arXiv/bioRxiv | preprint 投稿 | 中 |
| KAKEN DB | 過去申請書参照 | 高（B2 のため）|
| ORCID | 著者識別 | 中 |
| OpenAlex | 引用追跡 | 中 |
| Semantic Scholar | 関連論文発見 | 中 |
| Crossref | DOI 解決 | 中 |
| Zenodo | データ・コード公開 | 低 |

## 横断観察

- **学術出版は段階多くて遅い**。M3E は草稿生成 + 査読対応 + 改訂 に注力
- **査読対応が二度目の本丸**。コメント取込 → 改訂版生成のループ
- **申請書ワークフローは論文と独立**。テンプレ更新と業績自動更新が鍵
- **CSL（Citation Style Language）採用で引用形式問題は解決**
- **Zotero 連携が引用管理の現実解**
- **DOI 取得は出版社任せでよい**、自分で取るのは Zenodo のデータ・マップ程度
- **査読者ペルソナによる自己査読** は C カテゴリ（AI 深化）と接続できる強力機能

## 論点

- **Au1. 投稿フロー** — 手動 / ブラウザ拡張 / API 直接
- **Rv1. 査読対応モデル** — コメントノード化 / 改訂履歴 / time travel
- **Pr1. 投稿先別プロファイル** — 同梱 / 自作 / コミュニティ
- **Cit. 引用形式** — CSL 採用、Zotero 連携
- **Pn. 査読者ペルソナ** — C エージェントとの統合
- **Wf. 申請書ワークフロー** — テンプレ更新方式、業績自動取込
- **Doi. DOI 取得** — マップ自体を Zenodo に、論文は出版社任せ
- **Oa. OA 戦略** — Green OA リポジトリ連携
- **Cb. 共著者連携** — マップ共有 / 出力物共有 / 双方向リンク
