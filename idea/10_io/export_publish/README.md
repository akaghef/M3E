# 出力・公開（Export / Publish）— ブレインストーミング

M3E マップを「外に出す」体験設計のブレスト。
研究者用途では **入力（capture）と並ぶ最重要チャネル**。
project_projection_vision（世界モデル → 射影で科研費出力）に直結する本丸テーマ。

レビュー保留のため、各論点について **複数の選択肢を並べ、決定はしない**。

## 目的（一言）

マップに溜め込んだ思考・知識・意思決定ログを、**外部世界（読者・査読者・申請窓口・聴衆・将来の自分）に
読める形に射影** する。マップは内部表現、出力物はその場面ごとの「投影面」。

## 方針

- 採用判断はしない
- 実装は考えない（最後の MVP ファイルでだけボリューム感に触れる）
- 出力先（論文／申請書／スライド／動画／Web／ブログ／SRS／RDF）を **横並びに比較**
- 学術出版業界の慣習（IMRAD / 査読 / DOI / 引用 / フォーマット制約）を踏まえる
- プライバシー（公開時の機微情報マスキング）は L1〜L5 への参照で扱う
- 既存ブレスト（slideshow / capture_ingest / automation_obstacles）と接続

## 論点一覧（後で ID 単位で選ぶ）

- **Ex. 出力単位の粒度** — マップ全体／subtree／タグセット／クエリ結果
- **Tp. テンプレート方式** — 固定テンプレ／可変テンプレ／LLM プロンプト合成／ユーザ DSL
- **Vw. 中間ビュー** — IMRAD / 章節 / Q&A / 時系列 / クラスタ … マップにビューを乗せるか別 file か
- **LLM. LLM の介入度** — 0（純テンプレ展開）／1（要約のみ）／2（文章生成）／3（構成も AI）
- **Cit. 引用・出典管理** — ノード属性 / 別 .bib 連携 / Zotero 統合 / DOI 解決
- **Fmt. 出力フォーマット** — Markdown / LaTeX / HTML / PPTX / PDF / Reveal.js / mp4 / Anki .apkg / .ttl
- **Vo. ボイス・トーン制御** — フォーマル/カジュアル/技術系の切替、誰に向けて書くか
- **Ms. マスキング戦略** — 自動／半自動／手動承認、ノード属性 vs ルールベース（L2/L4/L5 連動）
- **Pv. プライバシー境界** — 公開Vault と private を分離、書き出し時に境界チェック（L3）
- **Cy. 双方向 vs 一方向** — 出力した後の修正をマップに戻すか、出力は throw-away か
- **Au. 出版ワークフロー連動** — 査読依頼 / OAリポジトリ投稿 / プレプリント / DOI 取得
- **In. 段階公開** — 完全公開 / 限定公開（リンク知る人）/ 査読者のみ / 自分だけ
- **Re. 再現性** — 同じマップから何度出力しても同じ結果か、生成ログを残すか
- **Vn. バージョニング** — 出力物の v1, v2 を区別、マップのスナップショットと紐付け
- **Up. アップデート伝播** — マップを更新したら過去出力物を再生成するか、差分通知だけか
- **Cb. コラボ系出力** — 共著者と「同じマップから自分の章だけ出す」体験
- **Bd. 業界フォーマット制約** — ジャーナル毎のスタイル、申請書の文字数制限、動画長制限

## キーメッセージ（ブレスト前提）

- **マップ＝内部表現、出力物＝射影**。同一マップから複数射影が出る前提で考える
- **出力単位 = subtree が現実解**。マップ全体を1つの論文にするのは大きすぎる
- **LLM 介入度はユーザが選べるべき**。0（純テンプレ）から 3（自由生成）まで連続的に
- **査読・申請書の "形式制約" は M3E の強み**。構造化されたマップは構造化された出力に強い
- **プライバシー境界が出力時の最大リスク**。capture 時より公開時にこそ機微情報が漏れる
- **"出力したら終わり" にせず、出力物の反応を再びマップに取り込むループ** を設計したい
