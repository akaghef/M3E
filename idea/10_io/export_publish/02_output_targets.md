# 02. 出力先カタログ — B1〜B8 の比較

各出力先について、**何が出るか／読者は誰か／業界慣習／LLM 介入度／実装難度** を並べる。
採否は決めない。

## 全体比較表

| ID | 出力先 | 主読者 | 形式 | 慣習・制約 | LLM 介入度 | 実装難度 | 推し度 |
|----|-------|-------|------|----------|----------|--------|-------|
| B1 | 論文ドラフト | 査読者・学術界 | LaTeX / Markdown | IMRAD・引用・図番号 | 中（要約・接続文） | 高 | ⭐⭐ |
| B2 | 申請書 | 配分機関・審査員 | docx / PDF | 字数制限・項目厳密 | 中（圧縮要約） | 高 | ⭐⭐ |
| B3 | スライド | 聴衆 | PPTX / Reveal.js | 1枚1メッセ・図中心 | 低〜中 | 中 | ⭐ |
| B4 | 動画＋ナレ | 聴衆（非同期） | mp4 / WebM | 長すぎNG・字幕推奨 | 高（読み上げ文） | 高 | △ |
| B5 | 静的サイト | 不特定多数 | HTML/CSS | SEO・OGP | 低 | 中 | ⭐ |
| B6 | ブログ記事 | 読者 + 自分 | Markdown | タイトル・タグ・冒頭リード | 中 | 低 | ⭐ |
| B7 | SRS カード | 自分 | Anki .apkg / CSV | Q&A・短文 | 中（質問生成） | 低 | ⭐ |
| B8 | 知識グラフ | 他システム | RDF/JSON-LD/SKOS | W3C 仕様準拠 | なし | 中 | (基盤) |

## B1. 論文ドラフト自動生成 ⭐⭐

### 何が出るか
- IMRAD 構造の LaTeX or Markdown 雛形
- セクション見出し + 各節の "種" 文章 + 引用プレースホルダ
- 図キャプション・表・参考文献リスト

### 読者と慣習
- **査読者**: 厳密な構成、論理飛躍なし、引用根拠あり、再現性
- **共著者**: 編集しやすいフォーマット（LaTeX なら .tex、共有なら Overleaf）
- **ジャーナル特有**: スタイルファイル（.cls）、文字数、図数、引用形式

### IMRAD ビューの作り方候補
| 案 | 内容 | 良い点 | 悪い点 |
|----|------|------|------|
| B1a | マップノードに `imrad: introduction` 等のタグ | 既存マップに後付けしやすい | タグ漏れで欠落しがち |
| B1b | 別途 `imrad-view.json` で構造定義 | 構造とマップを分離 | 二重管理 |
| B1c | 専用 "論文 subtree" を作る（手動配置） | 明示的、確実 | 手間が大きい |
| B1d | LLM が IMRAD を推論して提案 → ユーザ承認 | ゼロから書かなくていい | 精度依存 |

### LLM 介入レイヤー
- L0: ノード text を貼り付けるだけ（純テンプレ）
- L1: 接続文・段落導入のみ AI 生成
- L2: 各節を AI が再構成（人間は校正）
- L3: 構成・論理展開も AI（倫理・捏造リスク）

### 関連論点
- Cit（引用管理）と密接、Zotero 統合（H4）と組合せると強い
- ジャーナルスタイル多様性 → 出力プロファイルを切替可能に？

## B2. 科研費・申請書テンプレ生成 ⭐⭐

### 何が出るか
- 科研費の決まったセクション（研究目的・方法・期待効果・準備状況・業績）
- 字数制限内に収まった文章（800字、1200字 等）
- 必要な図・表（研究計画図、ガントチャート）

### 読者と慣習
- **配分機関の審査員**: 短時間で読む、要点が冒頭、専門外でも分かる
- **形式厳守**: フォーマット崩れは即減点
- **業績欄**: 過去論文・受賞をリスト化、定型句多い

### 申請書テンプレ候補
| 案 | 内容 | 良い点 | 悪い点 |
|----|------|------|------|
| B2a | 科研費「基盤C」固定テンプレ + マップから抽出 | 即実用 | 他制度に転用しにくい |
| B2b | 制度別プロファイル（基盤A/B/C・若手・挑戦的等） | 多制度対応 | プロファイル整備コスト |
| B2c | テンプレを「マップ上のノード」として持つ | ユーザが自分で作れる | 学習コスト |
| B2d | 公募ごとに DSL 定義 → AI が拡張 | 柔軟性最大 | 仕様策定が重い |

### 圧縮の課題
- 字数制限内に収めるための要約は LLM 必須
- "重要度" 属性で何を残すかランキング → 落とすものを明示
- ユーザが手動で「削るところ」を選べる UI が必要

### 関連
- project_projection_vision の本丸
- B1 と多くを共有（IMRAD vs 申請書フォーマット）
- 過去申請書を時系列で蓄積 → 学習素材化（自分用 LoRA 的）

## B3. スライド自動生成 ⭐

### 何が出るか
- PPTX または Reveal.js（HTML）形式のスライド
- マップ subtree → 1ノード1スライド or 1subtree1スライド
- カメラワーク（slideshow との連携）

### 読者と慣習
- **聴衆**: 1分1枚ペース、図中心、テキスト最小
- **配信用**: 字幕、PDF 配布版

### 構造変換候補
| 案 | 変換 | 良い点 | 悪い点 |
|----|------|------|------|
| B3a | 1ノード = 1スライド | シンプル | スライド数膨大化 |
| B3b | 1subtree = 1スライド（タイトル + 子のbullet） | 適度な圧縮 | subtree粒度に依存 |
| B3c | slideshow ツアー定義 = スライド順序 | 既存資産活用 | slideshow 依存 |
| B3d | テンプレ別: title slide / chart / quote | リッチ | テンプレ整備コスト |

### slideshow との関係
- slideshow（idea/slideshow/）はマップ上のカメラワーク
- B3 はそれを「外部ファイル」として書き出すバージョン
- → 同じツアー定義を流用、出力先だけ切替（マップ内 / PPTX / Reveal.js）

## B4. ナレーション付き動画書き出し △

### 何が出るか
- マップツアー（カメラ動作）の画面録画 mp4
- 音声合成によるナレーション（ノードの details を読み上げ）
- 字幕 .srt

### 読者と慣習
- **YouTube / 学会非同期発表**: 5〜15分、字幕必須、サムネ重要
- **自分用学習**: ながら聞き、倍速再生

### 技術選択肢
| 案 | 動画生成 | 音声合成 | 良い点 | 悪い点 |
|----|---------|---------|------|------|
| B4a | ヘッドレス Chrome 録画 | OS の TTS | 軽い | 音質低い |
| B4b | Puppeteer + ffmpeg | ElevenLabs / 11.ai | 高品質 | API コスト |
| B4c | Remotion (React 動画) | OpenAI TTS | プログラマブル | 学習コスト |
| B4d | 出力は「動画用素材」のみ（人間が編集） | - | 失敗が無い | 自動化メリット薄 |

### 慎重論
- 動画は失敗が目立ちやすい（音ズレ・読み間違い）
- "静止画 + 音声" のシンプル形式から始める
- 投資対効果は B1/B2/B3 より低い（推し度 △）

## B5. Web 公開（静的サイト）⭐

### 何が出るか
- HTML/CSS/JS の静的サイト（GitHub Pages / Netlify / Cloudflare Pages デプロイ）
- ノード 1 つ = 1 ページ or タグ別ページ
- 検索・ナビゲーション・OGP

### 読者と慣習
- **検索流入**: SEO（タイトル・meta・URL構造）
- **SNS シェア**: OGP 画像・タイトル
- **Web アクセシビリティ**: WCAG 配慮

### 公開戦略候補
| 案 | 公開範囲 | 良い点 | 悪い点 |
|----|---------|------|------|
| B5a | publish タグ付きノードのみ | 厳格、漏れにくい | タグ付け手間 |
| B5b | private タグ以外全部 | 楽 | 漏洩リスク |
| B5c | 別 workspace を「公開用」として分離 | 物理分離で安全 | 二重管理 |
| B5d | 動的フィルタ（ロール別公開） | 柔軟 | 複雑 |

### マスキング（L2/L4）
- 公開前に LLM が機微情報を検知（L5）
- ユーザ確認モーダル
- 自動マスキング（人名 → 役職、固有名詞 → 一般化）

## B6. Markdown ブログ生成 ⭐

### 何が出るか
- 1記事 = 1 Markdown ファイル
- frontmatter（title, date, tags, draft）
- 既存ブログツール（Hugo / Astro / Zola）に投入可能

### 読者と慣習
- **ブログ読者**: タイトル・冒頭リードで読む読まない判断
- **検索エンジン**: 見出し階層、内部リンク

### 抽出戦略候補
| 案 | 抽出元 | 良い点 | 悪い点 |
|----|-------|------|------|
| B6a | publish タグ + 重要度高 のノード | 質重視 | 量出にくい |
| B6b | 週次レビュー subtree から自動 | 定期発信 | 内輪話になりがち |
| B6c | ユーザが「これブログ化」と1キー指定 | 意図的 | 忘れがち |
| B6d | LLM がマップ全体から「記事ネタ」提案 | ネタ枯れない | ノイズ多い |

### 既存ツール統合
- Hugo / Astro / Zola → frontmatter フォーマットを合わせる
- Dev.to / Medium API → 直接投稿
- note.com → API 経由

## B7. Anki/SRS カード書き出し ⭐

### 何が出るか
- Anki .apkg ファイル または CSV
- カード = (front, back, tags) の組
- メディア（画像、音声）も同梱可

### 読者と慣習
- **自分**: 短時間で復習、moble で
- **チーム共有**: 用語集、研究室の知識基盤

### カード生成候補
| 案 | 生成方法 | 良い点 | 悪い点 |
|----|---------|------|------|
| B7a | ノード text → front, details → back | 機械的、確実 | カード品質微妙 |
| B7b | LLM が穴埋め問題自動生成 | カード品質高い | API コスト・誤り |
| B7c | ユーザが質問形式でノード書き、それをカード化 | 設計時から考慮 | 既存マップに弱い |
| B7d | "重要度 5" のノードのみ自動でカード化 | 厳選 | 手動属性付け必要 |

### J カテゴリ（SRS 統合）との関係
- J1（SRS 統合）は「マップ内で復習」アプローチ
- B7 は「外部 Anki に流す」アプローチ
- 両立可能（マップ内 + Anki 同期）

## B8. 知識グラフ標準形式エクスポート (基盤)

### 何が出るか
- RDF/Turtle (.ttl) ファイル
- JSON-LD (Schema.org / SKOS / FOAF)
- SKOS（概念体系の語彙）

### 読者と慣習
- **他システム**: 機械可読
- **W3C 標準準拠**: vocab を選ぶ（DCMI, SKOS, FOAF, BIBO 等）

### 用途
| 案 | 用途 | 良い点 | 悪い点 |
|----|-----|------|------|
| B8a | Wikibase / Wikidata 連携 | 公共知識基盤 | 公開リスク |
| B8b | 学術知識グラフ（OpenAlex 連携） | 引用ネットワーク化 | 仕様学習必要 |
| B8c | 自分用バックアップ（標準形式） | ベンダロック回避 | 過剰品質 |
| B8d | 他 PKM ツール移行用（Logseq, Obsidian） | データ portability | マッピング難 |

### 哲学的価値
- マップを「ベンダロックされない形式」で持つ保険
- 将来的にセマンティック Web 連携、AI エージェント間共有

## 横断観察

- **B1, B2 は本丸、B3, B6 は容易、B4 は重い、B7, B8 は基盤**
- **B1〜B6 は LLM 必須、B7 は半必須、B8 は不要**
- **B1, B2, B5 はプライバシー高リスク**（マスキング必須）
- **B3, B5, B6 は slideshow / web 系で技術スタックが近い**
- **B7, B8 は別ツールへ流すための portability 軸**
- **マップ → 中間ビュー → 出力物の二段階変換が共通**
