# 06. MVP / 段階導入 / 未決質問

「採否は決めない」前提で、**仮に最小実装するならどう始めるか** の見取り図。
project_projection_vision の半年デッドライン（〜2026-10）を意識。

## MVP の絞り込み

### 全 8 出力先（B1〜B8）から MVP 候補を選ぶ

採点軸（高いほど MVP 向き）:
| 軸 | 重み |
|----|-----|
| ユーザ価値（akaghef にとって）| ×3 |
| 実装容易性 | ×2 |
| ビジョン整合性（projection_vision）| ×3 |
| 既存資産活用 | ×2 |
| プライバシーリスク低さ | ×1 |

| 出力先 | 価値 | 容易 | ビジョン | 資産 | プライバシ | 合計 |
|-------|-----|-----|--------|-----|---------|-----|
| B1 論文 | 5 | 2 | 5 | 3 | 2 | 41 |
| B2 申請書 | 5 | 2 | 5 | 3 | 2 | 41 |
| B3 スライド | 4 | 4 | 2 | 5 (slideshow) | 3 | 35 |
| B4 動画 | 2 | 1 | 1 | 2 | 3 | 16 |
| B5 静的サイト | 3 | 3 | 2 | 3 | 1 | 24 |
| B6 ブログ | 3 | 5 | 2 | 3 | 2 | 31 |
| B7 SRS | 3 | 5 | 1 | 2 | 4 | 27 |
| B8 RDF | 2 | 4 | 2 | 4 | 5 | 28 |

→ **B1 / B2 が同点首位、B3 / B6 が次点**

### 推し: B6 → B2 → B1 の順で着手

理由:
- **B6 で「マップ → Markdown 出力」のパイプライン基盤を作る**（最も簡単）
- **B2 で「テンプレ + 字数制約」を学ぶ**（申請書の本丸へ）
- **B1 で IMRAD・引用・LaTeX に拡張**（論文の本丸）
- B3 は slideshow ブレストと並行で枝として進む

別案: B2 を最優先（projection_vision 直結）も合理的。

## MVP 1: B6 Markdown 出力（最小）

### スコープ
- マップ subtree を 1 つ指定
- Markdown ファイルを生成（frontmatter + 本文）
- LLM 介入なし（L0）
- マスキングなし（自分用のみ）

### ボリューム見積
- 約 100〜200 行
- マップ走査 + Markdown レンダリング
- 既存 m3e-map.json 構造を読むだけ

### コマンド例（仮）
```
m3e export --target md --node <node_id> --out post.md
```

### 出力サンプル
```markdown
---
title: ノードタイトル
date: 2026-04-16
tags: [research, m3e]
---

ノード本文...

## 子ノード1

子ノード1の text...

## 子ノード2

...
```

### 達成条件
- `outputs/<output_id>.md` にファイル生成
- マップ snapshot ID をメタとして記録
- 再実行で同じ出力（再現性）

## MVP 2: B2 申請書テンプレ（小）

### スコープ
- 科研費「基盤C」テンプレ 1 つ固定（Tp1）
- マップから「研究目的」「方法」「期待効果」相当 subtree を抽出
- 字数制限を表示（超過時は警告のみ、自動圧縮はオプション）
- LLM 介入度 L0 or L1（要約のみ）

### ボリューム見積
- 約 300〜500 行
- テンプレ定義 + マップ抽出 + 字数チェック
- LLM API 呼出は L1 でのみ

### 段階
- W1: テンプレ + マップ抽出（L0）
- W2: 字数チェック・超過警告
- W3: 字数超過時の LLM 圧縮（L1）
- W4: 業績欄の自動取込（過去論文ノードから）

### 達成条件
- `outputs/kakenhi_kibanC_<date>.md` を生成
- 各セクションが字数制限内 or 警告
- マップとの双方向リンク（出力物から元ノード ID 辿れる）

## MVP 3: B1 論文ドラフト（中）

### スコープ
- IMRAD ビュー定義（タグベース、Cv1）
- LaTeX 出力 + Markdown 出力の両方（プロファイル切替）
- 引用は CSL 経由（最初は APA 1 つだけ）
- LLM 介入度 L1（接続文のみ）
- 図・表は **手動配置**（自動生成は範囲外）

### ボリューム見積
- 約 500〜1000 行
- IMRAD 抽出 + LaTeX レンダリング + 引用処理
- Zotero 連携は別フェーズ

### 段階
- P1: IMRAD タグ付け UI（マップ側）
- P2: 抽出 + LaTeX 雛形生成（L0）
- P3: LLM による接続文生成（L1）
- P4: 引用形式変換（CSL）
- P5: pdflatex ビルド検証

### 達成条件
- `outputs/paper_<id>.tex` を生成
- pdflatex でビルド成功
- 引用が正しいフォーマット

## MVP 4: B3 スライド（slideshow と統合）

### スコープ
- slideshow ツアー定義 → Reveal.js HTML 出力
- 既存 slideshow ブレストの成果物を流用
- ノード text → スライド内容、details → speaker note

### ボリューム見積
- 約 200〜400 行
- slideshow が実装済みなら流用部分多い

### 段階
- S1: ツアー定義の読込
- S2: Reveal.js テンプレレンダリング
- S3: PPTX 出力（python-pptx 等使用）

### 注意
- slideshow ブレストの実装次第で先後が変わる

## 共通基盤（最初に作るべき）

すべての出力先で共通のコンポーネント:

### Common 1. マップ走査 API
- subtree 抽出、タグフィルタ、属性取得
- 既存の m3e-map.json 構造に対するクエリ

### Common 2. メタデータ管理
- output_id, snapshot_id, profile, llm_level
- `outputs/.meta.json` として一元管理

### Common 3. プロファイル loader
- profile を読込、テンプレ + 設定を返す
- `profiles/` フォルダ配下に同梱プロファイル

### Common 4. LLM ラッパ（介入度切替）
- L0: 何もしない
- L1: 要約 API
- L2+: 文章生成 API
- ローカル LLM 切替対応（プライバシー）

### Common 5. マスキングフィルタ
- Ms1（属性ベース）を最初に
- Ms2（正規表現）を補助で

### Common 6. ファイル出力 + バックアップ
- `outputs/<format>/<output_id>.<ext>`
- 既存 backup.ts と接続

## 未決質問（答えを出すべき）

### Q1. B1 と B2 のどちらを先にやるか
- B2 推し: projection_vision 直結、akaghef の immediate ニーズ
- B1 推し: 構造的に B2 より一般的、汎用性高い
- 折衷: B6 を先、B2 を本命、B1 を最後

### Q2. LLM 介入度のデフォルト
- L0: 安全、つまらない
- L1: 控えめで実用、推奨
- L2+: 一気に賢いが倫理リスク
- 選択肢: 出力先別にデフォルト変えるか、全体で 1 つか

### Q3. ローカル LLM vs クラウド LLM
- プライバシー → ローカル（Ollama, LM Studio）
- 性能 → クラウド（Claude, GPT）
- 案: 介入度 L1 まではローカル、L2 以上はクラウド（ユーザ承認）

### Q4. プロファイルの作成方法
- 同梱の少数で始めるか、ユーザ自作前提か
- 推し: 科研費基盤C・若手・挑戦的の 3 つ同梱、Nature 等は別フェーズ

### Q5. マップ → Markdown の階層変換ルール
- 子ノード = h2 か、子ノード = bullet か
- 案: 設定可能、デフォルトは深さ 2 まで heading、それ以下は bullet

### Q6. マスキングの自動 vs 半自動
- 完全自動: 漏洩リスクあり
- 半自動: 確認手間
- 推し: 半自動デフォルト、自動オプションあり（自分用 SRS のみ）

### Q7. 出力物を git commit するか
- する: 履歴管理、共有しやすい
- しない: ストレージ肥大、マップとの二重管理
- 推し: ユーザ選択、デフォルトは公開出力のみ commit

### Q8. 既存 backup.ts との関係
- backup.ts はマップのバックアップ
- 出力物は別物として扱う？それとも統合？
- 推し: 統合せず、別フォルダ `outputs/`

### Q9. Zotero 連携のタイミング
- B1 で必要だが、後回しにすると痛い
- 推し: B1 着手時に最低限の CSL 対応、Zotero 双方向同期は別フェーズ

### Q10. 申請書「業績欄」の自動更新
- 過去論文ノードから自動抽出
- 案: ノードに `publication: true, year: 2024, venue: "Nature"` 属性
- 同梱の業績テンプレ + マップから埋める

### Q11. 出力物のレビュー UI
- CLI 出力のみで OK か、Web UI 必要か
- MVP 1 は CLI でいい
- B5/B6（公開系）は Web プレビューほしい

### Q12. 共同研究者との出力共有
- Phase 6 までに必要か
- 推し: MVP 段階では自分のみ、共有は後

## 横断観察 / 気づき

- **共通基盤を最初に作れば、出力先追加は段階的に楽になる**
- **B6 → B2 → B1 の順は「機能の複雑度順」と一致** していて学習曲線が緩やか
- **メタデータ管理（output_id 等）が地味だけど最重要**。これがないと再現性も伝播も難しい
- **LLM 介入度の連続スライダー** は早期から実装した方が後で楽
- **プライバシー基盤（Ms1, Ms5）は MVP 2 までに最低限ほしい**（B2 は機微情報多い）
- **dogfood 機会は申請書応募（おそらく秋）** → このタイミングを意識した実装順
- **ローカル LLM 対応は重要差別化要因**（policy_privacy 整合）
- **slideshow と並行進行は意識的に**（B3 が浮かないように）

## 戦略的観察

### 「投影」ビジョンとの整合
project_projection_vision は「世界モデル → 射影で科研費出力」。
本ブレストの構造は **完全に整合** している:
- マップ = 世界モデル
- 中間ビュー = 射影面
- 出力物 = 射影結果
- LLM 介入度 = 射影の "歪み許容度"

→ MVP 2（B2 申請書）が最も "射影" の本質を体現する。

### dogfood のタイミング
- 2026 秋の科研費応募シーズン（10〜11 月）
- それまでに B2 が動けば「自分の応募で使う」最強の dogfood
- 失敗しても自分が困るだけ（他者影響なし）
- → Phase 6（10 月）の dogfood は計画的に

### M3E の差別化要因
- 他ツール（Notion 等）は「線形ドキュメント出力」のみ
- M3E は「グラフ → 多形式射影」
- B1〜B8 を1ツールで覆うのは現状唯一無二の可能性

### 失敗時の落としどころ
- B1 が間に合わない → 手動 LaTeX で済ます（既存ワークフロー）
- B2 が間に合わない → 過去応募書を流用、新規はゼロから
- 最低限 B6（Markdown 出力）だけでも価値あり
