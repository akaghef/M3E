# 01. Concept — Capture / Ingest とは何か

ノードを M3E マップに「入れる」までの体験全体を設計対象とする。
入力チャネル列挙だけでは終わらない。**「入れる→保持→triage→定着」までの一連の流れ**
を 1つの体験単位として捉える。

---

## C1. Capture / Ingest / Triage の3段階モデル

「入力」は単一の動作ではなく、3層に分解できる。

| 段階 | 何をする | 主体 | 摩擦許容度 |
|---|---|---|---|
| **Capture** | 思いついた瞬間に「箱」に入れる | 人間 | ゼロ近く要 |
| **Ingest** | 箱から M3E scratch ノードに変換 | システム | 自動でよい |
| **Triage** | scratch を本来の場所に移動・分類 | 人間（or AI 補助） | やや高くてよい |

### 案 C1-a: 3層を完全分離
- 利点: 入力時に「どこに入れるか」を考えなくていい
- 欠点: scratch が肥大化、後で処理する人格が必要
- 既存例: GTD の Inbox、Things の Inbox

### 案 C1-b: Capture 時に最低限分類（タグ1つだけ）
- 利点: 後の triage が楽
- 欠点: 入力摩擦が増える
- 既存例: Drafts の Action

### 案 C1-c: AI が即時 triage を試みる
- 利点: 人間の介入最小
- 欠点: 誤分類のリスク、信頼感が育たないと使われない
- 既存例: Reflect の AI tagging

### 案 C1-d: 後で triage する責任を別人格（夜の自分・週次レビュー）に渡す
- 利点: 「今」の自分は楽
- 欠点: triage が溜まると挫折
- 既存例: Weekly Review（GTD）

**推し度**: C1-a を基本、C1-c を opt-in。C1-d は週次レビュー機能（G2 と接続）で補完。

## C2. 既存ツールに対する M3E の差別化軸

「Capture ツール」は既に飽和している。M3E が新規に作る価値は何か。

| 既存ツール | 強み | M3E が超えるべき/補完すべき点 |
|---|---|---|
| Drafts (iOS) | 起動0.5秒、Action豊富 | M3E はマップ構造に直結できる |
| Apple Notes | OS 統合、即起動 | 構造化が弱い |
| Obsidian Quick Add | プラグイン無限 | グラフ表示は静的 |
| Notion Quick Capture | 仕分け先指定が強い | UI 重い |
| Logseq | block ベース、Outliner | グラフ操作が弱い |
| Roam Research | リンク優先 | capture 体験は普通 |
| Reflect | AI tagging | データ主権が弱い |
| Tana | supertag、構造強 | 学習コスト高 |

### M3E の差別化候補
- **D-a**: 「入れた瞬間にマップ上で動く」可視化（取り込み演出）
- **D-b**: マップ構造への自動接続提案（C4 と統合）
- **D-c**: 研究者ワークフロー特化（PDF/論文/対話を1級市民に）
- **D-d**: 「世界モデル→射影」の入口として位置付け（project_projection_vision）
- **D-e**: クライアント暗号化前提の capture（policy_privacy）

**推し度**: D-c + D-d が研究者向けとして筋。D-a は楽しさドライバ。

## C3. 取り込み速度を成功指標にするか

### 案 C3-a: 「思いついてから3秒以内に scratch 化」を KPI 化
- 利点: 設計判断が明確になる
- 欠点: 速度を犠牲にすべき場面（暗号化・AI処理）と衝突

### 案 C3-b: チャネル別に SLA を分ける
- 音声: 1秒以内に「録音中」フィードバック
- ホットキー: 200ms 以内に入力欄
- メール: 60秒以内に scratch 反映
- PDF: 「処理中」表示OK、結果は分単位

### 案 C3-c: 速度より「入った確信」優先
- フィードバック（音・通知・色）を最重視
- 速度は二次

**推し度**: C3-b が現実的。チャネル特性を尊重。

## C4. 「壊れててもいい・後で直せばいい」許容度

Capture 時点での品質基準をどう設定するか。

| 案 | 入力品質要求 | 例 |
|---|---|---|
| C4-a 完璧主義 | タイトル + 本文 + タグ必須 | Notion |
| C4-b ゆるい | 1行テキストでOK | Drafts |
| C4-c 超ゆるい | 空でもOK（後で書き足す） | Apple Notes 新規作成 |
| C4-d 構造優先 | 必ず親ノード指定 | Tana |
| C4-e 後付け推奨 | scratch は壊れてて当たり前 | M3E 既存 scratch 設計 |

**推し度**: C4-e（既存設計の延長）。triage 段階で形を整える前提。

---

## C5. 取り込み元の出典（source）をどこまで残すか

### 案 C5-a: 全チャネルで `captured_via` メタを必須に
- 利点: 後で「どこから来たか」追跡可能
- 欠点: メタデータ肥大

### 案 C5-b: source URL / アプリ名のみ
- 例: `source: "https://example.com"` `source_app: "Drafts"`

### 案 C5-c: タイムスタンプ + チャネルだけ
- メタデータ最小

### 案 C5-d: AI 関与チャネルでは prompt も保存
- AI 取り込みの再現性確保

**推し度**: C5-a + C5-d。後で「あれどこから来たっけ」問題を防ぐ。

## ユースケース集（具体化）

### UC1. 散歩中の音声メモ
**シーン**: akaghef が散歩中、研究テーマについて口頭でアイデア出し。
**フロー**: スマホの常駐音声 → 句点で自動分割 → 帰宅後 PC で triage。
**チャネル**: A1（音声）+ A9（モバイル共有）
**未解決**: 句点検出の精度、長時間録音の電池

### UC2. 論文 PDF を一気に取り込み
**シーン**: 査読論文 PDF を3本入手、章節構造を自動マップ化したい。
**フロー**: PDF をドラッグ → 章/節/段落で自動ノード化 → 引用文献も子ノード。
**チャネル**: A3（PDF）
**未解決**: PDF 構造解析の限界、figure の扱い

### UC3. Claude との対話を後でマップに
**シーン**: Claude.ai で30分議論、要点だけマップに残したい。
**フロー**: 対話 export → AI 要約 → 主要 claim をノード化。
**チャネル**: A8（AI チャット履歴）
**未解決**: どのプラットフォームの export 形式に対応するか

### UC4. ホワイトボードの写真
**シーン**: 研究室ミーティング後、ホワイトボードを撮影。
**フロー**: 写真 → OCR → 構造推定（矢印・枠検出）→ ノード+リンク。
**チャネル**: A4（OCR/画像）
**未解決**: 構造推定の精度、手書き OCR の精度

### UC5. ブラウザで気になった記事をクリップ
**シーン**: arXiv の abstract、関連研究として保存したい。
**フロー**: 拡張 → 既存ノード「文献レビュー」配下に新規ノード生成。
**チャネル**: A2（ブラウザ拡張）
**未解決**: 親ノードの指定 UI、abstract 抽出の安定性

### UC6. メールで自分宛にメモ送信
**シーン**: 出先、スマホからアイデアをメール送信、PC 戻ったら scratch にある。
**フロー**: 専用アドレスへ送信 → IMAP 監視 → scratch 化。
**チャネル**: A5（メール）
**未解決**: メールサーバ運用、暗号化

### UC7. git commit メッセージから知識ノード
**シーン**: M3E 開発中、commit に記録した設計判断をマップに残す。
**フロou**: post-commit hook → 特定 prefix のメッセージを scratch 化。
**チャネル**: A7（コード）
**未解決**: prefix 規約、双方向同期するか

### UC8. Slack の自分発言を「これ重要」リアクションでマップ化
**シーン**: 研究室 Slack で議論、後で残したい発言に reaction。
**フロー**: bot が reaction を監視 → 該当発言を scratch に。
**チャネル**: A6（Slack）
**未解決**: 認証、プライベートチャンネルの扱い

### UC9. 朝のシャワー後、メニューバーから一行
**シーン**: 起きてすぐ、PC 開いてメニューバーから1行投入。
**フロー**: ホットキー → 小ウィンドウ → Enter で scratch。
**チャネル**: A10（常駐 widget）
**未解決**: 起動速度、フォーカス奪取

### UC10. iOS 共有シートから即捕捉
**シーン**: Twitter で見かけた論文リンク、即マップへ。
**フロー**: 共有 → M3E アプリ → タグ選択 → 完了。
**チャネル**: A9（モバイル共有）
**未解決**: アプリ実装コスト、PWA で代替可能か

## C6. 取り込み「演出」の方針

「入った」という感覚をどう作るか。

| 案 | 演出 | 雰囲気 |
|---|---|---|
| C6-a 無音・無アニメ | 何も起きない、ログだけ | プロ向け |
| C6-b ミニマル | 0.3秒のフェードトースト | バランス |
| C6-c 楽しい | ノードがマップにふわっと着地 | ゲーム感 |
| C6-d リッチ | 着地 + 音 + 周囲ノードが一瞬光る | 中毒性 |
| C6-e 後で表示 | 取り込み時は何もせず、次回起動時に「12件届いてます」 | バッチ向け |

**推し度**: チャネル別に C6-b（ホットキー）/ C6-c（音声・写真）/ C6-e（メール・Slack）の混合。

---

## C7. 取り込み失敗時の体験

| 案 | 失敗時挙動 |
|---|---|
| C7-a 静かに失敗 | エラー隠蔽、ログのみ |
| C7-b 即通知 | トースト「保存失敗」 |
| C7-c ローカル退避 | オフライン時は localStorage に貯めて後で送信 |
| C7-d 再送 UI | 失敗キューを表示、手動再送 |
| C7-e dead letter ノード | 失敗内容を専用 scratch に書く |

**推し度**: C7-c + C7-e。データロス絶対回避。
