# 位置づけ

本ドキュメントは AlgLibMove（MATLAB → Julia 移植）の**直接の作業項目ではない**。
ただし、移植時に「簡約化 API（`calc`, `C`, `calcComplete` 等）は信頼できない負債だ」
と短絡的に断じてしまうと、本質を見誤る。

計算代数における簡約化は **信頼性の問題ではなく、計算時間との戦略的トレードオフ**
であり、言語を変えても消えない本質的設計課題である。Julia 側で素朴に
「正規形を型で保証」のような設計を入れると、逆に性能破壊を招きうる。

移植後、あるいは移植と並行して「AlgLibMove 2.0 の簡約化戦略」を検討する際の
問題意識の出発点として、ここに記録する。

---

# 問題の構造

## トレードオフの両極

### Eager（毎演算後に簡約）
- **利点**: 各中間値が正規形。式の項数がコントロールされ、後段の演算が軽い。
  デバッグ容易（途中式が読める）
- **欠点**: 簡約そのもののコストが積み上がる。
  特に「`simplify` が高価」な領域（symbolic coefficient を含む場合など）で顕著。
  書き換え規則の評価と不動点判定が毎回走る
- **代表例**: CAS の素朴な実装、MATLAB の `sym` の演算

### Lazy（放置して必要時だけ簡約）
- **利点**: 簡約コストを払うのは「結果を見る」ときだけ。
  代数的な再配置で**途中で 0 になる項**を持ち越しても、最終簡約で消せば元は取れる
- **欠点**: 項数が指数的に爆発しうる。
  特に積演算 `mtimes` は項数 × 項数 で O(n²) なので、
  lazy のまま積を重ねると**中間表現のメモリが破綻**する
- **代表例**: Mathematica の `HoldForm`, lazy expression tree

### どちらも単独では壊れる

- Eager は「簡約コスト > 演算コスト」の領域で負け
- Lazy は「項数爆発 > 簡約コスト」の領域で負け
- **境界は問題・データ・計算パターンごとに違う**

## 書き換え系としての根本的困難

- 一般に「完全に簡約されたか」の判定は**停止性問題に触れる**
  （Knuth-Bendix が常に終わるとは限らない、など）
- `calcComplete` が「10 回リトライ + warning("simplify not completed")」を出すのは、
  **信頼性不足ではなく、判定が原理的に難しいことへの pragmatic な対応**
- Julia 側で「型で正規形を保証」と書くと、この原理的困難を型システムに
  押し付けることになり、**正規形の定義が曖昧なまま型だけ導入**すれば
  嘘の型（値は正規形でない `Normalized{T}`）を生む

## 「いつ簡約すべきか」はコール側の責任

AlgLibMove の現状（推測）：

- `calc` の呼び出しタイミングは明文化されていない
- 各 classdef の実装者が経験則で `plus`, `mtimes` 内に差し込んでいる
- ユーザコードも適宜呼ぶ必要があるが規約が不明
- **経験則が値を持っている**: MATLAB 実装者の試行錯誤の成果が
  「どこで `calc` を呼ぶか」という散在した判断として埋め込まれている

この経験則は、コードを読まないと取り出せない形で暗黙化している。
これは**ドキュメント負債**ではあるが、**判断そのものは妥当**である可能性が高い。

# 設計軸の列挙（解ではなく選択肢）

以下は相互排他ではなく組み合わせ可能。

## 軸1: 正規形を型で表現

```julia
struct Normalized{T}  x::T  end
simplify(x::T)::Normalized{T}
```

- **pros**: `Normalized` を要求する API は正規形前提で書ける。契約が明示的
- **cons**: 「正規形とは何か」を型ごとに厳密に定義する必要。
  部分的簡約（`removeZero` だけ済んでいる）状態をどう表現する？
- **注意**: AlgLibMove の `removeZero`, `combineTerm`, `replace`, `calcComplete` が
  示唆するように、**簡約には段階がある**。二値の `Normalized` では粗すぎる可能性
- **変種**: 段階型 `PartiallyNormalized{Stage,T}`（Stage ∈ ZeroRemoved, TermsCombined, FixedPoint）

## 軸2: Lazy 表現 + 遅延評価

- 式木（AST）として保持し、要求時にのみ簡約
- `Symbolics.jl` や `SymbolicUtils.jl` の流儀
- **pros**: 簡約コストは 1 回に集約できる。途中の書き換えが記録に残る
- **cons**: メモリ使用量。項数爆発の直接の受け皿

## 軸3: 戦略をオブジェクト化

```julia
with_simplify_policy(:threshold, threshold=100) do
    # この中の演算は項数 100 超で自動簡約
end
```

- **pros**: 同じ代数操作コードが、外側の戦略で挙動を変えられる。
  profile で最適戦略を選べる
- **cons**: 戦略が動的に効くと reasoning しにくい。
  データフロー解析の余地が減る

## 軸4: 複雑度メトリクスを第一級市民に

- `term_count(x)`, `depth(x)`, `complexity(x)` を型ごとに定義
- 演算内で閾値チェック → 必要なら簡約
- **pros**: eager/lazy の中間が自然に書ける。経験則をコードに落とせる
- **cons**: 閾値のチューニングが実データ依存。汎用閾値は存在しない

## 軸5: Profile-driven policy

- 代表ワークロードをベンチマーク化
- 戦略 × データ × サイズ の組み合わせで実測
- **現状の MATLAB 版の経験則を、定量データに置き換える作業**が先

## 軸6: 代数構造の活用

- Hopf 代数, 群代数, テンソル代数など、**構造が簡約規則を決める**
- 汎用 CAS の simplify ではなく、**構造特化の正規形**（basis representative,
  normal-ordered monomial 等）で定義
- **pros**: 「完全簡約」の判定が構造的に明確になる。停止性も言える場合がある
- **cons**: 構造ごとに設計コスト。横断的な抽象と噛み合わせる難しさ

# 現行 MATLAB 版から学ぶべき観察点

移植前に拾うべき情報（再実装フェーズで調査）:

1. **`calc` / `C` / `removeZero` / `combineTerm` が呼ばれる場所のパターン**
   - どの演算（`plus`, `mtimes`, `Delta`, `S`, ...）の末尾で呼ばれているか
   - クラスごとに一貫性はあるか、バラバラか
2. **`calcComplete` が必要だったユースケース**
   - どの計算で「1 回の calc では足りない」が生じたか
   - これは書き換え規則の合流性が足りない兆候
3. **性能プロファイル**
   - `simplify` / `replace` / `combineTerm` の各コストの実測
   - 現行版で「遅い」「爆発する」と知られているワークロード
4. **既存の経験則**
   - コメントや Docs/ に残された「いつ呼ぶか」の知見
   - 口伝になっている知識があれば記録

これらは**コードの表面からは見えない資産**なので、移植前に聞き取り/調査しておく
価値がある。

# この問題を直接扱わない場合の最小防衛ライン

AlgLibMove 移植で**簡約化戦略の再設計に踏み込まない**場合でも、以下は守る:

1. **MATLAB 版の `calc` 呼び出しタイミングは忠実に再現**する
   - 経験則の価値を壊さない
   - 「Julia なら型で保証できるから呼ばなくていい」は罠
2. **命名だけは整理**する（B' 節参照）
   - `calc`, `C`, `calcComplete` を `simplify`, `simplify!`, `normalize`,
     `canonicalize` など意図の分かる名前に
   - ただし**意味は変えない**。改名だけ
3. **ドキュメントで経験則を明文化**
   - 「この関数は末尾で simplify を呼ぶ」「ユーザは積の後で明示的に
     simplify を呼ぶべき」等、暗黙知を docstring に落とす
4. **Normalized 型や lazy 化などの大改造は保留**
   - 本ドキュメントが扱う設計課題として、別プロジェクトで扱う
