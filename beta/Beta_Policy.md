# Beta方針

## 位置づけ

Betaは、MVPで成立した最小価値を維持したまま、実運用に近い利用で破綻しない状態へ引き上げる段階とする。
MVPが「使える最小構成の成立」を目的とするのに対し、Betaは「継続利用しても壊れにくく、判断可能な品質にする」ことを目的とする。

## 基本方針

### 1. MVPを置き換えず、運用可能性を上に積む

BetaではMVPの定義を消さない。
MVPで成立させた中核体験を土台として、保存信頼性、編集の安定性、操作継続性、復帰容易性を強化する。

### 2. 新規拡張より、既存体験の安定化を優先する

Betaの主目的は機能数の拡大ではない。
すでに重要と判断された操作フローが、長時間利用・繰り返し利用・中規模データ利用でも安心して扱えることを優先する。

### 3. 「触れる」から「任せられる」への移行を目指す

MVPでは編集・保存・表示が成立していることに価値がある。
Betaでは、ユーザーがデータ消失や状態破損を強く警戒せずに作業を続けられることを目標にする。

### 4. 実運用を意識した品質軸を導入する

Betaでは以下の品質軸を明示的に扱う。

- データ安全性
- 保存と復元の一貫性
- 編集操作の予測可能性
- ナビゲーションの疲れにくさ
- 一定規模での性能維持
- 問題発生時の原因把握しやすさ

### 5. Betaで検証するのは「価値」より「継続利用可能性」

MVPで価値仮説の初期確認を行った前提で、Betaでは継続利用時の摩擦を重点的に見る。
つまり「便利そうか」ではなく、「毎回安心して使えるか」「途中で壊れないか」「再開しやすいか」を判断基準にする。

## Betaで扱うテーマ

### 1. 安定した直接編集

ビュー上での追加・編集・削除・再配置を、迷いなく継続できる状態にする。
操作結果が視覚的にもデータ的にも一貫していることを重視する。

### 2. 保存・復元の信頼性向上

保存成功、復元成功、異常時の保全、再読込後の整合性をBetaの中心テーマとする。
「保存したはずが戻らない」「読み直したら崩れる」を起こさないことを優先する。

### 3. 操作継続性の改善

パン、ズーム、フォーカス移動、選択維持など、作業の流れを止めないための操作性を高める。
短時間のデモ向けではなく、実際の作業時間に耐えることを目指す。

### 4. 中規模利用での性能確認

ノード数が増えた場合でも、編集・移動・保存の主要操作が著しく悪化しないことを確認対象とする。
性能は理論値ではなく、実際の操作感と待ち時間で評価する。

### 5. 問題の見える化

異常時に「何が起きたかわからない」状態を減らす。
最低限の状態把握、エラー把握、復旧判断ができるようにして、運用時の不安を下げる。

## Betaでまだ主目的にしないもの

以下はBetaで全面展開の主対象にしない。

- AI提案や自動生成を中心にした高度機能
- 大規模な情報設計の再編
- アーキテクチャ先行の広範囲リライト
- 多数の周辺機能追加

これらは、Betaで基盤品質を確認した後の次段階で扱う。

## 完了イメージ

Beta完了時点では、MVPの中核体験を保ったまま、次の状態に近づいていることを目標とする。

- 主要編集操作が安定している
- 保存と復元に対する不安が小さい
- 中規模データでも実用上の操作継続が可能
- 問題が起きても状況を把握しやすい
- 次段階の拡張を載せても崩れにくい土台がある

## Betaの一文定義

Betaは、MVPで成立した価値を壊さずに、継続利用に耐える信頼性と操作安定性を与える段階である。

## Commit Policy

Betaでは、特にSQLite導入のような保存基盤の変更が入るため、コミットは小さく、戻しやすく、意図が追いやすい単位を原則とする。

### 基本原則

- 1コミット1意図を守る
- リファクタと挙動変更を同じコミットに混ぜない
- スキーマ変更、保存処理変更、UI変更、ドキュメント更新をできるだけ分離する
- 途中段階でもローカルで動作確認できる単位で積む
- データ安全性に関わる変更は、通常変更より細かく刻む

### SQLite着手時の扱い

SQLite導入はBetaの基盤変更として扱い、以下の順で小さく積むことを推奨する。

1. 依存追加と接続基盤
2. スキーマ定義と初期化
3. 読み取り経路の接続
4. 書き込み経路の接続
5. 既存保存導線との切替または併存整理
6. エラー処理と復旧導線
7. 検証とドキュメント更新

### コミットメッセージ方針

コミットメッセージは imperative にし、変更の主語を明確にする。
特に保存基盤では、「何を安全にしたのか」「どの層を変えたのか」がわかる表現を優先する。

例:

- `add sqlite connection bootstrap`
- `define initial sqlite schema for nodes`
- `load viewer data from sqlite repository`
- `persist node edits through sqlite`
- `add rollback handling for sqlite save failures`
- `update beta docs for sqlite migration plan`

### Betaで避けるコミット

- 大きな一括コミット
- 挙動変更を含む rename-only に見えるコミット
- 検証不能な途中状態のコミット
- 保存方式変更と無関係UI調整の抱き合わせコミット

### ドキュメント連動

SQLite関連コミットでは、実装完了だけでなく、現在の保存方式、切替状態、既知制約をドキュメントで追えるようにする。
Betaでは「実装したこと」より「今どこまで安全に切り替わったか」が判断できることを重視する。
